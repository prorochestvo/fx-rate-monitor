# Task Breakdown

## Overview

`cmd/collector/main.go` and `cmd/notifier/main.go` currently call `log.Fatalf` whenever
the per-runner error aggregation at the end of `main()` is non-nil. `r.Run(ctx)` already
swallows nothing — it joins per-source errors and returns them up — and the underlying
services already persist each per-source failure to `execution_history` (collector) and
the notification pool tables (notifier). So today's behaviour double-reports a per-source
failure as a process-level fatal exit (`exit 1`), which:

- terminates the local `make run` chain mid-pipeline when, e.g., 3 upstreams are
  unreachable (this became routinely reproducible after plan 008 added per-URL
  fail-fast in the collector),
- generates false "service crashed" alerts from cron wrappers in production, even
  though every per-source failure is already recorded in the DB and visible in the
  UI and Telegram notifications.

The fix is a single, narrowly-scoped change at the composition root of each binary:
treat the aggregated runner error as informational (log it, exit 0), while keeping every
setup-phase failure (logger, env config, DB open, schema check, repository construction,
runner construction, Telegram bot init, notifier `Vacuum`) strictly fatal.

This change does **not** touch the runner interface, `internal/application/collection`,
`internal/application/notification`, the repository layer, or any tests. Both binaries
receive identical treatment in one PR so the exit-code contract stays symmetric.

## Assumptions

- Per-source / per-event failures are already persisted by the application layer:
  - collector → `execution_history` rows with `success=false` plus the captured error
    message,
  - notifier → `rate_user_events` rows transition state, and failed dispatches land in
    the notification pool's failed view.

  No additional persistence is required to make exit code 0 safe — the operator can
  still observe failures through `GET /api/errors/execution`, `GET /api/notifications/failed`,
  and the existing UI surfaces.
- The exit-code contract is **not** currently documented in `CLAUDE.md` (verified by
  reading the file). Therefore no docs change is needed as part of this plan. If a future
  plan introduces an explicit "exit codes" section, it should incorporate the contract
  stated below.
- Cron wrappers and the `make run` target rely on POSIX exit-code semantics (0 = success,
  non-zero = failure). No wrapper today distinguishes "soft" from "hard" failure, so
  a single 0/non-zero split is sufficient.
- `cmd/collector/main.go` and `cmd/notifier/main.go` have no unit tests today and adding
  tests for `func main` is an anti-pattern — verification is manual.
- The string "execution:" is the existing log-line prefix used by both binaries for this
  branch; keep it for log-grep continuity (operators currently match on it).
- Plan 008 (`260519.0002.collector-url-failfast.md`) is already completed and in
  `plans/completed/`; this plan builds on the visibility it created but does not
  modify any of its code.

## Tasks

### Task 1: Downgrade collector aggregated-runner error from fatal to informational

- **Description:**
  In `cmd/collector/main.go`, replace the post-loop `log.Fatalf` with a `log.Printf`
  so that per-source failures returned by `r.Run(ctx)` do not abort the process.
  The setup phase (lines 51-116) keeps every `log.Fatalf` exactly as-is.

  Concrete edit, current code at `cmd/collector/main.go:118-130`:

  ```go
  ctx := context.Background()

  errs := make([]error, 0, len(runners))
  for _, r := range runners {
      if err = r.Run(ctx); err != nil {
          errs = append(errs, err)
      }
  }
  if err = errors.Join(errs...); err != nil {
      log.Fatalf("execution:  %s", err)
  }

  log.Println("execution: done")
  ```

  Becomes:

  ```go
  ctx := context.Background()

  errs := make([]error, 0, len(runners))
  for _, r := range runners {
      if err = r.Run(ctx); err != nil {
          errs = append(errs, err)
      }
  }
  if err = errors.Join(errs...); err != nil {
      log.Printf("execution: completed with errors: %s", err)
  }

  log.Println("execution: done")
  ```

  Notes:
  - `log.Println("execution: done")` MUST still fire on every path so that ops dashboards
    and tail-log monitors see a consistent "this run finished" marker regardless of
    per-source outcomes.
  - The doubled space in the existing `"execution:  %s"` format string is corrected to
    a single space as part of this edit; this is an opportunistic cleanup, not a
    behaviour change.
  - No new imports, no new helpers, no signature changes. `errors.Join` stays.

- **Acceptance Criteria:**
   - [ ] `cmd/collector/main.go` no longer contains `log.Fatalf("execution: ...` in the
         post-loop block (line ~127). The setup-phase `log.Fatalf` calls at lines
         57, 67, 75, 84, 91, 96, 101, 113 are unchanged.
   - [ ] The replacement uses `log.Printf` with the literal format string
         `"execution: completed with errors: %s"`.
   - [ ] `log.Println("execution: done")` still executes unconditionally after the
         error-logging branch.
   - [ ] `errors.Join` still aggregates the per-runner errors (the slice + join shape
         is preserved so future runners are handled identically).
   - [ ] `go build -o ./build/collector ./cmd/collector` succeeds with `CGO_ENABLED=0`.
   - [ ] `make lint` (go vet + forbidden-imports check) passes.
   - [ ] Manual smoke: with at least one upstream blocked (e.g. block bcc.kz via `/etc/hosts`
         or rely on a known-flaky source), running `./build/collector` exits with status
         code `0` and writes `execution: completed with errors: ...` followed by
         `execution: done` to stdout.

- **Pitfalls:**
  - Do not also remove `errors.Join` — even though only one runner exists today
    (`collectionRateAgent` from `buildRunners`), the slice abstraction is intentional
    for future runners and a follow-up plan may add e.g. a vacuum runner. Preserve it.
  - Do not touch the `defer c.Close()` block — `log.Fatalf` calls `os.Exit(1)` which
    skips deferreds, so the previous behaviour silently leaked the DB handle on the
    error branch. Switching to `log.Printf` actually fixes this side-effect; do not
    "fix" it again by manually closing.
  - Do not change `log.Fatalf("runners: runners building is failed: %s", err)` at
    line 113 — `buildRunners` failure is construction, not runtime, and must remain
    fatal.
  - The doubled-space typo in the current log message is a low-risk fix to do
    in the same edit, but if any operator grep matches `"execution:  "` (two spaces)
    they would silently miss the new line. The new prefix `"execution: completed with errors:"`
    differs from the old enough that no grep alias should overlap; document this in
    the commit body so log-monitoring rules can be updated if any exist.

- **Complexity:** Easy.

### Task 2: Apply the identical downgrade to the notifier

- **Description:**
  In `cmd/notifier/main.go`, mirror Task 1 exactly. The setup phase (lines 42-118)
  including `dispatchAgent.Vacuum(ctx)` keeps every `log.Fatalf` unchanged. Only the
  post-`Vacuum` aggregation block at lines 120-129 is downgraded.

  Current code at `cmd/notifier/main.go:120-131`:

  ```go
  var errs []error
  if err = checkAgent.Run(ctx); err != nil {
      errs = append(errs, err)
  }
  if err = dispatchAgent.Run(ctx); err != nil {
      errs = append(errs, err)
  }
  if err = errors.Join(errs...); err != nil {
      log.Fatalf("execution: %s", err)
  }

  log.Println("execution: done")
  ```

  Becomes:

  ```go
  var errs []error
  if err = checkAgent.Run(ctx); err != nil {
      errs = append(errs, err)
  }
  if err = dispatchAgent.Run(ctx); err != nil {
      errs = append(errs, err)
  }
  if err = errors.Join(errs...); err != nil {
      log.Printf("execution: completed with errors: %s", err)
  }

  log.Println("execution: done")
  ```

  Rationale for keeping `Vacuum` fatal: `Vacuum` is a one-shot reaper that runs before
  the loop and is part of the agent's startup contract, not per-notification work — if
  it fails, the agent is in an indeterminate state and exiting is correct. The runtime
  loop is `checkAgent.Run` + `dispatchAgent.Run`.

- **Acceptance Criteria:**
   - [ ] `cmd/notifier/main.go` no longer contains `log.Fatalf("execution: ...` in the
         post-loop block (line ~128). The setup-phase `log.Fatalf` calls at lines
         47, 53, 57, 63, 71, 75, 81, 85, 89, 93, 107, 112, 116 are unchanged.
   - [ ] The replacement uses `log.Printf` with the same literal format string used
         in collector: `"execution: completed with errors: %s"`. The wording is
         identical across binaries so cron/log-monitoring rules can match both with
         one pattern.
   - [ ] `log.Println("execution: done")` still executes unconditionally.
   - [ ] `errors.Join` is preserved.
   - [ ] `dispatchAgent.Vacuum(ctx)` at line 115 stays fatal-on-error (line 116).
   - [ ] `go build -o ./build/notifier ./cmd/notifier` succeeds with `CGO_ENABLED=0`.
   - [ ] `make lint` passes.
   - [ ] Manual smoke: with the bot temporarily configured to reject sends (e.g. invalid
         chat ID for one subscription) or with no network, running `./build/notifier`
         exits with status code `0` and emits `execution: completed with errors: ...`
         followed by `execution: done`.

- **Pitfalls:**
  - Symmetry with collector matters. The reviewer should diff the two changes and
    confirm the format string and surrounding structure are identical.
  - Do not collapse the two `if err = X.Run(ctx); err != nil` branches into a loop —
    `checkAgent` and `dispatchAgent` have different concrete types and the current
    explicit form is clearer at the composition root.
  - As with collector, switching `log.Fatalf` → `log.Printf` means the deferred
    `db.Close()` now runs on this branch, which is desirable; do not bypass it.

- **Complexity:** Easy.

### Task 3: Manual end-to-end verification

- **Description:**
  Both binaries are composition roots with no unit tests; verification is manual and
  consists of three checks.

  1. **Compilation:** both binaries build cleanly with the project default flags.
     ```
     CGO_ENABLED=0 go build -o ./build/collector ./cmd/collector
     CGO_ENABLED=0 go build -o ./build/notifier  ./cmd/notifier
     ```
     Per CLAUDE.md, binaries land in `./build/` — do not run bare `go build ./cmd/...`.

  2. **`make test`:** the existing suite still passes. There are no main-package tests
     so this is mostly a regression net for any accidentally-shared code path.
     ```
     make test
     ```

  3. **Exit-code contract:** confirm both binaries exit `0` when a runner returns
     a non-nil error.

     For collector — easiest path is to rely on the natural flakiness of one of the
     remote sources, or to temporarily block one host in `/etc/hosts`:
     ```
     # in another shell, e.g. block one slow upstream:
     # 127.0.0.1 bcc.kz   (added to /etc/hosts; revert after)
     ./build/collector ; echo "exit=$?"
     # expect: exit=0
     # expect: log line "execution: completed with errors: ..."
     # expect: log line "execution: done"
     ```

     For notifier — point `TELEGRAMBOT_DSN` at an admin chat ID that does not exist,
     or rely on whatever pending events are in the local DB to fail dispatch:
     ```
     ./build/notifier ; echo "exit=$?"
     # expect: exit=0
     # expect: log line "execution: completed with errors: ..."
     # expect: log line "execution: done"
     ```

  4. **`make run`:** the full chain (migrator → collector → notifier → web) no longer
     short-circuits when an individual source/notification fails. Run it once with the
     network available, then once with one upstream blocked, and confirm the web server
     comes up in both cases.

- **Acceptance Criteria:**
   - [ ] Both `go build` invocations succeed and produce binaries under `./build/`.
   - [ ] `make test` is green.
   - [ ] At least one collector run is executed with a deliberately failing source and
         observed to exit `0` with the documented log lines.
   - [ ] At least one notifier run is executed with a deliberately failing dispatch
         and observed to exit `0` with the documented log lines.
   - [ ] `make run` succeeds end-to-end with at least one upstream unavailable; the web
         server reaches "listening" log line.

- **Pitfalls:**
  - Don't permanently edit `/etc/hosts` — revert after the test. Or stage the block
    inside a netns / pf rule on macOS if you prefer not to touch `/etc/hosts`.
  - Don't rely on `make test` to catch this regression — there is no test that
    invokes `main()`. The exit-code contract is verified by running the binary.
  - If a sources's outage produces *zero* runner-level errors (e.g. the application
    layer absorbs all of them into `execution_history` without returning anything),
    you won't trigger the new branch. In that case force the branch by adding a
    temporary `return errors.New("forced")` in `collection.RateAgent.Run` for one
    smoke run only (revert before committing). Document this fallback in the PR
    description if used.

- **Complexity:** Easy.

## Execution Order

Tasks 1 and 2 are independent edits to two separate files and can be applied in any
order or in parallel by a single engineer. Task 3 strictly depends on both binaries
being updated, since the verification covers both.

Recommended sequence (single PR):
1. Task 1 — `cmd/collector/main.go` edit.
2. Task 2 — `cmd/notifier/main.go` edit (review the diff side-by-side with Task 1 to
   confirm the log format string is identical).
3. Task 3 — build, `make test`, manual exit-code checks for each binary, then a single
   `make run` smoke pass with one upstream blocked.
4. Commit. Suggested message body:
   ```
   feat(collector,notifier): tolerate per-source failures, keep exit code 0

   Per-source and per-notification failures are already persisted to
   execution_history and the notification pool; they're operational events,
   not crashes. Downgrade the post-loop log.Fatalf in both composition roots
   to log.Printf so a partial failure no longer terminates the binary with
   exit 1. Setup-phase fatals (env, DB, migrations, repositories, bot client,
   Vacuum) are unchanged. Log format is identical across both binaries:
   "execution: completed with errors: %s".
   ```

## Risks

- **Silent regressions in cron alerting.** If any environment runs the collector or
  notifier under a wrapper that alerts on exit-code != 0 *as the only signal*, those
  alerts will go quiet for per-source failures after this change. That is the intended
  behaviour, but operators must know to watch `execution_history` /
  `GET /api/errors/execution` and the notification-failed surfaces for those signals
  instead. Mitigation: call this out explicitly in the commit body and in any
  ops-channel announcement.
- **Log-grep rules.** If any external monitoring tail-greps `"execution:  "` (with the
  doubled space from the old format) or `Fatalf execution:`, those rules will need
  updating. The new prefix is `"execution: completed with errors:"`. Mitigation: PR
  description must list the old and new strings.
- **Deferred-close behavioural shift.** Today `log.Fatalf` on the post-loop branch
  skips the deferred `db.Close()`. After this change, the close runs. Risk is
  near-zero (SQLite close is idempotent and well-behaved), but it is a behaviour
  change worth mentioning.
- **Vacuum vs runtime classification.** Treating `dispatchAgent.Vacuum(ctx)` as setup
  rather than runtime is a judgement call. If `Vacuum` later evolves to perform
  per-event work that can legitimately fail per-event, this classification should be
  revisited (out of scope for this plan — flag for a future plan if the agent grows).

## Trade-offs

- **No "soft" exit code (e.g. exit 2 for partial failure).** Considered and rejected:
  cron wrappers and `make run` today have no support for a non-binary exit-code
  contract, and adding one only to remove it later if the convention doesn't catch on
  is more risk than the operator value it would deliver. A single 0/non-zero split
  is enough because the per-source state is already first-class in the DB.
- **No structured logging or metrics emission.** Considered and rejected for scope: the
  goal is a one-line change to exit-code semantics, not an observability overhaul.
  Metrics for per-source failure rates already exist via `execution_history` and the
  HTTP routes documented in CLAUDE.md.
- **No refactor of `r.Run(ctx)` to return a richer error type.** Considered and
  rejected for scope: the runner interface and application-layer aggregation are
  working as designed. Per-source detail is in the DB; the joined error at the
  composition root is for diagnostic logging only, and `errors.Join`'s string form
  is sufficient for that.
- **Apply both changes in one PR rather than two.** Considered alternatives:
  collector-only first, notifier follow-up. Rejected because the change is trivial
  and the symmetry (identical log format, identical structure) is a feature — splitting
  invites format drift between the two binaries. A single small PR is the lower-risk
  choice here.
- **Keep `errors.Join` despite there being one runner in collector and two in notifier
  today.** Considered and rejected to inline-simplify: the slice + join shape is the
  abstraction that lets future runners be added without revisiting this branch, and
  it costs nothing.
- **Skip docs update.** Verified that `CLAUDE.md` does not document an exit-code
  contract for these binaries today. Adding one purely to record this change would
  be scope creep; if a future plan introduces an "Exit codes" section, it should
  capture: 0 = setup ok (runners may have logged partial failures); non-zero = setup
  failed or unexpected panic.
