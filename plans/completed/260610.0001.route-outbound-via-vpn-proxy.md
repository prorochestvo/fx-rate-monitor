# Task Breakdown

## Overview

Route every outbound HTTP request the service makes through a local HTTP proxy
at `http://127.0.0.1:1701` **except** calls to the Telegram Bot API. The proxy
is the same address on dev and prod (it is a per-host local VPN proxy, not a
remote one). No proxy auth is in scope: the operator disables auth on the proxy
side, so the service never sends `Proxy-Authorization`.

The mechanism is the standard Go proxy env triplet:

- `HTTPS_PROXY=http://127.0.0.1:1701` (covers `https://` targets)
- `HTTP_PROXY=http://127.0.0.1:1701` (covers `http://` targets)
- `NO_PROXY=api.telegram.org` (exempts Telegram)

Most of the service's HTTP clients already share `http.DefaultTransport`, whose
`Proxy` field is `http.ProxyFromEnvironment` by default. They pick up the
triplet for free with zero code change. Two code sites need explicit fixes:

1. `internal/tools/rateextractor/rateextractor.go` builds a bare `&http.Transport{}`
   (Proxy=nil), so it currently ignores the env. Fix it to fall back to
   `http.ProxyFromEnvironment` when the explicit `proxyURL` override is empty.
2. The chromedp allocator in two files (`internal/tools/rateextractor/chromedp.go`
   and `internal/application/rulegen/chromedpfetcher.go`) drives Chromium, which
   does **not** read `HTTPS_PROXY` from env. Add a `chromedp.ProxyServer(...)`
   option resolved from the same env at construction time.

Out of band: operator updates `/opt/monitor/.{stage,prime}_monitor.env` with the
triplet, plus the in-repo `.env.example` for local dev. The cron wrappers that
launch `cmd/collector` and `cmd/notifier` already source the env file (they
need `SQLITEDB_DSN`), so they pick up the triplet automatically.

## Assumptions

- The local proxy at `http://127.0.0.1:1701` accepts plain `CONNECT` without
  auth and forwards to the public internet. Operator owns its lifecycle (start
  on boot, restart on failure); the service treats it as infrastructure.
- The same proxy address is reachable on the dev workstation and on both
  stage/prime hosts. There is no per-env override: one value in each env file
  is enough.
- `api.telegram.org` is the only host the service must reach directly. No
  current AI provider (Groq, OpenRouter, OpenAI), no rate source, and no
  operator tool needs a direct path.
- `http.ProxyFromEnvironment` reads the env **once per process** and caches the
  result. That matches the deploy model: every binary is restarted on config
  change, no hot-reload of the proxy URL is required.
- Chromium's `--proxy-server` flag uses the same URL syntax we already use for
  the env vars (`http://host:port`), so a single resolved string drives both.
- No code currently relies on the explicit `proxyURL` parameter of
  `NewRateExtractor` in production wiring (`cmd/collector`, `cmd/web`,
  `cmd/notifier`, `cmd/doctor`). It is kept for backward compatibility and
  test-override use; if a production caller does pass it, that wins over env.
- `make lint` enforces forbidden imports but does not perform any proxy-
  related checks; no lint rule needs adding.

## Tasks

### Task 1: Make `NewRateExtractor` fall back to `http.ProxyFromEnvironment`
- **Description:** In `internal/tools/rateextractor/rateextractor.go` the
  constructor builds `transport := &http.Transport{}` at line 43 with no
  `Proxy` field. Set `transport.Proxy = http.ProxyFromEnvironment` as the
  default. Keep the explicit `proxyURL` parameter: when it is non-empty,
  continue to call `url.Parse` and assign `transport.Proxy = http.ProxyURL(parsed)`
  so explicit overrides still win (useful for tests). Update the godoc on
  `NewRateExtractor` to describe both paths and to note that Telegram-bound
  HTTP is bypassed via `NO_PROXY=api.telegram.org` from the operator's env.
  Do **not** invent a new constructor — extend the existing one.
- **Acceptance Criteria:**
   - [ ] `transport.Proxy` is `http.ProxyFromEnvironment` when `proxyURL == ""`.
   - [ ] `transport.Proxy` is `http.ProxyURL(parsed)` when `proxyURL != ""`.
   - [ ] Godoc on `NewRateExtractor` explicitly documents the env-fallback
         behaviour and the `HTTPS_PROXY` / `HTTP_PROXY` / `NO_PROXY` triplet
         it honours.
   - [ ] No change to the function signature, no new exported identifier.
   - [ ] Unit test (see Task 5) covers env-fallback, override-precedence, and
         `NO_PROXY` bypass scenarios.
- **Pitfalls:**
   - `http.ProxyFromEnvironment` caches the env on first call inside the
     stdlib `httpproxy` package. Tests that use `t.Setenv` to change the
     value mid-process MUST build a fresh transport per subtest; sharing one
     transport across subtests will mask the bug.
   - Do not swap the field for a closure that re-reads env per request — the
     stdlib already does the right amount of caching, and re-reading per
     request would change semantics in unexpected ways.
   - Keep the explicit-override branch first in the `if` chain; the env
     fallback is the default, not the primary intent.
- **Complexity:** Easy
- **Code Example:**
  ```go
  transport := &http.Transport{
      Proxy: http.ProxyFromEnvironment, // honours HTTPS_PROXY / HTTP_PROXY / NO_PROXY
  }
  if proxyURL != "" {
      parsed, err := url.Parse(proxyURL)
      if err != nil {
          return nil, fmt.Errorf("parse proxy URL: %w", err)
      }
      transport.Proxy = http.ProxyURL(parsed) // explicit override wins
  }
  ```

### Task 2: Add a small helper that resolves the chromedp proxy URL from env
- **Description:** Add an unexported package-level helper
  `resolveProxyFromEnv() string` in
  `internal/tools/rateextractor/chromedp.go`. It reads `HTTPS_PROXY` first,
  falls back to `HTTP_PROXY`, and returns the value (or `""` when neither
  is set). Trim whitespace before returning. This helper is the single
  source of truth that the chromedp allocator queries at construction time.
  Per the user's `feedback_dedup_coincidental_vs_invariant` memory, do NOT
  export it or move it to a shared package yet: the second consumer
  (`internal/application/rulegen/chromedpfetcher.go`) gets a copy of the
  same 5-line helper local to its own file. If a third consumer ever
  appears the duplication becomes worth de-duping; until then, both copies
  stay co-located with the chromedp allocator they configure.
- **Acceptance Criteria:**
   - [ ] `resolveProxyFromEnv` returns the trimmed value of `HTTPS_PROXY`
         when set and non-empty.
   - [ ] Returns the trimmed value of `HTTP_PROXY` when `HTTPS_PROXY` is
         empty/unset.
   - [ ] Returns `""` when both are empty/unset.
   - [ ] Helper is unexported.
   - [ ] Identical copy lives in `internal/application/rulegen/chromedpfetcher.go`
         (file-local, unexported).
- **Pitfalls:**
   - `os.Getenv` returns `""` for unset and for explicitly-empty values; do
     not distinguish. Both mean "no proxy".
   - Resist the urge to also honour `NO_PROXY` here. Chromium gets `NO_PROXY`
     bypass via the `--proxy-bypass-list` companion option, which is a
     separate concern; for this task the bypass for chromedp is fine to
     handle by simply pointing `chromedp.ProxyServer` at the proxy and
     trusting the proxy itself to forward Telegram-bound traffic (which we
     do not generate from chromedp anyway). If a future task needs a
     Telegram-bypass for chromedp, add `chromedp.Flag("proxy-bypass-list", ...)`
     there.
- **Complexity:** Easy
- **Code Example:**
  ```go
  func resolveProxyFromEnv() string {
      if v := strings.TrimSpace(os.Getenv("HTTPS_PROXY")); v != "" {
          return v
      }
      return strings.TrimSpace(os.Getenv("HTTP_PROXY"))
  }
  ```

## Risks

- **Stdlib proxy cache surprises tests.** `http.ProxyFromEnvironment` uses a
  `sync.Once` in the stdlib's `httpproxy` package — the env it sees on
  first invocation is the env it remembers for the process lifetime. Tests
  that flip env vars across subtests may observe stale config. Mitigation:
  Task 5 explicitly calls this out and provides the
  `NewRateExtractorWithHTTPClient` injection fallback if the simple
  `t.Setenv` pattern misbehaves under `go test -race`.

- **Chromedp `ProxyServer` semantics.** Chromium's `--proxy-server` flag
  treats `http://host:port` as "HTTP proxy for all schemes" — but if the
  user changes the proxy later to a SOCKS5 endpoint, the URL form changes
  (`socks5://...`) and chromedp must be re-checked. Today we only support
  the HTTP CONNECT proxy at `127.0.0.1:1701`, so this is a non-issue;
  document it as a known assumption in the godoc on the helper.

- **`HTTPS_PROXY` env name case-sensitivity.** Go's stdlib is case-
  insensitive on the env name on most platforms but NOT all (Windows is
  lenient; some shells may pass through lowercase versions). The deploy
  host is Linux and the env file is operator-controlled, so this is not
  a production risk, but local dev on macOS shells that prefer
  `https_proxy` (lowercase) may behave differently. The stdlib actually
  honours both — Mitigation: document the uppercase form in `.env.example`
  and CLAUDE.md, since that is the convention every operator already
  knows.

- **Proxy down → service degraded silently.** If the local proxy crashes
  the collector and notifier produce nothing but connection-refused
  errors, persisted to `execution_history`. This is captured by the
  existing failure-tracking endpoints, but an operator who is not
  watching the logs may not notice for hours. **Out of scope for this
  plan** — adding a "proxy reachable?" healthcheck is a separate task.
  Flag it in the PR description.

- **Telegram bypass typo.** If the operator misspells `NO_PROXY` (e.g.
  `NOPROXY` or `no_proxy=api.telegram.com`), the bot will route through
  the proxy, which may or may not work depending on the proxy's
  upstream. Mitigation: Task 6 includes the EXACT three lines in
  `deploy/README.md` so the operator can copy-paste verbatim.

- **Allocator option-list test brittleness.** Strategy A in Task 5
  (count-based assertion) breaks if a future change adds another
  allocator option, which would silently invalidate the "expected delta"
  the test asserts. Mitigation: also assert the SPECIFIC delta value
  (the proxy URL string) by walking the slice and applying each option
  to a real `*chromedp.ExecAllocator` if chromedp's public API permits.
  If it does not, the count test is acceptable as a baseline; the smoke
  test in Task 7 covers the end-to-end behaviour.

## Trade-offs

- **Decision 1: env triplet over a custom DSN.** Chose `HTTPS_PROXY` /
  `HTTP_PROXY` / `NO_PROXY` over inventing a `PROXY_DSN=...` env var
  parsed by `dsninjector.Unmarshal`. Reasons:
  - Free pickup by every `http.Client` that uses `http.DefaultTransport`.
  - `NO_PROXY` gives Telegram bypass without writing a single line of
    code.
  - Operators (and humans skimming the systemd unit env file) recognise
    the convention immediately.
  - Trade-off: we lose the per-binary, structured-config style of
    `SQLITEDB_DSN` / `TELEGRAMBOT_DSN`. Acceptable because the proxy is
    a host-level concern, not a per-service concern.

- **Decision 2: duplicate the 5-line `resolveProxyFromEnv` helper instead
  of extracting it.** Aligned with the user's
  `feedback_dedup_coincidental_vs_invariant` memory: two near-identical
  copies of trivial code are cheaper than a shared dependency edge
  between `internal/tools/rateextractor` and
  `internal/application/rulegen`. If a third consumer appears we revisit.

- **Decision 3: keep the legacy `proxyURL` parameter on `NewRateExtractor`.**
  Could have removed it now that env is the default path; chose to
  preserve it as an explicit test-time override seam (tests pass a real
  `httptest.Server` URL without polluting process env). Trade-off: two
  proxy mechanisms coexist (explicit param + env fallback). The godoc
  must make the precedence rule unambiguous.

- **Decision 4: read env at construction time, not per request.** Could
  have stored the proxy URL on the struct so `Run` re-evaluates env each
  call. Chose construction-time read because: every binary in this
  project is short-lived (one-shot collector/notifier ticks) or
  restarted via systemd on env change. Hot-reload is not a use case here.

- **Decision 5: Strategy A (count-based test) for chromedp option
  verification.** Chromedp's `ExecAllocatorOption` is a function type
  with no introspection hook. Strategy B (running options against a
  test double) is invasive and depends on internals. Count + smoke test
  catches every regression that matters in this plan; if a future
  refactor changes the option list shape, the test fails and the
  developer updates the baseline. Cheap to maintain.
