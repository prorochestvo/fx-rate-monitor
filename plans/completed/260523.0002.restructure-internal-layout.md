# Task Breakdown

## Overview

Three independent, purely structural package relocations in the Go module
`github.com/seilbekskindirov/monitor`. **Zero logic or behaviour changes** — every
move is a `git mv` of files plus the mechanical rewrite of import paths and (where a
directory name changes) nothing else. The package *clause* in each file stays the same
in all three cases; only the import *path* the rest of the tree uses changes.

The three relocations, in the order recommended for execution:

1. `internal/application/labelfmt` → `internal/tools/labelfmt`
   (smallest blast radius: 2 importers, no build tags, server-only, native test fully
   covers it).
2. `pkg/api` → `internal/api`, then remove the now-empty `pkg/` directory
   (23 files import it, but the package body has no logic and no build tags).
3. `internal/wasm/{apiclient,application,dom,ui}` → `cmd/wasm/{apiclient,application,dom,ui}`
   (largest: 32 files, 5054 loc, three `//go:build js && wasm` files that native tooling
   cannot see — needs the dedicated wasm verification gate).

Rationale for the structure (agreed with the user): *code shared by multiple binaries
lives in `internal/`; code with exactly one consumer lives next to that consumer.*
`internal/api` is shared (server gateway + wasm client), so it stays in `internal/`.
The wasm packages have a single consumer (`cmd/wasm/main.go`), so they move next to it.
`labelfmt` is a cross-layer utility, so it moves to `internal/tools/` alongside the
other utilities.

## Assumptions

- The working tree is clean before each task and after each task the tree is committed
  separately, so any single relocation can be reviewed or reverted in isolation
  (one task = one commit).
- No external module imports this one. Verified: there is exactly one Go module
  (`go.mod` root) and `pkg/api` has no out-of-repo consumer, which is the entire
  justification for demoting it from `pkg/` to `internal/`.
- `go 1.26` toolchain; builds run `CGO_ENABLED=0`; the wasm artifact is built with
  `GOOS=js GOARCH=wasm`.
- Editor / IDE auto-import-rewrite is **not** relied upon. All rewrites are specified as
  explicit, greppable string replacements so a junior developer can perform and verify
  them mechanically.
- The package identifiers do not change in any task:
  - `pkg/api` declares `package api`; after the move `internal/api` still declares
    `package api`.
  - `internal/application/labelfmt` declares `package labelfmt`; unchanged after the move.
  - the four wasm packages declare `package apiclient` / `application` / `dom` / `ui`;
    unchanged after the move.
- `cmd/wasm/main.go` stays exactly where it is. Only the four subpackages move *under*
  the `cmd/wasm/` directory. This is why CI needs no edit (see Task 3 / Risks).

### Verified facts baked into this plan

- Module path: `github.com/seilbekskindirov/monitor`.
- `internal/application` contains **no** `.go` files directly — it is a directory of
  subpackages only. There is therefore no `package application` at that path and **no
  identifier collision** with the wasm `package application` after it moves to
  `cmd/wasm/application`. Confirmed: no single Go file imports both a wasm `application`
  package and a server `internal/application/*` subpackage.
- `labelfmt` importers (exactly 2):
  `internal/application/notification/message.go`,
  `internal/application/service/telegramapi.go`.
- `pkg/api` importers (23 files): `cmd/wasm/main.go`;
  `internal/gateway/httpV1/handlers/{handlers.go,handlers_test.go}`;
  and 20 files across `internal/wasm/{apiclient,application,ui}` (which themselves move
  in Task 3 — so doing Task 2 before Task 3 means those 20 import sites get rewritten
  to `internal/api` in Task 2, and their *containing directory* changes in Task 3; both
  steps are mechanical and independent).
- `internal/wasm` subpackage importers (22 files): `cmd/wasm/main.go` plus the wasm
  packages importing each other.
- Build tags present in wasm (must be preserved byte-for-byte):
  - `//go:build js && wasm` — `apiclient/fetcher_wasm.go`, `dom/fetch.go`, `dom/event.go`
  - `//go:build !js` — `apiclient/client_integration_test.go`, `apiclient/fetcher_http_test.go`
- **The three `//go:build js && wasm` files are invisible to native `go build`/`go vet`/
  `go test`.** A broken import inside them is caught **only** by the
  `GOOS=js GOARCH=wasm go build` step (the `make build` wasm line) or `make test-wasm`.
  This is the single most important verification nuance in this plan.
- The server (`cmd/web`, `internal/gateway`, `internal/application/service`) does **not**
  import any `internal/wasm` Go package. It serves the *compiled* `app.wasm` +
  `wasm_exec.js` as static assets via `//go:embed static` in `cmd/web/main.go`. Moving
  the wasm *source* does not touch that embed (the embed points at
  `cmd/web/static/app.wasm`, an output artifact, not at source packages).
- `Makefile` references to the moving paths: lines **122–123** of the `test-wasm` target
  (`./internal/wasm/dom/... ./internal/wasm/apiclient/...`). This is the **only**
  Makefile reference to a moving path and the only non-Go edit required across all three
  tasks.
- CI workflows `.github/workflows/{prime.yml,stage.yml}` build the wasm via
  `... go build -o ./cmd/web/static/app.wasm ./cmd/wasm/main.go` (prime.yml:86,
  stage.yml:89). That path (`cmd/wasm/main.go`) does **not** move, so **CI needs no
  edit**. Neither workflow runs `test-wasm` or references `pkg/api`/`labelfmt`/
  `internal/wasm/<subpkg>` directly.
- `internal/gateway/swagger/` contains a single hand-maintained `swagger.go` with **no**
  `pkg/api` reference. `make swagger` regenerates from annotations in `cmd/web/main.go`,
  not from `pkg/api`. **No swagger change is required** by any task.
- `pkg/` contains only `pkg/api`. After Task 2, `pkg/` is empty and is removed.
- Baseline `CGO_ENABLED=0 go build ./...` is green before any change.

### Task 1: Move `internal/application/labelfmt` → `internal/tools/labelfmt`

- **Description:**
  `labelfmt` is a 152-loc human-readable label *formatter* — a cross-layer utility, not
  application orchestration. It belongs under `internal/tools/` next to the project's
  other utilities, not under `internal/application/`. The package clause stays
  `package labelfmt`; only its import path changes from
  `…/internal/application/labelfmt` to `…/internal/tools/labelfmt`.

  Steps:
  1. Move the directory, preserving git history:
     ```bash
     git mv internal/application/labelfmt internal/tools/labelfmt
     ```
     (Both `label.go` and `label_test.go` move together.)
  2. Find every import site:
     ```bash
     grep -rl "seilbekskindirov/monitor/internal/application/labelfmt" --include="*.go" .
     ```
     Expected result (exactly two files):
     `internal/application/notification/message.go`,
     `internal/application/service/telegramapi.go`.
  3. Rewrite the import path in those two files
     (`…/internal/application/labelfmt` → `…/internal/tools/labelfmt`). The imported
     identifier used at call sites (`labelfmt.<X>`) does **not** change, because the
     package name is unchanged.
  4. `go fmt ./...` (gofmt re-sorts the import block; the new path may sort to a
     different line within the group).

- **Acceptance Criteria:**
  - [ ] `internal/application/labelfmt/` no longer exists; `internal/tools/labelfmt/`
        contains `label.go` and `label_test.go`.
  - [ ] `grep -rn "internal/application/labelfmt" .` returns nothing.
  - [ ] The two importers reference `…/internal/tools/labelfmt` and call sites are
        unchanged (`labelfmt.<X>` still resolves).
  - [ ] `make test` is green (native, `-race`, `CGO_ENABLED=0`). This package is
        server-only and fully native-compiled, so `make test` is a complete gate here.
  - [ ] `make build` is green (no behaviour change; sanity that nothing else broke).
  - [ ] `git diff --stat` shows only renames + the two import-line edits — no body diff
        in `label.go`/`label_test.go` beyond gofmt-neutral movement.

- **Pitfalls & edge cases:**
  - Do **not** rename the package or any exported identifier; this is a path move only.
  - gofmt will reorder the import group — that is expected and not a logic change.
  - Confirm there are exactly two importers before editing; if `grep` shows more, stop
    and re-verify scope (the plan assumes two).

- **Complexity:** Easy

### Task 2: Move `pkg/api` → `internal/api` and remove the empty `pkg/`

- **Description:**
  `pkg/api` is a pure-DTO wire-contract package (JSON request/response structs, no logic,
  no dependencies). It currently lives in `pkg/` (public, externally importable), but
  there is no external consumer and the project is a single Go-only module. Moving it
  under `internal/` makes the boundary honest: the compiler forbids external coupling and
  the contract becomes a private detail free to reshape. The package clause stays
  `package api`; only the import path changes from `…/pkg/api` to `…/internal/api`.

  Steps:
  1. Move the directory:
     ```bash
     git mv pkg/api internal/api
     ```
  2. Remove the now-empty `pkg/` directory (it contained only `api/`):
     ```bash
     rmdir pkg
     ```
     (git tracks files, not empty dirs; once `pkg/api` is moved, `pkg/` has no tracked
     content. `rmdir` removes the leftover empty directory from the working tree.)
  3. Find every import site:
     ```bash
     grep -rl "seilbekskindirov/monitor/pkg/api" --include="*.go" .
     ```
     Expected: 23 files —
     - `cmd/wasm/main.go`
     - `internal/gateway/httpV1/handlers/handlers.go`
     - `internal/gateway/httpV1/handlers/handlers_test.go`
     - 20 files under `internal/wasm/{apiclient,application,ui}` (these still live under
       `internal/wasm/` at this point; Task 3 moves their directory afterward).
  4. Rewrite the import path in all 23 files
     (`…/pkg/api` → `…/internal/api`). The imported identifier (`api.<X>`) is unchanged
     because the package name stays `api`.
  5. `go fmt ./...`.

- **Acceptance Criteria:**
  - [ ] `pkg/` no longer exists; `internal/api/` contains all eight DTO files
        (`me.go`, `subscriptions.go`, `events.go`, `notifications.go`, `stats.go`,
        `sources.go`, `history.go`, `rates.go`).
  - [ ] `grep -rn "monitor/pkg/api" .` and `grep -rn '"pkg/api"' .` return nothing.
  - [ ] All 23 importers reference `…/internal/api`; call sites (`api.<X>`) unchanged.
  - [ ] `make build` is green — and this matters here because the wasm packages import
        the moved `api` package. None of the three `js && wasm` files import `pkg/api`
        directly, but `make build` runs the `GOOS=js GOARCH=wasm` step that compiles
        `cmd/wasm/main.go`'s full transitive graph, including the wasm packages that DO
        import `api`, so a missed rewrite there surfaces at the wasm build.
  - [ ] `make test` is green (native, `-race`, `CGO_ENABLED=0`).
  - [ ] `make test-wasm` is green if Node is available (it compiles `dom` + `apiclient`
        under the js runtime; `apiclient` imports the moved `api` package). If Node is
        unavailable, the `make build` wasm step is the required substitute gate.
  - [ ] No file outside `internal/api/` declares `package api` for these DTOs (no
        accidental package-name collision introduced).

- **Pitfalls & edge cases:**
  - The package name `api` is generic. Confirm no other package in the tree is named
    `api` and imported alongside this one without an alias (verified: none today). If a
    future file imports both, an alias would be needed — out of scope here, just be aware.
  - 20 of the 23 importers are wasm files that move in Task 3. Doing Task 2 first means
    you rewrite their import path now, and Task 3 only changes their *directory*. This is
    deliberate and keeps each task's diff coherent (Task 2 = "api path", Task 3 = "wasm
    location"). Do not try to combine them.
  - Removing `pkg/` with `rmdir` only succeeds if it is truly empty. If `rmdir` fails,
    something else lives under `pkg/` that was not in scope — stop and report.
  - The `cmd/web/main.go` `//go:embed static` is unaffected (it embeds compiled assets,
    not `pkg/api`).

- **Complexity:** Easy (mechanical, but 23 files — verify the count before and after).

### Task 3: Move `internal/wasm/{apiclient,application,dom,ui}` → `cmd/wasm/{...}`

- **Description:**
  The entire WASM browser frontend (~5054 loc, 32 files) is a separate program whose only
  consumer is `cmd/wasm/main.go`. Per the agreed principle (single-consumer code lives
  next to its consumer), the four subpackages move from `internal/wasm/` to
  `cmd/wasm/`. `cmd/wasm/main.go` itself does not move. After the move, `internal/wasm/`
  is empty and is removed, shrinking `internal/` so it reads again as the layers of the
  one server app. Package clauses (`apiclient`, `application`, `dom`, `ui`) are unchanged;
  only import paths change from `…/internal/wasm/<pkg>` to `…/cmd/wasm/<pkg>`.

  Steps:
  1. Move each subpackage, preserving build-tagged files verbatim:
     ```bash
     git mv internal/wasm/apiclient   cmd/wasm/apiclient
     git mv internal/wasm/application cmd/wasm/application
     git mv internal/wasm/dom         cmd/wasm/dom
     git mv internal/wasm/ui          cmd/wasm/ui
     rmdir internal/wasm
     ```
     `git mv` moves files byte-for-byte, so the `//go:build` lines at the top of
     `apiclient/fetcher_wasm.go`, `dom/fetch.go`, `dom/event.go` (`js && wasm`) and
     `apiclient/{client_integration_test.go,fetcher_http_test.go}` (`!js`) are preserved.
  2. Find every import site:
     ```bash
     grep -rl "seilbekskindirov/monitor/internal/wasm" --include="*.go" .
     ```
     Expected: 22 files — `cmd/wasm/main.go` plus the wasm packages importing each other.
     **Crucially, this includes the three `js && wasm` files**, which `grep` finds
     (it reads text, ignoring build tags) even though native `go vet` cannot compile them.
  3. Rewrite every occurrence of `…/internal/wasm/` → `…/cmd/wasm/` across those 22 files
     (e.g. `…/internal/wasm/dom` → `…/cmd/wasm/dom`). Imported identifiers
     (`apiclient.`, `application.`, `dom.`, `ui.`) are unchanged.
  4. Update the Makefile `test-wasm` target — the only non-Go edit in the whole plan.
     Lines 122–123 currently read:
     ```make
     		./internal/wasm/dom/... ./internal/wasm/apiclient/...
     ```
     Change to:
     ```make
     		./cmd/wasm/dom/... ./cmd/wasm/apiclient/...
     ```
  5. `go fmt ./...`.

- **Acceptance Criteria:**
  - [ ] `internal/wasm/` no longer exists; `cmd/wasm/{apiclient,application,dom,ui}/`
        contain all 32 files with build tags intact
        (`grep -rn "//go:build" cmd/wasm/` shows the same five tagged files as before).
  - [ ] `grep -rn "internal/wasm" .` returns nothing (Go files, Makefile, anywhere).
  - [ ] Makefile `test-wasm` references `./cmd/wasm/dom/... ./cmd/wasm/apiclient/...`.
  - [ ] `make build` is green — this is the **load-bearing gate** for this task because
        its `CGO_ENABLED=0 GOOS=js GOARCH=wasm go build -o ./cmd/web/static/app.wasm
        ./cmd/wasm/main.go` step is the only thing that compiles the three
        `js && wasm` files. A native-only check would silently miss a broken import in
        them.
  - [ ] `make test` is green (native, `-race`, `CGO_ENABLED=0`) — covers all
        non-js-tagged wasm files (`apiclient` 4, `application` 5, `dom` 1, `ui` 5
        production files plus their `_test.go`).
  - [ ] `make test-wasm` is green if Node is available (compiles `dom` + `apiclient`
        including the js-tagged `dom/fetch.go`, `dom/event.go`,
        `apiclient/fetcher_wasm.go` under the js runtime and runs their tests). This is
        the strongest gate for the js-tagged code; the `make build` wasm step is the
        fallback when Node is absent.
  - [ ] `cmd/web` / `internal/gateway` / `internal/application/service` are untouched
        (the server never imported `internal/wasm`); `cmd/web/main.go`'s
        `//go:embed static` still resolves and `app.wasm` is still produced at the same
        path.
  - [ ] No import cycle introduced (`go build ./...` and the wasm build both succeed;
        `internal/api` imported by both `internal/gateway` and `cmd/wasm/*` is a tree,
        not a cycle).

- **Pitfalls & edge cases:**
  - **The three `js && wasm` files are the trap.** Native `make test` / `go vet` will
    pass even if their import paths are wrong, because the native build excludes
    `GOOS=js` files. You **must** run the `GOOS=js GOARCH=wasm` build (via `make build`
    or `make test-wasm`) to validate them. Treat a green native run as necessary but
    **not sufficient** for this task.
  - Preserve the `//go:build` first-line comments exactly. `git mv` does this; if any
    file is recreated by hand instead of moved, the tag (and the blank line after it,
    required by Go) must be reproduced verbatim. Also preserve the `!js` tags on the two
    test files.
  - Making `cmd/wasm/` "fat" (a command directory holding four sub-packages, not just
    `main.go`) is acceptable Go convention: subpackages under a `cmd/<x>/` directory are
    a normal way to keep a binary's private code co-located, and `internal/`-style
    encapsulation is not required because nothing outside this module imports it. This is
    the user's accepted rationale (single consumer → co-located).
  - `cmd/wasm/application` and the server's `internal/application/*` subpackages share no
    import path and no co-import, so the duplicated *directory name* "application" is only
    a reading concern, not a compile concern. The package identifiers (`application` for
    the wasm one; `collection`/`notification`/`service`/etc. for the server ones) do not
    collide. No alias needed.
  - Do the `pkg/api` → `internal/api` rewrite (Task 2) **before** this task so the wasm
    files' `api` import is already correct; here you only change their `internal/wasm` →
    `cmd/wasm` path. If Task 3 is somehow done first, the wasm files would still import
    `…/pkg/api` and Task 2 would have to rewrite them under their new `cmd/wasm/`
    location — same end state, but the recommended order keeps each diff cleaner.

- **Complexity:** Medium (mechanically simple, but 22 files + the js-tagged blind spot
  make verification the hard part).

## Execution Order

Recommended order: **Task 1 → Task 2 → Task 3**. The three are technically independent
(no task depends on another's output to compile), so they could be done in any order, but
this sequence is ordered by **ascending blast radius and verification difficulty**, which
minimises the chance of conflating a real break with a path-rewrite artifact:

1. **Task 1 (labelfmt)** first — smallest (2 importers), server-only, fully covered by
   native `make test`. A clean, low-risk warm-up that exercises the move-and-rewrite
   procedure with the simplest possible verification gate.
2. **Task 2 (pkg/api → internal/api)** second — more importers (23) but still no build
   tags and no logic. Doing it before Task 3 means the 20 wasm import sites get their
   `api` path corrected while still in their original directory, so Task 3's diff is
   purely a directory move.
3. **Task 3 (wasm → cmd/wasm)** last — largest, and the only task with `js && wasm` files
   that native tooling cannot validate. Doing it last means the only remaining variable
   when the wasm-specific gate (`make build` / `make test-wasm`) runs is the wasm
   directory move itself, making any failure easy to localise.

Each task is its own atomic commit (one move = one reviewable, revertible commit). Run
the full gate (`make build` **and** `make test`, plus `make test-wasm` for Tasks 2 and 3
when Node is available) after each task before committing the next.

Parallelisation note: because the three are independent, they *could* be split across
people, but the shared rewrite of the 20 wasm files (Task 2 edits their `api` import;
Task 3 moves their directory) means doing 2 and 3 concurrently on the same files invites
merge churn. Keep them sequential.

## Risks

- **js/wasm blind spot (highest risk).** A wrong import in a `//go:build js && wasm` file
  (`apiclient/fetcher_wasm.go`, `dom/fetch.go`, `dom/event.go`) passes native
  `make test`/`go vet` silently. Mitigation: Task 3's gate **mandates** the
  `GOOS=js GOARCH=wasm` build (`make build` wasm step) and, where Node exists,
  `make test-wasm`. Do not sign off Task 3 on a native-only green.
- **Stale path references outside Go.** Verified today: the only non-Go reference to a
  moving path is Makefile lines 122–123 (Task 3). CI workflows reference only
  `cmd/wasm/main.go` (unchanged) and contain no `pkg/api`/`labelfmt`/`internal/wasm/<pkg>`
  references; swagger references none. Residual risk: a reference added between now and
  execution. Mitigation: re-run
  `grep -rn "internal/wasm\|pkg/api\|internal/application/labelfmt" Makefile .github/`
  immediately before each task.
- **Lost build tags.** If any file is hand-recreated instead of `git mv`d, a dropped
  `//go:build` line (or the required trailing blank line) silently changes which files
  compile under which GOOS. Mitigation: use `git mv` only; verify with
  `grep -rn "//go:build" cmd/wasm/` post-move (Task 3) — five tagged files expected.
- **Empty-directory cleanup.** `rmdir pkg` (Task 2) and `rmdir internal/wasm` (Task 3)
  fail loudly if anything unexpected remains, which is the desired behaviour — treat a
  failure as a signal that scope was wider than assumed and stop.
- **Import cycle.** None expected: `internal/api` is a leaf DTO package imported by both
  `internal/gateway` and `cmd/wasm/*`, which is a fan-in, not a cycle. Mitigation: the
  build itself enforces this — `go build` fails on a cycle, so the existing gate covers it.

## Trade-offs

- **`pkg/api` → `internal/`, not kept public.** Chose `internal/api` over leaving it in
  `pkg/` because there is no external consumer and a single module. `internal/` makes the
  compiler enforce that the wire contract is a private, reshapeable detail. The cost —
  the package can no longer be imported by a hypothetical future external module without
  being moved back — is acceptable given the project is and intends to remain a single
  Go module. If a public API package is ever genuinely needed, that is a deliberate
  future decision, not something to pre-provision now.
- **Wasm under `cmd/wasm/` ("fat command"), not a `frontend/` top-level.** Chose
  co-location with the single consumer over a separate top-level frontend tree. This keeps
  the agreed principle consistent (single consumer → next to consumer) and shrinks
  `internal/` to read as the server's layers. The cost is that `cmd/wasm/` holds four
  subpackages rather than just `main.go`, which is slightly heavier than a typical
  `cmd/<x>/`; this is idiomatic enough (subpackages under a command are a recognised
  pattern) and was explicitly accepted by the user.
- **`labelfmt` → `internal/tools/`, not left under `internal/application/`.** Chose to
  classify it as a cross-layer utility (it formats display labels for two different
  application packages) rather than application orchestration. The cost is one more entry
  under `internal/tools/`; the benefit is that `internal/application/` holds only genuine
  business-logic packages.
- **Three separate commits, not one bundled move.** Chose atomic per-relocation commits
  so any single move can be reviewed or reverted independently, at the cost of three
  round-trips through the build/test gate instead of one. Given the js/wasm verification
  subtlety in Task 3, isolating that move from the trivial Tasks 1–2 is worth the extra
  gate runs.
