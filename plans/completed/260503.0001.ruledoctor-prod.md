# 003 — ruledoctor production integration (STEP 2)

## Overview

STEP 1 (`plans/002-ruledoctor-mvp.md`) closed: Groq Llama 3.1 8B Instant
extracts the correct value on 45/45 pairs across the NBK table layout, the
BCC div-card layout, and the (chromedp-prerendered) Bankffin React layout.
CSS selectors verify on 89%, regex on 60%. Pluggable `Generator` interface
with four providers (`OllamaClient`, `AnthropicClient`, `ClaudeCodeClient`,
`GroqClient`), `Snipper`, `Cleaner`, `BuildPrompt`/`ParseExtraction`, and
`Verify` are all green and offline-tested.

This plan integrates the proven extraction pipeline into the running system
**without** broadening scope. Specifically:

- Add a generic `extraction_rules` table (target-kind agnostic — designed so
  weather / commodities / arbitrary structured-from-HTML use cases plug in
  with no schema change).
- Wire the collector to consult `extraction_rules` first, falling back to the
  inline rules already on `rate_sources` so live monitoring is never broken
  during rollout.
- Ship a `cmd/ruledoctor` CLI that reads target identifiers, fetches HTML,
  runs the existing pipeline, verifies, and emits **one** SQL artifact per
  run that the user reviews and applies manually.
- Expose a "rule broken" signal the collector raises when extraction fails N
  times in a row, and that ruledoctor consumes to pick targets to regenerate.

Out of scope (deferred): pre-render / chromedp / XHR-pivot integration —
will be `plans/004-ruledoctor-prerender.md`. Scheduled / GitHub Actions /
VPS execution — same future plan family.

## Assumptions

1. **Where ruledoctor runs.** User's Mac, manually (`make ruledoctor-run …`).
   No cron, no GH Actions, no VPS. The Mac always has `GROQ_API_KEY` (and
   optionally `ANTHROPIC_API_KEY`) in the environment.
2. **How rules ship to prod.** ruledoctor writes a `.sql` file; the user
   `scp`s it to the prod VPS and applies it with the existing `sqlite3`
   binary. Schema migrations stay owned by Go `Migration()` per
   `CLAUDE.md` — only **data** ships as SQL artifacts.
3. **Rollout discipline.** The new `extraction_rules` table COEXISTS with
   the existing `rate_sources.rules` JSON column for the duration of this
   plan. Migration to "rules-table-only" is a follow-up and not in this plan.
   The collector checks `extraction_rules` first; if no active rule is found
   it falls back to `rate_sources.rules`. This keeps every currently working
   source running unchanged and lets us promote sources to LLM-generated
   rules one at a time.
4. **Target identifier semantics.** For `kind = "rate"`, the target identifier
   equals `rate_sources.name` (an existing primary key). For future kinds
   (e.g. `"weather"`) the consumer interprets the identifier however it
   wants — the rules table itself holds an opaque string.
5. **Failure threshold.** Three consecutive failed `execution_history` rows
   for the same source mark the active rule "broken" (visible to
   ruledoctor). Three is small enough to react quickly, large enough to ride
   out one or two transient network blips.
6. **Default provider.** `RULEDOCTOR_PROVIDER=groq`,
   `RULEDOCTOR_MODEL=llama-3.1-8b-instant`. Cascade to a stronger provider
   on verification failure is **optional** and gated behind a flag (kept
   simple in v1: single-shot, no cascade).
7. **No new runtime dependencies.** `goquery` is already vendored; `chromedp`
   and the Anthropic / Groq HTTP clients are already in go.mod via the
   ruledoctor MVP. Nothing new.

### Task 3: Add `ExtractionRuleRepository` with append-only migration

- **Description:** New file `internal/repository/extractionrule.go`,
  modelled exactly on `RateSourceRepository`:
  - `NewExtractionRuleRepository(db) (*ExtractionRuleRepository, error)`
    runs its own migration on construction.
  - `Migration()` returns a single keyed entry
    `extraction_rules_001_table_initiate` whose DDL creates the table and
    indexes (see schema below). Append-only — never edit this entry.
  - Methods needed by Tasks 5–7:
    - `RetainExtractionRule(ctx, *ExtractionRule) error` — INSERT or UPDATE
      by ID (same upsert pattern as `RetainRateSource`).
    - `ObtainActiveRulesByTarget(ctx, kind, targetID string) ([]ExtractionRule, error)`
      — returns every `status='active'` row for that (kind, target), one
      per label, in stable order (`label ASC`). Hot path for the collector
      (one source has many pairs). Empty slice (not error) when none.
    - `ObtainActiveRuleByLabel(ctx, kind, targetID, label string) (*ExtractionRule, error)`
      — returns the single active row for the (kind, target, label) triple,
      or `nil, nil` if none. Used by `InstallActiveRule` to detect prior
      occupant of that slot.
    - `ObtainAllRulesByTarget(ctx, kind, targetID string) ([]ExtractionRule, error)`
      — read-only history; ordered `generated_at DESC`. Used by the CLI's
      `list-rules` subcommand and by future UI surfaces.
    - `MarkRuleStatus(ctx, id string, status ExtractionRuleStatus) error` —
      transitions; used to flip `active → superseded` when a new rule is
      installed and `active → broken` when the failure detector trips.
    - `ObtainBrokenTargets(ctx, kind ExtractionRuleKind) ([]string, error)` —
      returns distinct target IDs with at least one rule in status `broken`
      and no currently-`active` rule for the SAME label. Consumed by
      ruledoctor's `regenerate-broken` sub-command. (Implementation note:
      `SELECT DISTINCT target_id FROM extraction_rules WHERE target_kind=?
      AND status='broken' AND NOT EXISTS (SELECT 1 FROM extraction_rules e2
      WHERE e2.target_kind = extraction_rules.target_kind AND e2.target_id =
      extraction_rules.target_id AND e2.label = extraction_rules.label AND
      e2.status='active');`)

  Schema (final column names belong to the engineer):

  ```sql
  CREATE TABLE IF NOT EXISTS extraction_rules (
      id               TEXT NOT NULL PRIMARY KEY,
      target_kind      TEXT NOT NULL,
      target_id        TEXT NOT NULL,
      label            TEXT NOT NULL DEFAULT '',
      source_url       TEXT NOT NULL,
      method           TEXT NOT NULL,
      pattern          TEXT NOT NULL,
      provider_tag     TEXT NOT NULL,
      context_hash     TEXT NOT NULL DEFAULT '',
      status           TEXT NOT NULL DEFAULT 'active',
      generated_at     INT  NOT NULL,
      last_verified_at INT,
      notes            TEXT NOT NULL DEFAULT ''
  );
  CREATE INDEX IF NOT EXISTS idx_extraction_rules_active
      ON extraction_rules (target_kind, target_id, status);
  CREATE INDEX IF NOT EXISTS idx_extraction_rules_lookup
      ON extraction_rules (target_kind, target_id, generated_at DESC);
  CREATE UNIQUE INDEX IF NOT EXISTS uniq_extraction_rules_active_per_label
      ON extraction_rules (target_kind, target_id, label) WHERE status = 'active';
  ```

  The partial unique index enforces "at most one active rule per
  (kind, target, label)" at the DB level — multiple labels for the same
  source coexist (NBK has 39 pairs, one rule per pair), but a second
  active rule for an already-occupied (kind, target, label) is rejected.
  Single-datum kinds (where one URL → one extracted value) use `label=''`
  and the index degenerates to "one active rule per (kind, target)".

- **Acceptance Criteria:**
  - [ ] Migrator finds and applies the new migration on first boot.
  - [ ] Re-running on an already-migrated DB is a no-op.
  - [ ] `RetainExtractionRule` round-trips every field including `Label` and
        nullable `LastVerifiedAt`.
  - [ ] Inserting a second `active` row for the same (kind, target, label)
        returns an SQLite uniqueness error. Two `active` rows for the same
        (kind, target) but DIFFERENT labels coexist without error.
  - [ ] `ObtainActiveRulesByTarget` returns an empty slice (NOT an error)
        when no rules exist; otherwise returns rules in `label ASC` order.
  - [ ] `ObtainActiveRuleByLabel` returns `nil, nil` (NOT an error) when no
        rule exists for the (kind, target, label) triple.
  - [ ] Tests in `internal/repository/extractionrule_test.go` cover: insert,
        upsert (update), per-label active lookup, multi-label active listing,
        status transitions, broken-target listing, and the unique-active
        constraint (both same-label rejection AND different-label coexistence).
        Use the existing `stubSQLiteDB` helper and the `var _ db = ...`
        compile-time check.
  - [ ] At least one DB-failure subtest per public method, modelled on
        `internal/repository/ratevalue_test.go`'s `mockFailDB` pattern.
        Required because every existing repo in the project carries that
        coverage; new repo must match the convention.
  - [ ] One top-level `Test*` per method, scenarios as `t.Run` subtests
        (per CLAUDE.md).
- **Pitfalls:**
  - SQLite stores booleans / times as INTEGER. Follow the `execution_history`
    pattern (`time.Time.Unix()` for storage, `time.Unix(...).UTC()` on read),
    not the `rate_values` RFC3339-string pattern. Unix epoch makes
    `ORDER BY generated_at DESC` cheap.
  - `LastVerifiedAt` is nullable. Use `sql.NullInt64` on the Go side.
  - The partial unique index (`WHERE status = 'active'`) is supported by
    SQLite ≥ 3.8.0 — modernc.org ships much newer than that.
  - When upserting an `active` rule, callers must call `MarkRuleStatus(prev,
    "superseded")` first inside the same transaction — design the API so the
    engineer is forced to think about this. Suggest a higher-level repo
    method `InstallActiveRule(ctx, *ExtractionRule)` that wraps both
    operations in a single tx.
- **Complexity:** Medium
- **Code Example:**
  ```go
  func (r *ExtractionRuleRepository) InstallActiveRule(ctx context.Context, rule *ExtractionRule) error {
      tx, err := r.db.Transaction(ctx)
      if err != nil {
          return errors.Join(err, internal.NewStackTraceError())
      }
      defer printRollbackError(tx)

      // Demote any currently-active rule for the same target.
      _, err = tx.ExecContext(ctx,
          "UPDATE "+extractionRuleTableName+" SET "+statusField+" = ? "+
          "WHERE "+kindField+" = ? AND "+targetField+" = ? AND "+statusField+" = ?;",
          ExtractionRuleStatusSuperseded, rule.TargetKind, rule.TargetID, ExtractionRuleStatusActive)
      if err != nil { return errors.Join(err, internal.NewTraceError()) }

      // Insert the new one.
      // ... INSERT, then tx.Commit() ...
  }
  ```

### Task 5: Teach `RateAgent` (and `RateExtractor`) to consult `extraction_rules` first

- **Description:** In `internal/application/collection/rateagent.go` and
  `internal/tools/rateextractor/rateextractor.go`, change the per-source
  execution path so that:

  1. Before applying `source.Rules`, look up active rules via
     `ExtractionRuleRepository.ObtainActiveRulesByTarget("rate", source.Name)`.
     This returns a slice — one rule per pair label.
  2. If the slice is non-empty, build a `[]RateSourceRule` from it (one
     rule per label) and use that instead of `source.Rules`. Each label
     becomes the pair name on the resulting `RateValue`. For each rule,
     the method dispatches as today:
     - For `MethodCSS`: `goquery` (the same code path as
       `internal/ruledoctor/verify.go`'s `applyCSS`). Extract trimmed text
       from the first match, then run it through the existing
       trim/comma-replace + `strconv.ParseFloat` tail.
     - For `MethodRegex`: existing `fetchRegexPage` path.
     - For `MethodJSONPath`: existing `extractJSONPath` path.
     - For `MethodParseFloat`: existing path.
  3. If the slice is empty, fall back to `source.Rules` (current
     behaviour, untouched). All-or-nothing fallback: we do NOT splice
     `extraction_rules` and inline rules together. A source has either
     migrated to the new table (some labels covered) or it hasn't.
     Partial coverage is a configuration error the user resolves by
     either adding the missing labels to the targets file and re-running
     ruledoctor, or deleting the partial active rows.
  4. On success, the agent updates `LastVerifiedAt = now()` on the rule
     that produced the value (cheap UPDATE keyed by ID). On failure, the
     agent does NOT mutate the rule — Task 7 handles broken-status
     promotion based on `execution_history` aggregation, not on a single
     failure.

  Add a new `extractionRuleRepository` interface in `rateagent.go`:

  ```go
  type extractionRuleRepository interface {
      ObtainActiveRulesByTarget(ctx context.Context, kind domain.ExtractionRuleKind, targetID string) ([]domain.ExtractionRule, error)
      TouchVerifiedAt(ctx context.Context, ruleID string, when time.Time) error
  }
  ```

  Inject it via `NewRateAgent`. Keep `rateExtractor.Run`'s signature; the
  rule resolution happens inside the agent, and the agent passes the
  resolved per-label rule slice into the extractor (or calls a new
  `RunWithRules` to avoid mutating the source struct in place).

- **Acceptance Criteria:**
  - [ ] Unit test: source has both inline `Rules` and ≥1 active CSS rule in
        `extraction_rules` → the CSS rules are used; inline `Rules` ignored.
  - [ ] Unit test: source has only inline `Rules`, empty
        `extraction_rules` → inline rules are used (back-compat).
  - [ ] Unit test: source has 3 active rules (one per label) — all 3 are
        applied, producing 3 `RateValue` rows with the right `Pair` per
        rule's `Label`.
  - [ ] Unit test: source has an active CSS rule pointing at a selector that
        matches no nodes → `execution_history` records `success=false` with
        a clear error string. No panic from goquery / cascadia (use the
        existing `safeFind` recover idiom from `verify.go`).
  - [ ] Unit test: successful run updates `LastVerifiedAt` on the producing
        rule to within 5s of `time.Now()`.
  - [ ] `rateagent_test.go` mocks satisfy the new interface, with the
        `var _ extractionRuleRepository = &mockX{}` compile-time check.
- **Pitfalls:**
  - **Do not** mutate `source.Rules` in place — fresh `[]domain.RateSourceRule`
    slice per run, or thread the rule directly without going through that
    field.
  - The CSS branch returns the FIRST match's text. The cleaner used in
    ruledoctor is irrelevant here — the LIVE page is fetched, not the
    cleaned/snipped version. The selector was generated against the cleaned
    fragment but `Verify` already proves it works against the original; we
    rely on that invariant.
  - Guard against a goquery panic on a selector containing pseudo-classes
    cascadia rejects at parse time. Wrap with `safeFind`.
  - `TouchVerifiedAt` must NOT fail the collector run on an UPDATE error —
    log and continue (the rate value was already saved).
- **Complexity:** Medium
- **Code Example:**
  ```go
  // resolveRules returns the rules to apply (per-label) and a parallel slice
  // of *ExtractionRule for the LastVerifiedAt UPDATE on success. When the
  // returned []ExtractionRule is empty, the caller uses source.Rules.
  func (a *RateAgent) resolveRules(ctx context.Context, source domain.RateSource) ([]domain.RateSourceRule, []domain.ExtractionRule, error) {
      active, err := a.extractionRuleRepository.ObtainActiveRulesByTarget(
          ctx, domain.ExtractionRuleKindRate, source.Name,
      )
      if err != nil {
          return nil, nil, errors.Join(err, internal.NewTraceError())
      }
      if len(active) == 0 {
          return source.Rules, nil, nil
      }
      out := make([]domain.RateSourceRule, 0, len(active))
      for _, r := range active {
          out = append(out, domain.RateSourceRule{
              Pair:    r.Label,
              Method:  r.Method,
              Pattern: r.Pattern,
          })
      }
      return out, active, nil
  }
  ```

### Task 7: Failure detector — promote a rule to `broken` after N failures

- **Description:** Add a small detector that runs at the end of every
  collector invocation (after `RateAgent.Run` returns):

  ```go
  // BrokenRulePromoter inspects recent execution_history per source and,
  // if the last N rows are all failures and an active extraction_rule exists
  // for that source, marks it broken.
  type BrokenRulePromoter struct { ... }
  func (p *BrokenRulePromoter) Run(ctx context.Context) error
  ```

  Place it under `internal/application/collection/brokenrule.go` (same
  package as `RateAgent`). N = 3 (constant
  `brokenRuleConsecutiveFailureThreshold = 3`). Algorithm:

  1. For every active rule (`extraction_rules.status = 'active'`), pull the
     last 3 `execution_history` rows for `source_name = target_id`.
  2. If all 3 are `success = 0`, call `MarkRuleStatus(rule.ID, "broken")`.
  3. Log the transition; do not error the run if a single mark fails.

  Wire into `cmd/collector/main.go` after the `RateAgent.Run` call.

- **Acceptance Criteria:**
  - [ ] Unit test: 3 consecutive failures + active rule → rule becomes
        `broken`. Re-running the promoter is a no-op (already broken).
  - [ ] Unit test: 2 failures + 1 success → rule stays `active`.
  - [ ] Unit test: failures but no active rule for that source → no
        repository call, no error.
  - [ ] Unit test: <3 history rows total → rule stays `active` (we don't
        promote on ambiguous evidence).
- **Pitfalls:**
  - Edge: a brand-new source with no history. `ObtainLastNExecutionHistoryBySourceName`
    returns 0 rows → do not promote.
  - Iterating over every active rule is fine for the project's volume
    (~10–20 sources). If it grows, the index on
    `(target_kind, target_id, status)` keeps it cheap; no extra optimisation
    needed in this plan.
  - Do NOT re-use the "successOnly" flag of
    `ObtainLastNExecutionHistoryBySourceName` — we need both successes and
    failures to compute the streak. Add a new variant or pass `successOnly=false`
    (it already supports that).
- **Complexity:** Medium

### Task 8: Add `cmd/ruledoctor` CLI

- **Description:** New binary `cmd/ruledoctor/main.go`. Manual-run from the
  user's Mac. Subcommands (`flag.NewFlagSet`-based; one `main` is fine, no
  cobra):

  - `generate --target=<name>[,<name>...]` — fetch HTML for each target
    (via the existing static `http.GET` only — pre-render is a future plan;
    if the page returns no rate-shape number, log a warning and skip), run
    the existing pipeline (`Clean` → `SnipForPair` → `BuildPrompt` →
    `Generator.Generate` → `ParseExtraction` → `Verify`), and append an
    INSERT-or-UPDATE statement to the run's SQL artifact.
  - `regenerate-broken` — read `ObtainBrokenTargets("rate")`, then run
    `generate` for each.
  - `list-rules [--target=<name>]` — read-only; prints active rules.

  **Source list: local JSON file.** The CLI does NOT touch the prod SQLite
  DB or any HTTP API. It reads a flat JSON file (default
  `./configs/ruledoctor-targets.json`, override with `--targets-file=<path>`)
  describing every target it knows about. Format:

  ```json
  [
    {
      "target_kind": "rate",
      "target_id":   "nbk_kzt",
      "source_url":  "https://nationalbank.kz/rates/...",
      "labels":      ["EUR / KZT", "USD / KZT", "RUB / KZT"]
    }
  ]
  ```

  `labels` are the per-pair anchors the snipper uses (one rule is generated
  per label). The `--target=<name>` flag, when present, filters by
  `target_id` after the file is parsed. A missing or empty file is a hard
  error with a clear "create configs/ruledoctor-targets.json" message.

  This is the manual-control-phase shape; revisit (DB-backed source list,
  REST API, etc.) is deferred. Decision recorded in
  `plans/002-ruledoctor-mvp.md` lineage.

  Configuration via env (consistent with collector / notifier):

  - `RULEDOCTOR_PROVIDER` (default `groq`), `RULEDOCTOR_MODEL` (default
    `llama-3.1-8b-instant`), `GROQ_API_KEY` / `ANTHROPIC_API_KEY`,
    `RULEDOCTOR_TIMEOUT`, `PROXY_URL` — all already honoured by the existing
    integration test code; reuse the construction helpers (extracted from
    `extract_test.go` if needed; do not import test code).
  - `RULEDOCTOR_OUT_DIR` (default `./tmp/ruledoctor-out/`) — where the SQL
    artifact lands. Filename:
    `YYYYMMDD-HHMMSS-<NNNN>-ruledoctor.sql` where `NNNN` resets to `0001`
    each day.

  Output format (one file per CLI invocation, regardless of how many
  targets were processed):

  ```sql
  -- ruledoctor run 2026-05-04T12:34:56Z provider=groq:llama-3.1-8b-instant
  -- targets: nbk_kzt (3 labels), bcc_kzt (1 label)
  -- This file modifies extraction_rules ONLY. Schema is owned by Go migrations.

  BEGIN TRANSACTION;

  -- nbk_kzt label="EUR / KZT": verified css ok=true regex ok=true value="542.16"
  UPDATE extraction_rules
     SET status = 'superseded'
   WHERE target_kind = 'rate' AND target_id = 'nbk_kzt'
     AND label = 'EUR / KZT' AND status = 'active';

  INSERT INTO extraction_rules (id, target_kind, target_id, label, source_url,
      method, pattern, provider_tag, context_hash, status,
      generated_at, last_verified_at, notes)
  VALUES ('R20260504123456Z123T<uuid-1>', 'rate', 'nbk_kzt', 'EUR / KZT',
      'https://nationalbank.kz/rates/...', 'css',
      'tr:has(td:contains("EUR / KZT")) td:nth-child(4)',
      'ruledoctor:groq:llama-3.1-8b-instant',
  ```

  Each (target, label) combination produces one statement-pair
  (UPDATE old active for that exact label + INSERT new). Different labels
  for the same `target_id` coexist as multiple active rows — the partial
  unique index is keyed on `(target_kind, target_id, label)`. **Verification gate:** ONLY rules that pass `Verify.ValueMatches` AND
  (`CSSMatches` OR `RegexMatches`) reach the SQL artifact. Failures are
  written as `-- skipped` comments inside the same file plus a non-zero
  exit code on the CLI so the user notices.

  The CLI also writes a sibling `.report.txt` next to the `.sql` with the
  per-target verification matrix from the integration test (re-using
  `Verify`).

  **No DB writes from the CLI.** The artifact is the only side-effect
  besides logs.

  - [ ] `regenerate-broken` reads `ObtainBrokenTargets("rate")` and runs

  - [ ] `list-rules` prints `target_id | label | method | provider_tag | generated_at | status`
        without writing anything.
  - [ ] Smoke test under `cmd/ruledoctor/` (`main_test.go`) covers the SQL

        `Extraction` JSON. No network, no LLM. Runs in `make test`.
- **Pitfalls:**
  - Single-quote escaping in SQL: any pattern that includes a `'` must be
    doubled. Write a small `sqlQuote(s string) string` helper and use it
    for every value. Do NOT use `fmt.Sprintf("'%s'", x)` directly.
  - The CLI must NOT WRITE to the project SQLite DB. All schema changes
    flow through Go `Migration()`; all rule-data changes flow through the
    `.sql` artifact and manual `sqlite3 < file`. Read-only access via
    `ObtainBrokenTargets` (for `regenerate-broken`) and
    `ObtainAllRulesByTarget` / `ObtainActiveRulesByTarget` (for
    `list-rules`) is permitted. The binary opens the DB read-only:
    `sqlitedb.NewSQLiteClient` followed by an immediate
    `db.Conn().ExecContext(ctx, "PRAGMA query_only = ON;")` on every

    invariant is that no Retain/Install path is reachable from any CLI
    subcommand). Source-list lookup remains JSON-file-only; the DB is
    only consulted for the broken / list flows.
  - JSON parse errors must surface clearly (filename + offset). Don't try
    to recover or accept partial input.
  - Always wrap statements in a single `BEGIN; ... COMMIT;` so a partial
    apply on prod can't leave half-installed rules.
  - Provider construction: extract `newClient` from `extract_test.go` into
    a non-test helper (e.g. `internal/ruledoctor/providers.go`) so both
    the integration test and the CLI use the same code path. Do not import
    `_test.go` files.

## Risks

- **Two source-of-truth problem during transition.** While
  `rate_sources.rules` and `extraction_rules` coexist, two engineers can
  reasonably disagree about which path runs. The fallback rule is
  unambiguous (active rule wins, else inline) but must be documented in
  `rateagent.go` with a header comment. Risk → low if Task 5's tests
  exercise both branches.
- **SQL artifact applied twice.** Re-applying yesterday's `.sql` would
  insert duplicate rule rows with different IDs (UUID-based). The active
  one would still be the latest, but the table grows. Acceptable; the
  unique partial index prevents two `active` rows. Mention in the SQL file
  comment header that re-applying is safe but bloats history.
- **goquery version drift.** The CSS branch depends on goquery / cascadia
  parsing identically in `verify.go` and `rateextractor.go`. They use the
  same library version (single go.mod). Risk → low.
- **Provider rate limits during a regenerate-broken run.** Groq free tier
  is 14400 req/day. A run over 20 broken rules at ~1 req each is fine. If
  the user ever runs ruledoctor in a tight loop the existing
  `parseGroqRetryAfter` backoff handles it.
- **Targets file drift.** The JSON targets file is hand-maintained on the
  Mac and not synced with prod's `rate_sources`. Risk: ruledoctor
  generates a rule for a `target_id` that no longer exists in prod, so the
  applied SQL silently sits unused. Mitigation: the SQL file header lists
  every `target_id` written; user spot-checks against prod source list
  before `sqlite3 < file`. Promotion to a DB-backed source list is a
  follow-up plan once volume justifies it.
- **Single-shot generation, no cascade.** v1 deliberately does not cascade
  to a stronger model on verification failure. If 8B's 89% CSS verify rate
  proves too low in practice, a follow-up plan adds the cascade behind a
  `RULEDOCTOR_FALLBACK_PROVIDER` env flag — the architecture (Task 11) is
  already shaped to allow it.

## Trade-offs

- **Generic `extraction_rules` table vs columns on `rate_sources`.** Chose
  the table. Reason: the user explicitly flagged future non-rate use cases
  (weather, etc.). Columns on `rate_sources` would solve "rate" cleanly and
  force a duplicate `weather_extraction_rules` table later. The cost of the
  generic shape is one extra JOIN-equivalent lookup per source per
  collector run — negligible for the project's volume (~20 sources). The
  partial unique index keeps `ObtainActiveRulesByTarget` to a single
  index hit.
- **Coexist with old inline rules vs replace.** Chose coexist. Reason:
  zero-downtime promotion. Replacing would require migrating every existing
  source's rule into the new table in the same change, with no fallback if
  the migration is wrong. Coexisting lets the user move sources one at a
  time and roll back any individual one by deleting the active rule row.
  Phasing out inline rules entirely is a future plan.
- **Manual `.sql` artifact vs direct DB write.** User decision —
  manual-control phase. Trade-off accepted: extra step per run, but the
  user reviews exactly what hits prod, and apply is `sqlite3 < file` (no
  new admin endpoint, no auth surface). Maps cleanly onto the "Mac is
  authoritative, prod VPS is dumb" architecture.
- **Single SQL file per CLI invocation vs one per target.** Single file.
  Reason: a multi-target run's failures and successes belong in one
  reviewable artifact. Splitting per-target multiplies file management for
  no clarity gain.
- **`broken` threshold N=3.** Trade-off: faster reaction (N=2) risks
  promoting a rule to broken on a transient network blip; slower reaction
  (N=5) leaves the user staring at silently-failing collection longer.
  Three is a defensible middle. Promoted to a constant the user can tune
  later without re-deploying schema.
- **No cascade in v1.** Trade-off: simpler, faster, fewer LLM credentials
  in scope. We accept that ~10–15% of generated rules may need a manual
  retry with a stronger model (the user re-runs `make ruledoctor-run` with
  `PROVIDER=anthropic`). If the failure rate hurts in practice, the
  cascade plug-in slot is already shaped (Task 11).
- **Pre-render deferred.** Trade-off: targets that are JS-only (Halykbank,
  Bankffin without `cmd/ruledoctor-fetch`) cannot be onboarded by
  `make ruledoctor-run` alone. The user pre-renders manually for now via
  `make ruledoctor-fetch` and feeds the saved HTML in. A follow-up plan
  formalises this. Acceptable because it does not block the static-HTML
  majority (NBK, BCC static, etc.).
