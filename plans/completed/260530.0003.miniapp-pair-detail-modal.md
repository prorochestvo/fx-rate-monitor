# Task Breakdown

## Overview

Refactor the Telegram Mini App subscriptions screen (`/app/subscriptions.html`,
driven by `cmd/wasm`) from a single dense scroll into a two-level "list + detail
overlay" model:

1. The sparkline-list card collapses each pair row to **two text rows only**:
   the pair label and a single spread/delta indicator. The per-series B/A value
   line and the SVG sparkline disappear from the collapsed view.
2. Clicking a pair row opens a **modal overlay** that surfaces everything the
   collapsed view hides — per-series prefix + latest + delta % + sparkline SVG,
   the spread (when applicable), the last grab timestamp, and the matching
   subscription condition badges joined client-side from `state.Items`.
3. The "Show / Hide subscriptions" toggle button (`#me-list-toggle`), the
   search bar (`#me-search`), and the paginated subscription card list
   (`#me-subs-list` + `#me-subs-pagination`) are **removed**. With per-pair
   details now reachable via the modal, the redundant subscription list under
   the chart no longer earns its real estate.

The change is WASM-only: HTML structure moves into the Go renderer, CSS lives
in `subscriptions.html`, and one new field is added to `MeSubscriptionsState`
to track which pair (if any) the modal is currently open for. The chart DTO
(`MeChartResponse`) is **not** modified — condition badges are joined
client-side against the already-loaded `state.Items` by canonical pair label.

The modal is rendered via the existing innerHTML / state-redraw pattern (same
as the current sparkline slot), not via the native `<dialog>` element or
CSS-only `:target`. This keeps the dismissal logic (✕ button, backdrop click,
Escape key) inside the same Go controller that already owns the page, and
avoids the `<dialog>` element's awkward interaction with Telegram WebApp's
viewport on iOS.

## Assumptions

- The chart response and subscription list are both loaded by the existing
  `runRenderMeSubscriptions` flow. The modal join (pair → conditions) operates
  purely against the in-memory `MeSubscriptionsState.Items`, which today is
  paginated. **Trade-off:** if a user's pair is on page 2 of subscriptions but
  the chart shows all pairs, the modal will display the per-pair details with
  an empty `badges` block. This is acceptable for v1 because (a) the list +
  pagination is being removed and (b) the typical user has < 10 subscriptions
  in total. If this becomes a real issue, the fix is to bump `LoadInitial`'s
  page size to a large value (e.g. 100) — not a server-side change. See Risks.
- Condition badges in `MeSubscriptionRow.Conditions` are already stringified
  on the server (e.g. `">1.05"`, `"<2.00"`). The modal reuses them verbatim
  with `dom.Escape`.
- "Last grab" for the modal is derived from the latest `MeChartPoint.Timestamp`
  across the row's series (taking the max). This avoids adding a new DTO field
  and is consistent with what the sparkline already plots. When all series are
  sparse-with-no-points, the modal omits the line.
- Keyboard focus trapping inside the modal is **out of scope**. Telegram Mini
  Apps are touch-first; the cost of a correct focus trap (saving and restoring
  `document.activeElement`, intercepting Tab) is higher than the benefit for a
  modal whose only interactive element is the ✕ button. We add `role="dialog"`,
  `aria-modal="true"`, and `aria-labelledby` so screen readers announce the
  pair label correctly. Escape-to-close gives keyboard users a clean exit.
- Telegram theme variables (`var(--tg-theme-*)`) must continue to drive all
  colors. The hardcoded `#1D9E75` / `#378ADD` / `#A32D2D` / `#3B6D11` constants
  from `internal/domain/ratepair/ratepair.go` stay on the inline `style="color:..."`
  attributes (already the case in `chart.go`) since they are role colors, not
  theme colors.
- The boot loader markup in `subscriptions.html` and the auth-failure path are
  not touched.

## Risks

- **Modal badge accuracy depends on `state.Items` being loaded for the right
  page.** The list pagination is removed in this plan, so `LoadInitial` fetches
  one page (size = 10) and that's it. If a user has > 10 subscriptions across
  multiple pairs and the matching pair is on page 2+, the modal renders without
  badges. Mitigation: bump page size to a large constant (e.g. 200) in
  `NewMeSubscriptionsPage(..., 200)` from `main.go` — cheap, but adds a few
  KB to the initial response.
- **`closest()` polyfill in old WebViews.** Telegram Mini Apps on very old
  Android WebViews may lack `Element.prototype.closest`. Risk is low (`closest`
  has been in WebView since Chrome 41 / 2015). If we get reports, fall back to
  manual parent traversal.
- **Modal Escape handler is a document-level listener.** If a future feature
  adds another `keydown` listener on `document` (e.g. a global search shortcut),
  the order of dispatch becomes load-bearing. Annotate the listener with a
  comment and consider an event-bus pattern if this happens.
- **Reduced functionality vs old screen.** The old screen had a searchable,
  paginated subscription list. Users who relied on that explicit list now get
  the chart's pair rows as their primary navigation. If we have telemetry,
  glance at `me-list-toggle` click rate before merging. If it's > 30 % of
  sessions, reconsider keeping a collapsed list section accessible via a
  smaller, less prominent "All subscriptions" link.
- **CSS animation on `prefers-reduced-motion: no-preference`.** Telegram's
  in-app browser ignores some media queries on older iOS. The animation is
  cosmetic — failure mode is a hard pop-in, which is acceptable.

## Trade-offs

- **State-driven modal (innerHTML redraw) over native `<dialog>`.** Chose
  innerHTML to stay consistent with the existing redraw pattern
  (`redrawChart`, `redrawList`). The native `<dialog>` element would give us
  free focus management and ESC-to-close, but its API is imperative
  (`showModal()`, `close()`) and would force the Go controller to call into
  JS to open/close it — a layering violation. Cost of the chosen path: we
  re-implement ESC and focus-restoration manually.
- **Conditions joined client-side over server-side enrichment.** Adding a
  `Conditions []string` field to `MeChartPairRow` would have given us a
  guaranteed-complete join but required (a) a DTO change, (b) a server-side
  repository query for the join, and (c) test churn across `internal/dto` and
  `internal/gateway`. The client-side join uses data already loaded for the
  list (now repurposed for the modal) at the cost of badge accuracy for very
  long subscription lists. Accepted trade-off given typical user has a
  handful of subscriptions; documented under Risks.
- **Removing the list / search / pagination entirely** instead of keeping
  them collapsed. Justification: the modal now exposes the per-pair detail
  (price, delta, sparkline, conditions, timestamp) that the list previously
  carried. Keeping both would duplicate information and dilute the design.
  If we miss the list later, the modal can grow a "View all subscriptions"
  link that opens a separate screen — cleaner than half-hiding the list under
  a toggle.
- **Per-series independent SVG y-axes in the modal** instead of a shared
  y-axis. Independent axes make small percentage moves visible regardless of
  series scale. Shared axis would make BID/ASK overlap easier to read but
  flatten a series whose absolute values differ. The collapsed view drops
  SVGs entirely, so the modal is the only place this choice matters.
- **No focus trap inside the modal.** Adding one means tracking
  `document.activeElement` at open, intercepting Tab to cycle within the modal,
  and restoring focus on close. Telegram Mini Apps are touch-first and the
  modal has exactly one focusable control (the ✕ button). Cost-benefit does
  not justify the code. Escape-to-close gives keyboard users a clean exit.
  If a future iteration adds form controls inside the modal, revisit this.
