# Task Breakdown

## Overview

Unify the web frontend so that a single entry point (`cmd/web/static/index.html`) serves both audiences:

- **Authenticated Telegram Mini App users** — the page behaves exactly as the current `cmd/web/static/tbot-miniapp/subscriptions.html`: it calls `window._wasm.renderMeSubscriptions()` which hits the `/api/me/...` endpoint family. Visual rhythm, sparkline list, modal, history pagination — all unchanged.
- **Unauthenticated browser visitors** — the page calls a new `window._wasm.renderPublicSubscriptions()` that loads `GET /api/public/rates/chart?page=N&limit=L` and renders a paginated (20 per page) sparkline list of **all currency pairs across all active sources** over the last 7 days, using the existing `ui.RenderSparklineSlot` rendering plus a pagination row underneath.

The current operator-facing dashboard (`renderSources` / `renderSourceDetail` / `renderErrors`) is preserved verbatim and relocated to `cmd/web/static/admin/index.html`, served at the URL path `/admin/`. The WASM function-table registrations for those screens remain in place; `cmd/web/main.go` already embeds the entire `static/` tree via `//go:embed static`, so the move is a pure file relocation with zero Go changes required to reach it.

The dispatcher decision is JS-only and minimal: at boot, inspect `window.Telegram.WebApp.initData`; if non-empty call `renderMeSubscriptions`, otherwise call `renderPublicSubscriptions`. The page-URL `?initData=` dev fallback continues to live inside the WASM `readInitData()` so a developer can still drive the /me view from a normal browser tab without touching the dispatch JS (see Task 2 trade-off note).

`cmd/web/static/tbot-miniapp/` is deleted in this work. The operator will update the BotFather Mini App URL manually to point at the site root (`https://<host>/`). The `webAppURL` constant baked into `cmd/web/main.go:94` must also be updated to drop the `/tbot-miniapp/subscriptions.html` suffix so any deep-links the bot emits go to the new root.

## Assumptions

- The `chart` application service is the right home for the new public listing. It already owns the sparkline downsampling kernel, the BID/ASK grouping, the canonical sort, and reuses `ratepair.Less`. Adding a public method here keeps both endpoints honest about ordering and chart math.
- A new repository method on `RateSourceRepository` is the right way to enumerate distinct active `(base, quote, kind)` triples. `ObtainAllRateSources` returns the full row set including inactive sources and dozens of metadata columns we don't need for chart enumeration; a focused query keeps the public endpoint's working set small.
- The 7-day window (`ratepair.ChartWindow`) is identical for /me and /public — no new constant needed.
- Public visitors do not need per-pair history. The existing `OpenPairModal → OpenHistory` flow on `MeSubscriptionsPage` requires the user's initData; the public modal must close that door cleanly (chosen approach: option **b**, a separate slimmer modal that omits the "View history" button entirely — see Task 2).
- `bucketCount = 12` and the 7-day window are not page-size sensitive; we slice the **rows** by page, but every returned row carries its full 12-bucket sparkline. Pagination only paginates the list, not the time series.
- The operator did not ask for caching or rate limiting on the new public endpoint, and explicitly excluded it from scope; treat the endpoint's QPS as bounded by the existing reverse-proxy / Cloudflare layer.
- The plan respects the no-PII rule: the new endpoint reads only `rate_sources` and `rate_values`, both of which are pair-keyed and source-keyed — zero `chat_id`, zero `username`, zero `user_id`. Confirmed below in Task 1's acceptance criteria.
- The existing `static/` FileServer handler in `cmd/web/main.go:182-231` is mounted at `/` and serves any file under `static/` by URL path. Therefore moving the old `index.html` to `static/admin/index.html` makes `/admin/` resolve via the built-in directory-index behaviour of `http.FileServer` (which serves `index.html` when the URL ends in `/`). No new route is needed.

  1. **Repository** — add `ObtainDistinctActivePairTriples(ctx context.Context) ([]domain.SourcePairKey, error)` on `*repository.RateSourceRepository` (`internal/repository/ratesource.go`). The `SourceName` field on the returned `SourcePairKey` is needed because `chart.Service` looks up time-series via `ObtainValuesForPairsSince`, which is keyed by source name (BID and ASK for the same pair may come from different sources).   2. **Application** — refactor `chart.Service.ObtainMeChart` (`internal/application/chart/service.go`) to extract steps 4–10 of its pipeline (the value loading, grouping, downsampling, pair-row building, sorting) into an unexported helper `buildPairRows(ctx context.Context, allKeys []domain.SourcePairKey, uniquePairs []ratepair.Pair) ([]PairRow, error)`.      Add a new exported method:

     ```go

     type PublicChart struct {
         Pairs []PairRow
     }

     // ObtainPublicChart enumerates every distinct (base, quote, kind) triple
     // across active sources, downsamples each into a 7-day sparkline, groups
     // BID/ASK into one PairRow per canonical pair, sorts via ratepair.Less,

     func (s *Service) ObtainPublicChart(ctx context.Context, page, limit int64) (*PublicChart, int64, error)
     ```

- **Pitfalls:**
  - **Kernel extraction collisions.** `ObtainMeChart`'s steps 4–10 currently capture local state (`subs`, `sourceMeta`, `seen` map). Make sure the extracted helper takes `allKeys []domain.SourcePairKey, uniquePairs []ratepair.Pair` as inputs and is otherwise pure — no references to `subs` or any user-subscription state. The /me path computes `allKeys` from subscriptions; the /public path computes `allKeys` directly from the new repo method. This is the load-bearing line of separation.
  - **Pagination total semantics.** The total returned must match the number of rows the frontend can paginate through, not the raw triple count. BID and ASK for the same canonical pair collapse to ONE row (per `buildPairRow`), so `total = len(pairRows)` AFTER grouping, BEFORE slicing.
  - **Sort stability.** Slicing after sorting is correct; slicing before sorting is a stability bug across page boundaries. `sort.Slice` is not stable — if two pairs sort to equal `ratepair.Less` order, page boundaries could flicker. `ratepair.Less` uses canonical pair as a tiebreaker, but two BID-only rows with the same canonical pair (one from active sources and one from inactive) could theoretically tie. In practice the new repo method filters to active so this is moot, but document it in the helper's godoc.
  - **Off-by-one in pagination.** `offset = (page - 1) * limit`. When `offset >= len(rows)`, return an empty slice and the actual total — do NOT 404. The frontend should be able to render "no results on this page" gracefully.
  - **Source name leak.** `domain.SourcePairKey.SourceName` is included in the loader's return — that is fine internally but the public DTO must NOT expose source names per the spec ("just rate data and source metadata"). Re-reading the spec: "no chat_id, no username, nothing user-keyed. Just rate data and source metadata" — source names are explicitly OK ("source metadata"). The `MeChartPairRow` shape does not contain source names anyway (the user-facing label is "USD/KZT", not the source list), so this resolves itself. No action needed but call it out in review.
  - **Empty system edge case.** A fresh database with zero active sources must return `{"window":"7 days","page":1,"limit":20,"total":0,"pairs":[]}`, NOT null and NOT 404. The handler should construct `pairRows := make([]dto.MeChartPairRow, 0, ...)` so the JSON encoder emits `[]` and not `null`.
  - **`make build` rebuilds the WASM.** Because the new endpoint will be called from `cmd/wasm/apiclient`, both the host binary and the WASM bundle need rebuilding to keep the API client in sync. `make build` already does this; flag it in the PR.

- **Complexity:** Medium

- **Code Example:**

  ```go
  // internal/application/chart/service.go (sketch — kernel extraction)

  // buildPairRows runs the shared downsampling, BID/ASK grouping, and sort
  // pipeline used by both ObtainMeChart and ObtainPublicChart. The caller
  // owns the assembly of allKeys (the per-source query targets) and
  // uniquePairs (the deduped (base, quote, kind) triples for which series
  // should be built); buildPairRows handles the rest.
  func (s *Service) buildPairRows(
      ctx context.Context,
      allKeys []domain.SourcePairKey,
      uniquePairs []ratepair.Pair,
  ) ([]PairRow, error) {
      if len(allKeys) == 0 {
          return []PairRow{}, nil
      }

      now := s.now()
      since := now.Add(-ratepair.ChartWindow)

      rawValues, err := s.values.ObtainValuesForPairsSince(ctx, allKeys, since)
      if err != nil {
          return nil, err
      }
      // ... rest of steps 5–10 from ObtainMeChart go here, byte-for-byte.
  }

  // ObtainPublicChart returns one page of the system-wide sparkline list.
  // total is the count of PairRows before slicing — note BID+ASK for the
  // same canonical pair collapse to one row, so total ≤ len(triples).
  func (s *Service) ObtainPublicChart(ctx context.Context, page, limit int64) (*PublicChart, int64, error) {
      if page < 1 {
          page = 1
      }
      if limit < 1 {
          limit = 20
      }

      triples, err := s.publicSources.ObtainDistinctActivePairTriples(ctx)
      if err != nil {
          return nil, 0, err
      }
      if len(triples) == 0 {
          return &PublicChart{Pairs: []PairRow{}}, 0, nil
      }

      allKeys := triples
      uniquePairs := dedupePairTriples(allKeys)

      rows, err := s.buildPairRows(ctx, allKeys, uniquePairs)
      if err != nil {
          return nil, 0, err
      }

      total := int64(len(rows))
      offset := (page - 1) * limit
      if offset >= total {
          return &PublicChart{Pairs: []PairRow{}}, total, nil
      }
      end := offset + limit
      if end > total {
          end = total
      }
      return &PublicChart{Pairs: rows[offset:end]}, total, nil
  }
  ```

  ```html
      var go = new Go();
      WebAssembly.instantiateStreaming(fetch('/app.wasm'), go.importObject)
        .then(function(result) { go.run(result.instance); resolve(); })
  ```

  **Dispatcher trade-off note (the spec asked for this to be documented):** the dispatcher only inspects `window.Telegram.WebApp.initData`. The page-URL `?initData=` fallback that `cmd/wasm/main.go:readInitData()` already supports remains intact — but the dispatcher does NOT call into `readInitData` to make its decision. Consequence: when a developer opens `http://localhost:8080/?initData=...` in a regular browser tab, the dispatch JS sees no `window.Telegram` object and routes to `renderPublicSubscriptions`. To reach the /me view from a normal browser tab during local dev, the developer must either (a) set `window.Telegram = { WebApp: { initData: '...' } }` in DevTools before page load, or (b) we add a `?view=me` URL parameter to the dispatch JS. The user explicitly asked us to lean toward the simpler form and let `readInitData` keep handling the dev fallback "by hand-editing or via a future toggle" — so we stay with form (a) and ship a one-line console message in the public path nudging devs toward the right escape hatch. No additional JS complexity.

## Risks

- **WASM bundle size growth.** The new public page reuses most of the existing UI helpers, but the new application controller, new UI renderers, new API client method, and (most of all) the bundled `dto.PublicChartResponse` add bytes. Mitigation: confirm growth ≤ 30 KB compressed in `app.wasm.gz`. If growth is meaningful, consider extracting shared modal logic before merge.
- **Public endpoint cold-start cost.** Enumerating every active triple and downsampling each into 12 buckets is a `O(triples × samples)` operation per request. With 20–30 sources and a 7-day window, that's measured in tens of thousands of rows. SQLite-on-NVMe handles this in tens of ms, but a fresh DB connection plus the `ObtainValuesForPairsSince` query can spike to ~100 ms. The endpoint is paginated but the chart math runs on the FULL row set per request before slicing — there is no "cheap page 5" path. **Mitigation acknowledged but out of scope per the spec** (no caching, no rate limiting in this plan). Re-visit if real-world traffic justifies it.
- **Telegram WebApp script CSP.** The new `index.html` keeps the inline `https://telegram.org` script tag for the Mini App arm. A future tighter CSP that strips that origin would break the Mini App path. Document the load-bearing CSP origins in a comment next to the `<script src="https://telegram.org/...">` line.
- **Operator forgets to update BotFather.** Until BotFather is updated, the Mini App button in the chat client still routes to the old URL. Once `tbot-miniapp/` is deleted that URL 404s — meaning the Mini App stops working for users who click the button from BotFather's stored URL, even though the bot's own keyboard URL (driven by `webAppURL`) is correct. Mitigation: PR description includes the BotFather step as a required post-deploy action; add it to the deploy runbook in Task 4.
- **Hash-router collision on `/admin/`.** The admin dashboard uses hash-based routing (`#/sources/<name>`, `#/errors`). Hash routing is independent of path routing, so `/admin/#/sources/foo` works correctly. But shareable links from operators that paste a hash-only URL (`#/sources/foo`) into a browser will now land on the public guest page (because they hit `/`, not `/admin/`). Document the new URL form in CLAUDE.md / README.
- **Test flakiness from sort-tie ordering.** `sort.Slice` is unstable. If two pairs sort equal under `ratepair.Less`, page boundaries could flicker across test runs. Fix it now by using `sort.SliceStable` in the extracted kernel even though `ObtainMeChart` currently uses `sort.Slice` — call this out in review as a deliberate improvement.

## Trade-offs

- **DTO option A over B.** Chose a separate `dto.PublicChartResponse` over generalising `MeChartResponse`. Rejected B because it ties the wire shape of an authenticated user-keyed endpoint to a public list-all endpoint; future changes to either contract drag the other. Cost of A: a few lines of duplicate struct definition. Benefit: clean separation, no risk of accidentally exposing pagination fields in the /me response or accidentally requiring them.

- **Modal option (b) over (a).** Chose a slim public modal that omits the history button entirely over a "same modal with the button hidden" approach. Rejected (a) because hidden-by-CSS still ships the HTML/handlers and leaks intent; option (b) is honest about what the guest path can do. Cost: a new `RenderPublicPairModal` function (small; ~50 lines of HTML templating). Benefit: no auth-dependent code path in the guest controller, no dead JS handlers, simpler to test.

- **Kernel extraction vs inline duplication.** Chose to extract steps 4–10 of `ObtainMeChart` into a shared helper. Rejected inline-copy because the extraction memo from CLAUDE.md (`feedback_dedup_coincidental_vs_invariant` in agent memory) says to dedup genuine invariants but let coincidentally-similar code diverge. Here the downsampling/grouping/sort pipeline IS a genuine invariant — both /me and /public must produce identical sparkline shapes for the same underlying data. Letting them diverge would introduce a visual inconsistency where the same pair looks different to a guest than to a logged-in user. Extraction is the right call.

- **Dispatcher in JS over WASM.** Chose to dispatch in the inline `<script>` block instead of always calling `_wasm.renderEntry()` and letting WASM decide. Rejected the all-WASM path because it would require WASM to know whether to render the Mini App boot loader or the guest loader (different first-paint markup), forcing both bundles' UI code paths to load on every request. JS dispatch lets us route at the cost of ~20 lines of inline script. Cost: dispatch logic lives in two languages (JS + Go). Benefit: simpler bundle loading and no flash of wrong content.

- **`/admin/` route via directory-index over explicit handler.** Chose to rely on `http.FileServer`'s built-in `index.html` behaviour rather than adding an explicit handler. Rejected an explicit handler because it adds code for zero benefit — the embedded FS already supports it. Cost: future readers might not realise `/admin/` works without grepping for it. Mitigation: document the convention in the CLAUDE.md HTTP Routes section.

- **No auth on `/admin/`.** Chose to leave the admin dashboard reachable by URL with no HTTP-layer auth. Rejected adding basic-auth or a token gate because (a) the user did not ask for it and the spec is explicit about scope, and (b) it's the existing posture — moving the file doesn't change the security model. The implicit contract is that `/admin/` is gated by network access (private VPC, Cloudflare Access, or whatever the operator chooses). Document this in CLAUDE.md so a future reader doesn't assume the move added auth.
