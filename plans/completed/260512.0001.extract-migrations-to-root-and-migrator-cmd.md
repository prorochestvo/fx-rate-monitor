# Task Breakdown

## Overview

Today each repository file in `internal/repository/*.go` embeds its own `CREATE TABLE` /
`CREATE INDEX` statements as a `map[string]string` returned via a `Migration()` method
that is fed into `sqlitedb.NewMigrator(...)` as a `source`. Each long-running binary
(`cmd/web`, `cmd/collector`, `cmd/notifier`) calls `Migrator.Run(ctx)` during startup,
which means schema is silently bootstrapped on first boot and there is no operational
control over schema changes.

Goal:

1. Move every existing migration into a project-root directory `./migrations/` as
   standalone `.sql` files, named after the existing migration keys so deployed
   `__schema_migrations` rows continue to match.
2. Create a new root-level Go package `./migrations/` (file `embed.go`, `package migrations`)
   that exposes the `.sql` files via `//go:embed *.sql` as an exported
   `var MigrationsFS embed.FS`. The directory is simultaneously the SQL source-of-truth
   and a tiny Go package that publishes them.
3. Refactor `sqlitedb.Migrator` to consume an external `fs.FS` (no longer self-embedded);
   `cmd/migrator` and tests pass `migrations.MigrationsFS` explicitly.
4. Introduce a dedicated `cmd/migrator` binary that imports the new package and is the
   only thing allowed to mutate schema in any environment.
5. Stop the service binaries from running migrations. They must fail fast when the
   schema is missing or out of date.
6. Replace bare table/column identifiers inside SQL strings in `internal/repository/`
   with Go `const` declarations so renames become a compile-time and grep-time concern,
   not a runtime surprise.

After this change, deployment is a two-step process: run `cmd/migrator`, then start the
service. The migrator binary is self-contained — it embeds the root-level `migrations`
package's FS at build time — and needs nothing but `SQLITEDB_DSN` to run.

## Assumptions

- The existing migrator at `internal/infrastructure/sqlitedb/migrator.go` currently
  embeds `.sql` files itself via `//go:embed migrations/*.sql` and tracks applied files
  in the `__schema_migrations` table keyed by filename. The applied-files contract is
  preserved; the `//go:embed` directive moves out of the migrator package.
- Production SQLite databases already contain `__schema_migrations` rows whose `name`
  column matches the existing keys (`rate_source_001_table_initiate`,
  `rate_value_001_table_initiate`, `rate_user_event_001_table_initiate`,
  `execution_history_001_table_initiate`, `rate_user_subscription_001_table_initiate`).
  Filenames of the new `.sql` files MUST equal those keys (plus `.sql`) so applied
  migrations are not re-run.
- Migrations live at the project root: `./migrations/*.sql` plus `./migrations/embed.go`
  (`package migrations`). The Go file exports `var MigrationsFS embed.FS` populated by
  `//go:embed *.sql`.
- `cmd/migrator` imports the root `migrations` package and passes
  `migrations.MigrationsFS` into `sqlitedb.NewMigrator(...)`. The binary is
  self-contained — the FS is embedded at build time. No `--migrations-dir` flag.
- `sqlitedb.Migrator` no longer carries its own `//go:embed`. The embedded directory
  `internal/infrastructure/sqlitedb/migrations/` and the empty
  `001_init.sql` placeholder are deleted entirely.
- Module path is `github.com/seilbekskindirov/monitor`; the new package's import path
  is `github.com/seilbekskindirov/monitor/migrations`.
- DSN is read via `dsninjector.Unmarshal("SQLITEDB_DSN")`. Format:
  `sqlite://<path-to-db-file>`.
- Repository unit tests that today rely on `NewMigrator(db, repo)` to bootstrap schema
  must switch to a test helper that applies `migrations.MigrationsFS` instead.

## Tasks

### Task 1: Inventory every `Migration()` source in `internal/repository/*.go`

- Description: Catalogue every `(key, sql)` pair returned by `Migration()` across the
  five repositories:
    - `internal/repository/ratesource.go`
    - `internal/repository/ratevalue.go`
    - `internal/repository/rateuserevent.go`
    - `internal/repository/executionhistory.go`
    - `internal/repository/rateusersubscription.go`

  Write the inventory to `./tmp/migration-inventory.md` as a numbered list:
  filename it will become, table it creates, and exact SQL body. This is the source of
  truth for Task 2; do not paraphrase the SQL.
- Acceptance Criteria:
    - [ ] `./tmp/migration-inventory.md` exists.
    - [ ] One entry per `(key, sql)` pair across the five repositories.
    - [ ] Each entry lists: target filename (`<key>.sql`), table name, full SQL body
          copied verbatim (including trailing semicolons and indentation).
    - [ ] Count of entries equals the sum of `len(m)` returned by each `Migration()`.
- Pitfalls & edge cases:
    - A repository may register more than one entry (e.g. initial `CREATE TABLE` plus a
      later index). Do not skip secondary entries.
    - The `Migration()` map iteration order is undefined; capture the keys explicitly,
      do not rely on Go's map walk order.
    - Repositories may build SQL via `fmt.Sprintf` or string concatenation; expand to
      the final literal before recording.
- Complexity: Easy.

### Task 2: Create root-level `./migrations/` package with `.sql` files + embed

- Description: For each inventory entry from Task 1, create a `.sql` file under the
  project-root `./migrations/` directory. Filename MUST exactly equal the migration key
  plus `.sql` (e.g. `rate_source_001_table_initiate.sql`). File contents are the exact
  SQL body from the inventory.

  Additionally, create `./migrations/embed.go` with:

  ```go
  // Package migrations exposes the canonical SQL migration files via embed.FS so
  // every consumer (cmd/migrator, repository tests) reads from a single source.
  package migrations

  import "embed"

  //go:embed *.sql
  var MigrationsFS embed.FS
  ```

  Delete the entire existing directory
  `internal/infrastructure/sqlitedb/migrations/` (including the empty
  `001_init.sql`). The new package is the single source of truth.
- Acceptance Criteria:
    - [ ] Exactly one `.sql` file exists in `./migrations/` per inventory entry.
    - [ ] `./migrations/embed.go` exists with `package migrations` and the exported
          `MigrationsFS embed.FS`.
    - [ ] Filenames match the existing migration keys (no rename, no reordering).
    - [ ] `internal/infrastructure/sqlitedb/migrations/` no longer exists in the repo.
    - [ ] `go build ./migrations/` succeeds and `embed.FS` lists every `.sql` file.
    - [ ] Running `cmd/migrator` (after Task 4) on a database that already has rows in
          `__schema_migrations` from production performs zero `INSERT` / `CREATE`
          statements.
- Pitfalls & edge cases:
    - The migrator sorts files by filename (alphabetical). Verify alphabetic order
      matches intended apply order. Numeric prefixes like `001` keep ordering stable
      within a table; the `_` between table name and number means cross-table order
      groups by table name — confirm this is acceptable (it is, because every existing
      migration is `_001_table_initiate` with no inter-table dependencies today).
    - Preserve trailing newlines and final semicolons — the migrator splits statements
      naively and an unterminated last statement may be skipped.
    - `//go:embed` paths are resolved relative to the Go source file; the directive
      `*.sql` is correct only because `embed.go` sits in the same directory as the
      `.sql` files. Do NOT move `embed.go` into a subdirectory.
    - Do NOT introduce SQLite-version-specific syntax that the embedded driver does
      not support; copy verbatim.
- Complexity: Easy.

### Task 3: Refactor `sqlitedb.Migrator` to consume an external `fs.FS`

- Description: Remove the self-embedded migrations from
  `internal/infrastructure/sqlitedb/migrator.go` and change the constructor signature
  to accept an `fs.FS` (or `embed.FS`) explicitly:

  ```go
  func NewMigrator(db committer, fsys fs.FS, sources ...source) (*Migrator, error)
  ```

  Internally `newDefaultMigrations` reads `fs.ReadDir(fsys, ".")` and iterates `.sql`
  entries. Drop the `//go:embed migrations/*.sql` directive and the `migrationsFS`
  package variable.

  All call sites (`cmd/migrator`, tests) must pass `migrations.MigrationsFS` from the
  new root package.
- Acceptance Criteria:
    - [ ] `NewMigrator(db, migrations.MigrationsFS)` compiles and runs end-to-end
          against a fresh DB and applies every file in the FS.
    - [ ] The `//go:embed` directive and `migrationsFS` variable are gone from
          `migrator.go`.
    - [ ] Running `Migrator.Run(ctx)` twice on the same DB is a no-op on the second
          call (idempotency).
    - [ ] `TestMigrator_Run` covers: fresh DB applies all files; second run applies
          nothing; corrupt SQL aborts the transaction without partial commits. The
          test passes an in-memory `fstest.MapFS` (not the real `MigrationsFS`) so the
          test is isolated from production schema changes.
    - [ ] All compile sites of `NewMigrator` are updated in this task (or in Task 7
          for service binaries, which are removed there).
- Pitfalls & edge cases:
    - `fs.ReadDir(fsys, ".")` requires the FS root to be the directory containing the
      `.sql` files. `embed.FS` produced by `//go:embed *.sql` satisfies this.
    - Apply must happen inside a single transaction per file so a failure does not
      leave half-applied schema.
    - Do not introduce an `io/fs` walk — flat directory is sufficient and matches the
      current contract.
- Complexity: Medium.

### Task 4: Create `cmd/migrator` binary

- Description: New `cmd/migrator/main.go` that:
    1. Reads `SQLITEDB_DSN` via `dsninjector.Unmarshal("SQLITEDB_DSN")`. `log.Fatalf`
       on missing value.
    2. Opens the SQLite client via the same constructor `cmd/web` uses
       (`sqlitedb.New(dsn)` or equivalent — match the existing pattern).
    3. Imports `github.com/seilbekskindirov/monitor/migrations` and calls
       `sqlitedb.NewMigrator(db, migrations.MigrationsFS)`.
    4. Invokes `Run(ctx)` where `ctx` is
       `signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)`.
    5. On success: logs `migrator: applied N migration(s)` and exits 0.
    6. On failure: logs the error to stderr and exits non-zero.

  No HTTP server, no Telegram client, no business-logic imports allowed. Import list
  should be limited to: `dsninjector`, `sqlitedb`, the new root `migrations` package,
  `context`, `log`, `os`, `os/signal`, `syscall`.

  Locked decision: migrations are embedded via the root `migrations` package's
  `MigrationsFS` (option 4 from the design discussion). `.sql` files are NOT duplicated
  into `cmd/migrator/`; the directory `./migrations/` is simultaneously the SQL source
  and the Go package that publishes them. The binary is self-contained — one `scp` of
  one file — without sacrificing the "migrations live in the project root" property.
- Acceptance Criteria:
    - [ ] `make build` produces `./build/migrator`.
    - [ ] `./build/migrator` against a fresh DB applies all files and records them in
          `__schema_migrations`.
    - [ ] Second invocation is a no-op (logs `applied 0 migration(s)`).
    - [ ] `go list -deps ./cmd/migrator` shows no imports from `internal/application`,
          `internal/gateway`, `internal/domain`, or `internal/repository`.
    - [ ] `go list -deps ./cmd/migrator` includes
          `github.com/seilbekskindirov/monitor/migrations`.
    - [ ] Binary exits with non-zero status when `SQLITEDB_DSN` is unset.
    - [ ] Binary handles SIGINT cleanly (cancels in-flight transaction; no partial
          state).
- Pitfalls & edge cases:
    - Do not call `os.Exit` from inside deferred functions; deferred `db.Close()`
      will not run. Use a small `run() error` helper and `os.Exit(1)` in `main` after
      logging.
    - Log format must be greppable for the deploy script; keep prefixes stable
      (`migrator: ...`).
    - The `migrations` package must NOT depend on any project package — it is a leaf
      package that contains only the `embed.FS` declaration. Adding logic here would
      create import cycles down the line.
- Complexity: Medium.

  know nothing about schema bootstrapping after this task.

  Update unit tests that previously called `sqlitedb.NewMigrator(db, repo)` to use a

- Acceptance Criteria:
    - [ ] No `Migration()` method exists on any repository type in
          `internal/repository/`.
    - [ ] `grep -R "CREATE TABLE" internal/repository/` returns zero matches.
    - [ ] `grep -R "CREATE INDEX" internal/repository/` returns zero matches.
    - [ ] All repository unit tests pass under `make test`.
    - [ ] Test helper exists and is used by every repo test that needs a populated
          schema.
- Pitfalls & edge cases:
    - Tests that today pass a repository as a `source` to `NewMigrator` will fail to
      compile. Update them in the same commit.
    - Some tests rely on a particular table existing at suite start; ensure the helper
      runs before any `t.Run` subtests via `t.Helper()` + explicit invocation in
      `TestMain` or per-test setup.
    - Do not delete the migrator's `source` interface or variadic parameter — even if
      unused after this task, removing it is out of scope and would break the public
      signature unnecessarily.
- Complexity: Medium.

### Task 6: Replace inline column/table identifiers with Go constants in `internal/repository/`

- Description: After `Migration()` is removed, every SQL string in
  `internal/repository/*.go` (`SELECT`, `INSERT`, `UPDATE`, `DELETE`, `WHERE` clauses,
  `ORDER BY`, `JOIN`, etc.) must reference table and column names via `const`
  declarations defined in the same file, never as bare quoted identifiers inside the
  query string.

  Rationale: when a column is renamed, the Go compiler plus a project-wide grep on the
  constant name catch every reference. With raw string literals, only runtime errors
  surface them — typically after deploy.

  Convention:
    - One block of `const` per file, near the top, grouping the table name and its
      columns.
    - Constant names use the form `tableRateSource`, `colRateSourceID`,
      `colRateSourceName`. (Match whatever short prefix style the file already uses; if
      none exists, use `tableX` / `colX`.)
    - Queries are assembled via `fmt.Sprintf` or string concatenation; do not pull in
      a query builder library.
- Acceptance Criteria:
    - [ ] `grep -RnE '"[A-Z]+ +[a-z_]+ +(FROM|INTO|SET|JOIN) +[a-z_]+"' internal/repository/`
          (or equivalent) returns zero matches where the table/column identifier is a
          bare literal not derived from a `const`. A targeted grep test or audit note
          is acceptable evidence; record the exact command used in the PR description.
    - [ ] Every table and column referenced by `internal/repository/` has a
          corresponding `const` declared in the same file.
    - [ ] All repository unit tests still pass.
    - [ ] No new dependency added to `go.mod`.
- Pitfalls & edge cases:
    - `fmt.Sprintf("SELECT %s FROM %s", colX, tableY)` is more verbose than the
      original literal; this verbosity is the point.
    - Watch for parameterised placeholders (`?`) — keep them as literal `?` inside
      `fmt.Sprintf` format strings (no `%s` for them) so SQL injection is not
      reintroduced.
    - Index column lists inside `CREATE INDEX` do not appear here — those are in the
      `.sql` files now and out of scope.
    - Some files use multi-line raw strings (backticks) for queries; the same rule
      applies — assemble with `fmt.Sprintf` or break into concatenated pieces.
- Complexity: Medium.

### Task 7: Stop running migrations from `cmd/web`, `cmd/collector`, `cmd/notifier`

- Description: Remove every call to `migrator.Run(ctx)` (and the constructor that
  produced the migrator) from the three service `main.go` files. Replace with an
  explicit fail-fast schema check.

  **Locked decision: fail-fast at startup**. Each service performs a single cheap query
  against `__schema_migrations` immediately after opening the DB:

  ```text
  SELECT COUNT(*) FROM __schema_migrations;
  ```

  If the table does not exist OR the count is zero, the service logs a clear error
  (`schema not initialised — run cmd/migrator before starting <service>`) and exits
  non-zero. This is preferred over `SELECT 1 FROM <table> LIMIT 0` because it does not
  bake a specific table name into every service and it surfaces "migrator never ran at
  all" as the actual error, which is the most likely operator mistake.

  No service should compare the row count to an expected number — that would couple
  service binaries to migration count and create false positives on rolling deploys.
  Existence + non-empty is sufficient signal; missing schema is the failure mode we
  guard against.
- Acceptance Criteria:
    - [ ] `cmd/web/main.go`, `cmd/collector/main.go`, `cmd/notifier/main.go` no longer
          import `sqlitedb.NewMigrator` and no longer call `Run`.
    - [ ] Each service performs the `SELECT COUNT(*) FROM __schema_migrations` check
          after DB open and before any other DB work; on failure it `log.Fatalf` with
          the message above.
    - [ ] An integration-style test (or manual check noted in the PR) confirms each
          service exits non-zero against an empty DB.
    - [ ] `deploy/` systemd units gain `ExecStartPre=` invoking `./build/migrator`
          (path must match the installed location) for each service unit.
- Pitfalls & edge cases:
    - Running multiple services in parallel against the same DB during a deploy: the
      `ExecStartPre=` hook will be invoked once per unit. The migrator is idempotent
      so this is safe, but logs will show "applied 0 migration(s)" twice — document.
    - The `__schema_migrations` table is created by the migrator itself. Services must
      handle the "table does not exist" case gracefully (SQLite returns a clear error
      that we should translate to the friendly message).
    - Do not silently re-run migrations as a fallback — that is the exact behaviour we
      are removing.
- Complexity: Medium.

## Risks

- **Production DB already has these migrations applied under existing names.**
  Mitigation: Task 2 acceptance requires byte-equal filenames to the existing
  `__schema_migrations` keys. Verification step: dump `__schema_migrations.name` from
  a staging or production DB, sort, and diff against `ls migrations/*.sql | sed
  's/.sql$//' | sort`. The two lists MUST be identical before Task 7 ships.
- **Drift between original embedded SQL and the new `.sql` files.** Mitigation: Task 1
  copies the inventory verbatim; Task 2 uses that inventory as the only source. Spot
  check by booting a service against a freshly-migrated empty DB and running the full
  test suite.
- **Tests break en masse when `Migration()` disappears.** Mitigation: introduce the
  test helper as part of Task 5, in the same commit that deletes `Migration()`, so
  the build is never red on `main`.
- **Operator forgets the migrator step during deploy.** Mitigation: Task 7 adds
  `ExecStartPre=` to systemd units and Task 7's fail-fast check makes the failure
  loud and immediate rather than a silent empty-DB boot.
- **Stale `cmd/migrator/migrations/` copy diverges from the canonical directory.**
  Mitigation: locked by the option-4 layout — the `.sql` files exist in exactly one
  location (`./migrations/`) and are exposed via the exported `MigrationsFS embed.FS`
  in the same directory's Go package. No duplication possible.
- **Root-level Go package collides with future tooling expectations.** Mitigation: the
  `migrations` package is a leaf — only `embed.go` with an `embed.FS` declaration. It
  imports nothing and exports one symbol. Any future contributor expecting "root has
  only `cmd/` and `internal/`" gets one extra directory to learn about; documented in
  CLAUDE.md (Task 8).

## Trade-offs

- **Schema-near-queries vs centralised migrations.** We lose the "schema lives next to
  the queries that read it" property. Accepted: schema becomes a first-class artifact
  that is reviewable in isolation, runnable outside the binary, and version-controlled
  as plain SQL. Engineers will need to grep two directories instead of one to trace a
  column; Task 6 (constants in repositories) gives back most of the grep ergonomics.
- **Embed vs disk-loaded migrations.** Chose `//go:embed`, located in a root-level
  `migrations` package rather than inside the migrator or the `cmd/migrator` binary.
  Pros: one binary, one `scp`, no environment-specific configuration, no "wrong
  migrations dir on prod" failure mode AND `.sql` files visibly live at the project
  root where a new contributor expects them. Cons: applying a hotfix requires a binary
  rebuild; you cannot patch schema by editing a file on the host. The root-level Go
  package is a slight layout oddity (only `cmd/` and `internal/` exist today), but
  it is the only way to satisfy both "embed" and "files at root" simultaneously —
  `//go:embed` cannot reach files outside its own package directory.
- **Fail-fast vs lazy migrations.** Chose fail-fast at service startup. Pros:
  deployment errors surface immediately, services never run against an unmigrated DB,
  blast radius is contained. Cons: a missed `ExecStartPre=` step causes a hard failure
  rather than self-healing. Accepted because self-healing has historically masked
  deploys where the migrator was the *wrong version* — much harder to diagnose than a
  loud "schema not initialised" message.
- **Constants for every identifier vs raw SQL ergonomics.** Chose constants
  everywhere. Pros: rename safety, compile-time references, single grep finds all
  call sites. Cons: queries become `fmt.Sprintf` salads and are harder to read at a
  glance. Accepted because the alternative (silent runtime errors after a column
  rename) is the more expensive failure mode. Scope is limited to
  `internal/repository/`; service and gateway layers remain untouched.
