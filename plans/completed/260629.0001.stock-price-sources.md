# Task Breakdown

## Overview

Add stock-price monitoring modeled exactly on the existing FX currency-rate
system: operator-seeded sources (no user-defined tickers), multi-source per
ticker, and user subscriptions through the existing `/api/me/subscriptions`
flow with the existing condition types (delta / interval / daily / cron). The
first increment ships two keyless sources and no new secrets:

- **AAPL** via Yahoo Finance v8 chart JSON
  (`https://query1.finance.yahoo.com/v8/finance/chart/AAPL`,
  `chart.result[0].meta.regularMarketPrice`), `fetcher_kind=plain`,
  `method=json`. Requires a browser-like `User-Agent` or Yahoo returns 403.
- **CCBN** (Bank CenterCredit on KASE) via static HTML scrape of
  `https://kase.kz/en/investors/shares/CCBN`, `fetcher_kind=plain`,
  `method=regex`, extracting the "Last deal price". KASE renders the number as
  `4 619,00` (space thousands separator, comma decimal).

A stock's last price is a single number, so it maps onto the existing
`rate_values` row (base=ticker, quote=currency, `price REAL`) with no schema
change. The pairs are `AAPL/USD` and `CCBN/KZT`.

### What the code review established (verified against the repo, not assumed)

1. **Number format `4 619,00` is already handled.** `applyRulesAndStore`
   (`internal/tools/rateextractor/rateextractor.go:234-235` and `:266-267`)
   strips all spaces and converts every comma to a dot before
   `strconv.ParseFloat`, both inside the `parse_float` branch and in the
   unconditional end-of-pipeline normalization. `4 619,00` -> `4619.00`. **No
   new extraction method is needed.** The only requirement is that the KASE
   regex capture class includes the comma (existing FX patterns use
   `([0-9][0-9 .]*)`, which excludes `,`).

2. **No schema migration is required.** `rate_sources.kind` is `TEXT NOT NULL`
   with **no `CHECK` constraint** (`migrations/202605.001...:8`), so a new
   `LAST` value is accepted by the column as-is. `RateSourceOptions` is
   serialized as JSON into the existing `options TEXT` column
   (`internal/repository/ratesource.go:192`), so adding a `Headers` field needs
   no DDL. The plan needs **one** migration (009, the seed); 010 is reserved
   and unused.

3. **A Go-level allowlist is the single biggest blocker.**
   `rateSourceQueryContext` (`internal/repository/ratesource.go:441-445`)
   rejects any `kind` other than `BID`/`ASK` and returns an error for the
   **entire** result set. This query backs the collector, the notifier,
   `/api/sources`, `/api/me/subscriptions` enrichment, and the chart. A single
   seeded `LAST` row would break all of them. (The single-row path
   `ObtainRateSourceByName` has no such check, so subscription *creation* still
   works.)

4. **Two fetchers hardcode a non-browser `User-Agent`.** The collector
   (`rateextractor.go:187`) and the audit tool
   (`internal/application/sourceaudit/fetcher.go:77`) both send
   `Beacon/1.0 (...)`. Yahoo needs a browser UA. Both must learn per-source
   headers for collection to succeed and for `doctor audit` to validate the
   Yahoo row.

5. **`LAST` ripples through the notifier and the Mini App** because some sites
   branch on `== BID` (so `LAST` falls into the wrong `quote/base` branch) and
   others on `== ASK` (so `LAST` works by default). Every site is enumerated in
   the tasks below.

The functional MVP is Tasks 1-5 (collect -> subscribe -> notify). Task 6
(Mini App chart/history display + equity grouping) is a display enhancement and
can be shipped as a second increment.

## Assumptions

- **`LAST` kind, not reuse of `BID`.** A stock last-trade price is neither a buy
  nor a sell quote; reusing `BID` would mislabel the UI and collide with any
  future stock bid/ask-spread feature. This is a judgment call; the cheaper
  "reuse `BID`" alternative is documented in Trade-offs for the operator to
  override.
- **No new external secret in the MVP.** Yahoo and KASE are keyless. Finnhub
  (`/quote?symbol=AAPL&token=KEY` -> `.c`) is a documented future source only.
- **Stock prices fit the existing plausibility bounds** `(0, math.MaxInt32]`
  (`rateextractor.go:25-28`). AAPL (~200) and CCBN (~4619) are well inside.
- **Existing condition types are sufficient.** `delta` / `interval` / `daily` /
  `cron` (`internal/domain/rateusersubscription.go:201-210`) operate on a price
  float and a clock; they are kind-agnostic. No new condition type.
- **The collector cron cadence is at least every 6h** (every existing seed uses
  `interval='6h'`). Stock sources are seeded at `6h` for safety; the operator
  may lower this only if the collector cron ticks more frequently.
- **No new health-check inspector for stock sources.** `cmd/web` never fetches
  upstreams; its inspectors are DB + Telegram (`cmd/web/main.go:139-141`). The
  36 FX sources have no per-source inspector — their liveness is observed via
  `execution_history` (surfaced by `/api/sources` and `/api/errors/execution`).
  Stock sources follow the same model. Rationale is expanded in Trade-offs.
- **Migration 009 is additive and idempotent** (`INSERT OR IGNORE`); it touches
  no existing row and needs no DB wipe. Once applied its filename is immutable
  per the repo migration rule.
- The exact KASE regex must be authored against the live page during
  implementation; the plan supplies a template and requires a saved fixture.

    req.Header.Set("User-Agent", "Beacon/1.0 (+https://github.com/seilbekskindirov/beacon)")

  - The chromedp fetcher (`chromedp.go`) is out of scope: both MVP stock

- **Acceptance Criteria:**

    the read path: a source seeded with `{"headers":{"User-Agent":"X"}}`
    reads back with `Options.Headers["User-Agent"] == "X"`; a source with

  - [ ] A `rateextractor` test (extend `TestRateExtractor_Run` or add the

## Risks

- **Yahoo fragility / ToS-gray.** The v8 endpoint is unofficial, can change
  shape, rate-limit, or block by UA/IP at any time. Mitigations: `6h` cadence,
  per-source UA, `execution_history` surfaces failures, and the source can be
  disabled via `PATCH /api/sources/{name}/active` without code changes. Accept
  the fragility (explicit decision).
- **KASE DOM drift.** The regex anchors on page markup that may change or
  localize. Mitigation: a saved fixture test plus `doctor audit` for live
  re-validation; the regex is operator-maintainable in the seed.
- **Allowlist blast radius.** If Task 1 is missed or partial, a single `LAST`
  row takes down the collector, notifier, `/api/sources`, `/api/me/subscriptions`,
  and the chart simultaneously (the list query errors for the whole set).
  Enforce execution order: Task 1 merges and deploys before migration 009 runs.
- **Forward-looking dedup gap.** `LAST -> MAX` is a placeholder; when a second
  feed for the same ticker is added (e.g. Finnhub AAPL), MAX biases toward the
  staler/higher feed. Revisit to "most recent" at that point (tracked in the
  code comment and Trade-offs).
- **Cache keyed by URL, not headers.** A future pair of sources sharing a URL
  with different headers would cross-contaminate the response/tombstone cache.
  Safe for MVP (unique URLs); documented in code.
- **Migration immutability.** Once 009 applies anywhere it cannot be renamed;
  get the row content right before tagging a release.
- **Interface signature change (Task 5).** Widening `Fetcher.Fetch` ripples to
  the fake and callers; compile-time assertions and the build catch omissions,
  but a careless merge could leave a stale fake.

## Trade-offs

- **`LAST` kind vs reuse `BID` (chose `LAST`).** Reusing `BID` would skip
  Task 1's allowlist change, Task 4 entirely, and shrink Task 6 (BID is already
  handled everywhere, and `BID -> base/quote` happens to render `AAPL/USD`
  correctly). It is materially less code and lower risk. It was rejected because
  the `kind` column feeds labels, coloring, dedup semantics, and the public
  `/api/sources` output; tagging a last-trade price as `BID` mislabels the UI
  (a "BID" badge and fiat grouping for a stock) and would collide with any
  future stock bid/ask-spread feature. `LAST` is the honest model and the ripple
  is fully enumerated and bounded. **Operator decision point:** if shipping
  speed outweighs semantic correctness, reuse `BID` and drop Tasks 1 (repo) and
  4.
- **Dedup `LAST -> MAX` vs most-recent (chose MAX for MVP).** Most-recent is the
  semantically correct pick for N feeds of the same quantity but requires
  threading the observation timestamp through `SubscriptionSnapshot` and the
  digest handler. MVP has no same-ticker collision at all, so MAX (deterministic,
  zero new plumbing) is sufficient and is documented as revisit-on-second-feed.
- **Chart/Mini App as Phase 2 (deferrable).** Tasks 1-5 deliver the full
  functional feature; Task 6 is display polish. Splitting lets the operator ship
  subscriptions+notifications first and validate the keyless sources before
  investing in the larger frontend ripple (chart slot, history slot, WASM,
  category).
- **No new health inspector (chose execution_history).** The health-check rule
  targets a *service's own* dependencies (DB, Telegram), which `cmd/web` probes.
  The collector's scraped upstreams (now 38, including the two stock sources)
  are monitored via `execution_history` / `/api/sources` / `/api/errors/execution`,
  exactly like the 36 FX sources. Adding Yahoo/KASE probes to `/health/check`
  would be inconsistent (web never fetches them) and a flakiness/DoS vector on a
  readiness endpoint hit by uptime monitors.
- **Finnhub out of MVP (chose secret-free first increment).** Finnhub needs an
  API key — the project has no upstream-data secret today. Keeping it out lets
  the first increment ship with no new secret surface. Documented path to add it
  later: a key'd source whose token is injected via `dsninjector` (like
  `BEACON_TELEGRAMBOT_DSN`), with the key carried in `options.headers` or a new
  env var, reusing the same `LAST` kind and per-source-header machinery.
- **Equity classification by symbol set vs by `kind==LAST` (Task 6, open).**
  A symbol set (like `metalSymbols`) duplicates ticker knowledge between the
  migration and `ratepair`; deriving the category from `kind==LAST` avoids the
  duplication but conflates "is an equity" with "is a last-price kind". Decide
  during Task 6; the symbol-set approach matches the existing metal precedent.

## Post-Review Divergences & Resolutions

Recorded per the "keep plans honest" rule. The MVP shipped (Tasks 1-5 + 7) with a
green `make test` (-race + WASM); Task 6 (Mini App chart/history/equity grouping)
remains deferred. The following diverged from the plan as written and were resolved
during the five-lens review and the targeted re-review:

1. **`RateSourceKind` constants typed (plan said: keep untyped).** The constants were
   given the named `RateSourceKind` type. Three review lenses confirmed no ripple (the
   only raw-string kind comparisons live in `cmd/wasm`, which compares JSON-derived
   literals, not the Go constants). Net type-safety improvement; accepted.
2. **`chart/history.go` LAST handling pulled into the MVP (plan deferred it to Task 6).**
   `/api/me/rates/history` is a live endpoint and routed a LAST price into the `Ask`
   slot (mislabeled). A minimal fix routes LAST to the `Bid` slot (consistent with
   `pairLabel` and the notifier fold treating LAST like BID). The dedicated `Last` DTO
   field and full rendering remain Task 6.
3. **`rulegen.Fetcher` widened for per-source headers (plan put rulegen out of scope).**
   The adapter silently dropped `options.headers`, so `doctor rulegen` on a
   header-dependent source would fail with an opaque 403. The interface now threads
   headers through every implementer, making `doctor rulegen` consistent with
   `doctor audit`.
4. **Response reads bounded with `io.LimitReader` (new, from the security lens).** Both
   the collector (`rateextractor`) and audit (`sourceaudit`) fetchers cap reads at 10 MB
   — an OOM defense made concrete by the first large-HTML source (KASE ~540 KB) on the
   Raspberry Pi target.
5. **`RateSourceOptions.Headers` secret-storage warning (new, from the security lens).**
   A godoc warning forbids storing secrets in `Headers` (plaintext in the DB and in
   git-tracked migrations). This SUPERSEDES the earlier Trade-off note that a future
   Finnhub key could live in `options.headers`: it must not — inject auth via `BEACON_*`
   env vars / `dsninjector`.
6. **`IsDeltaSatisfied` zero-threshold bug fixed (new, correctness).** A pre-existing bug
   (`absDelta >= threshold` fired every tick when threshold=0) was surfaced by the new
   notifier test. Fixed with a `threshold==0 -> absDelta > 0` branch (symmetric via
   `math.Abs`). Affects all `delta` subscriptions, FX included; behavior is tightened
   (no spurious re-fire on an unchanged price), not broken.
7. **Fixtures saved under `testdata/`** (`kase_ccbn_last_deal.html`, `yahoo_v8_aapl.json`),
   as the plan required.

### Empirically validated (live, during review)

- The KASE regex matches the live page markup
  (`class="last-deal"><div ... class="value"> 4 630,00 `) and captures the "price of the
  last deal"; KASE returns 200 to the default `Beacon/1.0` User-Agent (no header needed).
- Yahoo v8 returns 200 + `regularMarketPrice` with the browser User-Agent and currently
  also returns 200 to the default UA, so the per-source browser UA is DEFENSIVE (Yahoo's
  UA/IP filtering is intermittent), not a hard current requirement.

### Explicitly deferred (rejected for the MVP, with rationale)

- A `FetchOptions` struct in place of the positional `headers` param — YAGNI; no second
  per-source fetch knob is on the horizon.
- Including headers in the `rateextractor` response/tombstone cache key — unreachable in
  the MVP (every source has a unique URL); documented inline.
- The dedicated `Last` DTO field and Mini App chart/history rendering — Task 6.
- CRLF header validation at seed-parse — Go's transport already rejects invalid headers
  safely at wire-write; not worth adding the `golang.org/x/net` dependency.
