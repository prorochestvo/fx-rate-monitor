# Task Breakdown

## Overview

Plan 010 (`plans/completed/260530.0003.miniapp-pair-detail-modal.md`) shipped
earlier today. It collapsed each pair row to two text lines (pair label on
line 1, `Spread X.XX%` / `Δ X.XX%` on line 2) and moved the per-series SVG
sparkline into the pair-detail modal. After using the result the user has
decided we cut too aggressively. This plan partially walks plan 010 back and
extends it:

1. **Restore the SVG sparkline on the list row.** The collapsed 2-line text
   block stays exactly as plan 010 left it; the SVG returns as a sibling to
   the text block on the right. Two-series rows draw BID + ASK polylines on a
   shared y-frame; single-series rows draw one polyline. The per-series value
   line that plan 010 deleted does **not** return to the list — it lives only
   in the modal.
2. **Strip the SVG out of the modal.** The modal becomes a text-only detail
   sheet (per-series prefix + latest + delta, spread line when applicable,
   last-grab line, condition badges). No `<svg>` anywhere inside the modal.
3. **Add a per-pair history view, opened from a button at the bottom of the
   modal.** History is per canonical pair, aggregated across all of the
   user's sources for that pair, paginated, newest first. One history entry
   per `rate_values` row: timestamp + source name, BID line, ASK line, with
   per-direction delta percent vs the previous chronological sample from the
   same source.

The change spans both ends of the stack:

- **Backend** — new endpoint `GET /api/me/rates/history`, new DTO
  `dto.MeHistoryResponse`, new repository query, new service. Same Telegram
  WebApp HMAC auth as `/api/me/rates/chart`. 499 on context cancellation
  mirrors the chart handler patch from `87d2e9b`.
- **Frontend (WASM)** — new apiclient method, new state fields on
  `MeSubscriptionsPage`, new `RenderPairHistory` UI function, modal renderer
  rewrites (delete SVG, add History button, swap to history view in place),
  `renderSparklineRow` re-adds the SVG, `subscriptions.html` CSS additions.
- **Tests** — repository + handler tests for the new endpoint, application +
  UI tests for the new client surface and renderers, regression tests that
  prove the SVG is back on the list and gone from the modal.

The history view renders as a **swap inside the same modal card**, not a
separate route. The History button replaces the modal body with the history
list; a back button at the top of that body returns to the detail view. The
modal `OpenPair` continues to drive the modal's visibility — `HistoryOpen`
is a sub-state of it. Closing the modal (`ClosePairModal`) resets
`HistoryOpen` so the next open starts on detail, not history.

History fetch is **lazy**: it fires only when the user clicks the History
button, not when the modal opens. The user may never click it, and the
history query is the most expensive operation in this plan. Eager fetch
would add a round-trip to every modal open for no benefit on the common
path.

## Assumptions

- The user-decisions block in the orchestrator prompt is authoritative: 2-line
  text + SVG on the right in the list; text-only modal; per-pair history
  aggregated across sources; vertical history rows with timestamp + source
  header line and stacked BID/ASK lines.
- The chart service `internal/application/chart/Service.ObtainMeChart` already
  owns the (subscription → source → direction → label) resolution that the
  history endpoint needs. We extend that package rather than duplicating its
  resolution logic in a sibling. The new method
  `Service.ObtainMeHistory(ctx, userID, pair, page, limit)` lives in
  `internal/application/chart/history.go` alongside `service.go`. Trade-off
  recorded under Trade-offs.
- Delta computation runs in **Go**, not in SQL. The window-function path
  (`LAG()`) is supported on `modernc.org/sqlite` but the per-pair history
  query returns at most `len(sources)*limit` rows. Computing the delta in Go
  after the query is trivial (one pass per source) and avoids a second SQL
  variant we have to test. Trade-off recorded under Trade-offs.
- Pagination defaults: **`page=1`, `limit=20`, max `limit=100`**. Mirrors the
  existing `parsePageSize` helper's intent (small default, hard cap).
- "Pair → sources" resolution in the new history query reuses the user's
  subscriptions, the same way the chart does: load the user's subscriptions,
  resolve their `source_name` to `(base, quote, kind)`, filter to those whose
  canonical pair label equals the requested `pair` query string, then load
  `rate_values` for those source names.
- Empty / non-matching pair returns **200 with empty `Items` and `Total: 0`**,
  not 404. The user may have unsubscribed between opening the modal and
  clicking History, and the UI should render a humane empty state rather than
  surfacing 404. 400 is reserved for genuinely malformed input (missing pair,
  bad page, bad limit).
- The handler does **not** validate the pair string against the user's
  current subscriptions before issuing the query. The query already filters
  by subscription join; an unsubscribed pair returns an empty page set, which
  is the correct response.
- The new endpoint is allowed to log the pair label, source name, and rate
  values. None of them are PII. The handler **must not** log the
  `X-Telegram-Init-Data` value (same constraint as existing me-handlers).
- The history button does not change layout when clicked; it only triggers
  state change + redraw. The modal swap renders the history body inside
  `.me-pair-modal-body` so the close-X and the card chrome remain in place.
- The history fetch reuses `MeSubscriptionsPage` rather than spawning a
  separate page controller. Adding a sibling controller (`MeHistoryPage`)
  would force `main.go` to juggle two stateful controllers for one screen.
  The state additions are small enough that one controller still reads
  cleanly.

    // MeHistoryResponse is the JSON envelope returned by GET
    // /api/me/rates/history. Items is sorted newest first.

    declarations. Do not introduce inline string literals for column names.

  - [ ] `ObtainHistoryForPairsPaged` is exported with the documented godoc.
  - [ ] Empty `pairs` returns `([]domain.RateValue{}, 0, nil)` without
        issuing a query.
  - [ ] Returns rows sorted `timestamp DESC, id DESC`.
  - [ ] Returns the un-paginated `COUNT(*)` as the second return.
  - [ ] Uses `r.db.ReadOnlyTransaction(ctx)`.
  - [ ] All column / table references go through the `rateValue*FieldName`

  - Add method:
    ```go
    // ObtainMeHistory returns paginated rate-collection events for the
    // calling user's subscribed sources that match the given canonical pair
    // label (e.g. "USD/KZT"). Results are sorted newest first.
    //
    // page is 1-based. limit is bounded by the caller; the service does no
    // re-bounding. When the user has no subscriptions matching pair, the
    // result has zero items and Total=0 (not an error).
    //
    // The returned MeHistoryResult always groups BID and ASK observations
    // for the same (source, timestamp) into one MeHistoryRow. Delta percent
    // is computed against the previous sample for the same (source, direction)
    // chain in the unpaginated stream — meaning each row's delta compares
    // against the chronologically previous sample of that direction from
    // that source, not against the previous row in the page.
    func (s *Service) ObtainMeHistory(
        ctx context.Context, userID, pair string, page, limit int64,
    ) (*MeHistoryResult, error)
    ```
  - Return type:
    ```go
    type MeHistoryResult struct {
        Pair  string
        Total int64
        Items []MeHistoryRowResult
    }
    type MeHistoryRowResult struct {
        SourceName  string
        SourceTitle string
        Timestamp   time.Time
        Bid         *float64
        Ask         *float64
        BidDeltaPct *float64
        AskDeltaPct *float64
    }
    ```
  - Implementation outline (pseudocode):
    ```
    1. Load subscriptions via subs.ObtainRateUserSubscriptionsByUserID.
    2. Build the source-name set.
    3. Resolve via sources.ObtainRateSourcesByNames.
    4. Build SourcePairKey list filtered to sources whose canonical label
       equals pair (use the same labelBase/labelQuote/pairGroupKey helpers
       that service.go already exports/uses).
    5. If no matching keys: return &MeHistoryResult{Pair: pair, Total: 0,
       Items: []MeHistoryRowResult{}}, nil.
    6. Call history.ObtainHistoryForPairsPaged(pairs, limit, offset).
       (offset = (page-1) * limit, clamped to >= 0.)
    7. Group rows by (source_name, timestamp) — when BID and ASK appear at
       the same (source, timestamp), they collapse into one MeHistoryRowResult.
    8. Compute per-direction deltas: load ONE EXTRA row per source-direction
       chain that precedes the current page (to anchor the first sample's
       delta). Optimization left for later — for v1, compute delta only
       between rows present in the current page; the first row in any
       (source, direction) chain has nil delta. This is documented as a
       known limitation.
    9. Populate SourceTitle from the resolved source map.
    10. Return the result.
    ```
  - The pair → label matching uses the existing helper `pairGroupKey` from
    `service.go` (already unexported). Exporting it is unnecessary — `history.go`
    lives in the same package.
- **Acceptance Criteria:**
  - [ ] `ObtainMeHistory` exists with the documented godoc.
  - [ ] Returns empty result (not an error) when the user has no
        matching subscriptions.
  - [ ] Returns 499-equivalent error when ctx is canceled (the underlying
        sql calls return `context.Canceled`; the handler is responsible for
        the 499 mapping).
  - [ ] Groups same-(source, timestamp) BID + ASK rows into one
        `MeHistoryRowResult`.
  - [ ] Delta percent for the first sample in a (source, direction) chain
        within the page is `nil`, not zero.
  - [ ] Test file `internal/application/chart/history_test.go` exists with
        `TestService_ObtainMeHistory` and subtests:
        - "no subscriptions returns empty page and total zero"
        - "pair not matching any subscription returns empty page"
        - "single source single direction page 1"
        - "two sources two directions are interleaved by timestamp"
        - "delta is nil for first observation in chain"
        - "delta is computed against previous observation in same chain"
        - "pagination offset and limit apply"
        - "context cancellation propagates as error"
- **Pitfalls & edge cases:**
  - The existing `Service.now` is unused by the history method (history is
    not time-windowed in this version). Leave it alone.
  - `pairGroupKey` casing: pair labels are upper-case `BASE/QUOTE`. The
    handler should upper-case the incoming `pair` query parameter before
    passing it through, so a `?pair=usd/kzt` is treated identically to
    `?pair=USD/KZT`.
  - Same-source rows with the SAME timestamp but DIFFERENT directions exist
    when a collector emits both directions in the same scrape. The group
    must collapse them; never emit two rows for the same (source, timestamp).
  - If a source emits BID only, `Ask` is nil; if it emits ASK only, `Bid` is
    nil. Both fields nil is a programming error — assert in tests.
- **Complexity:** Hard

## Risks

- **Delta gap at page boundaries.** Computing delta within the page only
  means row 1 of page 2 will have a `nil` delta even though row 1 of the
  full history has a predecessor. This is the documented v1 limitation
  (Task 3, point 8). If the user complains, the fix is to load one extra
  row per (source, direction) chain before page boundaries — a 10-line
  change that we explicitly defer.
- **SQLite expression-tree limit.** Repository query uses `IN (...)` with
  one tuple per (source, base, quote) triple subscribed by the user. A
  user with ~333+ unique subscribed sources on one pair would overflow
  SQLite's 1000-term default. Vanishingly unlikely; documented in Task 2.
- **Mobile width and SVG squeeze.** The flex layout in Task 12 may not
  behave correctly on very narrow Telegram WebView viewports (e.g.
  iPhone SE landscape with system fonts at 200%). QA on real devices
  before merging.
- **Modal swap UX surprise.** Tapping "History" replaces the detail view
  in-place. Users may expect a new screen / route. The back button is
  prominent enough but watch for confusion in QA feedback.
- **Backwards-compat with deployed clients.** The `MeRatesHistory`
  endpoint is purely additive — no existing client sees it. The
  `appchart.Service` constructor signature changes; the only known
  caller is `cmd/web/main.go`. Both endpoints (chart and history)
  continue to work in isolation.
- **History fetch on slow networks.** Lazy fetch means the user sees a
  loading skeleton between clicking History and seeing rows. The modal
  open is unchanged. Acceptable for v1.
- **Stale `HistoryItems` after a modal close + reopen of a different
  pair.** `ClosePairModal` resets history state (Task 7) so this is
  guarded — assert in tests.

## Trade-offs

- **Extending `internal/application/chart` over a new `internal/application/history` package.**
  Chosen because the resolution logic (subscriptions → sources →
  canonical pair labels) is already encapsulated in the chart service.
  Forking the package would duplicate `pairGroupKey`, `dedupePairTriples`,
  `uniqueSourceNames`, and the `SourcesLoader` / `SubscriptionsLoader`
  contracts. Cost: the package name `chart` will be slightly misleading
  once it owns both chart and history methods; the godoc on the package
  declaration must be updated to say "chart and history" or similar.
- **Delta computation in Go after the SQL query, not via `LAG()`.**
  `LAG()` is supported in `modernc.org/sqlite` 3.39+ which we already
  use, but the Go path is simpler to test and avoids a SQL variant we
  have to keep in sync with the column constants. With `limit <= 100`
  the working set is trivial. If the page grows much larger we can
  switch.
- **Lazy history fetch vs eager prefetch on modal open.** Lazy chosen.
  The history endpoint is the most expensive in this plan and the
  majority of modal opens are not followed by a History click (assumed,
  not measured — record this assumption and revisit if the user reports
  noticeable loading delay).
- **In-modal swap vs separate screen for history.** In-modal swap chosen
  to avoid touching the page-router structure for what is essentially a
  drill-down. Reading the back button copy ("← Back to details") is the
  affordance that keeps users oriented.
- **Pagination defaults `page=1, limit=20, max=100`.** 20 is small enough
  to render fast on Telegram WebView, large enough to cover roughly a
  half-day of activity for a noisy source. The 100 cap is the standard
  defensive bound applied across our other endpoints.
- **Returning 200 + empty page over 404 when no subscription matches the
  requested pair.** 404 would force the UI into an error path for a
  state that is functionally "no rows," which the empty-state copy
  already handles cleanly. 404 is reserved for "this route does not
  exist." Documented in Task 4.
- **Reusing the existing `MeSubscriptionsPage` controller for history
  state vs a sibling `MeHistoryPage`.** Reuse chosen because the WASM
  layer already manages exactly one state controller per route. Adding
  a sibling would force `main.go` to coordinate two controllers for the
  same screen, including their loading flags. The state additions are
  small enough that one controller still reads cleanly.
- **No focus management in the history view.** Same precedent as plan
  010's modal: Telegram Mini Apps are touch-first; the cost of cycling
  focus through the back / prev / next buttons is higher than the
  benefit. Escape and the back button cover keyboard exit.
