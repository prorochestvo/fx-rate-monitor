# Task Breakdown

## Overview

Bundle 3 of plan 008 (commit `8a3583e`, 2026-05-22) added FK constraints with
`ON DELETE CASCADE` on `rate_values`, `rate_user_subscriptions`, and
`rate_user_events` pointing at `rate_sources(name)`, plus a `busy_timeout=5000`
PRAGMA aimed at smoothing concurrent-writer contention. Both protections rely
on per-connection PRAGMAs that are not actually applied to every connection in
the pool.

Concretely, `internal/infrastructure/sqlitedb/sqlclient.go`:

```go
db.SetMaxOpenConns(7)                       // pool of 7 connections
// ...
for _, pragma := range []string{
    "PRAGMA foreign_keys=ON;",
    "PRAGMA journal_mode=WAL;",
    "PRAGMA busy_timeout=5000;",
} {
    if _, err := db.Exec(pragma); err != nil { ... }
}
```

`db.Exec` checks out one anonymous connection from the pool, runs the PRAGMA,
and returns the connection. PRAGMA `foreign_keys` and PRAGMA `busy_timeout` are
**per-connection**, so the other six connections opened lazily by
`database/sql` are left with the SQLite defaults (`foreign_keys=OFF`,
`busy_timeout=0`). `journal_mode=WAL` is persistent in the database file
header and therefore unaffected.

Symptoms in production today:

- FK constraints are enforced on roughly 1/7 of write paths and skipped on
  6/7. Orphan inserts (e.g. an `INSERT INTO rate_values` with a misspelled
  `source_name`) succeed silently on six connections out of seven and are
  rejected on the seventh — non-deterministic by connection scheduling.
- `busy_timeout=5000` only protects writes that happen to land on the same
  one connection. The other six get `SQLITE_BUSY` immediately under
  contention, which was the exact failure mode Bundle 3 set out to fix.
- `RemoveRateSource` is not currently wired into any handler, so the
  CASCADE path is dormant. The `INSERT/UPDATE` FK enforcement gap is the
  active half of the bug.

Test coverage gap that masks the issue:

- Every test that exercises an in-memory SQLite via `stubSQLiteDB` sets
  `SetMaxOpenConns(1)`. With one connection, the PRAGMAs always apply and
  the bug is invisible.
- `TestSourceRepository_RemoveSource/delete` only verifies that the
  parent row is gone after `RemoveRateSource`. It never asserts that
  dependent rows in the three child tables were also deleted. The CASCADE
  contract written into CLAUDE.md ("deleting a source destroys all
  dependent rows") has no test.

This plan fixes the PRAGMA wiring, adds regression tests that would have
caught the gap, and audits stage/prime for orphans accumulated under the
broken regime.

## Assumptions

- `modernc.org/sqlite` accepts PRAGMA DSN parameters of the form
  `?_pragma=foreign_keys(1)&_pragma=busy_timeout(5000)` and applies them
  to every new connection opened by the pool. (To be verified against the
  driver version pinned in `go.mod` before implementation.)
- It is acceptable to keep `db.SetMaxOpenConns(7)` in production. The
  problem is not the pool size — it is the PRAGMA wiring.
- Stage and prime are the only production-like databases. There are no
  long-running developer copies that need separate orphan audits.
- The fix can ship without a schema migration. PRAGMAs are connection-time
  settings, not on-disk state.

## Tasks

### Task 1: Apply PRAGMAs via DSN, not via single `db.Exec`

- Description: change `NewSQLiteClient` / `NewSQLiteClientEx` to pass
  `foreign_keys` and `busy_timeout` through the DSN string accepted by
  `modernc.org/sqlite`, so the driver re-applies them on every new
  connection opened by the pool. `journal_mode=WAL` may stay as a single
  `db.Exec` because it is persistent in the database file header.
- Acceptance Criteria:
  - Every connection returned by the pool reports `PRAGMA foreign_keys` as
    `1` and `PRAGMA busy_timeout` as `5000`, verified by a test that opens
    the client, sets `SetMaxOpenConns(N>1)`, drains N concurrent
    connections, and reads the PRAGMA values on each.
  - `NewSQLiteClientEx` no longer relies on `db.Exec("PRAGMA foreign_keys=...")`
    for correctness — it may keep the call for `journal_mode=WAL` only.
  - The DSN construction is encapsulated in one helper; callers continue
    to pass the bare path/DSN from `SQLITEDB_DSN` unchanged.
- Pitfalls & edge cases:
  - `dsninjector.DataSource.Database()` returns the path portion. The
    helper must append `_pragma` query parameters without breaking
    existing DSN formats. Verify against both `sqlite://./build/x.db`
    and `:memory:` shapes used in tests.
  - The driver applies PRAGMAs in DSN order. Put `busy_timeout` before
    `foreign_keys` so the FK enforcement check (which itself can trip a
    busy-wait under contention) already has the retry window.
  - The Timeout invariant in CLAUDE.md (busy_timeout < Go Timeout) still
    holds. Do not raise busy_timeout without raising Timeout first.
- Complexity: Easy

### Task 2: Regression test — FK rejection under multi-connection pool

- Description: add a test in `internal/infrastructure/sqlitedb/` that
  opens an in-memory client with `SetMaxOpenConns(4)`, applies the
  current migrations, and concurrently attempts four `INSERT INTO
  rate_values` rows where `source_name` does not exist in `rate_sources`.
  All four inserts must return `FOREIGN KEY constraint failed`. This test
  fails on `main` today and passes after Task 1.
- Acceptance Criteria:
  - The test forces parallel use of distinct pool connections (use
    `errgroup.Go` with `sync.WaitGroup` barrier; do not rely on
    scheduling luck).
  - Failure message asserts a SQLite FK error, not a generic "execution
    failed".
  - Runs under `-race` and in the standard `make test` flow.
- Pitfalls & edge cases:
  - `:memory:` databases are per-connection by default; the test must use
    `file::memory:?cache=shared` or a temp file path so the four
    connections see the same schema.
  - Goroutine leak hygiene: ensure all goroutines exit before
    `t.Cleanup` closes the DB.
- Complexity: Medium

### Task 3: Regression test — cascade delete on `RemoveRateSource`

- Description: add a test that creates a `rate_sources` row plus child
  rows in all three child tables (`rate_values`,
  `rate_user_subscriptions`, `rate_user_events`), invokes
  `RemoveRateSource`, then asserts that each child table has zero rows
  for the deleted `source_name`. Test must run under
  `SetMaxOpenConns(N>1)` so it exercises the real pool wiring.
- Acceptance Criteria:
  - Three independent assertions, one per child table, each comparing
    the row count to 0.
  - Test fails on `main` today (pool=1 obscures the bug, but
    `RemoveRateSource` itself plus the FK definition pass on pool=1, so
    the test needs pool>1 to be meaningful — document this in a comment).
  - Lives in `internal/repository/ratesource_test.go` as a subtest of
    `TestSourceRepository_RemoveSource`.
- Pitfalls & edge cases:
  - Child rows must satisfy NOT NULL constraints — supply realistic
    fixtures (see existing tests in `ratevalue_test.go`,
    `rateusersubscription_test.go`).
  - Other parallel subtests in the same file already use
    `stubSQLiteDB(t)` with pool=1. Either give this subtest its own
    helper or extend `stubSQLiteDB` with a pool-size option.
- Complexity: Medium

### Task 4: Audit stage and prime for orphan child rows

- Description: before Task 1 ships, run a read-only audit on both stage
  and prime to surface child rows whose `source_name` no longer matches
  any `rate_sources` row. Once Task 1 lands, those orphans will continue
  to sit in the database (FK is only enforced on write) but new
  insert/cascade attempts will surface them. Decide per orphan whether
  to delete or backfill the parent.
- Acceptance Criteria:
  - Three audit queries are documented in `deploy/README.md` (or a new
    `deploy/orphan-audit.sql`), one per child table:
    ```sql
    SELECT child.source_name, COUNT(*)
      FROM <child_table> child
      LEFT JOIN rate_sources s ON s.name = child.source_name
     WHERE s.name IS NULL
     GROUP BY child.source_name;
    ```
  - Audit runs against stage and prime. Results recorded in this plan
    (append a "Audit results" section before moving to completed).
  - If orphans exist, follow-up commit cleans them up before Task 1 is
    deployed, or the deploy is staged so the cleanup precedes restart.
- Pitfalls & edge cases:
  - `rate_user_events.source_name` is nullable; treat `NULL` as
    "not an orphan" — only rows with non-null source_name that do not
    join belong in the report.
  - Run audit from a snapshot or via `sqlite3 -readonly` to avoid
    blocking the live writer.
- Complexity: Easy

### Task 5: Update CLAUDE.md after Task 1

- Description: the current `### Database` section claims "All connections
  set three PRAGMAs at open time (in `NewSQLiteClientEx`)". After Task 1
  this is true; until Task 1 lands it is false. Update the wording to
  reflect the DSN-based wiring and drop the implicit claim that
  `db.Exec` is sufficient.
- Acceptance Criteria:
  - The sentence describing how PRAGMAs are applied mentions the DSN
    pathway and names `_pragma` parameter syntax.
  - The `busy_timeout` invariant note stays.
- Pitfalls & edge cases: none.
- Complexity: Easy

## Execution Order

1. Task 4 (audit) — runs against current production state, must precede
   any code change so we know what orphans we are inheriting.
2. Task 1 (PRAGMA fix) — landed only after audit results are reviewed.
3. Task 2 + Task 3 (regression tests) — committed together with Task 1
   so the fix is covered.
4. Task 5 (CLAUDE.md) — same commit as Task 1.

## Risks

- **Driver behaviour with `_pragma` DSN params** — the documented syntax
  for `modernc.org/sqlite` has shifted historically. If the pinned
  version does not honour `_pragma`, fall back to a registered driver
  with a connect hook. This adds ~30 lines but is functionally
  equivalent.
- **Pre-existing orphans in production** — Task 4 may surface rows that
  Task 1 will then start protecting against. Cleaning orphans is a
  separate decision (delete vs backfill parent); the audit produces the
  data, not the policy.
- **Test-prod parity** — keeping test pool at 1 was convenient but
  produced exactly this blind spot. Task 2 / Task 3 force pool>1 in at
  least these new tests. We are not retrofitting pool=N across the
  entire test suite — only where it would catch a class of bug that
  pool=1 hides.

## Trade-offs

- **DSN params vs registered driver with connect hook.** DSN is one
  line; a registered driver is the more general escape hatch but is
  overkill for three PRAGMAs and adds an import side effect across the
  binary. Picking DSN.
- **Audit before fix vs audit alongside fix.** Audit first preserves the
  option to backfill rather than delete, which a hard FK enforcement
  pass cannot do retroactively. Cheap, worth the round trip.
- **`RemoveRateSource` exposure.** Not in scope. The method stays with
  its godoc warning until somebody actually wants to delete a source
  from a handler.

## Outcome (2026-05-23)

- **Task 1 done.** `connectionOptions` appends
  `?_pragma=busy_timeout(5000)&_pragma=foreign_keys(1)` to the DSN
  before `sql.Open`. `NewSQLiteClient` uses the helper; the
  modernc.org/sqlite v1.29.9 `Open` hook re-applies both PRAGMAs on
  every new pool connection. `NewSQLiteClientEx` keeps its `db.Exec`
  PRAGMA loop for the legacy single-connection test path; godoc spells
  out the production-vs-test split. `journal_mode=WAL` continues to be
  set once via `db.Exec` because it is persisted in the database file
  header.
- **Task 2 done.** New `TestSQLitePoolPerConnectionPragmas` in
  `internal/infrastructure/sqlitedb/sqlclient_pool_test.go` covers two
  subtests: per-connection PRAGMA inspection across four reserved pool
  slots, and four concurrent transactions racing an orphan-FK INSERT
  through distinct pool connections. Sanity-verified by temporarily
  no-op'ing the helper: both subtests went red with `foreign_keys=0`
  and an accepted orphan INSERT, respectively, then turned green after
  the fix was restored.
- **Task 3 done.** New `cascade removes dependent rows` subtest of
  `TestSourceRepository_RemoveSource` exercises the full CASCADE
  contract across `rate_values`, `rate_user_subscriptions`, and
  `rate_user_events`. The subtest pins pool slot 1 in a parked
  transaction so the cascade workload opens fresh connections; the
  fresh slots only get `foreign_keys=ON` via the DSN driver hook, so
  the post-condition assertions on child row counts directly observe
  the fix. Sanity-verified: removing the DSN PRAGMA parameters made
  the cascade assertions fail with `Should be zero, but was 1`.
- **Task 4 skipped.** Operator wiped stage to start from a clean
  schema, so the orphan inventory was unnecessary. Prime status was
  not relevant at this stage. If the project later moves to a prod
  database with pre-existing data, run the three audit queries
  (`SELECT child.source_name FROM <child> LEFT JOIN rate_sources ...`)
  before deploying further FK-affecting migrations.
- **Task 5 done.** `CLAUDE.md` Database section rewritten to describe
  the DSN-based PRAGMA pathway, the `_pragma=` query parameter syntax,
  and the rationale for keeping `journal_mode=WAL` as a one-shot
  `db.Exec`. The `busy_timeout < Timeout` invariant is preserved.

`make test` is green; no other tests required updates.
