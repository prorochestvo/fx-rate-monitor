# Task Breakdown

## Overview

Telegram rate notifications produced by `internal/application/notification/ratecheckagent.go` and rendered by `buildAlertMessage` in `internal/application/notification/message.go` currently emit one bullet per fired `RateUserSubscription`, with no indication of which trigger (delta / interval / daily / cron) caused the alert. When a single user has multiple subscriptions on the same source/currency-pair/kind, they receive duplicate or visually indistinguishable bullets.

This plan adds:

1. **Per-(source, base, quote, kind) deduplication** in `RateCheckAgent.Run`: every fired subscription on the same dedup key for the same user folds into a single `alert`, while every fired subscription still has its `LatestNotifiedRate`/`UpdatedAt` retained via `RetainRateUserSubscription`.
2. **Trigger-icon labels** appended after the existing price/delta/forecast portion of each bullet, using the agreed Unicode glyphs `Δ`, `⏱`, `⌚`, `⏲`, with compact condition-value text.
3. **Shared label-formatting helpers** extracted into a small package so both `internal/application/service/telegramapi.go` and `internal/application/notification/message.go` use the same formatter.

The change is internal and additive: no new tables, no new HTTP routes, no new env vars.

## Assumptions

- The dedup key is the tuple `(SourceName, BaseCurrency, QuoteCurrency, CurrencyKind)`, scoped per `(UserType, UserID)`. `SourceName` already implies `(Base, Quote, Kind)` in the current schema, but the wider key keeps the code defensive against future shared sources and matches the user's stated intent.
- Stable icon order in the rendered bullet is **delta, interval, daily, cron** regardless of the order in which subscriptions were evaluated.
- When the same condition type fires more than once on the same dedup key for the same user, only one icon is shown for that type, with a representative value chosen as follows:
  - **delta** — lowest threshold (most sensitive trigger that fired).
  - **interval** — shortest `time.Duration` parsed from `ConditionValue`.
  - **daily** — earliest `HH:MM:SS` lexicographically.
  - **cron** — earliest weekday number (Sun=0 .. Sat=6) extracted from the cron expression.
- The notification package may import a sibling helper package; it must NOT import `internal/application/service` directly (that would couple the notification cycle to the Telegram-bot-API service surface).
- Existing public constants in `message.go` (`telegramBotArrowUp`, `telegramBotArrowDown`, `telegramBotForecast`, `telegramMaxMessageLen`) stay where they are; the new icon constants live alongside them.
- `make test` continues to be the gate. No CGO is introduced.

- **Acceptance Criteria:**
  - [ ] New file `internal/application/labelfmt/label.go` exposes `SubscriptionConditionLabel`, `IntervalLabel`, `CronWeekdayLabel` with identical behaviour to the originals.
  - [ ] All call sites in `internal/application/service/telegramapi.go` use `labelfmt.*`; the unexported helpers are removed.
  - [ ] New `internal/application/labelfmt/label_test.go` covers each function with one `Test*` per exported function and scenarios as `t.Run` subtests, matching the project test style. Tests cover at minimum: each `ConditionType` for `SubscriptionConditionLabel`, the `24h`/`168h`/passthrough branches for `IntervalLabel`, and a valid expression + a malformed expression for `CronWeekdayLabel`.
  - [ ] `make test` passes; existing tests in `internal/application/service` continue to pass without modification (behaviour is unchanged).
  - [ ] `make lint` (`go fmt` + `go vet`) is clean.

- **Pitfalls:**
  - The helpers currently return phrasing tailored to the Telegram inline-keyboard UI (e.g. `"daily at 06:00 UTC"`, `"weekly on Monday (UTC 09:00)"`). Do **not** change that wording — the bot UI tests will fail. The compact bullet variant for notifications is built in Task 3 by composing `IntervalLabel` / `CronWeekdayLabel` with new format strings, not by editing these.
  - `cronWeekdayLabel` returns the raw expression on parse failure. Preserve that — callers depend on it as a stable fallback.
  - Keep the package name `labelfmt` lowercase one word; do not split across files unless the file grows beyond ~120 lines.

### Task 2: Add trigger-icon constants and a compact-label helper in the notification package

- **Description:**
  In `internal/application/notification/message.go`, alongside the existing `telegramBotArrowUp` / `telegramBotArrowDown` / `telegramBotForecast` constants, add four named constants:

  ```go
  const (
      triggerIconDelta    = "Δ" // U+0394
      triggerIconInterval = "⏱" // U+23F1
      triggerIconDaily    = "⌚" // U+231A
      triggerIconCron     = "⏲" // U+23F2
  )
  ```

  Add an unexported helper `triggerLabel(condType domain.SubscriptionConditionType, condValue string) string` returning the compact bullet form:
  - `delta`    → `"Δ ≥10%"` (no space between `≥` and number, raw `condValue` followed by `%`)
  - `interval` → `"⏱ 4h"` (uses `labelfmt.IntervalLabel`)
  - `daily`    → `"⌚ 06:00"` (first 5 chars of `condValue` if length ≥ 5; otherwise the full value)
  - `cron`     → `"⏲ Mon"` (first three letters of `labelfmt.CronWeekdayLabel(condValue)`)

  Unknown condition types return an empty string — the caller treats empty as "no icon".

  This helper does **not** know how to combine multiple labels; that's the deduper's job in Task 3.

- **Acceptance Criteria:**
  - [ ] Four exported-style-but-unexported constants added with the exact glyphs from the spec.
  - [ ] `triggerLabel` returns the values described above for the four known condition types and `""` for unknown.
  - [ ] New tests in `message_test.go` (single top-level `TestTriggerLabel` with subtests per condition type and an "unknown" subtest) follow the project's one-test-per-function rule. Tests use `t.Parallel()` and `require`.
  - [ ] `make test` passes.
  - [ ] No new third-party imports.

- **Pitfalls:**
  - The Unicode glyphs must be the exact code points listed in the background section. Reviewers can paste them through `go run -gcflags='-m'`-style sanity checks — but the simplest verification is to grep the file for `Δ`/etc., or rely on the test using string equality with the constant.
  - `Δ ≥10%` deliberately omits the space between `≥` and the number (compact bullet variant). The non-compact `SubscriptionConditionLabel` keeps `Δ ≥ 10%%` — different format, different consumer. Don't unify the two.
  - For cron, `cronWeekdayLabel` returns full names like `Monday`. Take the first 3 runes (`Mon`, `Tue`, ...) — bytes are safe here because the names are ASCII, but use `[:3]` after a length check.

- **Complexity:** Easy

- **Code Example:**
  ```go
  func triggerLabel(condType domain.SubscriptionConditionType, condValue string) string {
      switch condType {
      case domain.ConditionTypeDelta:
          return fmt.Sprintf("%s ≥%s%%", triggerIconDelta, condValue)
      case domain.ConditionTypeInterval:
          return fmt.Sprintf("%s %s", triggerIconInterval, labelfmt.IntervalLabel(condValue))
      case domain.ConditionTypeDaily:
          v := condValue
          if len(v) >= 5 {
              v = v[:5]
          }
          return fmt.Sprintf("%s %s", triggerIconDaily, v)
      case domain.ConditionTypeCron:
          name := labelfmt.CronWeekdayLabel(condValue)
          if len(name) >= 3 {
              name = name[:3]
          }
          return fmt.Sprintf("%s %s", triggerIconCron, name)
      default:
          return ""
      }
  }
  ```

  ```go
  Triggers []alertTrigger // ordered by stable icon order (delta, interval, daily, cron)

  }
  ```

  In `buildAlertMessage`, after the existing price/delta/forecast section is appended to `val` but before `values = append(values, val)`, append the trigger labels:

  Do not change the existing bullet prefix, currency-pair logic, delta/forecast sections, or message-splitting logic.

- **Acceptance Criteria:**

  - [ ] `buildAlertMessage` emits the trigger labels at the end of each bullet, separated from the previous content by a single space, e.g. ` • <b>USD/KZT</b>: 470.46 (1.20 🔼) Δ ≥10% ⌚ 06:00`.
  - [ ] Bullets with no triggers (empty slice) are byte-identical to today's output. This is verified by an updated `TestBuildAlertMessage` subtest "empty triggers — output unchanged" that uses `require.Equal` against the existing format.
  - [ ] New `TestBuildAlertMessage` subtests cover:

  - [ ] All pre-existing `TestBuildAlertMessage` subtests still pass; any whose substring assertions need adjustment because the bullet now ends with extra content remain valid (most use `strings.Contains`, which is unaffected).

- **Pitfalls:**
  - The 2048-char split logic in `buildAlertMessage` measures against `len(message)` for the per-source block, not per-bullet. Adding ~30 bytes per bullet can shift the boundary; the existing "messages split above 2048 chars" test case may need tightening or relaxing (currently asserts `Equal(t, 2, len(msgs))`). Verify this and update only if the boundary genuinely shifts; do NOT silently weaken the assertion.

  - Empty-string returns from `triggerLabel` (unknown condition type) must be filtered out so we don't emit `" "` runs.

### Task 4: Dedup fired subscriptions in `RateCheckAgent.Run` and retain every fired subscription

- **Description:**
  Replace the current per-subscription `append` to `telegramBotAlerts[subscription.UserID]` (lines 91-105 of `internal/application/notification/ratecheckagent.go`) with a fold step that groups by `(SourceName, BaseCurrency, QuoteCurrency, CurrencyKind)` per `UserID`.

  Concretely, inside the `for _, source := range sources` loop, after fetching `subscriptions`:

  1. Initialize a `map[string]map[string]*alert` shaped as `userID → dedupKey → *alert` scoped to this source.
  2. Iterate `subscriptions`. For each fired subscription:
     - Compute `dedupKey := source.Name + "|" + base + "|" + quote + "|" + string(kind)` (the source already pins three of those, so in practice this is `source.Name`, but follow the spec).
     - If the key has no `*alert` yet, create one with the current `alert` field values, an empty `Triggers` slice, and store the pointer.
     - Append `alertTrigger{ConditionType, ConditionValue}` to the existing `*alert.Triggers`.
     - Always call `subscription.LatestNotifiedRate = currentValue` and `RetainRateUserSubscription` for **every** fired subscription, not only the first. Errors here aggregate into `errs[source.Name]` exactly as today.
  3. After the subscription loop, for each `(userID, alerts)` pair: sort each `alert.Triggers` into the canonical order (delta, interval, daily, cron) — collapsing duplicates of the same condition type per the dedup rules in Assumptions — and append the dereferenced `alert` value to `telegramBotAlerts[userID]`.

  Preserve every other behaviour: error wrapping with `internal.NewTraceError`, the `default:` branch for unsupported `UserType`, the per-chat message-build/retain loop at the bottom of `Run`.

- **Acceptance Criteria:**
  - [ ] Dedup logic groups fired subscriptions by `(SourceName, BaseCurrency, QuoteCurrency, CurrencyKind)` per `UserID` per source-iteration.
  - [ ] When N subscriptions on the same dedup key for the same user fire, exactly one `alert` is appended to `telegramBotAlerts[userID]` and its `Triggers` length equals `min(N, 4)` (4 = number of distinct `ConditionType`s).
  - [ ] `RetainRateUserSubscription` is called for **every** fired subscription, including those whose trigger collapsed into an existing alert. A new test in `ratecheckagent_test.go` named `t.Run("multiple subs same dedup key — all retained", ...)` asserts the mock's retained-call count equals the number of fired subscriptions.
  - [ ] `Triggers` arrives at `buildAlertMessage` in canonical order regardless of subscription evaluation order.
  - [ ] When the same condition type fires more than once on the same dedup key, only one `alertTrigger` is emitted for that type, choosing:
    - delta → lowest numeric threshold (`strconv.ParseFloat`; on parse failure prefer the existing entry).
    - interval → shortest `time.ParseDuration` (on parse failure prefer the existing entry).
    - daily → lexicographically-earliest `ConditionValue`.
    - cron → smallest weekday digit (parts[4] of `strings.Fields`); on parse failure prefer the existing entry.
  - [ ] All existing `TestRateCheckAgent_Run` subtests pass. The "two sources same user — consolidated into one event" test (lines 297-324) MUST still pass — it covers cross-source consolidation, which is unaffected by the within-source dedup change.
  - [ ] `make test` passes with `-race`.

- **Pitfalls:**
  - **Mutable subscription copies.** The loop ranges over `subscriptions` by value. Mutating `subscription.LatestNotifiedRate` and passing `&subscription` to `RetainRateUserSubscription` is fine because each iteration gets a fresh copy — but if the dedup logic introduces a closure over `subscription`, the loop-variable-aliasing bug pattern can creep in. Use Go ≥ 1.22 semantics (per-iteration scope) or explicitly copy into a local.
  - **Retain-before-or-after dedup decision.** Retain MUST happen for every fired subscription regardless of whether it added a new trigger or was collapsed into an existing one. The simplest correct shape: do the retain in the per-subscription loop body unconditionally after `IsDue` returns true; do the dedup/append separately.
  - **Error pathway.** If `RetainRateUserSubscription` fails for one subscription on a dedup key, the alert for that key has already been built. Today's behaviour is to record the error in `errs[source.Name]` and continue; preserve that — don't drop the alert.
  - **Default arrow/forecast fields** (e.g. `Delta`) come from the first subscription that lands in the bucket. Since all subscriptions on the same dedup key see the same `currentValue` in the same run, `Delta` and `CurrentPrice` will be identical across them, so first-write-wins is correct. Document this in a comment so a future reader doesn't think it's a bug.
  - **Tests use loop-only mocks.** `mockCheckSubscriptionRepository.RetainRateUserSubscription` currently doesn't count calls. Either add a counter field to the mock or capture the retained `*RateUserSubscription` slice (deep-copy, like `mockCheckEventRepository.RetainRateUserEvent` already does at lines 367-371). Prefer the latter — it composes with future tests.

- **Complexity:** Medium

- **Code Example:**
  ```go
  type dedupBucket struct {
      a        alert
      triggers map[domain.SubscriptionConditionType]string // condType → ConditionValue (collapsed by rule)
  }

  buckets := make(map[string]map[string]*dedupBucket) // userID → dedupKey → bucket

  for _, sub := range subscriptions {
      delta := currentValue - sub.LatestNotifiedRate
      ok, err := sub.IsDue(now, delta)
      if err != nil { /* aggregate */ continue }
      if !ok { continue }

      // 1. Always retain — fired subs must advance LatestNotifiedRate.
      sub.LatestNotifiedRate = currentValue
      if err := a.rateUserSubscriptionRepository.RetainRateUserSubscription(ctx, &sub); err != nil {
          errs[source.Name] = errors.Join(errs[source.Name], errors.Join(err, internal.NewTraceError()))
          // fall through — still emit the alert
      }

      if sub.UserType != domain.UserTypeTelegram {
          errs[source.Name] = errors.Join(errs[source.Name], fmt.Errorf("unsupported user type: %s", sub.UserType))
          continue
      }

      key := dedupKey(source, source.Kind) // "name|base|quote|kind"
      userBuckets := buckets[sub.UserID]
      if userBuckets == nil {
          userBuckets = make(map[string]*dedupBucket)
          buckets[sub.UserID] = userBuckets
      }
      b := userBuckets[key]
      if b == nil {
          b = &dedupBucket{
              a:        alert{ /* fields from source + currentValue + delta + now */ },
              triggers: make(map[domain.SubscriptionConditionType]string),
          }
          userBuckets[key] = b
      }
      b.triggers[sub.ConditionType] = collapse(b.triggers[sub.ConditionType], sub.ConditionValue, sub.ConditionType)
  }

  // After the source loop, flatten buckets into telegramBotAlerts in canonical trigger order.
  ```

## Risks

- **Trigger-icon glyph rendering on user devices.** `Δ`, `⏱`, `⌚`, `⏲` render reliably on iOS/Android/desktop Telegram, but font fallback on minimal Linux clients can show tofu boxes. Mitigation: the spec is locked (do not propose alternatives); if a user reports tofu, swap to a different glyph in `message.go` constants only — no other code changes needed.
- **Boundary shift in the 2048-char split test.** Adding ~30 bytes per bullet can flip the existing assertion from `Equal(t, 2, len(msgs))` to 3. Mitigation: re-verify and update the test case with an exact, intentional new value rather than `Greater(t, len(msgs), 1)` so future regressions are caught.
- **Forgotten retain on collapsed subscriptions.** This is the single highest-impact bug if missed: a delta sub whose trigger collapses into a daily sub's bucket and never has its `LatestNotifiedRate` advanced will fire on every subsequent run, spamming the user. Mitigation: Task 5's `subRepo.retained` length assertion is the canary; do not weaken or remove it.
- **Cross-package cycle.** Importing `internal/application/service` from `internal/application/notification` would couple the notification cycle to the Telegram inline-keyboard service. Mitigation: `internal/application/labelfmt` is a leaf package — verified by `go list -deps` showing no application-layer imports.

## Trade-offs

- **`labelfmt` as a new package vs. inlining in `notification`.** Chose extraction. Inlining (or duplicating helpers) keeps the diff smaller but creates two sources of truth for label phrasing and risks them drifting (e.g. someone adding a new condition type in only one place). The new package is ~50 lines and has clear ownership.
- **Notification importing service vs. extracting to a shared package.** Chose extracting to `labelfmt`. Importing `service` would pull the entire Telegram-API surface (`tgbotapi`, integration tests, callback parsing) into the notification cycle's dependency graph, slowing builds and creating an obvious smell.
- **Compact bullet variant (`Δ ≥10%`) vs. reusing `SubscriptionConditionLabel` (`Δ ≥ 10%%`).** Chose compact. The bot-UI variant is verbose because it is shown in a settings menu where space is plentiful; the bullet variant is shown inline next to a price and must stay short. Two formats, one set of underlying parsers.
- **Dedup at the agent (Task 4) vs. dedup at the message builder.** Chose agent. The retain-every-fired-subscription invariant is at the agent layer — pushing dedup down to the message builder would force every caller to also wire retain logic, which is the opposite of the desired separation of concerns.
- **First-trigger-wins for collapsed same-condition-type duplicates vs. most-specific-wins.** Chose most-specific (lowest threshold / shortest interval / earliest time / earliest weekday). This matches user intent — the most-sensitive trigger is the one that "really" fired — and is deterministic regardless of subscription evaluation order.
