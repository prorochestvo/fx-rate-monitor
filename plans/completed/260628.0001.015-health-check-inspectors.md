# Task Breakdown

## Overview

Replace the hardcoded `/healthz` endpoint (single DB ping, static JSON response) with the
full inspector pattern described in the project health-check skill:

- `/ping` — dumb liveness; touches no dependency; always 200 `{"status":"ok"}`.
- `/health/check` — real readiness; runs DB and Telegram inspectors under a 3 s timeout;
  503 when any dependency is down, with per-component JSON body.
- `/healthz` kept as an alias for `/ping` for backward compatibility with any existing
  monitors, but the deploy gate is updated to gate on `/health/check`.

## Assumptions

- The `web` binary is the only binary that needs this endpoint (collector/notifier/doctor
  do not expose HTTP). Inspectors are wired only from `cmd/web/main.go`.
- The `/health/check` endpoint is reachable only from localhost (the CI deploy gate uses
  `http://localhost:8000/health/check`), so no bearer-token auth is added.
- The Telegram bot client's `Ping` ignores the context (upstream `tgbotapi` limitation);
  the inspector participates in the aggregate timeout at the loop level but the underlying
  HTTP call inside `GetMe` may block beyond the timeout in degenerate cases. Acceptable.

## Tasks

### Task 1: Inspector package

- Description: Create `internal/application/inspector/inspector.go` with the `Inspector`
  interface, the `Agent` aggregator (`NewAgent`, `CheckUp`), and two adapters:
  `DBInspector` (wraps `SQLiteClient.Ping`) and `TelegramInspector` (wraps
  `TelegramBotClient.Ping`). Each adapter defines a thin interface (`dbPinger`,
  `botPinger`) so tests substitute fakes without touching production types.
- Acceptance Criteria: Package compiles standalone with `CGO_ENABLED=0`. All inspector
  and agent methods are covered by tests with parallel subtests. `Agent.CheckUp` returns
  within `3×timeout` even when one inspector blocks.
- Pitfalls & edge cases: Sequential execution — a blocking inspector exhausts the context,
  so subsequent inspectors also fail immediately. This is correct and bounded behaviour,
  not a bug.
- Complexity: Easy

### Task 2: DTO additions

- Description: Add `HealthCheckResponse` and `HealthServer` structs to
  `internal/dto/health.go` for the `/health/check` wire format.
- Acceptance Criteria: JSON tags match `{"status": bool, "server": {...}, "services": {...}}`.
- Complexity: Easy

### Task 3: Handler changes

- Description: In `internal/gateway/httpV1/handlers/handlers.go`:
  - Remove `CheckUP` from `rateService` interface.
  - Add `healthCheckAgent` interface (`CheckUp(ctx context.Context) (bool, map[string]string)`).
  - Add `healthAgent healthCheckAgent`, `serverVersion string`, `serverStart time.Time`
    fields to `Handler`; update `NewHandler` to accept them.
  - Remove `Healthz` handler. Add `Ping` (always 200) and `HealthCheck` (delegates to
    agent, 200/503) handlers.
- Acceptance Criteria: `NewHandler` compiles. `Ping` returns 200 `{"status":"ok"}` with no
  dependency calls. `HealthCheck` returns 200 when agent is healthy, 503 when not, with
  JSON body including `status`, `server`, and `services` keys.
- Pitfalls & edge cases: `healthAgent` nil-guard in `HealthCheck` returns 503 so existing
  tests that pass nil for the agent do not panic.
- Complexity: Easy

### Task 4: Route constants

- Description: Add `Ping = "/ping"` and `HealthCheck = "/health/check"` to
  `internal/gateway/httpV1/routes/routes.go`. Keep `Healthz = "/healthz"` — it now
  maps to the `Ping` handler.
- Complexity: Easy

### Task 5: Router and gateway wiring

- Description:
  - Add `healthCheckAgent` interface and new params to `NewRouter` in
    `internal/gateway/httpV1/router.go`; register `Ping` at both `/ping` and `/healthz`,
    register `HealthCheck` at `/health/check`.
  - Mirror the same param additions in `NewGateway` in `internal/gateway/gateway.go`.
- Complexity: Easy

### Task 6: `cmd/web/main.go` wiring

- Description: Import the inspector package, create `serviceStart := time.Now()` early in
  `main()`, instantiate `DBInspector` and `TelegramInspector`, wrap them in `NewAgent`,
  pass `BuildVersion`, `serviceStart`, and the agent to `NewGateway`.
- Complexity: Easy

### Task 7: Test updates

- Description: In `handlers_test.go`:
  - Replace `TestHealthz` with `TestPing` and `TestHealthCheck`.
  - Remove `CheckUP` from `mockRateService` (no longer in interface).
  - Add `mockHealthAgent` + `var _ healthCheckAgent = (*mockHealthAgent)(nil)`.
  - Update all `NewHandler` calls to pass the 3 new trailing params.
  Write `inspector_test.go` with parallel subtests for `Agent.CheckUp`,
  `DBInspector`, and `TelegramInspector`.
- Complexity: Medium

### Task 8: CI and doc updates

- Description:
  - `.github/workflows/release.yml`: change `HEALTH_URL` to `/health/check`.
  - `deploy/README.md`: update prose and curl example from `/healthz` to
    `/health/check`; mention `/ping` as the liveness probe.
  - `README.md`: update HTTP-routes mention.
  - `CLAUDE.md`: replace the `Healthz` route entry with `/ping` + `/health/check`.
- Complexity: Easy

## Execution Order

Tasks 1 → 2 → 3 → 4 → 5 → 6 → 7 → 8 (each depends on the previous).

## Risks

- Many `NewHandler` call sites in the test file need updating; a missed site causes a
  compile error. Replace-all on the common suffix patterns is safe.
- The `TelegramBotClient.Ping` ignores the context, so the Telegram inspector is not
  truly bounded. Noted; acceptable given the upstream library constraint.

## Trade-offs

- **`/healthz` kept as a `/ping` alias** rather than removed: zero breakage risk for any
  existing monitoring script or systemd `ExecStartPost` that already probes `/healthz`,
  while the deploy gate moves to `/health/check` for real readiness.
- **Sequential inspector loop** over concurrent goroutines: simpler, matches the skill's
  reference implementation, sufficient for two inspectors at 3 s total budget.
