# Task Breakdown

## Overview

The Telegram bot's `[📈 Latest updates]` callback (`cbLatest`) currently emits a
bullet list of subscriptions formatted ad-hoc in `service.TelegramApi.handleLatestUpdates`
(`internal/application/service/telegramapi.go:151`). The scheduled notifier
(`notification.RateCheckAgent.Run`, `internal/application/notification/ratecheckagent.go:97`)
emits a much nicer aligned table — same data, cross-source-deduped, timezone-aware.
This plan replaces the ad-hoc bullet list with the notifier's table renderer.

Per upstream decisions (do not re-litigate):

- Header is plain `FX rates` (no hashtag — on-demand is not a trigger fire).
- Delta basis is `currentValue - LatestNotifiedRate`, mirroring `ratecheckagent.go:137`.
  The existing first-fire guard (`Delta == 0 || Delta == CurrentPrice` → blank cell)
  in `buildRows` (`message.go:108`) applies as-is.
- The renderer in `internal/application/notification/message.go` is the source of
  truth — no duplication. We expose what we need from `notification` and call it.
- Cross-source dedup matches the daily message: same `(base, quote, kind)` across
  multiple sources for one user → one row, extremum-selected (BID→MAX, ASK→MIN).
- Timezone comes from `rate_user_profiles.timezone`, UTC fallback. Use the same
  resolution path as `RateCheckAgent.resolveUserTimezone` (`ratecheckagent.go:295`).
- All caller subscriptions are shown, not just those whose triggers would fire.

### Chosen design — single exported renderer in `notification`

I pick **option (b)** from the task brief: introduce a single new exported entry
point in `notification` that takes a snapshot input (subscriptions + their
currently-known rates + source metadata) and returns the rendered message parts.
The bot service builds the snapshot and calls this entry point; `RateCheckAgent.Run`
is refactored to use the same helper, sharing the dedup/render path.

Why (b) over (a):

- Option (a) (export `alert` + `buildAlertMessage`) exports two raw types and
  forces every caller to repeat the dedup-bucket fold. Two callers now, more
  later — duplicated dedup logic is exactly the failure mode this change is
  meant to prevent.
- Option (b) keeps `alert` / `dedupBucket` unexported and surfaces one well-named
  API. The cost is a slightly bigger refactor of `ratecheckagent.go`.
- The dedup logic in `ratecheckagent.go` lines 132–210 is the exact piece that
  must not be re-implemented by hand in the bot service. Centralising it removes
  the risk of the two surfaces drifting (e.g. the BID-MAX/ASK-MIN tie-break,
  the "retain even when collapsed" invariant).

The new entry point lives in a new file `internal/application/notification/render.go`
to keep `message.go` focused on the low-level table primitives and `ratecheckagent.go`
focused on the scheduled-run side effects.

### Snapshot input shape

The exported function takes a flat slice of per-subscription snapshots plus the
run timestamp and timezone:

```go
// SubscriptionSnapshot pairs one subscription with the source metadata and the
// currently-known rate value needed to build a row. Sources with no current
// rate must be omitted by the caller (we cannot fabricate one).
type SubscriptionSnapshot struct {
    Subscription domain.RateUserSubscription
    Source       domain.RateSource
    CurrentPrice float64
}

// BuildSubscriptionDigest renders all caller subscriptions into one or more
// Telegram HTML message parts using the same aligned-table format the scheduled
// notifier produces. Cross-source dedup on (BaseCurrency, QuoteCurrency, Kind)
// matches RateCheckAgent.Run: BID keeps MAX price, ASK keeps MIN price, ties
// resolved by first-seen. The header is the plain "FX rates" title — no
// hashtag — because the digest is on-demand, not a trigger fire.
//
// Returns an empty slice when snapshots is empty (the caller is responsible
// for the "no subscriptions yet" empty-state UX).
func BuildSubscriptionDigest(now time.Time, loc *time.Location, snapshots []SubscriptionSnapshot) ([]string, error)
```

This signature is deliberately narrower than what `RateCheckAgent.Run` needs:
the agent additionally has to decide which subscriptions are due, retain them,
and queue events. Those concerns stay in `ratecheckagent.go`. The shared piece
is "given a deduplicated bag of alerts, render them" — which `BuildSubscriptionDigest`
encapsulates.

### Where the on-demand "build alerts for this user" logic lives

In `notification/`, not in `service/telegramapi.go`. The bot service is a thin
caller: load subs, load each subscribed source's latest rate, build snapshots,
call `BuildSubscriptionDigest`, send the parts. All dedup/extremum logic lives
in `notification` next to the existing render pipeline.

## Assumptions

- The renderer entry point is added without breaking the existing notifier path:
  `RateCheckAgent.Run` is refactored to compose the same shared helper rather
  than calling `buildAlertMessage` directly. Acceptance tests in `message_test.go`
  continue to assert the same outputs (header, alignment, hashtags) because the
  scheduled-notifier code path adds hashtags via its own header builder before
  rendering — see Task 2 for the exact split.
- `subscriptionRepository` already exposes `ObtainRateUserSubscriptionsByUserID`,
  which is the existing read path for the bot. The plan keeps that interface
  and adds **separate** narrow read-only interfaces for the new dependencies
  (`rateValueRepository`, `rateSourceRepository`, `rateUserProfileRepository`)
  on `TelegramApi`, mirroring the pattern in `RateCheckAgent`.
- `RateSourceRepository.ObtainRateSourcesByNames` already exists
  (`internal/repository/ratesource.go:75`) and is the right batch lookup —
  one query for all the user's subscribed sources instead of N+1.
- The Telegram message size guard (`telegramMaxMessageLen = 2048`) in
  `message.go:16` is fine to inherit. A heavy user with dozens of subscriptions
  receiving multiple Telegram messages on `/Latest updates` is acceptable.
- We are NOT introducing on-demand triggers/hashtags. The bot digest never
  emits `#DELTA` / `#DAILY` / etc. — `BuildSubscriptionDigest` calls a header
  builder that hard-codes the "FX rates" title without inspecting triggers.

### Task 2: Refactor `RateCheckAgent.Run` to share `foldIntoBuckets`

- **Description:**
  In `internal/application/notification/ratecheckagent.go`, replace the inline
  bucket-folding block (lines ~136–210) with a call to the shared helper

  - Trigger accumulation (`b.triggers[sub.ConditionType] = collapseConditionValue(...)`).
  The shared helper handles only the dedup-bucket fold (extremum selection,
  price/delta coherence). The agent calls it once per due subscription with
  a per-subscription snapshot.

  produces the smallest diff in `ratecheckagent.go`:

    triggers into the bucket the helper just touched.

    must accumulate triggers in a parallel structure (extra plumbing). Reject
    this if it makes the agent significantly noisier.

  The retain-on-collapse invariant (`ratecheckagent.go:147-155`) stays in the
  agent — it is a side effect outside the renderer's concern.
- **Acceptance Criteria:**
  - [ ] `ratecheckagent.go` no longer contains an inline bucket map fold;
    the dedup-key formation and extremum selection are delegated to the
    helper from Task 1.
  - [ ] All existing tests in `internal/application/notification/` pass
    unchanged (no test should need to be rewritten — this is a refactor).
    Specifically, the cross-source dedup tests for the agent must still pass.
  - [ ] `RateCheckAgent.Run` still emits the same alert ordering, prices,
    deltas, source attribution (`SourceName`), and trigger lists as before
    on identical inputs. Verify by running `make test`.
  - [ ] No change to the agent's public surface (`NewRateCheckAgent` signature,
    repository interfaces).
- **Pitfalls:**
  - The agent's existing test fixtures depend on the exact extremum tie-break
    (strict `>` vs `>=`). The helper must use strict `>` / `<` to keep
    "first-seen wins" behaviour.
  - The agent attributes `alert.SourceName` to the source that contributed

  - Do NOT move trigger accumulation (`b.triggers[...] = collapseConditionValue(...)`)

    Mixing them would either pull `collapseConditionValue` semantics into the
    digest (wrong) or leave dead code in the helper (smell).

### Task 4: Rewrite `handleLatestUpdates` to use `BuildSubscriptionDigest`

- **Description:**
  Replace the body of `handleLatestUpdates` (`telegramapi.go:151`) with the
  following flow:
  1. Load subscriptions: `h.subRepo.ObtainRateUserSubscriptionsByUserID(...)` —
     existing call, unchanged.
  2. Empty subs → existing "You have no subscriptions yet." empty-state, unchanged.
  3. Collect the distinct set of source names from the subs.
  4. Batch-load source metadata: `h.sourceRepo.ObtainRateSourcesByNames(ctx, names)`.
     The result is a `map[string]domain.RateSource`.
  5. For each subscription, look up its source in the map.
     - If the source is missing (deleted, or never existed), skip the sub.
     - Optionally also skip when `Source.Active == false` — see "Edge cases"
       below for the decision.
  6. For each surviving subscription, load the most recent rate value via
     `h.rateValueRepo.ObtainLastNRateValuesBySourceName(ctx, sub.SourceName, 1)`.
     - If no rows, skip that subscription (no current price → no row).
     - To avoid N+1 in the common case, dedup the source-name lookups: many
       subs can share one source name. Build a `map[sourceName]float64`
       (current price) first, then iterate subs once.
  7. Build `[]notification.SubscriptionSnapshot` from the surviving entries.
  8. Resolve timezone via the helper in Task 6 (or inline if Task 6 is folded
     here — see Task 6 description).
  9. Call `notification.BuildSubscriptionDigest(time.Now().UTC(), loc, snapshots)`.
  10. Empty digest output (all subs filtered out by missing source / missing
      rate) → fall back to a clear "No rates available yet" message (NOT
      "no subscriptions" — the user has subs, they just have no data).
  11. Send each part with `h.sendOrEditWithKeyboard`. The first part edits
      the original message (when msgID > 0); subsequent parts are appended
      as new sends. The current `sendOrEditWithKeyboard` only handles one
      message — extend the flow to send N parts:
      - `msgID > 0`: edit with part[0], then send parts[1:] as new messages.
      - `msgID == 0`: send all parts as new messages.
  12. The back-button keyboard attaches only to the **last** part so the user
      can tap «Back» from the most recent bubble.
- **Acceptance Criteria:**
  - [ ] `handleLatestUpdates` no longer formats text inline; all rendering
    goes through `notification.BuildSubscriptionDigest`.
  - [ ] Output contains `<pre>` block with aligned columns (verified by
    substring assertion in tests — see Task 5).
  - [ ] Output header is `FX rates\n<timestamp>` — no `#DELTA` / `#DAILY` /
    other hashtag (verified by substring negative assertion).
  - [ ] When the user has no subscriptions, the existing
    "You have no subscriptions yet." empty-state is preserved byte-for-byte.
  - [ ] When the user has subscriptions but every one is filtered out
    (source missing or no rate value), a clear message like
    "No rate data available yet." is shown — distinct from the no-subs case.
  - [ ] When the repo returns an error, the existing
    "⚠️ Failed to load subscriptions." message is preserved.
  - [ ] When multiple parts are emitted (heavy user, >2KB), all parts reach
    the user; the «Back» keyboard is attached only to the last part.
  - [ ] Source lookups are batched (one `ObtainRateSourcesByNames` call,
    not N).
  - [ ] Rate-value lookups are deduplicated by source name (no N+1 when
    the user has multiple subs on the same source).
- **Pitfalls:**
  - The current handler always returns exactly one message via
    `sendOrEditWithKeyboard`. Splitting into N parts changes the message
    shape. Tests that assert "exactly one html message" must be rewritten
    to assert "at least one" / specific part count.
  - When editing the original message (msgID > 0) AND emitting >1 parts,
    you cannot edit one msg into multiple. Strategy: edit msgID with part[0],
    then `SendHTMLMessage` parts[1:]. The mock client's existing
    `editedTexts` and `htmlMessages` slices accommodate this exactly.
  - Don't run rate-value queries inside the sub loop; build the
    `map[sourceName]float64` first by iterating distinct source names.
  - **Decision needed for "source inactive" filtering**: Recommend including
    inactive sources whose last rate is still recent enough that
    `ObtainLastNRateValuesBySourceName` returns a row. Rationale: from the
    user's perspective an inactive source is still a subscribed pair until
    they delete the sub; hiding it surprises them. Reject only when the
    source row is missing entirely (deleted). Document this in code.
- **Complexity:** Medium

### Task 6: Add timezone resolution helper to the bot service

  "dedup only genuine invariants; let coincidentally-similar code diverge",
  these are coincidentally similar — they happen to share semantics today,

  `log.Printf`). Extracting a shared helper now would couple the two
  artificially. Revisit if a third caller appears.
- **Acceptance Criteria:**
  - [ ] `resolveUserTimezone` exists on `*TelegramApi` and is called from
    `handleLatestUpdates`.
  - [ ] When `h.profileRepo` is nil, returns nil (no panic, no log spam).
  - [ ] When the profile repo returns `ErrNotFound`, returns nil without

  - [ ] Unknown timezone strings produce one `log.Printf` line and return nil.

## Risks

- **Cross-source dedup drift.** The refactor in Task 2 moves a 70-line block
  out of `ratecheckagent.go`. If `foldIntoBuckets` is not byte-identical in
  behaviour, the scheduled notifier breaks silently — wrong prices in user
  notifications. Mitigation: keep the agent's existing dedup tests passing
  unchanged (acceptance criterion for Task 2). Use strict `>` / `<` for
  extremum tie-break.
- **N+1 query risk in the digest.** A user with 20 subs on 20 sources runs
  20 rate-value queries. Task 4 mitigates by deduplicating by source name
  first, but heavy users with many distinct sources still pay N round-trips
  to `ObtainLastNRateValuesBySourceName`. Acceptable for v1 — most users
  have <5 distinct sources — but worth a follow-up to add a batch
  `ObtainLatestRatesBySourceNames(names []string)` if telemetry shows
  this is hot.
- **Multi-part edit semantics.** The current bot UX edits the original
  callback message in place. With >1 parts, only the first part replaces
  the bubble; subsequent parts appear as new bubbles. This is a visible
  UX change — confirm it is acceptable. Alternative: send all parts as
  new messages (never edit) when parts > 1 — debatable, both are defensible.
  Recommend "edit first, send rest" for minimum surprise.
- **Backward compatibility — none required.** The bot's `[📈 Latest updates]`
  output is a presentation layer; no caller programmatically parses it.
  The only persistent contract is the database (unchanged) and the renderer's
  HTML output (which Telegram clients render — text changes are fine).

## Trade-offs

- **Decision 1: Option (b) over option (a) for renderer exposure.** Picked
  a single high-level entry point (`BuildSubscriptionDigest`) over exposing
  raw `alert` / `buildAlertMessage`. Trade-off: more refactor work in
  Task 2; net win: dedup logic lives in one place, only one new exported
  symbol to maintain.
- **Decision 2: Three new narrow interfaces on `TelegramApi`** instead of
  passing a "facade" with all three methods. Matches the existing pattern
  in `RateCheckAgent` and keeps mocks small. Cost: three mock structs in
  tests instead of one. Acceptable — the project already pays this cost
  consistently.
- **Decision 3: Do not factor out `resolveUserTimezone` into a shared
  helper.** The agent's version writes to `a.logger`, the bot's writes to
  `log.Printf`. Different logging surfaces, twenty-line bodies, two
  callers. Per the project memory rule on dedup, accept the duplication
  until a third caller appears.
- **Decision 4: Skip subscriptions with missing source or missing current
  rate; do not synthesise a placeholder row.** Showing
  "USD/KZT (no data yet)" leaks dead sources to the user; better to omit
  silently and surface "No rate data available yet." only when *every*
  subscription is filtered out.
- **Decision 5: Send multi-part output as edit(part[0]) + send(parts[1:]).**
  Preserves the in-place edit UX for the common case (one part) while
  still showing all data for heavy users. Alternative considered: always
  send new messages on multi-part — rejected because it makes the common
  one-part case worse for no gain.
