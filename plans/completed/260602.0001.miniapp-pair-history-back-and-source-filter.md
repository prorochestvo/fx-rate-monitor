# Task Breakdown

## Overview

Two UX changes for the Telegram Mini App per-pair history view:

1. **Remove the inline `← Back` button** inside the history view and overload
   the modal header X (`#me-pair-modal-close`) to act as "back to detail" while
   `HistoryOpen=true`, and as "close modal" otherwise. This mirrors the
   existing Escape-cascade behavior. Backdrop click stays as a fast exit
   (always closes the whole modal).
2. **Add a source filter** to the history view. The user can scope the listed
   events to a single source (e.g. "Halyk only"). The filter is server-side:
   `GET /api/me/rates/history` gains an optional `source_name` query parameter,
   and `Total` reflects the filtered count so pagination math stays correct.

Both changes are layered:

- Change 1 lives entirely in WASM (UI + main.go delegated click handler).
- Change 2 touches: repository SQL, application service signature, HTTP
  handler, apiclient URL builder + method, WASM page state, WASM UI renderer.

Change 1 is the smallest and ships first because it requires no API change
and is independent of Change 2; if anything goes wrong with Change 2 the X
behavior is already in place.

## Assumptions

- The header X behavior change is purely additive: backdrop click is left
  untouched as a fast escape hatch. User confirmed this.
- Reopening the modal (closing then opening a different pair) resets the
  source filter. `OpenHistory` zeroes `SelectedSource`. Cross-pair filter
  persistence is not a goal — different pairs have different source sets and
  preserving the filter would lead to confusing empty pages.
- Unknown `source_name` (typo, deleted source, source the user has no
  subscription for) returns a 200 with empty `Items` and `Total=0`. We do not
  return 400 because (a) the existing "no history yet" empty-state UX already
  handles this gracefully and (b) the dropdown is server-sourced so a 400
  would indicate an internal bug, not user error — surfacing it as 400 only
  helps an attacker probe source names.
- The dropdown values are the canonical `rate_sources.name` (the same string
  the server stores and the WASM client passes through unchanged). Display
  uses `SourceTitle` from rows we already have.
- `source_name` is not PII (it's a public bank label like "halyk", "kkb") so
  it is fine to include in access logs and Referer headers. No special
  redaction needed.
- The existing in-flight stale-fetch guard (`targetPair` snapshot in
  `LoadHistory`, `cmd/wasm/application/me_subscriptions.go:168`) must be
  extended to cover `SelectedSource` too. Without this, a slow first fetch
  for filter A landing after the user has switched to filter B will overwrite
  state with rows that don't match the now-current selection.

### Source-set decision (Option A vs Option B)

**Picked: Option A** — derive the dropdown from the current page's
`HistoryItems`, plus the union of any sources the user has seen on previously
loaded pages of the *same* `OpenPair` session.

Rationale: Option B (add `sources` array to the response or a
subscription-backed list) is cleaner semantically but adds either a query or a
non-trivial join on every history request, and the dropdown is a quality-of-
life filter, not a correctness primitive. The "previously loaded pages"
accumulation costs one `map[string]string` in state and gives the user a
filter UX that grows monotonically as they paginate, which is good enough.
If users complain about "I know source X has data for this pair but it's not
in the dropdown until I find it", upgrade to Option B as a follow-up.

If the engineer hits a concrete edge case that makes Option A worse than
Option B (e.g. discovers the union-accumulation cannot survive
`LoadSparklineChart` clearing modal state), escalate before writing more code
— do not silently flip to Option B without confirmation.

### UI element — chip-list vs select

**Picked: chip-list**. The existing modal already uses inline chips for the
"✕ All" pattern in subscription condition badges (search `me-pair-modal-*`
for the precedent), and a chip row gives users an at-a-glance view of which
source is selected without requiring a click to open a dropdown. With ≤ 10
sources per pair in realistic data this fits inline; if a pair ever has > 10
sources, the chip row wraps to two lines — still better than a `<select>`
because the active selection stays visible. The selected chip carries an
extra class so it can be styled distinct (a single CSS rule already
established in the modal).

A leading "All" chip (id `me-pair-history-source-all`) means
`SelectedSource = ""`. Each source chip carries
`data-source="<canonical-name>"` so a single delegated click handler in
`cmd/wasm/main.go` can read the dataset and dispatch.

## Risks

- **Stale-fetch race after filter change (highest)**: documented in
  Task 3 pitfalls and tested in Task 4. If the `targetSource` snapshot
  is missed during implementation, the UI will show wrong-source rows
  after a fast filter change. Mitigated by an explicit race-condition
  test that fails without the guard.
- **Single-source dropdown UX**: with Option A's "derive sources from
  loaded items", users on a slow connection see no chips until the
  first response lands. Acceptable — the empty-state copy ("Loading…")
  covers it.
- **`KnownSources` shrinks/stales when subscriptions change in another
  Telegram tab**: the user keeps the old chip set until they close +
  reopen the modal. Acceptable; documented in the field godoc.
- **Backward compatibility**: the `source_name` query param is
  *optional* on the server. Old WASM bundles continue to work without
  passing it. The DTO (`MeHistoryResponse`) is unchanged. No migration.
- **Interface change blast radius**: the
  `meChartService.ObtainMeHistory` signature change forces every test
  mock to update. `grep -rn "ObtainMeHistory(" ./internal ./cmd` before
  implementation to make sure no caller is missed. Compile errors at
  build time will catch any miss; this is annoying, not dangerous.
- **The history view's existing godoc on `RenderPairHistory` lists the
  back button as layout bullet 1**. Forgetting to update the godoc leaves
  a load-bearing lie. Acceptance criteria flag this explicitly.
- **Re-render on filter change without server response**: if the user
  clicks a chip and nothing visibly changes for 200ms, they may
  double-click. The implementation should toggle `HistoryLoading` true
  immediately on `SetHistorySource` so the skeleton renders, not after
  the network round-trip completes.
- **No DOM-level test coverage for `main.go`**: the WASM main is not in
  the host `make test` run. The application-layer tests cover state
  transitions, but the wiring between `#me-pair-history-source-all`
  clicks and `SetHistorySource` is exercised only manually / in
  production. Document this gap in the task; do not pretend
  otherwise by inventing a fake DOM.

## Trade-offs

- **Option A (current page sources) vs Option B (subscription-backed
  sources) for the chip list**: chose A for zero added API surface. The
  cost is "filter UX feels incomplete on the first page until you
  paginate". If users hit this in the field, B is a clean follow-up.
- **Chip-list vs `<select>`**: chose chips for at-a-glance state
  visibility and consistency with the existing modal's chip pattern.
  Cost: chips wrap to multiple lines when sources > ~6 on a narrow
  viewport. Acceptable for the realistic source count per pair.
- **Service-level pre-filter vs SQL-level WHERE on `source_name`**:
  chose service-level for zero repository signature churn and clearer
  test layering. Cost: at very high subscription counts the repository
  still issues an IN clause covering all matching keys including the
  filtered-out ones, but only after the service has narrowed
  `matchingKeys`. So the SQL operates on the post-filter slice — there
  is no extra database cost. The "cost" is purely conceptual: someone
  reading the SQL won't see the filter. Document the indirection in the
  service godoc.
- **`SelectedSource` as `string` (`""` = no filter) vs `*string`**:
  chose `string` to match the project's existing convention for absent
  query strings (`q string` in `MeSubscriptions`). Cost: cannot
  distinguish "explicitly cleared" from "never set"; we don't need to.
- **Header X overload vs separate icon for "back"**: chose overload to
  reuse the established mental model from the existing Escape cascade.
  Cost: discoverability — first-time users may expect X to close the
  whole modal even from history. Backdrop click remains the always-on
  "close everything" escape hatch, so this cost is bounded.
- **Storing `SelectedSource` in `MeSubscriptionsState` vs in a separate
  `historyFilterState`**: chose the existing state struct to keep
  re-render logic in one place. Cost: the state struct keeps growing.
  Acceptable until the struct hits 15+ fields or we add a second view
  that needs filtering — then split.
- **`OpenPairModal` defensive zeroing of filter fields**: redundant with
  `ClosePairModal`'s zeroing during a clean close-then-open cycle, but
  cheap insurance against any future code path that opens a new pair
  without closing the previous one. Trade-off: a few lines of "looks
  redundant" code in exchange for an invariant that's easier to reason
  about.
