# Task Breakdown

## Overview

Long-period sparklines (90d / 180d / 360d) currently render as a flat
horizontal line on the right edge of every chart because the system has
roughly seven days of collected data. The fixed `bucketCount = 12` in
`internal/application/chart/service.go` divides the requested window
(e.g. 360 days) into 12 buckets ≈ 30 days wide; with only the last week
populated, all real samples crowd into the last 1–2 buckets and the
polyline degenerates into a horizontal segment.

The user has chosen Option A: keep the period chips honest about what
the user *requested*, but cap the *rendered* X-axis to the actual data
coverage and surface that truth in the chart header. The fix is
per-series (each pair row independently caps to its own first sample),
because each row already has its own sparkline and the rows do not
share an X-axis visually.

Two coordinated edits:

1. **Service layer** (`internal/application/chart/service.go`): inside
   `buildSeriesRow`, compute an effective `since` as
   `max(requestedSince, firstSampleTimestamp)`. Bucket over the
   effective window, not the requested one. When coverage is already
   ≥ requested period, behaviour is identical to today.
2. **Wire contract** (`internal/dto/me_rates.go`): each
   `MeChartSeries` gains an `EffectiveDays` field carrying the integer
   number of days actually covered by the series. Zero when the series
   is sparse / no-data (header keeps the requested window label in that
   case). Field is per-series because BID and ASK for the same pair
   can have different first-sample timestamps.
3. **WASM client** (`cmd/wasm/ui/chart.go`): when the row's max
   `EffectiveDays` across its series is strictly less than the
   requested period, the header subtitle changes from
   `"… · last 360 days"` to `"… · last 7 days (max available)"` using
   the largest `EffectiveDays` of any series in the chart. When every
   series is sparse / no-data the subtitle keeps the requested period
   so the header does not contradict the chip the user just clicked.

The period chips themselves are untouched — they remain the user's
*request*. We just cap how we draw the X-axis and tell the truth in the
header.

## Assumptions

- `bucketCount = 12` stays a constant. The intent is to spread 12
  buckets across whatever effective window we have, not to vary the
  bucket count by period.
- `EffectiveDays` is a per-series field, not per-pair or per-chart. The
  sparkline-list layout already gives each pair its own X-axis (each
  row has its own SVG), so two pairs in the same response can have
  different effective windows and the UI will not look inconsistent.
- For the header subtitle, the WASM client takes the *maximum*
  `EffectiveDays` across all series of all visible pairs as the chart's
  reported coverage. Header text reflects the longest covered window
  rather than the shortest, on the basis that the user is most likely
  to ask "is my chart actually showing 360 days?" — answering with the
  best-covered series is honest without being misleadingly pessimistic.
- "Effective coverage" is measured from `firstSampleTimestamp` to
  `now`, rounded down to whole days. A series with one sample
  (sparse=true) gets `EffectiveDays = 0` because there is no covered
  *range* to report.
- The handler does **not** need its own validation logic for the new
  field — it is a pass-through in the DTO marshal block.
- No `PublicError` paths are introduced. None of these failures are
  user-actionable; if data coverage is short, the chart honestly says
  so. No 400s, no error toast.
- Existing tests at `internal/application/chart/service_test.go` and
  `cmd/wasm/ui/chart_test.go` follow the "one `Test*` per method,
  scenarios as subtests" rule. New scenarios go as additional `t.Run`
  blocks inside the existing top-level test functions.

### Task 2: Compute effective `since` and `EffectiveDays` in `buildSeriesRow`

- **Description:** This is the load-bearing change. In
  `internal/application/chart/service.go:435` (`buildSeriesRow`),
  compute the effective lower bound of the window as
  `max(requestedSince, firstSampleTimestamp)`, then use it for both
  bucketing and `EffectiveDays`. Specifically:
  1. After `deduped := dedupeByTimestamp(vals)` and the `sparse`
      flag (line 451–455), determine `effectiveSince`:
      - If `len(deduped) == 0`: leave `effectiveSince = since`,
        `EffectiveDays = 0`, return as before (no-data path).
      - Else: `effectiveSince = since` if `deduped[0].Timestamp.Before(since)`
        (defensive — should not happen since the repository query filters
        by since, but keep the math safe), else
        `effectiveSince = deduped[0].Timestamp`.
  2. Recompute `window := now.Sub(effectiveSince)` and
      `step := window / time.Duration(bucketCount)`. Replace every
      remaining use of `since` in the bucketing loops with
      `effectiveSince`. The bucket-time computation
      `bucketTime := since.Add(time.Duration(i+1) * step)` becomes
      `effectiveSince.Add(...)`.
  3. Set `sr.EffectiveDays = int(window / (24 * time.Hour))` when at
      least one non-sparse point was produced. When `sparse == true`
      (one sample) or `len(points) == 0` (no data), leave
      `EffectiveDays = 0`.
  4. When `effectiveSince == since` (coverage ≥ requested period), the
      math is bit-identical to today. This is the regression-guard
      property that Task 4's tests will assert.
- **Acceptance Criteria:**
   - [ ] When `firstSampleTimestamp > since`, bucketing uses
         `effectiveSince = firstSampleTimestamp`. Verified by a test that
         constructs samples covering only the last N days inside a longer
         requested window and asserts the first bucket timestamp equals
         `effectiveSince.Add(step)` not `since.Add(step)`.
   - [ ] When `firstSampleTimestamp <= since` (coverage ≥ requested
         period), `effectiveSince == since` and every existing point /
         delta / bucket assertion in `service_test.go` still passes
         unchanged.
   - [ ] `SeriesRow.EffectiveDays` reflects the rounded-down day count
         of the effective window when at least two points exist.
   - [ ] `SeriesRow.EffectiveDays == 0` when `Sparse == true` or
         `len(Points) == 0`.
   - [ ] No new exported helpers — keep the logic inside
         `buildSeriesRow` since it is the only caller.
- **Pitfalls:**
   - Integer division: `time.Duration` divided by `24 * time.Hour`
     truncates toward zero, which is what we want (a 6.5-day window
     becomes 6 days, not 7). Resist the urge to round up.
   - `effectiveSince` must be propagated to the bucket-time loop. If
     you only change `step` but leave `since.Add(time.Duration(i+1) * step)`
     in the bucket-time math, the `Timestamp` on each `SparkPoint` will
     be wrong (it would point inside the empty pre-data range) and the
     existing "leading empty buckets are omitted" guarantee silently
     breaks.
   - The leading-empty-bucket logic at the bottom of the function
     becomes a no-op when `effectiveSince` is the first sample (the
     first bucket is always filled now). Leave the logic in place; it
     is a defensive belt against rounding edge cases.
   - Do **not** introduce a `nowFn` field on the struct (already a
     parameter). Do **not** discard errors. Do **not** add section
     divider comments.
- **Complexity:** Medium
- **Code Example:**
   ```go
   // After: sparse := len(deduped) < 2
   effectiveSince := since
   if len(deduped) > 0 && deduped[0].Timestamp.After(since) {
       effectiveSince = deduped[0].Timestamp
   }

   window := now.Sub(effectiveSince)
   if window <= 0 {
       // Defensive: should not happen since values are filtered by
       // ObtainValuesForPairsSince(ctx, _, since) and now > since.
       window = time.Hour
   }
   step := window / time.Duration(bucketCount)

   // ... bucket fill loop uses effectiveSince ...

   for i := 0; i < bucketCount; i++ {
       bucketTime := effectiveSince.Add(time.Duration(i+1) * step)
       // ... unchanged ...
   }

   // After computing Latest / DeltaPct:
   if !sparse && len(points) > 0 {
       sr.EffectiveDays = int(window / (24 * time.Hour))
   }
   ```

## Risks

1. **DTO `omitempty` drift.** `EffectiveDays` is an `int` without
   `omitempty`, so it is always present on the wire (zero when
   sparse). If someone later adds `omitempty`, the WASM client will
   silently read zero for a fully-covered series and start writing
   `"(max available)"` headers on charts where coverage is actually
   complete. Document this in the godoc.
2. **Per-series vs per-chart aggregation mismatch.** The WASM client
   picks the *max* per-series coverage for the subtitle. If a future
   PR groups series differently (e.g. one row showing 360d-covered
   USD/KZT next to 7d-covered XAU/USD), the single subtitle becomes
   ambiguous. The current sparkline-list layout sidesteps this because
   each row already has its own visual story; the header just gives a
   summary.
3. **Bucket-time off-by-one when `effectiveSince` is mid-day.** The
   existing right-edge bucket-time formula
   `since.Add(time.Duration(i+1) * step)` shifts to `effectiveSince`.
   For a first-sample timestamp like `2026-06-04 13:42:17`, the bucket
   times will no longer be midnight-aligned. The UI does not display
   these timestamps directly (they feed `SparkPoint.Timestamp`, which
   the renderer uses only for the X-axis position via `polyline`
   coords, not for display), so this is cosmetic at most. Worth a
   quick check in Task 4.
4. **Migration risk: none.** No schema changes. No new env vars. No
   new external dependencies. The DTO change is purely additive — old
   WASM clients reading a server with the new field ignore it; new
   WASM clients reading an old server see zero across the board and
   simply never render `"(max available)"`. Roll forward without
   coordination.

## Trade-offs

1. **Per-series `EffectiveDays` over per-chart.** Considered exposing a
   chart-level `EffectiveDays` field on `MeChartResponse` /
   `PublicChartResponse` instead. Rejected: each pair row is rendered
   independently, and a chart-level cap forces every row to align to
   the shortest-covered pair's window. That would make a USD/KZT row
   with 30 days of data look as flat as an XAU/USD row with 7 days of
   data, which is the opposite of what Option A asks for. Per-series
   keeps each row honest at the cost of one extra int per series on
   the wire (~8 bytes per series, negligible).
2. **Cap to first-sample timestamp, not to a discrete day boundary.**
   Considered snapping `effectiveSince` down to the start of the
   first-sample day (`time.Truncate(24*time.Hour)`). Rejected: the
   resulting day-aligned window makes bucket math nicer but introduces
   ghost leading empty buckets when the first sample is e.g. at
   13:42 — recreating the original problem at smaller scale. Capping
   at the actual first sample lets the polyline start at the left
   edge of the chart, which is the whole point of Option A.
3. **Keep `bucketCount = 12` fixed.** Considered varying bucket count
   by effective period (e.g. 12 buckets for 7d, 30 for 30d). Rejected:
   per the existing service.go comment, bucket count is a visual
   density choice independent of horizon. Changing it across periods
   would degrade short-horizon charts and is out of scope for this
   plan.
4. **Header subtitle uses the *max* per-series coverage, not the
   min.** Considered using the min (most conservative) or arithmetic
   mean. Rejected: the user is likely to ask "is my 360d chart real?"
   and the most honest answer is "the best-covered series shows X
   days; the rest have less". Reporting the min would understate
   coverage on mixed-quality data; reporting the mean is a fiction
   nobody can verify. Max keeps the subtitle interpretable: it is the
   ceiling of what the user is seeing, no more.
5. **Wire-level `Window` field stays untouched.** Considered changing
   `Window` to e.g. `"7 days (capped from 360)"`. Rejected: the wire
   field describes the *request*, which is what backend consumers
   (admin dashboard, scripts) actually care about. The "(max
   available)" disclaimer is a UI concern; only the WASM client should
   compute it from `EffectiveDays`. This keeps the JSON contract
   stable for any future consumer that does not need the UI overlay.

## Implementation notes (post-review)

The five-lens review surfaced one Major and several Minor / Nit findings.
Resolved findings are listed in commit `1483f26`. The following were
consciously deferred:

- **i18n of the subtitle**. The new `"last N days (max available)"` string
  is hardcoded English, joining the date label and the header that are
  already English-only. The project has no translation layer yet; a
  proper fix requires that layer first. Re-evaluate when the first
  localisation pass lands.
- **Subtitle wording**. `"(max available)"` was chosen in the plan and
  shipped verbatim. UX-wise it could read as a plan-tier limit rather
  than a data-coverage gap. Alternatives considered for a future tweak:
  `"(all data)"`, `"(no older data)"`. Decision punted to the next time
  the subtitle is touched.
- **Operator log signal when capping fires**. No log line or metric is
  emitted when `EffectiveDays < requestedPeriod`. With one operator and
  one instance the blind spot is acceptable. Revisit when source count
  or operator count grows. A `DEBUG`-level line at the handler
  marshalling site would be the minimum-cost fix.

Resolved in the fix commit (`1483f26`):

- Removed the now-unreachable `hasPrev` leading-empty-bucket guard in
  `buildSeriesRow`; documented the new invariant that bucket 0 is always
  populated when data exists.
- Clamped `EffectiveDays` to `>= 1` on non-sparse series so the zero
  value remains an unambiguous sentinel for sparse / no-data.
- Renamed local `max` accumulator in `effectiveDaysForChart` to `best`
  (avoids shadowing the Go 1.21+ builtin); deleted the stale
  `max(a, b int) int` helper and outdated comment in
  `cmd/wasm/ui/chart_test.go`.
- Wrapped the subtitle `dateLabel` in `dom.Escape` for consistency with
  the rest of the WASM HTML builder.
