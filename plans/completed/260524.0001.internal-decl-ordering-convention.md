# Task Breakdown

## Overview

Apply one fixed top-level declaration-ordering convention to every **non-test**
`*.go` file under `./internal`. This is a pure mechanical reordering of top-level
declarations within each file. **Zero behavior change**: no declaration body is
added, removed, renamed, or edited — only moved. After each file, `gofmt` stays
clean, the package still builds, and `make test` is green at the end.

The convention, applied per file:

1. Exported `const`/`var` plus the `New{Object}` constructor(s) — what a caller
   needs to create and use the object.
2. The object's struct definition.
3. The object's methods (prefer alphabetical; logical grouping may override).
4. Unexported `const`/`var`.
5. Auxiliary/helper structs (unexported support types) — between the unexported
   vars/consts and the unexported methods.
6. Unexported methods/functions (prefer alphabetical; logical grouping may
   override).

Multiple structs in one file: same layout, primary ("main") struct first, then
its satellites. Files with no object (free functions + a config/data type):
same spirit — exported type(s) and function(s) on top, then unexported
consts/vars, then aux structs, then unexported helpers.

This is reader-ergonomics work: public surface first, internals last. It does not
change the public API surface itself — only the order it appears in source.

## Assumptions

- The convention is a **target shape**, not a forced rewrite. A file that already
  conforms is a **no-op** and must be left **byte-identical** — no churn for
  nothing. Reorder only files that actually deviate.
- "Movement-only" means: the multiset of top-level declarations (and their bodies,
  comments, and doc comments) before and after a reorder is identical; only their
  order in the file changed. Imports are not reordered beyond what `gofmt`/`goimports`
  already enforce (they are already sorted; leave them alone).
- "Prefer alphabetical" for methods/unexported funcs is a tie-breaker, **not** a
  mandate. Where the existing file groups methods logically (e.g. `CheckUP` then
  the CRUD cluster, or a `Forecast`/helper pairing), that grouping may be kept.
  Do **not** alphabetize at the cost of separating a method from a tightly-coupled
  unexported helper that reads better adjacent — but default to alphabetical when
  there's no such pull.
- The "primary struct" in a multi-struct file is the one the file is named after /
  the one whose constructor and methods dominate the file. When ambiguous, keep the
  existing first-struct as primary rather than guessing.
- Build/test must be run **outside the sandbox** (or with the build cache
  reachable). During planning, sandboxed `go build ./...` failed only with
  `could not import ...: no such file or directory` cache-miss errors from
  vendored deps (gonum, chromedp, modernc/sqlite) — an environment artifact, not
  a code problem. `gofmt -l` works fine sandboxed and is the primary per-file
  gate.
- No file currently mixes scope concerns that would make "which object owns this
  method" ambiguous; the codebase is one-object-or-a-few-cohesive-objects per file.

## The convention, stated precisely (engineer reference)

Top-to-bottom within each file, **after** the fixed prologue:

**Fixed prologue (NEVER reordered, always in this order at the very top):**
0a. `//go:build` / `// +build` constraint line(s) — above the package clause,
    followed by a blank line. (None under `internal` today; preserve if present.)
0b. `// Package <name> ...` doc comment, glued to —
0c. the `package <name>` clause.
0d. the `import (...)` block. Leave import ordering exactly as gofmt has it.

**Body, in this order:**
0e. **Compile-time interface assertion(s)** (`var _ Iface = (*T)(nil)`) — placed at
   the **very top of the body, immediately after the `import` block**, above the
   constructor and struct. The assertion is the interface↔implementation contract
   and must be the first thing a reader sees, surfacing the file's interface
   dependency up front (user decision 2026-05-23). If a file asserts several types,
   group all assertions together here. The directive/explanatory comment above an
   assertion (e.g. `// Ensure interface is satisfied at compile time.`) travels with
   it. NOTE: this reverses the repo's prior habit of parking assertions at EOF.
1. Exported `const`/`var` declarations + `New{Object}` constructor(s) for the
   primary object.
2. Primary object's struct definition.
2b. **Unexported dependency/collaborator interfaces** (consumer-defined contracts the
   struct depends on — repository/client/extractor interfaces that are the type of a
   constructor param or struct field, e.g. `db`, `rateValueRepository`,
   `telegramClient`, `sqlAction`) — immediately after the struct (and its constructor),
   BEFORE the methods. Added mid-review (user decision 2026-05-24) after Lens E flagged
   `rateextractor.go`'s dependency interface buried ~80 lines from its struct. Same
   reasoning as the assertion rule: a dependency interface is a "what this object
   depends on" contract and belongs visible next to its type, not in the aux/unexported
   zone at the bottom. Multiple such interfaces stay a contiguous block in their
   existing relative order; in multi-struct files, place each after the struct that
   consumes it.
3. Primary object's methods (exported first, then unexported methods on the same
   receiver), prefer alphabetical within each visibility group.
4. For each additional ("satellite") struct, repeat 1–3 in declaration-significance
   order (most important satellite first).
5. Unexported package-level `const`/`var`.
6. Auxiliary/helper structs (unexported support types with no/minimal methods).
7. Unexported free functions / helpers (and `func init()`), prefer alphabetical.

**`//go:embed`** (none under internal today): the directive comment line must stay
**immediately above** its target `var`, with nothing inserted between them. Move
the directive+var as one inseparable unit.

**Const/var blocks** (`const (...)`, `var (...)`): move as an **atomic unit**.
Never split a block or reorder members inside it. Repository table/column-name
const blocks (`rateSourceTableName`, `rateValueTableName`, etc.) are exactly this
case — relocate the whole block, never crack it open.

**Comments travel with their declaration.** A doc comment directly above a decl,
and any preceding WHY/explanatory comment block attached to it, move **with** the
declaration. Never orphan a comment, never drop one, never rewrite one (CLAUDE.md:
"Preserve existing WHY-comments verbatim"). A comment separated from the next decl
by a blank line is a floating comment — rare; if encountered, leave it where it is
and flag it rather than guess what it attaches to.

## Tasks

The work is one repeated mechanical procedure applied across 22 package
directories. Tasks 2–23 are the per-directory batches; they are independent of
each other and could be parallelized, but are ordered leaf-package-first so an
accidental build break surfaces in the smallest blast radius (see Execution
Order). Task 1 is setup; Task 24 is the final gate.

### Task 1: Establish baseline and the movement-only verification harness
- **Description:** Before touching any file, capture the green baseline and define
  the per-file "movement-only" check the engineer will use throughout.
  - Confirm `gofmt -l ./internal` is empty (clean). Record it.
  - Run `CGO_ENABLED=0 go build ./...` and `make test` **outside the sandbox** (or
    with build cache reachable) to confirm a green starting tree. If `make test`
    is red **before** any change, STOP and report — do not start a reorder on a
    red tree (per pipeline rule, hand red trees to `gocode-testdoctor` first).
  - Define the per-file movement-only proof. Recommended cheap check: extract the
    sorted set of top-level declaration signatures before and after, and diff
    them — they must be identical. Example extractor (run per file):
    ```bash
    # crude but effective: top-level decl headers, sorted
    grep -nE '^(func |type |const |var )' ./internal/<pkg>/<file>.go \
      | sed -E 's/^[0-9]+://' | sort > /tmp/before_<file>.txt
    # ...reorder...
    grep -nE '^(func |type |const |var )' ./internal/<pkg>/<file>.go \
      | sed -E 's/^[0-9]+://' | sort > /tmp/after_<file>.txt
    diff /tmp/before_<file>.txt /tmp/after_<file>.txt   # MUST be empty
    ```
    (Put scratch files under `./tmp/`, never repo root — CLAUDE.md.) The sorted-set
    diff being empty proves no decl was added/removed/renamed; `gofmt` clean +
    `go vet` + tests prove bodies are intact.
- **Acceptance Criteria:**
  - [ ] `gofmt -l ./internal` confirmed empty and recorded.
  - [ ] `make test` confirmed green on the untouched tree (outside sandbox).
  - [ ] The movement-only check command is written down and reused verbatim per file.
  - [ ] Scratch artifacts live under `./tmp/`, not the repo root.
- **Pitfalls:** Starting on a red tree masks whether a later failure is yours.
  The sandbox build-cache-miss errors are NOT a red tree — distinguish
  `could not import ...: no such file` (environment) from real compile errors.
- **Complexity:** Easy

## Risks

- **Silent body edit during a large cut/paste** (highest risk in
  `service/`, `repository/`, `openai_compatible.go`). A reordered file that still
  compiles can still have a subtly altered body. Mitigation: the sorted-decl-set
  diff per file + manual spot-check of high-churn files + `make test`.
- **Orphaned or dropped comment.** A WHY-comment or doc comment left behind when
  its decl moves. Mitigation: treat comment+decl as one unit; the spot-check
  looks specifically for floating/duplicated comments. `errors.go`'s `init`
  comment and `composite.go`'s assertion comment are the named at-risk cases.
- **Cracking a `const`/`var` block** (repository table/column names, PRAGMA/DSN
  strings, prompt templates, scraped-label tokens). Splitting or reflowing breaks
  schema-rename safety, DB connectivity, LLM prompts, or scrape linkage.
  Mitigation: move blocks atomically; data-token rule from CLAUDE.md; never edit
  a non-English literal.
- **Accidentally touching a `*_test.go` file** (out of scope). Mitigation:
  Task 24 asserts zero test files in the diff.
- **Churn-for-nothing on already-conforming files.** Re-saving a conforming file
  through gofmt could still no-op, but reordering it pointlessly inflates the diff
  and risks introducing a body edit for zero benefit. Mitigation: identify
  conforming files first and leave them byte-identical (verify via absence from
  `git diff`).
- **Reviewer/merge cost.** A 70-file reorder is a large, low-information diff.
  Mitigation: per-directory commits (or one commit per batch) so review can
  proceed package-by-package, and a commit message that states "movement-only,
  no behavior change" with the verification evidence.
- **Build cache / sandbox false negatives.** Sandboxed `go build` reports
  spurious `could not import` cache misses. Mitigation: run real build/test
  outside the sandbox; do not interpret cache-miss noise as a code break.

## Trade-offs

- **Convention as target shape, not forced rewrite** (leave conforming files
  alone) over **rewrite-every-file-for-uniformity**: chose the former because
  CLAUDE.md forbids churn-for-nothing and a smaller diff is safer to verify and
  cheaper to review. The cost is that "did we apply it everywhere" requires the
  per-file sorted-decl check rather than a blanket "we rewrote all 70."
- **Per-directory batching (22 tasks)** over **one giant pass**: chose batching
  so build/test verify incrementally and a break has a small blast radius. The
  cost is more bookkeeping and more commits; acceptable given the file count.
- **"Prefer alphabetical, logical grouping may win"** over **strict
  alphabetical**: chose the softer rule because the existing code has meaningful
  method clusters (CRUD groupings, method+adjacent-helper pairings) that strict
  alphabetization would scatter, hurting the very readability this refactor is
  for. The cost is mild inconsistency between files; judged worth it.
- **Interface assertions at the very top of the file** over **keep at file bottom**
  (the current repo habit) and over **merely beside their struct**: the user's
  reasoning (2026-05-23) is that the assertion is the interface↔implementation
  *contract* — it must catch the eye first and surface the file's interface
  dependency before anything else, the same way the test-file convention puts
  `var _ Iface = &mock{}` above all tests. This is the one place the refactor
  changes an existing repo habit deliberately, and it is a stronger move than
  "near the struct": prominence at file top wins over mere adjacency.
- **Sorted-decl-set diff as the movement-only proof** over **trusting the
  compiler + tests alone**: chose the explicit set-diff because compile+green
  does not catch a silently altered body, and this refactor's entire value
  proposition is "zero behavior change." Cheap insurance for the one invariant
  that matters most.
