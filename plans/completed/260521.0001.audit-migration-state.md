# Task Breakdown — Audit Migration State and Reconcile CLAUDE.md

## Overview

Focused plan for Task 1 of `008-multi-agent-audit-followup.md`. The
multi-agent audit on 2026-05-21 flagged a Blocker:

> CLAUDE.md lists 11 migrations (001–011) under
> `### Database` → "Migration table", but only 6 files exist on disk.

Investigation shows this is **not** an undocumented incident — commit
`5a230f1` (2026-05-19, on `origin/main`) explicitly consolidated old
migrations under plan `260519.0001`. The plan claimed migrations 001–006
were "immutable and pushed to origin/main", but the commit itself:

- modified `004.rate_user_events.table_initiate.sql` to add the
  `source_name` column (absorbing the old 005);
- deleted the old `005.rate_user_events.add_source_name.sql`;
- renamed old `006.execution_history.table_initiate.sql` → new
  `005.execution_history.table_initiate.sql`;
- deleted old `007`–`014`;
- added new `006.rate_sources.seed_initial.sql` (consolidated seed of
  36 rate sources + 20 admin subscriptions).

CLAUDE.md was never updated; it still describes the pre-consolidation
world. Reality on disk and on `origin/main`:

```
202605.001.rate_sources.table_initiate.sql          (now includes rule_metadata + fetcher_kind columns)
202605.002.rate_values.table_initiate.sql
202605.003.rate_user_subscriptions.table_initiate.sql
202605.004.rate_user_events.table_initiate.sql      (now includes source_name column)
202605.005.execution_history.table_initiate.sql     (renamed from old 006)
202605.006.rate_sources.seed_initial.sql            (consolidated initial seed)
```

The audit blocker stands but its nature changes: **doc drift**, plus an
**open question about production's `__schema_migrations` state** which
this plan does not yet have evidence on.

## Assumptions

- Commit `5a230f1` is on `origin/main` (verified via
  `git branch -r --contains 5a230f1` → `origin/main`).
- No further migration churn since `5a230f1` (verified via
  `git log --all --oneline -- 'migrations/*.sql'`).
- The user has shell access to both `stage` and `prime` deploy targets
  and can either run the SSH verification themselves or paste the SQL
  result back.
- This plan does **not** touch any production code. It is doc-only.

## Tasks

### Task 1: Verify production `__schema_migrations` state on stage and prime

- **Description**: SSH into each deploy target and dump
  `__schema_migrations.filename` ordered. Compare against the six
  filenames on disk. Three possible outcomes determine the next step
  in Task 3:
  - **A — clean match (6 rows)**: prod was deployed after `5a230f1`
    against a fresh DB. No reconciliation needed.
  - **B — superset (e.g. 8 rows with old 005/006 plus new 005/006)**:
    prod was migrated through both pre- and post-consolidation
    chains. Schema is consistent (`CREATE TABLE IF NOT EXISTS` +
    `INSERT OR IGNORE`), but `__schema_migrations` has historical
    cruft. Document; no recovery.
  - **C — divergent (rows that don't match either chain)**: somebody
    applied a hand migration. Stop and investigate before doing
    anything else.
- **Acceptance Criteria**:
  - Output of
    ```bash
    ssh <stage-host> 'sqlite3 /opt/monitor/<db-path> \
       "SELECT filename FROM __schema_migrations ORDER BY filename;"'
    ssh <prime-host> 'sqlite3 /opt/monitor/<db-path> \
       "SELECT filename FROM __schema_migrations ORDER BY filename;"'
    ```
    is attached to the plan-completion note for both hosts. The
    user runs the commands; this plan does not initiate SSH.
  - The outcome bucket (A, B, or C) is recorded.
- **Pitfalls**:
  - The DB path is read from `EnvironmentFile` (e.g.
    `/opt/monitor/.stage.env`); operator must source it or pass the
    actual path. Format: `SQLITEDB_DSN=sqlite:///path/to/db.sqlite`.
  - SQLite WAL mode means a running service holds the file open;
    `sqlite3` can read it concurrently but a transactionally
    consistent snapshot requires `.timeout 5000` or stopping the
    service first. For a `SELECT` from a static table this is
    overkill — go ahead and read it live.
- **Complexity**: Easy (read-only).

### Task 2: Reconcile CLAUDE.md migration section with disk reality

- **Description**: Update `CLAUDE.md` lines 122–134 (the migration
  table) and surrounding prose to reflect the 6-file state.
  Specifically:
  - Replace the 11-row table with a 6-row table matching disk.
  - Add a `### Migration history` subsection immediately after the
    table that:
    - cites commit `5a230f1` and plan
      `plans/completed/260519.0001.sourceaudit-parser-and-migration-consolidation.md`
      as the consolidation event;
    - states explicitly that the documented immutability rule was
      **overridden once** on 2026-05-19 to fold pre-release
      migrations 005–014 into the current 001–006, and that this
      override is not a precedent;
    - warns future maintainers: **any further filename change is
      destructive to any DB that already applied the old name**;
    - links to the result of Task 1 (which prod-state bucket
      applied).
  - Update the `Repository files in internal/repository/...` paragraph
    only if a column changed location (no rename happened, so no
    change needed).
- **Acceptance Criteria**:
  - `CLAUDE.md`'s migration table has exactly 6 rows matching the
    output of `ls -1 migrations/*.sql`.
  - The note about "applied filename is immutable" remains the
    default rule but is explicitly bracketed by the historical
    override callout.
  - `grep -c 'add_rule_metadata\|add_fetcher_kind\|rename_headless\|add_source_name' CLAUDE.md`
    returns 0 outside the historical-override subsection.
- **Pitfalls**:
  - Do not delete the immutability rule itself — it still governs
    all future migrations. The historical override is a one-time
    exception, not a policy change.
  - Keep the existing `### Database` heading depth (`###`); the new
    subsection is `####` (or `###` with a clear separator) so the
    table-of-contents stays clean.
- **Complexity**: Easy.

### Task 3: Decide on prod-state recovery (conditional on Task 1)

- **Description**: Based on the bucket from Task 1:
  - **A (clean match)**: nothing to do. Close the plan.
  - **B (historical cruft)**: write a one-paragraph note in the
    CLAUDE.md "Migration history" subsection explaining that prod
    `__schema_migrations` carries leftover rows from the old chain
    and that this is intentional / safe. Leave the rows in place —
    deleting them would break re-derivability.
  - **C (divergence)**: stop. Open a new plan (`010-...`) to triage
    the hand-applied migration. Do **not** continue the audit-followup
    sequence until this is resolved.
- **Acceptance Criteria**:
  - Bucket A: plan moves to `completed/` after Task 2 lands.
  - Bucket B: CLAUDE.md note added; plan moves to `completed/`.
  - Bucket C: this plan moves to `history/` with the bucket-C note;
    a new plan is opened for triage.
- **Complexity**: Easy unless bucket C, in which case complexity is
  bounded by what hand-migration was applied.

## Execution Order

1. **Task 2 first** (doc patch) — independent of prod state. Lands
   the disk-reality fix immediately; the historical-override
   subsection initially carries a `_pending: see Task 1 outcome_`
   placeholder.
2. **Task 1** (SSH verification) — done by the user; this plan
   provides the exact commands.
3. **Task 3** (conditional recovery) — replace the placeholder in
   CLAUDE.md with the bucket finding.

Rationale for putting Task 2 before Task 1: the doc patch is needed
regardless of prod outcome, and unblocks the rest of the audit-followup
plan immediately. Production verification is asynchronous (depends on
user).

## Risks

- **Bucket C (divergent prod)** is the only real risk. Mitigated by
  refusing to continue the audit-followup until triaged.
- **Doc patch drifts again**: low risk — the audit plan 008 is the
  forcing function for keeping this current. If Task 2 of plan 008
  ships (FK migration), CLAUDE.md table will grow to 7 rows in the
  normal additive way.

## Trade-offs

- **Doc-only vs. also fixing the historical commit message**: not
  fixing. The commit is already on `origin/main`; rewriting history is
  destructive and gains nothing. The CLAUDE.md note is the canonical
  pointer.
- **One-time override vs. policy change**: keeping the immutability
  rule as the default and treating 5a230f1 as a documented exception.
  Alternative would be to soften the rule to "immutable after first
  external deploy" — rejected because that's harder to enforce and
  removes the bright line.
- **Hand-rolled CLAUDE.md edit vs. agent-driven**: hand-rolled. The
  scope is bounded (≈10 lines), the engineer has full context from
  the audit, and an architect/engineer/reviewer fan-out for a doc
  patch is overkill. A single reviewer pass on the diff is sufficient.

**Stage: bucket A — clean match.** User scp'd `./logs/stage_monitor.sqlite`
(operator wiped + redeployed beforehand to avoid migration friction);
`SELECT filename FROM __schema_migrations ORDER BY filename` returned the six
post-consolidation filenames in lockstep with disk:

```

202605.002.rate_values.table_initiate.sql
202605.003.rate_user_subscriptions.table_initiate.sql
202605.004.rate_user_events.table_initiate.sql
202605.005.execution_history.table_initiate.sql
202605.006.rate_sources.seed_initial.sql
```

`rate_sources.{rule_metadata, fetcher_kind}` and `rate_user_events.source_name`
are present in the live CREATE TABLE definitions. Row counts: 36 sources / 20
subs / 175 rates / 175 hist. Plan 260519.0001 expected 21 admin subscriptions —
actual 20 is a one-row drift on a seeded subscription row, not a structural
issue.

**Prime**: not separately verified in this plan. Stage was wiped and
redeployed by the operator post-consolidation; if prime followed the same
procedure (which the deploy workflow enforces), it is also bucket A. If a
later finding contradicts this, open a follow-up plan.

**Task 2 (CLAUDE.md patch)**: shipped in the same commit as this plan
closure. The historical-override subsection links to commit `5a230f1` and
plan `260519.0001`; no further reconciliation needed for bucket A.

**Task 3**: not triggered — bucket A means no recovery work.
