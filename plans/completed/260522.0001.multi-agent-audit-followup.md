# Task Breakdown — Multi-Agent Audit Follow-up

## Overview

Draft plan consolidating findings from a parallel project evaluation run
on 2026-05-21 by 3 architects (system, data, ops) and 5 reviewers
(correctness, tests, ops, security, performance). The audit found no
showstoppers but several Blocker-class issues that should be addressed
before further feature work, plus a meaningful tail of Major and Minor
items. This file is a roadmap — each Task below will be split into its
own focused plan when picked up.

Severity rubric used by the agents:
- **Blocker**: incorrect behaviour visible to users or operators, or a
  documentation/state inconsistency that risks data loss.
- **Major**: working today but fragile, with a clear correctness, perf,
  ops, or security cost.
- **Minor / Nit**: hygiene, style, dead code, doc drift.

## Assumptions

- The audit reflects the state of `main` at commit `178cd13`.
- Production / staging schema state is **not** verified yet — Task 1
  must run before any new migration is authored.
- `CLAUDE.md` is currently inconsistent with disk reality on migrations;
  the plan treats CLAUDE.md as the document to fix, not the disk.
- Off-main commits are free per project policy; push / branch ops still
  require explicit approval.

## Tasks

### Task 1: Audit production migration state and reconcile CLAUDE.md

- **Description**: CLAUDE.md lines 99–110 list migrations 005–011
  (add_source_name, add_rule_metadata, add_fetcher_kind,
  rename_headless_to_chromedp). On disk only 001–006 exist;
  `rule_metadata` and `fetcher_kind` are baked into
  `001.rate_sources.table_initiate.sql:12-13`. Either historical
  migrations were collapsed (violating the documented immutability
  rule) or production will diverge on next deploy.
- **Acceptance Criteria**:
  - `__schema_migrations` on both stage and prime is dumped and
    attached to the plan-completion file.
  - CLAUDE.md migration table matches reality on disk.
  - If a reset happened, document it under `## Migration history reset`
    inside CLAUDE.md with the date and rationale.
- **Pitfalls**: must not author a new migration before this audit is
  done — wrong assumptions will compound.
- **Complexity**: Easy (read-only, but coordination-heavy).

### Task 2: Schema integrity — foreign keys and busy_timeout

- **Description**: Add referential integrity now that
  `PRAGMA foreign_keys=ON` is being honoured at connect time, and set
  `busy_timeout` so concurrent writers don't see immediate
  `SQLITE_BUSY` errors.
  - New additive migration: add
    `REFERENCES rate_sources(name) ON DELETE CASCADE` (or `RESTRICT`)
    to `rate_values.source_name`, `rate_user_subscriptions.source_name`,
    `rate_user_events.source_name`.
  - `NewSQLiteClientEx` (`internal/infrastructure/sqlitedb/sqlclient.go`):
    add `PRAGMA busy_timeout=5000;` alongside the existing WAL and FK
    pragmas.
- **Acceptance Criteria**:
  - `RemoveRateSource` cascade-deletes (or refuses) and is covered by
    a repo-level test.
  - A concurrent write contention test reproduces the old `SQLITE_BUSY`
    behaviour without the pragma and passes with it.
- **Pitfalls**: SQLite cannot `ALTER TABLE ... ADD CONSTRAINT`. Each
  table must be rebuilt (`CREATE TABLE ... PRIMARY KEY (...) REFERENCES
  ...`, copy data, drop old, rename). Plan for a downtime window or do
  it inside a single transaction.
- **Complexity**: Hard (migration rewrite + correctness window).

### Task 3: Fix `ObtainSubscriptionSummaryBySource`

- **Description**: `rateusersubscription.go:277-280` joins
  `rate_user_events` by literal table and column names (breaks the
  const-discipline invariant) AND the JOIN condition is
  `e.source_name = s.source_name AND e.user_type = s.user_type` —
  missing `e.user_id = s.user_id`. With more than one subscriber per
  source, per-user aggregates collapse to source-wide totals.
- **Acceptance Criteria**:
  - Add `e.user_id = s.user_id` to the JOIN; verify counts per-user.
  - Either move the cross-table SQL into a typed helper that imports
    both repositories' column constants, OR build the SQL via
    `fmt.Sprintf` with both consts.
  - Regression test with two users on the same source.
- **Complexity**: Medium.

### Task 4: `UpdateRateSourceActive` — restore 404 contract and its test

- **Description**:
  - `service/raterestapi.go:78-84` returns a plain error when the
    source is missing; `handlers.go:363` expects
    `internal.ErrNotFound`. Caller sees 500 instead of 404.
  - The repository-level test for this method is currently commented
    out at `ratesource_test.go:385-437` with a `// TODO: rethink`.
- **Acceptance Criteria**:
  - Service wraps the not-found case with
    `fmt.Errorf("rate source %q: %w", name, internal.ErrNotFound)`.
  - The commented test block is either restored as a passing test or
    deleted entirely with a tracked-issue pointer.
  - Handler test asserts 404 for an unknown source name.
- **Complexity**: Easy.

### Task 5: Stop swallowing `RetainRateUserEvent` errors in `RateCheckAgent`

- **Description**: `ratecheckagent.go:167-177` increments `failCount`
  but discards `retainErr`. Failed event persistence is invisible to
  caller and to logs.
- **Acceptance Criteria**:
  - `retainErr` is logged with `chat_id` context AND appended to the
    `combined` slice that `Run` returns.
  - Test in `ratecheckagent_test.go` injects a failing event repo and
    asserts `Run` returns a non-nil error containing the retain
    failure.
- **Complexity**: Easy.

### Task 6: Fix chromedp shared-slice mutation and add scheme guard

- **Description**: Two issues on the same component.
  - `chromedp.go:105` and `rulegen/chromedpfetcher.go:96` do
    `append(chromedp.DefaultExecAllocatorOptions[:], …)` — relies on
    upstream array having no spare capacity. Replace with
    `slices.Clone`.
  - `fetchRenderedPage` navigates `source.URL` with no scheme check.
    `validateSourceURL` exists in `generator.go` but is only invoked
    on the rulegen path. Add scheme validation (`http` / `https` only)
    at the top of `fetchRenderedPage`.
- **Acceptance Criteria**:
  - Unit test confirms `file:///etc/passwd` and `javascript:` URLs
    return an error before any chromedp call.
  - `slices.Clone` is used; the global is never aliased.
- **Complexity**: Easy.

### Task 7: Migrator log filename and stale deploy docs

- **Description**:
  - `cmd/migrator/main.go:46` passes `"collector"` to
    `internal.NewLogger`, so migrator runs land in `collector.*.log`
    instead of `migrator.*.log`. Violates the binary-name-prefix rule.
  - `deploy/monitor.service` uses flags (`--config`, `--db`,
    `--interval`, `--log-format`) that no binary supports today and
    paths (`/etc/monitor/env`, user `monitor`) that don't match what
    CI deploys. `deploy/README.md` walks an operator through a path
    that cannot work.
- **Acceptance Criteria**:
  - Migrator logger name is `"migrator"`.
  - `deploy/monitor.service` either matches the real units under
    `configs/srv.*_monitor.service` or is deleted, and
    `deploy/README.md` is updated to point operators at the live unit
    files.
- **Complexity**: Easy.

### Task 8: Lifecycle — SIGTERM in collector/notifier, drop `time.Sleep` startup

- **Description**:
  - `cmd/collector/main.go:118` and `cmd/notifier/main.go:97` use
    `context.Background()` with no signal handler — SIGTERM mid-run
    kills the process silently. Wrap with `signal.NotifyContext` the
    way the migrator already does.
  - `cmd/web/main.go:203` uses `time.Sleep(10 * time.Millisecond)` as a
    startup barrier. Replace with a pre-bound `net.Listen` so the
    "ready" log fires only after the socket is actually accepting.
- **Acceptance Criteria**:
  - Collector and notifier emit a `shutdown: signal received` log line
    on SIGTERM and rollback any in-flight transaction cleanly.
  - The 10ms sleep is gone.
- **Complexity**: Medium.

### Task 9: SQLite backup and `/healthz`

- **Description**:
  - The entire service is one SQLite file with no documented backup or
    restore path. Add a nightly `sqlite3 "$DB" ".backup '...'"` (or
    `litestream` if preferred) with 14-day retention and a documented
    restore drill in `deploy/README.md`.
  - Add a `GET /healthz` handler on `cmd/web` that `db.Ping`s and
    returns 200 / 503. Wire it into systemd `ExecStartPost` or external
    monitoring as appropriate.
- **Acceptance Criteria**:
  - Nightly backup job is version-controlled (`deploy/cron/`) and
    documented.
  - Restore drill is exercised at least once on staging.
  - `/healthz` returns 503 when the DB is unreachable.
- **Complexity**: Medium.

### Task 10: Logging hygiene

- **Description**:
  - `internal/application/service/telegramapi.go:55-56` logs the full
    raw text of every inbound Telegram message — PII. Truncate to
    `len=` only, or hash.
  - `internal/logger.go:104` passes the literal
    `"lingocrm.product.api.error"` (a previous project's tag) into the
    Telegram alert hook. Replace with `"fx_rate_monitor.error"`.
  - `ratecheckagent` per-user `queued: N/M` log line is emitted
    unconditionally for every recipient — aggregate into one summary
    line per run.
- **Complexity**: Easy.

### Task 11: Read-only transactions for SELECT paths

- **Description**: Every repository read goes through `db.Transaction`
  (write-mode). `ReadOnlyTransaction` exists on the SQLite client but
  is never used by repositories. Convert all SELECT-only methods to
  `ReadOnlyTransaction`. Also: remove the explicit `tx.Rollback()`
  from read paths so the deferred `printRollbackError` is the single
  rollback site.
- **Acceptance Criteria**:
  - All SELECT-only methods across `internal/repository/*.go` use
    `ReadOnlyTransaction`.
  - No method calls both `tx.Rollback()` and
    `defer printRollbackError(tx)` in the same body.
- **Pitfalls**: A few repository methods do read-then-write inside the
  same transaction — those must stay on `Transaction`. Audit before
  bulk-renaming.
- **Complexity**: Medium.

### Task 12: Kill N+1 — `ListSources` and `ListMeSubscriptions`

- **Description**:
  - `handlers.go:87` issues one `ObtainLastNExecutionHistoryBySourceName`
    transaction per source on every `GET /api/sources`. Replace with a
    single bulk query returning the latest execution per source.
  - `handlers.go:639` (search path) calls `ObtainRateSourceByName` per
    group before pagination, then again per page item. Bulk-load once
    into a `map[string]*domain.RateSource`.
- **Acceptance Criteria**:
  - Both endpoints issue O(1) DB round-trips for the source lookup
    regardless of result size.
  - Handler tests assert the call count via a counting stub.
- **Complexity**: Medium.

### Task 13: chromedp allocator at `Run`-batch granularity

- **Description**: Today each chromedp source pays a fresh Chromium
  cold-start. Without committing to a full pool yet, hoist the
  allocator out of `fetchRenderedPage` to a shared scope inside one
  collector tick so N chromedp sources reuse one browser process.
- **Acceptance Criteria**:
  - One allocator per `Run`-batch (or per tick) instead of per URL.
  - Benchmark shows wall-clock improvement proportional to N for
    chromedp sources.
- **Complexity**: Hard.

### Task 14: Strip `?initData=` query-string fallback

- **Description**: `handlers.go:588` accepts `?initData=` when the
  `X-Telegram-Init-Data` header is absent. The HMAC-signed token ends
  up in access logs and `Referer` headers — a credential leak during
  its 24h validity window. Remove the fallback. The Telegram Mini App
  JS SDK always sends the header.
- **Acceptance Criteria**:
  - Query-string fallback removed; handler returns 401 when the header
    is absent.
  - Integration test updated.
- **Complexity**: Easy.

## Execution Order

Recommended order, grouped to minimise context-switches:

1. **Audit & schema wave**: Task 1 → Task 2 → Task 3.
   *Rationale*: nothing else makes sense until prod state is known and
   the schema bedrock is fixed.
2. **Correctness wave**: Task 4 → Task 5 → Task 6.
   *Rationale*: Blockers that are 5-line fixes; sweep them once the
   schema is in a known state.
3. **Ops wave**: Task 7 → Task 8 → Task 9 → Task 10.
   *Rationale*: turns "process is up" liveness into real observability
   and gets a backup story in before any further migration work.
4. **Perf wave**: Task 11 → Task 12 → Task 13.
   *Rationale*: requires the read-only transaction split (Task 11)
   before the N+1 fixes land their full benefit.
5. **Security & tests**: Task 14 → Task 15.
6. **Architecture polish**: Task 16.

## Risks

- **Task 2 (FK migration)**: SQLite table-rebuild is the only way to
  add FK constraints. A bug here corrupts every dependent table.
  Mitigation: do it inside a single transaction, dry-run on a copy of
  prod first, and ship Task 9 (backups) before Task 2.
- **Task 1**: if production really did skip migrations, the recovery
  path is manual. Coordinate a deploy freeze.
- **Task 11**: a SELECT method that secretly does a write after a
  conditional check will start failing if blindly switched to
  `ReadOnlyTransaction`. Read each method before renaming.
- **Task 13**: shared allocator across goroutines requires care —
  chromedp contexts are not safe to share concurrently; serialise per
  batch.

## Trade-offs

- **Bundle vs split**: each task above is small enough to be its own
  PR. Bundling Tasks 4–6 (the 5-line Blockers) into one PR is fine and
  preferred per the project's linear-history rule.
- **Migrations: rewrite vs additive**: Task 2 forces a rewrite of three
  tables. We accept the migration weight in exchange for restoring the
  FK contract, because the alternative ("just don't delete sources")
  is fragile and unenforced.
- **`?initData=` fallback removal (Task 14)**: removes a curl-based
  testing convenience. Acceptable — the WebApp JS SDK always sends the
  header, and the fallback is a security liability.
- **chromedp pool (Task 13)**: per-batch sharing is a stepping stone,
  not a full pool. Accepts simpler code now in exchange for a future
  follow-up if throughput requirements grow.
