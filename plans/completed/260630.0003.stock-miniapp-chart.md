# Task Breakdown

## Overview

Phase 2 of the stock-price feature: make equity sources render correctly in the
Mini App. The functional MVP (collect â†’ subscribe â†’ notify for `kind=LAST`
sources `US_YAHOO_LAST_AAPL_USD` and `KZ_KASE_LAST_CCBN_KZT`) already shipped on
this branch (see `plans/completed/260629.0001.stock-price-sources.md`, Tasks 1â€“5
+ 7). The deferred Task 6 (display) is this plan, regrounded in the
actually-shipped code, which diverged from the pre-MVP spec in three material
ways:

1. `RateSourceKind` is a typed string and `RateSourceKindLAST = "LAST"` already
   exists (`internal/domain/ratesource.go:67-78`). No domain change needed.
2. The chart **service** silently drops a `LAST` series:
   `buildSeriesRow` is called for every unique pair
   (`internal/application/chart/service.go:335`) but the `switch p.Kind`
   (`service.go:347-363`) has only `BID`/`ASK` cases, so the computed series for
   a `LAST` pair is discarded. The row still renders (label + empty `Series`),
   producing the "empty chart slot" gap.
3. The history endpoint already routes `LAST` into the **Bid** slot
   (`internal/application/chart/history.go:206-216`, an MVP review fix), so a
   stock's last price currently renders as a literal `BID 230.50` line in the
   history view (`cmd/wasm/ui/me_pair_history.go:81-87`). This is mislabeled, not
   broken; this plan reconciles it.

Scope: (1) dashboard sparkline chart, (2) per-pair history, (3) subscription
editor â€” equities grouped under a new `equity` category so they sort and display
distinctly from fiat and metal. No new migration (display only). No new external
dependency. The chart **DTO and gateway mapping need no change** for the chart
itself: `Kind`, `Color`, and `Category` are already string passthroughs
(`internal/gateway/httpV1/handlers/handlers.go:1100-1124`, `:1262-1286`); only
the history path adds a field.

## Assumptions

- **Equity classification by symbol set (recommended), mirroring `metalSymbols`.**
  Add `equitySymbols = {"AAPL", "CCBN"}` to `internal/domain/ratepair`. Rationale
  and the rejected `kind==LAST` alternative are in Trade-offs. This is an
  operator-overridable decision; the symbol set is the lower-ripple choice
  because `CategoryOf(base) â†’ Category` keeps its pure signature and the existing
  row-sort (`service.go:372-377`, which constructs a kind-less `Pair`) works
  unchanged.
- **Dedicated `last *SeriesRow` slot + `ColorLast`** in the chart `pairGroup`,
  not reuse of the `bid` slot. For an equity, `LAST` is the only series, so the
  slot choice is about code clarity, not behavior; a dedicated slot keeps `bid`
  meaning bid. See Trade-offs.
- **History gets a dedicated `Last *float64` field** end-to-end (application â†’
  DTO â†’ gateway â†’ renderer), not the shipped `LASTâ†’Bid` mapping. "Render
  correctly" is the whole point of Phase 2, and labeling a stock's last price as
  `BID` is exactly the defect this phase closes. The cheaper "keep Bid mapping"
  option is documented in Trade-offs; it cannot produce a correct label because
  `MeHistoryRow` carries no kind/category to switch on.
- **Equity sort position: after metal** (`fiat â†’ metal â†’ equity`), via an
  explicit `categoryOrder` map rather than the current lexicographic
  `catA < catB` compare (under which `"equity"` would sort first by spelling).
  The map makes order intentional and spelling-independent.
- **No wire-compat risk.** `Last`/`LastDeltaPct` are `omitempty` and absent for
  FX rows; the WASM client and server ship in one bundle per deploy, so there is
  no version skew.
- **The equity color hex is a design choice.** A distinct value (suggested amber
  `#D98E04`, colorblind-separable from the green BID / blue ASK) is recommended
  and pinned with a test; the exact value is the operator's to repaint.

  `equitySymbols` silently classifies as fiat (wrong band/category badge; the

- **WASM rebuild cost.** `make build` (`GOOS=js`) is slow on the Raspberry Pi

- **Color accessibility.** `ColorLast` must be distinguishable from the green BID

  separable; the pin-test enforces distinctness from both, and design can repaint.

## Trade-offs

- **Equity classification: symbol set vs `kind==LAST` (chose symbol set).**
  A symbol set mirrors the reviewed `metalSymbols` precedent, keeps classification
  in one place (metal and equity both by symbol, in `CategoryOf`), preserves
  `CategoryOf`'s pure `(base) â†’ Category` signature, and lets the existing
  kind-less row sort (`service.go:372-377`) work unchanged. Its cost is the
  two-place ticker duplication (migration + `ratepair`). The `kind==LAST`
  alternative avoids duplication (add a ticker via migration only) but: (a)
  splits classification (metal-by-symbol, equity-by-kind); (b) conflates "is an
  equity" with "quotes a last price" (breaks if a non-equity ever uses LAST or an
  equity ever gets BID/ASK); and (c) forces the row sort to become
  category-aware (it currently builds a `Pair` with no kind), i.e. *more* ripple,
  not less. **Operator override:** if tickers are expected to grow fast and
  migration-only additions matter more than classification consistency, derive
  the category from the group's `last != nil` in `buildPairRow` and sort by the
  row's already-computed `Category` field.
- **Chart LAST slot: dedicated `last` vs reuse `bid` (chose dedicated).** For an
  equity, LAST is the only series, so behavior is identical either way; a
  dedicated slot (~4 extra lines) keeps `bid` meaning bid and avoids a future
  reader puzzling over a "bid slot holding a LAST series." Reusing `bid` would
  save the struct field but muddy semantics for no functional gain.
- **History LAST: dedicated `Last` field vs keep `LASTâ†’Bid` (chose dedicated).**
  Keeping the shipped Bid mapping is cheaper (zero DTO/gateway/renderer change)
  but **cannot** produce a correct label: `MeHistoryRow` carries no kind, so the
  renderer can only print `BID`. Phase 2 exists to make stocks render correctly;
  shipping a `BID 230.50` line for AAPL defeats the purpose. The dedicated field
  is a bounded four-file slice. **Operator override:** if history correctness is
  acceptable to defer further, skip Task 4 and ship Tasks 1â€“3 + 5â€“6 (chart
  correct, history still mislabeled).
- **Equity color: dedicated `ColorLast` vs reuse `ColorBid` (chose dedicated).**
  Reusing `ColorBid` (teal) would render a stock line identically to a bank's
  buy price; a distinct color signals a distinct asset class at a glance. One
  constant feeds both the SVG and the modal prefix, so the cost is a single hex
  plus a pin-test.
- **Equity sort position: explicit `categoryOrder` map vs lexicographic (chose
  map, equity last).** The current `catA < catB` orders by string spelling;
  adding `"equity"` would sort it *first* (`"equity" < "fiat"`), an accidental
  order. An explicit `{fiat:0, metal:1, equity:2}` map makes order intentional
  and puts the new band at the bottom, leaving the familiar fiat-first layout
  intact.
