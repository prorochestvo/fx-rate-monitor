# Task Breakdown

## Overview

Replace the Mini App overlay chart (multi-line, common Y-axis, duplicate
legend entries) with a sparkline-list view: one row per unique currency
pair, auto-fit Y scale per row, category-aware coloring (fiats vibrant,
metals muted), deterministic colors per pair across sessions. The mock
lives next to this file as `009-fx-chart-sparkline-list-demo.html`.

Phase 1 (Telegram text-message dedupe + formatting) is already in
production. This plan covers the Mini App chart view (Phase 2) only.

A recon pass mapped the codebase before this rewrite. Key facts:

- **No reusable Phase 1 utilities exist.** The plan originally assumed
  `categoryOf` / `metalSymbols` / `PairCategory` / dedupe / sort live in
  a shared package. They do not. Phase 1 dedupe is inline inside
  `RateCheckAgent.Run()` (`internal/application/notification/ratecheckagent.go:111-222`),
  the sort in `message.go:76-80` is alphabetical only, and no category
  enum exists. Phase 2 builds these from scratch in a new shared
  package.
- **Existing chart endpoint and renderer go away.** The current
  `/api/sources/{name}/rates/chart` handler, the
  `ObtainRateValueChartBySourceName` repository method, the
  `apiclient.RatesChart` call, and `cmd/wasm/ui.RenderOverlayChart` are
  the only producers/consumers of the overlay view. All of them are
  removed; nothing else in the dashboard depends on them.
- **`/api/me/*` auth pattern is inline, not middleware.** Handlers call
  `h.validateInitData(initData, h.botToken, maxAge, h.nowFn())` directly
  (see `ListMeSubscriptions` in
  `internal/gateway/httpV1/handlers/handlers.go:561`). Phase 2 copies
  this pattern verbatim.
- **CSS variables in the Mini App are Telegram theme vars
  (`--tg-theme-*`), not the `--color-text-primary` etc. used in the
  demo HTML.** The demo's variable names are illustrative; the real
  implementation uses `--tg-theme-text-color`,
  `--tg-theme-hint-color`, `--tg-theme-secondary-bg-color`, etc., with
  fallbacks for clients that omit them.

## Assumptions

- Window is a fixed last-7-days rolling window. No period selector in
  the UI, no `?period=` query parameter on the new endpoint. The window
  lives as a single constant in the shared `ratepair` package so it can
  be retuned in one place.
- Downsampling uses **last-in-bucket** (the last sample in each evenly
  sized time bucket). Median was considered but rejected: the rightmost
  sparkline point should read as "current price", which last-in-bucket
  naturally delivers and median does not.
- Delta is reported as a percent: `(last - first) / first * 100`,
  computed over the points that fall inside the window. The
  same-direction sign drives the row's delta color (`ColorDeltaUp` if
  `>= 0`, `ColorDeltaDown` if `< 0`).
- BID and ASK for the same `(base, quote)` are two separate rows. The
  BID row is labeled `BASE/QUOTE` (e.g. `USD/KZT`); the ASK row is
  labeled `QUOTE/BASE` (e.g. `KZT/USD`). The displayed numeric value
  is **not inverted** for ASK — only the label flips, to communicate
  conversion direction.
- Sort order: canonical pair key `min(base,quote)/max(base,quote)`,
  ascending; within a pair, BID before ASK. Fiats before metals as a
  primary key. This keeps BID/ASK rows for the same pair adjacent.
- `ColorForPair` is keyed on `base` only. Two pairs sharing a base
  (e.g. `USD/KZT` and `USD/RUB`) share a color; acceptable for now —
  real users usually only hold one quote currency per base. Hashing
  `base+"/"+quote` is a later iteration if complaints surface.
- Sparse pair (fewer than two values in the window) renders as a flat
  line at the current value with `+0.0%` delta. Zero values renders an
  inline "no data" state in the row instead of a sparkline.

## Tasks

### Task 1: Shared `internal/domain/ratepair/` package

- **Description.** Create the new package
  `internal/domain/ratepair/` holding everything pair-classification
  related. Exported surface:
  - `Category` type (string-typed enum) with constants `CategoryFiat`
    and `CategoryMetal`.
  - `MetalSymbols` — a set (`map[string]struct{}`) of metal base codes:
    `GOLD`, `SILVER`, `PLATINUM`, `PALLADIUM`. Codes are uppercased on
    lookup; do not include `XAU`/`XAG` unless the data layer actually
    emits them — confirm during implementation by grepping rate-source
    rules and the `rate_values` table.
  - `CategoryOf(base string) Category` — uppercases input, returns
    `CategoryMetal` if in `MetalSymbols`, otherwise `CategoryFiat`.
  - `Pair` value object:
    `struct { Base, Quote string; Kind domain.RateKind }`.
  - `Less(a, b Pair) bool` — sort comparator. Sort key:
    `(category, canonicalPair, kindOrder)` where category ranks Fiat
    before Metal, `canonicalPair` is
    `min(base,quote)/max(base,quote)` uppercased, and `kindOrder` is
    BID before ASK.
  - `Dedupe(in []Pair) []Pair` — collapses duplicates by
    `(base, quote, kind)`, preserving the first occurrence's position.
    Caller sorts afterwards using `Less`.
  - `FiatPalette` and `MetalPalette` — `[]string` of hex colors:
    fiats `#378ADD`, `#7F77DD`, `#1D9E75`, `#D85A30`, `#D4537E`,
    `#639922`; metals `#BA7517`, `#5F5E5A`, `#993C1D`, `#888780`.
  - `ColorForPair(base string) string` — FNV-32a hash of uppercased
    base, modulo palette length for the appropriate category. Stdlib
    `hash/fnv` only (stable across Go releases).
  - `ColorDeltaUp` and `ColorDeltaDown` — `const string` hex
    `#3B6D11` and `#A32D2D`.
  - `ChartWindow` — `time.Duration` constant equal to
    `7 * 24 * time.Hour`.
- **Acceptance Criteria.**
  - `TestCategoryOf` covers fiat base, metal base, mixed-case input,
    empty string, unknown long string.
  - `TestColorForPair` table-driven: same input → same output across
    multiple invocations; fiat base produces a color from
    `FiatPalette`; metal base produces a color from `MetalPalette`;
    case-insensitive (`usd` and `USD` produce the same color).
  - `TestDedupe` covers: empty input, single-pair input, multiple
    identical entries, order preservation.
  - `TestLess` covers: fiat-before-metal, canonical pair ordering
    (USD/KZT and KZT/USD sit next to each other regardless of
    underlying direction), BID before ASK within the same canonical
    pair.
  - All tests `t.Parallel()`. One `Test<Identifier>` per identifier
    per CLAUDE.md.
  - Package has a `// Package ratepair ...` doc comment. Every
    exported identifier has a doc comment starting with its name.
- **Pitfalls & edge cases.**
  - Do not use `maphash` — its seed varies across processes; FNV is
    deterministic across releases and runs.
  - `Dedupe` is order-preserving by design; document this in the
    godoc.
  - `MetalSymbols` lookup is uppercase-only; document this.
- **Complexity.** Medium.

### Task 2: Repository method for bulk per-pair time series

- **Description.** Add a new method to
  `internal/repository/ratevalue.go`:
  ```go
  // ObtainValuesForPairsSince returns rate_values rows matching any
  // of the given (source, base, quote, kind) tuples whose timestamp
  // is >= since, ordered by timestamp ascending.
  func (r *RateValueRepository) ObtainValuesForPairsSince(
      ctx context.Context,
      pairs []domain.SourcePairKey,
      since time.Time,
  ) ([]domain.RateValue, error)
  ```
  - `domain.SourcePairKey` is a new value object:
    `struct { SourceName, BaseCurrency, QuoteCurrency string; Kind domain.RateKind }`.
    Place it in `internal/domain/` next to existing rate models.
  - Implementation uses one SQL statement with a parametrised
    `WHERE (source_name, base_currency, quote_currency, kind) IN (...)`
    expansion plus `AND timestamp >= ?` and
    `ORDER BY timestamp ASC, id ASC`.
  - All table and column names go through the existing repo `const`
    declarations. No string literals.
  - Runs inside `r.db.Transaction(ctx)`.
- **Acceptance Criteria.**
  - `TestRateValueRepository_ObtainValuesForPairsSince` covers:
    empty pairs slice → empty result, no error; single pair single
    match; single pair multiple matches across timestamps in
    ascending order; multiple pairs interleaved correctly; `since`
    filter excludes older rows; rows for deleted sources do not
    appear (FK-cascade behaviour).
  - Fixtures use the existing SQLite test harness pattern
    (see `internal/repository/ratevalue_test.go`).
- **Pitfalls & edge cases.**
  - Empty `pairs` short-circuits without hitting the DB to avoid an
    invalid `IN ()` clause.
  - SQLite expression-tree limit is ~1000 terms by default. Realistic
    users are far below; add a code comment so a future reader knows
    the limit exists if chunking ever becomes necessary.
- **Complexity.** Medium.

### Task 3: Application service `internal/application/chart/`

- **Description.** New package with a single exported service:
  ```go
  type Service struct { /* subs loader, values loader, now func */ }

  type SubscriptionsLoader interface {
      ObtainRateUserSubscriptionsByUserID(
          ctx context.Context, userType domain.UserType, userID string,
      ) ([]domain.RateUserSubscription, error)
  }

  type ValuesLoader interface {
      ObtainValuesForPairsSince(
          ctx context.Context, pairs []domain.SourcePairKey, since time.Time,
      ) ([]domain.RateValue, error)
  }

  func NewService(subs SubscriptionsLoader, values ValuesLoader,
      now func() time.Time) *Service

  func (s *Service) ObtainMeChart(
      ctx context.Context, userID string,
  ) (*MeChart, error)
  ```
  Pipeline inside `ObtainMeChart`:
  1. Load user subscriptions.
  2. Dedupe to unique `(base, quote, kind)` triples using
     `ratepair.Dedupe`.
  3. For each unique triple, collect the matching `source_name`s
     (a triple may be served by multiple sources — all contribute to
     a single sparkline row).
  4. `since = s.now().Add(-ratepair.ChartWindow)`.
  5. Call `values.ObtainValuesForPairsSince` once with the bulk pair
     list.
  6. Group returned values by `(base, quote, kind)`.
  7. For each group: when multiple sources have data at the same
     timestamp, pick the latest by source contribution timestamp; when
     timestamps differ, all values join the series sorted ascending.
  8. Downsample: divide the window into N equal time buckets
     (`N = 12` — a package-level const, tunable later); for each
     bucket take the **last** sample's value. Empty buckets carry the
     previous bucket's value (continuous line); leading empties before
     the first sample are omitted.
  9. Compute `delta_pct = (last - first) / first * 100` over the
     downsampled points; if fewer than two points, `delta_pct = 0`
     and the row is marked sparse so the renderer draws a flat line.
  10. Color via `ratepair.ColorForPair(base)`, category via
      `ratepair.CategoryOf(base)`.
  11. Sort rows via `ratepair.Less`.
  12. Return `MeChart{Pairs: ...}`.
- **Acceptance Criteria.**
  - Table-driven test covering: empty subscriptions, single pair with
    dense data, single pair with sparse data (one point), single pair
    with zero data in the window, multiple pairs in expected sort
    order, BID and ASK for same pair sit adjacent, dedupe of
    `(base, quote, kind)` collapsed across sources, delta sign math,
    metal vs fiat categorisation, bucket downsampling.
  - Mocks satisfy the loader interfaces with explicit
    `var _ SubscriptionsLoader = &fakeSubs{}` assertions.
  - No `time.Now()` calls inside the service body — only `s.now()`.
- **Pitfalls & edge cases.**
  - The "latest across sources at the same timestamp" rule is
    Phase-2 specific. Phase 1's notifier extremum (`MAX` for BID,
    `MIN` for ASK) does not apply here — different layer, different
    rule.
  - Bucket boundaries align to `since` and step by
    `ChartWindow / N`. Boundary convention: left-closed, right-open.
    Document in godoc.
  - The "no data" state surfaces as
    `Sparse=true, Points=nil, Latest=0, DeltaPct=0`. The handler still
    emits the row so the subscribed pair appears on the list.
- **Complexity.** Hard.

### Task 4: DTO, route, and handler

- **Description.**
  - Add to `internal/dto/` (a new `me_rates.go` if `rates.go` is

    ```go
    type MeChartResponse struct {
        Window string             `json:"window"` // e.g. "7d"
        Pairs  []MeChartPairSeries `json:"pairs"`
    }

    type MeChartPairSeries struct {
        Base     string         `json:"base"`
        Quote    string         `json:"quote"`
        Kind     string         `json:"kind"`     // "BID" | "ASK"
        Label    string         `json:"label"`    // "USD/KZT" or "KZT/USD"
        Color    string         `json:"color"`

        Latest   float64        `json:"latest"`
        DeltaPct float64        `json:"delta_pct"`
        Sparse   bool           `json:"sparse"`
        Points   []MeChartPoint `json:"points"`
    }

    type MeChartPoint struct {
        Timestamp time.Time `json:"timestamp"`
        Value     float64   `json:"value"`
    }
    ```
  - Add to `internal/gateway/httpV1/routes/routes.go`:
    `MeRatesChart = "/api/me/rates/chart"`.
  - Add `GetMeRatesChart(w, r)` handler in
    `internal/gateway/httpV1/handlers/handlers.go` next to
    `ListMeSubscriptions`. The handler:
    1. Reads `X-Telegram-Init-Data` header (no query-string fallback
       per CLAUDE.md privacy policy).
    2. Calls `h.validateInitData(initData, h.botToken,
       meChartMaxAge, h.nowFn())`. Reuse `meSubscriptionsMaxAge` if it
       already exists; otherwise define a sibling.
    3. Converts the returned `int64` to `string` and calls
       `ObtainMeChart`.
    4. Builds the JSON DTO. BID rows: `Label = base + "/" + quote`.
       ASK rows: `Label = quote + "/" + base`. The label flip is the
       handler's concern, not the service's.
    5. 401 on validation failure; 500 with the fallback error message
       on internal failure (per CLAUDE.md error policy).

    `MeSubscriptions` registration site).
- **Acceptance Criteria.**
  - Handler test covers: missing header → 401; invalid HMAC → 401;

    service-layer error returns 500 with the fallback message.
  - The handler test injects a fake `validateInitData` and a fake
    chart service.
  - Label-flip test asserts that for an ASK input
    `(base=USD, quote=KZT, kind=ASK)` the output is
    `label="KZT/USD"`.
  - DTO JSON round-trip test that decodes a response and matches the
    fixture.
- **Pitfalls & edge cases.**
  - The init-data header must not be logged. Verify the access-log
    writer omits it (the existing access log middleware already does;
    do not regress).
  - Use `internal.NewPublicError` only for the auth-failed branch;
    everything else returns plain errors so the controller uses the
    fallback string.
- **Complexity.** Medium.

## Risks

- **WASM testing.** The `cmd/wasm/ui` tests build under regular
  `GOOS`/`GOARCH` because they only manipulate strings. Confirm during
  Task 6 that new tests don't pull in `syscall/js` symbols; gate with
  build tags if they do.
- **Telegram theme fallback colors.** If the Mini App is opened in a
  client that does not pass theme variables, the page must remain
  legible. Fallback colors per the pitfall note are mandatory.
- **Performance under many subscriptions.** A user with 50
  subscriptions across 10 sources produces an IN-list of up to 500
  tuples — well under SQLite's 1000-term default. Log timing in the
  handler so we have data if a pathological user appears.
- **Color collision across pairs sharing a base.** `USD/KZT` and
  `USD/RUB` get the same color. Acceptable for this iteration; revisit
  if real complaints arrive.

## Trade-offs

- **One ratepair package vs split ratepair + ratecolor.** Chose one.
  Rationale: pair-category logic and pair-color logic are the same
  cognitive unit ("how we classify and present a pair"); two packages
  would force an import for one constant.
- **Hardcoded 7-day window vs `?period=` parameter.** Chose hardcoded.
  Rationale: UI drops the period selector for this iteration; a query
  parameter that only ever takes one value is dead weight. The
  `ratepair.ChartWindow` constant makes a future change trivial.
- **Last-in-bucket vs median for downsampling.** Chose last-in-bucket.
  Rationale: the rightmost sparkline point reads as "current price",
  which last-in-bucket delivers naturally and median does not. FX
  rates are low-noise so median's outlier suppression isn't worth its
  semantic distortion.
- **`ColorForPair` hashes `base` only.** Chose base-only hash.
  Rationale: real users typically hold one quote currency per base;
  hashing `base+"/"+quote` solves a problem nobody has yet. Easy to
  swap later — change is in one function.
- **Immediate removal of the overlay endpoint vs keep as a fallback.**
  Chose immediate removal. Rationale: only one consumer exists (the
  WASM frontend), and that consumer is also being rewritten in this
  PR. Dead code adds cognitive load and would rot.
