# Task Breakdown

## Overview

Rework the Telegram notification header so:
1. The first line is `#TAG1 #TAG2 FX rates` — one hashtag per distinct
   trigger type in the part (`#DELTA`, `#INTERVAL`, `#DAILY`, `#CRON`). The
   tags are clickable in Telegram clients, enabling per-trigger filtering.
2. The `📊` and the `Δ`/`⏰` badge row are removed — hashtags subsume them.
3. The second line is just the timestamp, with no `🕒` glyph.
4. The timestamp is rendered in the *user's* timezone, not UTC. Timezone is
   stored per user in a new `rate_user_profiles` table and captured from the
   Mini App on first visit via `Intl.DateTimeFormat().resolvedOptions().timeZone`.

The default fallback (no profile row or invalid IANA name) stays UTC so the
notifier never produces an unrenderable timestamp.

Example after change (assuming user TZ=Asia/Almaty, UTC+5):

```
#DELTA #INTERVAL FX rates
Sun 24 May, 14:57 +05

<pre>
USD/KZT  470.46  +2.60 ↑
</pre>
```

## Assumptions

- Telegram hashtags are uppercase ASCII — `#DELTA`, `#INTERVAL`, `#DAILY`,
  `#CRON`. Mixed-case (`#Delta`) would still be clickable but harder to
  filter on consistently; uppercase is the standard idiom.
- Tag order is canonical: `DELTA` first, then schedule-family in alphabetical
  order (`CRON`, `DAILY`, `INTERVAL`). Deterministic ordering keeps tests
  stable and avoids reorder-only diffs in a Telegram chat history.
- Per-user timezone is stored as the IANA name (e.g. `Asia/Almaty`,
  `Europe/Moscow`). Validation happens once at write time via
  `time.LoadLocation`; the read path treats lookup failure as "fall back
  to UTC" and never errors.
- The Mini App pickup endpoint accepts the timezone unconditionally and
  upserts the row. We trust the client because the worst outcome is a
  wrong timestamp in the user's own notifications.
- We embed `time/tzdata` in the `notifier` and `web` binaries so the
  deploy host does not need system tzdata installed. `notifier` is the
  primary consumer; `web` validates incoming timezones on the upsert.
- No backfill of historical events. New format applies from migration
  forward.

## Tasks

### Task 1: Migration for `rate_user_profiles`

- Description: Add a new table holding one row per (user_type, user_id) with
  IANA timezone and updated_at/created_at columns.
- Acceptance Criteria:
  - File `migrations/202605.010.rate_user_profiles.table_initiate.sql`
    creates the table with a composite PK `(user_type, user_id)`.
  - `migrator` applies it cleanly against a fresh DB and against the
    current production schema (idempotent re-run is no-op).
- Pitfalls & edge cases:
  - Filename is immutable once shipped — do not bikeshed.
  - SQLite has no native `TIMESTAMP`; use `TEXT` with ISO-8601 like the
    sibling tables.
- Complexity: Easy

### Task 2: Domain + repository for user profile

- Description: Add `RateUserProfile{UserType, UserID, Timezone, ...}` to
  `internal/domain`. Add `UserProfileRepository` to `internal/repository`
  with `UpsertRateUserProfile` and `ObtainRateUserProfileByUserID` methods.
- Acceptance Criteria:
  - Domain struct exists with godoc.
  - Repository follows the existing pattern (transactional queries,
    `const` table/column names, `var _ interface =` assertions in tests).
  - Upsert is `INSERT ... ON CONFLICT(user_type, user_id) DO UPDATE`.
  - Read returns `nil, internal.ErrNotFound` when the row is absent
    (caller treats that as "use UTC").
- Pitfalls & edge cases:
  - Composite PK requires named conflict target.
  - Validate timezone server-side (`time.LoadLocation`) on Upsert; reject
    bad input with `PublicError`.
- Complexity: Medium

### Task 3: HTTP endpoint for Mini App pickup

- Description: Add `POST /api/me/profile` (or extend an existing route)
  accepting `{"timezone": "Asia/Almaty"}` and upserting the profile row.
  Authenticated via Telegram WebApp HMAC like the other `/api/me/*`
  routes.
- Acceptance Criteria:
  - Route registered, handler tested via the same fake initData
    validator pattern used by `ListMeSubscriptions`.
  - Empty / invalid timezone → 400 with `PublicError("invalid timezone")`.
  - Success → 204 No Content.
- Pitfalls & edge cases:
  - Endpoint must not echo the user-supplied name back without validation
    — `time.LoadLocation` enforces both syntax and IANA-name presence.
  - Rate limit / size limit on the body is implicit (it's small JSON).
- Complexity: Medium

### Task 4: WASM client — capture & POST timezone

- Description: On Mini App mount, read
  `Intl.DateTimeFormat().resolvedOptions().timeZone` and POST it to
  `/api/me/profile` once, fire-and-forget. Errors are console.warn-only.
- Acceptance Criteria:
  - WASM main flow calls the upload exactly once per page mount.
  - Failure to upload does not block any other rendering.
  - Add an apiclient helper `UpdateProfile(ctx, initData, tz)` for
    testability.
- Pitfalls & edge cases:
  - `Intl` is available in every browser the Mini App supports; no
    fallback needed but guard with `IsUndefined()` defensively.
  - Skip the POST when the resolved timezone is empty.
- Complexity: Medium

### Task 5: Notification message rebuild

- Description: Rewrite `message.go` to:
  - Replace `📊 FX rates` + badge line with `#TAG... FX rates`.
  - Drop the `🕒` glyph from the timestamp line.
  - Accept an optional `*time.Location` and format the timestamp in it
    using `Mon 2 Jan, 15:04 -07` (ISO-style offset, not the literal `UTC`).
  - `buildAlertMessage` signature gains a `*time.Location` parameter; nil
    means UTC.
  - In `ratecheckagent.go`, look up the user's profile (via the new repo)
    before calling `buildAlertMessage` and pass the resolved `*time.Location`.
- Acceptance Criteria:
  - Header for a delta-only alert: `#DELTA FX rates`.
  - Header for mixed: `#DELTA #INTERVAL FX rates`.
  - Timestamp line carries no `🕒` and uses the user's offset suffix.
  - Update existing tests; add subtests for mixed-trigger tag order
    and timezone formatting.
- Pitfalls & edge cases:
  - `tableRow` width math (renderBlock) doesn't change — header reshape
    is isolated to `headerLines`.
  - The `unicode/utf8` width logic was the original reason to keep arrows
    on the row line; verify the body is unaffected.
- Complexity: Medium

### Task 6: Repository wire-up in `cmd/notifier`

- Description: Construct `UserProfileRepository` in the notifier composition
  root and inject into the rate-check agent.
- Acceptance Criteria:
  - Agent has a non-nil profile-repo field; nil-safe path still works
    in unit tests that don't supply one (default = UTC).
- Pitfalls & edge cases:
  - Inject the repo via a dedicated method, not a struct field rename,
    to keep prior test fixtures compiling.
- Complexity: Easy

## Execution Order

1. Task 1 — migration first (DB schema is the contract).
2. Task 2 — domain + repo (tested in isolation).
3. Task 5 — message rebuild (pure code, no infra deps; `*time.Location`
   parameter defaults to UTC so tests don't need the new repo).
4. Task 3 — HTTP endpoint (depends on repo).
5. Task 4 — WASM client (depends on endpoint).
6. Task 6 — notifier wiring (depends on repo + message rebuild).

## Risks

1. **Existing tests will need updates** — the badge-glyph and `🕒` assertions
   in `message_test.go` are now wrong. Update them, don't suppress.
2. **`time.LoadLocation` failure on the host** — guarded by importing
   `_ "time/tzdata"` in `cmd/notifier` and `cmd/web` so the lookup is
   binary-self-contained.
3. **Backward compat with current notifications in-flight** — the format
   change is per-message, not per-row. In-flight queued events will look
   inconsistent for ~one tick, then settle. Acceptable.
4. **Hashtag scraping ambiguity** — `#DELTA` could collide with unrelated
   user-content tags in the same chat. Low probability: the bot owns its
   conversation and the user picks them deliberately. Tag list is a
   closed set.

## Trade-offs

- **Per-user TZ vs single bot TZ**: Per-user is more work (table, endpoint,
  WASM pickup) but doesn't lock the deployment to one region. User asked
  for this explicitly; recording the rationale.
- **One row per user vs one per subscription**: One per user — the
  timezone is a property of the human, not the subscription. Composite PK
  `(user_type, user_id)` keeps the door open to non-Telegram channels later.
- **Tag list is closed**: We do NOT let users define custom tags. Tags
  identify the trigger type, not the source or the user's intent.
- **Tag ordering deterministic**: Alphabetical by family — Δ-only goes
  first when present, schedule-family in alphabetical order. Stable
  ordering makes tests deterministic and avoids visual jitter in chat
  history when the same alert recurs.
