# Task Breakdown — Telegram Mini App: My Subscriptions (read-only v1)

## Overview

Add a Telegram Bot main-menu button that opens a Telegram WebApp (Mini App) showing
**only the calling user's own subscriptions** with the latest rate value and last grab
timestamp per source. Read-only for v1: no add/delete from the Mini App. Must be paginated
(10 per page) and searchable across the user's own subscriptions.

The page is served from the existing `cmd/web` binary's embedded static file system. A new
HTTP endpoint returns the subscription page enriched with the latest rate value and
timestamp for each subscription, scoped to the Telegram user identified by the WebApp
`initData` HMAC-SHA256 signature using the bot token as the key (per Telegram WebApp
specification).

The implementation reuses existing repositories (`ObtainRateUserSubscriptionsByUserID`,
`ObtainLastNRateValuesBySourceName`, `ObtainRateSourceByName`). No DB schema change, no
new env var, no new binary.

## Assumptions

A1. **`API_DSN` does NOT exist today.** The user's brief says the public URL "comes from
    an existing env var `API_DSN` parsed via `dsninjector.Unmarshal` (already used
    elsewhere in `cmd/web/main.go`)" — verification of `cmd/web/main.go` shows only
    `SQLITEDB_DSN` and `TELEGRAMBOT_DSN`. There is no `API_DSN`, no `dsnAPI`, and no
    `Addr()` call in any cmd. The brief also says "don't introduce a new env var."
    These two requirements are mutually exclusive. **The plan resolves this by introducing
    `API_DSN` as a new env var parsed exactly the same way as the other two**, because
    the bot needs an externally-resolvable `https://` origin (Telegram refuses
    `http://`/loopback/IP-only WebApp URLs). The architect flags this as the single
    unavoidable deviation from the brief and recommends the user confirm before the
    engineer starts. A fallback option (Task 1.b) accepts the deviation cost: if the user
    insists on no new env var, the bot side simply skips the WebApp button when no public
    URL can be derived. **Awaiting user decision; default plan adds `API_DSN`.**

A2. **Bot token plumbing.** The bot token already lives inside
    `*integration.TelegramBotClient` (parsed from `TELEGRAMBOT_DSN.Addr()`), but is not
    exposed by the type. The plan adds a `BotToken() string` accessor and wires the token
    from `cmd/web/main.go` into the new HTTP handler via constructor injection — the
    handler does not re-read `TELEGRAMBOT_DSN`.

A3. **Telegram chat_id == Telegram user_id for direct chats.** This codebase already
    treats the chat_id as the subscription `user_id` (see
    `telegramapi.go:194 strconv.FormatInt(chatID, 10)`). For DM-only bots that equality
    holds. The Mini App's `initData.user.id` is the Telegram **user** ID and matches the
    same value, so the existing rows in `rate_user_subscriptions` (where
    `user_type='telegrambot'`, `user_id` is the chat ID stringified) will match without
    migration.

A4. **WebApp `initData.auth_date` skew tolerance: 24 hours.** Per the user's brief
    ("e.g. >24h"). Mini Apps stay open indefinitely after launch, so a tighter window
    breaks the UX. 24h is consistent with Telegram's own guidance.

A5. **Search is case-insensitive substring match across `source.title`,
    `source.name`, and `source.base_currency || '/' || source.quote_currency`.** This
    matches what the bot shows in `handleAddSourceList` and is what users will recognise.

A6. **Rendering grouping: by source title, sorted by most-recent grab.** Each row shows
    label/title, latest price, pair, condition, and "last grab" timestamp. A user may
    have multiple subscriptions on different conditions for the same source — they
    collapse into one row that lists each condition as a small badge. (Justified in
    Trade-off T2.)

A7. **One single embedded HTML file with vanilla JS+CSS, no build step, no WASM for v1.**
    Justified in Task 1 / Trade-off T1.

A8. **No CSRF protection beyond the HMAC signature itself.** The HMAC of `initData`
    already provides authenticated request origin from Telegram. The endpoint is GET-only
    and read-only, so CSRF surface is minimal. Adding CORS hardening is left for v2.

### Task 3: Implement Telegram WebApp `initData` HMAC validator

- **Description:** Create `internal/tools/tgwebapp/initdata.go` with one exported
  function:
  ```go
  // ValidateInitData verifies that initData was signed by Telegram with botToken,
  // returns the parsed user.id, and rejects payloads older than maxAge.
  // Uses HMAC-SHA256 with secret_key = HMAC_SHA256("WebAppData", botToken).
  func ValidateInitData(initData, botToken string, maxAge time.Duration, now time.Time) (userID int64, err error)
  ```
  Algorithm (per Telegram WebApp spec):
  1. URL-decode `initData` into key=value pairs.
  2. Extract and remove `hash`. The remaining pairs build `data_check_string`:
     keys sorted lexicographically, joined as `k=v` with `\n` separators.
  3. `secret_key = HMAC_SHA256("WebAppData", botToken)` (note: key is the literal
     string "WebAppData", token is the message — common implementation mistake).
  4. `expected_hash = HMAC_SHA256(data_check_string, secret_key)`, hex-encoded.
  5. `subtle.ConstantTimeCompare(expected_hash, hash)` — never `==`.
  6. Parse `auth_date` as Unix seconds; reject if `now.Sub(auth) > maxAge`.
  7. Parse `user` (URL-decoded JSON) and return its `id` as int64.
- **Acceptance Criteria:**
  - [ ] Public function signature is exactly as above. `now` is injected (not
        `time.Now()`) so tests are deterministic.
  - [ ] `internal/tools/tgwebapp/initdata_test.go` covers, as subtests of one
        `TestValidateInitData`:
        - happy path (valid hash, fresh `auth_date`, valid `user` JSON) → returns user
          ID, nil error.
        - tampered field after sign → error containing `"hash mismatch"`.
        - missing `hash` field → error containing `"missing hash"`.
        - `auth_date` older than `maxAge` → error containing `"expired"`.
        - missing or malformed `user` JSON → error containing `"user"`.
        - empty/blank `initData` → error.
        - empty `botToken` → error (defensive — never validate against empty key).
  - [ ] Uses `crypto/hmac`, `crypto/sha256`, `crypto/subtle`. No third-party deps.
  - [ ] Tests use `t.Parallel()` and `testify/require`.
  - [ ] One `TestValidateInitData` top-level function, all scenarios as `t.Run` subtests
        (per CLAUDE.md "One `Test*` per method" rule).
- **Pitfalls & edge cases:**
  - The two HMAC arguments are commonly swapped. The token is the **message**, the
    string `"WebAppData"` is the **key** for the first HMAC. Get this wrong and every
    real Telegram payload will be rejected.
  - `initData` arrives URL-encoded; values with `=` or `&` in them (the `user` JSON has
    both) must be `url.QueryUnescape`d **after** splitting on `&` and on the first `=`,
    not before.
  - Comparing hex strings with `==` leaks timing. Use `subtle.ConstantTimeCompare` on
    the decoded byte slices (or on equal-length hex strings).
  - `auth_date` is Unix seconds (int64), not RFC3339. A negative `now.Sub(auth)` (clock
    skew) should still pass — only positive ages exceeding `maxAge` are rejected.
- **Complexity:** Medium

#### Code example

```go
package tgwebapp

import (
    "crypto/hmac"
    "crypto/sha256"
    "crypto/subtle"
    "encoding/hex"
    "encoding/json"
    "errors"
    "fmt"
    "net/url"
    "sort"
    "strconv"
    "strings"
    "time"
)

func ValidateInitData(initData, botToken string, maxAge time.Duration, now time.Time) (int64, error) {
    if initData == "" {
        return 0, errors.New("tgwebapp: empty initData")
    }
    if botToken == "" {
        return 0, errors.New("tgwebapp: empty botToken")
    }

    values, err := url.ParseQuery(initData)
    if err != nil {
        return 0, fmt.Errorf("tgwebapp: parse initData: %w", err)
    }
    gotHash := values.Get("hash")
    if gotHash == "" {
        return 0, errors.New("tgwebapp: missing hash")
    }
    values.Del("hash")

    keys := make([]string, 0, len(values))
    for k := range values {
        keys = append(keys, k)
    }
    sort.Strings(keys)
    var sb strings.Builder
    for i, k := range keys {
        if i > 0 {
            sb.WriteByte('\n')
        }
        sb.WriteString(k)
        sb.WriteByte('=')
        sb.WriteString(values.Get(k))
    }

    secretKey := hmac.New(sha256.New, []byte("WebAppData"))
    secretKey.Write([]byte(botToken))
    mac := hmac.New(sha256.New, secretKey.Sum(nil))
    mac.Write([]byte(sb.String()))
    expected := hex.EncodeToString(mac.Sum(nil))

    if subtle.ConstantTimeCompare([]byte(expected), []byte(gotHash)) != 1 {
        return 0, errors.New("tgwebapp: hash mismatch")
    }

    authUnix, err := strconv.ParseInt(values.Get("auth_date"), 10, 64)
    if err != nil {
        return 0, fmt.Errorf("tgwebapp: invalid auth_date: %w", err)
    }
    if age := now.Sub(time.Unix(authUnix, 0)); age > maxAge {
        return 0, fmt.Errorf("tgwebapp: expired (age %s > %s)", age, maxAge)
    }

    var user struct{ ID int64 `json:"id"` }
    if err := json.Unmarshal([]byte(values.Get("user")), &user); err != nil {
        return 0, fmt.Errorf("tgwebapp: parse user: %w", err)
    }
    if user.ID == 0 {
        return 0, errors.New("tgwebapp: missing user.id")
    }
    return user.ID, nil
}
```

### Task 6: Add `GET /api/me/subscriptions` endpoint

- **Description:** New JSON endpoint returning the caller's own subscriptions enriched
  with the latest rate value and timestamp per source. **All identification is via
  Telegram WebApp `initData`** — the user does not pass a user ID directly.
  - Route: `GET /api/me/subscriptions`
  - Auth: header `X-Telegram-Init-Data: <initData>` (querystring is also accepted as
    fallback `?initData=...` because some Mini App contexts strip headers).
  - Query params: `q` (search string, optional), `page` (1-based, default 1),
    `page_size` (default 10, max 50 — guarded).
  - Response: `200 OK` JSON `{ items: [...], page, page_size, total }`.
  - Errors: `401 Unauthorized` for invalid/missing/expired initData;
    `400 Bad Request` for malformed query params; `500 Internal Server Error` via the
    existing `h.internalError` helper.
- **Acceptance Criteria:**
  - [ ] New route constant `MeSubscriptions = "/api/me/subscriptions"` in
        `internal/gateway/httpV1/routes/routes.go`.
  - [ ] New handler `(*Handler).ListMeSubscriptions` registered in
        `internal/gateway/httpV1/router.go` BEFORE the `/api/sources/...` block (no
        prefix conflict — `/me` and `/sources` differ — but maintain the file's existing
        registration grouping).
  - [ ] New DTO in `internal/gateway/httpV1/dto/dto.go`:
        ```go
        type MeSubscriptionRow struct {
            SourceName    string  `json:"source_name"`
            SourceTitle   string  `json:"source_title"`
            BaseCurrency  string  `json:"base_currency"`
            QuoteCurrency string  `json:"quote_currency"`
            Conditions    []string `json:"conditions"` // e.g. ["delta:5", "interval:6h"]
            LatestPrice   float64 `json:"latest_price,omitempty"`
            LatestAt      string  `json:"latest_at,omitempty"` // RFC3339; "" if no data
        }
        type MeSubscriptionsResponse struct {
            Items    []MeSubscriptionRow `json:"items"`
            Page     int64               `json:"page"`
            PageSize int64               `json:"page_size"`
            Total    int64               `json:"total"`
        }
        ```
        Note: `user_id` is NEVER returned (matches existing
        `SubscriptionDetailResponse` precedent at `dto.go` — comment explicitly says
        "UserID is never included").
  - [ ] Handler reads `X-Telegram-Init-Data` header, falls back to `?initData=` query
        param, calls `tgwebapp.ValidateInitData(...)`. On error returns `401` with
        `{"error":"unauthorized"}` (a single body shape — never echo the underlying
        message: it would leak HMAC details).
  - [ ] Handler converts `userID int64` → string and calls
        `subRepo.ObtainRateUserSubscriptionsByUserID(ctx, domain.UserTypeTelegram,
        strconv.FormatInt(userID, 10))`.
  - [ ] Subscriptions are grouped by `SourceName` into `MeSubscriptionRow` items.
        For each unique source: fetch the source via `ObtainRateSourceByName` (for
        `Title`, `BaseCurrency`, `QuoteCurrency`) and the latest rate via
        `ObtainLastNRateValuesBySourceName(ctx, name, 1)`. **Sequential
        per-source calls are acceptable for v1**: the user's brief specifies 10
        subscriptions per page, so the worst case is 10 SQLite calls per request,
        which is fine for SQLite. (Justified in Trade-off T3.)
  - [ ] Search filter `q` is applied **before pagination** as case-insensitive
        substring against `SourceTitle`, `SourceName`, and `BaseCurrency + "/" +
        QuoteCurrency`.
  - [ ] Pagination: 10 per page default, clamp to [1, 50]. `Total` is the count
        of distinct sources after `q` filtering.
  - [ ] Handler test covers, as subtests of `TestHandler_ListMeSubscriptions`:
        - `t.Run("rejects missing initData", ...)` → 401.
        - `t.Run("rejects bad hash", ...)` → 401.
        - `t.Run("happy path returns only caller's subs", ...)` — uses a fixture
          where two users have subscriptions; only the caller's are returned.
        - `t.Run("search filters by title", ...)`.
        - `t.Run("paginates", ...)` — fixture with 12 subs, page=2 returns 2 items.
  - [ ] Handler tests use a fake `tgwebapp` validator (inject the validator as a
        function field on the handler so tests don't need real signatures). See
        Pitfalls.
- **Pitfalls & edge cases:**
  - Hard-coding `tgwebapp.ValidateInitData` in the handler makes it untestable. Inject
    it as a function variable on `Handler`:
    ```go
    type Handler struct {
        rateService
        validateInitData func(initData, botToken string, maxAge time.Duration, now time.Time) (int64, error)
        botToken         string
        nowFn            func() time.Time
    }
    ```
    Default the function to `tgwebapp.ValidateInitData` and `nowFn` to `time.Now` in
    `NewHandler`. Override in tests.
  - The current `NewHandler(srvRate rateService)` signature is breaking: callers in

    EUR/KZT both as delta-5 and interval-6h) collapse into the same row with
    `Conditions: ["delta:5", "interval:6h"]`. Don't show one row per subscription.
  - When a source has zero rate values, `ObtainLastNRateValuesBySourceName` returns

  - SQLite latency for 10 sequential queries on a small DB is sub-millisecond, but

  - `ObtainRateSourceByName` returns `(*domain.RateSource, error)` and may return

## Risks

- **R1 — `API_DSN` ambiguity (already flagged in Assumption A1).** If the user wants
  zero new env vars, Tasks 2 and 5 collapse to "skip the WebApp button" and the Mini
  App is reachable only by typing the URL manually — defeats the purpose. **Block on
  user confirmation before Task 2 starts.**

- **R2 — HMAC implementation off-by-one.** Telegram's spec is unforgiving: any whitespace,
  any encoding mismatch, any wrong key order in the data-check string will silently fail
  100 % of real Telegram traffic while passing fabricated tests. Mitigate by adding one
  test fixture using a real `initData` payload from Telegram's Bot API documentation
  (the spec page contains a worked example). Cross-check the hex output byte-for-byte.

- **R3 — Telegram URL constraints.** WebApp URLs must be HTTPS, publicly reachable, and
  cannot be IP literals or localhost. If the deployment environment doesn't satisfy
  this, the button will be silently rejected by Telegram's client (no error feedback to
  the bot). Mitigation: document this in `CLAUDE.md` and include a manual smoke test in
  Task 7's acceptance.

- **R4 — Token leakage via logs.** Both `tbotapi.BotAPI`'s debug mode and any
  `errors.Join(err, ...)` chain that includes the token would be catastrophic.
  Mitigation: never include the token in error messages from `tgwebapp.ValidateInitData`,
  and verify `bot.Debug = false` (already set at `tbotclient.go:44`).

- **R5 — Telegram chat_id ≠ user_id assumption.** Holds for direct chats today but if
  the bot is ever added to groups, subscriptions stored under the group chat ID will
  not appear in the Mini App (because Mini Apps return user IDs, not chat IDs). The
  bot is currently DM-only and the brief is DM-scoped, so this is acceptable. Document
  in `CLAUDE.md` and add a TODO comment near the user-ID conversion.

- **R6 — Adding `API_DSN` may break the existing deploy.** Existing systemd units and
  `.env` files will lack `API_DSN` and the binary will refuse to start. Mitigation:
  release notes; alternatively make `API_DSN` optional and degrade gracefully (no
  WebApp button if absent) — recommended fallback.

## Trade-offs

- **T1 — Vanilla JS over WASM (Task 1).** The wasm binary is ~2 MB minimum and the page
  needs ~200 lines of DOM. Wasm pays its overhead when there's CPU-bound logic
  (charts, parsing) or when typed Go code prevents bugs in complex flows; neither
  applies here. The choice is reversible: the API endpoint is unchanged whether the
  client is JS or wasm. If the user later wants to migrate the entire `cmd/web/static/`
  surface to wasm, the Mini App page is one of the smallest stones to roll.

- **T2 — Group by source title, not by currency pair (Assumption A6).** A user
  subscribed to Halyk Bank EUR/KZT and Kaspi EUR/KZT cares more about which bank
  ("source") than which pair, because the pair is fixed by their interest in EUR/KZT
  generally. Grouping by pair would mix banks under one row, which loses information
  the user wants (which bank's rate is best?). The trade-off: if a user has many sources
  for the same pair (rare today), the page is longer than necessary.

- **T3 — N+1 DB calls per page load (Task 6).** The handler does 10 sequential SQLite
  queries (one `ObtainRateSourceByName` and one `ObtainLastNRateValuesBySourceName` per
  source — actually 20). For SQLite on the same machine, ~50 µs each → ~1 ms total,
  invisible to the user. A single JOIN query would be ~10× faster but adds a new
  repository method and SQL surface to maintain. Defer until measured P95 > 50 ms or
  per-page subscription count exceeds 50. Document the threshold so the v2 ticket has
  a clear trigger.

- **T4 — Auth via header (preferred) with query-string fallback (Task 6).** Headers are
  the correct place for auth credentials, but some webview environments strip custom
  headers from `fetch()`. Accepting the same value as `?initData=` query is a small
  hedge. The HMAC validates whichever transport carried the value, so security is
  unaffected. Cost: two code paths in the handler (one extra `if`).

- **T5 — `BotToken()` accessor exposes a secret (Task 4).** Unavoidable: the handler
  must HMAC against it. The accessor is unexported in the sense that it's only
  reachable inside this binary; nothing serializes the result. The alternative —
  re-reading `TELEGRAMBOT_DSN` in the handler — duplicates env parsing and violates
  the project's existing dependency injection convention.
