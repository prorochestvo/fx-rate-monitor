# Task Breakdown

## Overview

Collapse BID/ASK rows for the same canonical currency pair into a single
row containing both lines on one sparkline. Plan 009 shipped a row-per-
direction model (`USD/KZT` and `KZT/USD` as two adjacent rows with their
own sparklines and pair-hashed colors). This plan replaces that with a
pair-grouped model where BID and ASK render as two distinct series
sharing one Y-axis frame.

User-visible changes:

- **Color dots are removed** entirely (saves horizontal space).
- **Role-based colors**: BID is teal (`#1D9E75`), ASK is blue
  (`#378ADD`), uniform across all pairs. The per-pair hash colors from
  plan 009 (`ratepair.ColorForPair`, `FiatPalette`, `MetalPalette`)
  stay in the package for the future drill-in view, but are not used
  by this rendering path.
- **Two-line row layout**:
  ```
  USD/KZT                        Δ 0.29%
  B 487.55 +3.6%  A 488.95 +3.4%
  ```
  Upper-right `Δ` is the live relative spread
  `(ASK_latest - BID_latest) / BID_latest * 100`, shown only when both
  directions exist.
- **Single-direction rows** keep the same compact form using the full
  `BID`/`ASK` prefix (there is room):
  ```
  USD/KZT
  BID 487.55 +3.6%
  ```
- **Pair label is always canonical natural BID-form** (`USD/KZT`), even
  when only ASK is subscribed. The plan-009 label-flip rule (ASK →
  `QUOTE/BASE`) is removed.

## Assumptions

- BID and ASK for the same `(base, quote)` collapse into ONE row with
  two series.
- Canonical pair identity uses uppercased
  `min(base,quote)/max(base,quote)`.
- Pair display label is BID-natural direction (`base/quote` from the
  BID subscription if present, otherwise `quote/base` from the ASK
  subscription).
- Spread is **relative only**, formatted as `Δ X.XX%`. Computed when
  both BID and ASK series have a non-zero `Latest`. Otherwise omitted.
- Per-series period delta (`+3.6%`) remains
  `(last-first)/first*100` over the 7-day window — unchanged from plan
  009.
- Y-axis auto-fits per row across the union of all series values in
  that row. Baseline (zero line) drawn once if zero is in range.
- All-series-sparse → existing "no data" badge.
- Mixed-sparse (one direction has data, the other has < 2 points):
  draw both lines; the sparse direction is a flat horizontal line at
  its current value in its role color.

### Task 2: Redesign DTO

`internal/dto/me_rates.go` — replace the old shape entirely:

```go
type MeChartResponse struct {
    Window string           `json:"window"`
    Pairs  []MeChartPairRow `json:"pairs"`
}

type MeChartPairRow struct {
    Pair      string          `json:"pair"`       // "USD/KZT" canonical natural BID-style
    Category  string          `json:"category"`   // "fiat" | "metal"
    SpreadPct *float64        `json:"spread_pct,omitempty"`
    Series    []MeChartSeries `json:"series"`     // 1 or 2 entries
}

type MeChartSeries struct {
    Kind     string         `json:"kind"`      // "BID" | "ASK"
    Color    string         `json:"color"`     // ColorBid or ColorAsk
    Latest   float64        `json:"latest"`
    DeltaPct float64        `json:"delta_pct"`
    Sparse   bool           `json:"sparse"`
    Points   []MeChartPoint `json:"points"`
}

type MeChartPoint struct {
    Timestamp time.Time `json:"timestamp"`
    Value     float64   `json:"value"`
}
```

`MeChartPairSeries` (from plan 009) is removed.

### Task 3: Refactor application service

`internal/application/chart/service.go`:

- Group by canonical pair (uppercased `min(base,quote)/max(base,quote)`).
- Determine the `Pair` display label as BID-natural direction: if any
  BID subscription exists for the canonical pair, use
  `<BID.base>/<BID.quote>`. Otherwise (ASK only) use
  `<ASK.quote>/<ASK.base>` (inverted from ASK's storage direction).
- Build 1-2 `MeChartSeries` entries per row (BID and/or ASK).
- BID-before-ASK ordering within `Series` slice (deterministic).
- `SpreadPct`: set only when both directions exist and both have a
  non-zero `Latest`. Formula
  `(ASK.Latest - BID.Latest) / BID.Latest * 100`. Otherwise nil.
- Set `Color` per series from `ratepair.ColorBid` / `ratepair.ColorAsk`.
- Sort rows: fiat-before-metal, then canonical-pair-ascending.
- Update existing service tests for the new grouping; preserve the
  bucket-downsampling, forward-fill, leading-empty-omission coverage
  (those rules don't change).

### Task 5: Renderer rewrite

`cmd/wasm/ui/chart.go`:

- Remove the color-dot element from each row.
- Header line: `<pair label>` left-aligned, `Δ X.XX%` right-aligned
  (only when `SpreadPct` is non-nil).
- Value line:
  - Two-series: `B 487.55 +3.6%  A 488.95 +3.4%`. The `B` glyph is in

    prefix keeps the line compact on a phone screen.
  - Single-series: full prefix `BID 487.55 +3.6%` or `ASK 488.95 +3.4%`,

- Mixed-sparse row: both lines drawn; the sparse direction is a flat
  horizontal line in its role color at its `Latest`.
- Use the existing `strings.Builder` polyline-point pattern from plan
  009; do not regress on allocations.

## Risks

- DTO shape is a breaking change. Every test that asserts on the old
  `MeChartPairSeries` must be rewritten in lockstep, otherwise the
  build will be red for several tasks.
- Single-direction edge case (only BID or only ASK subscribed) must
  have explicit tests in both the service and the renderer — easy to
  forget when grouping is the main feature.
- Pair label rule change (canonical natural BID-form, even for ASK-only)
  flips behavior tested in plan 009. The "ASK → `KZT/USD`" assertion
  is now wrong.
- Role-color uniformity removes the "this is the USD/KZT pair" visual
  identifier. Spatial position + pair text label carry that load now;
  watch the visual review when sanity-checking in Telegram.

## Trade-offs

- **Role colors over pair-hash colors.** Pair-hash colors gave each
  row a unique identifier but produced collisions (the screenshot
  showed USD and KZT landing on the same green by FNV coincidence).
  Role colors clarify the BID/ASK distinction at the cost of unique
  pair identity. Accepted because pair identity is already in the
  label text.
- **Spread is relative only.** Absolute spread (`Δ 1.40`) carries no
  meaning without scale context (1.40 means very different things for
  EUR/USD at 1.08 and GOLD/KZT at 70,000). Relative spread is
  normalized.
- **Pair label always BID-natural.** Plan 009's "ASK rows labeled as
  `QUOTE/BASE`" rule communicated trade direction but doubled the row
  count visually. Single canonical label simplifies the screen at the
  cost of losing the directional hint — the `B`/`A` prefix in the value
  line carries that signal instead.
