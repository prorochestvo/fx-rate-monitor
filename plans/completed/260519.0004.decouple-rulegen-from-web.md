# Task Breakdown

## Overview

Rule generation (LLM-driven extraction-rule synthesis) currently lives in three
places: the standalone `cmd/rulegen` batch binary, an HTTP endpoint exposed by
`cmd/web` (`POST /api/sources/{name}/rules/generate`), and a Telegram `/regen`
bot command served by the same `cmd/web` process. The HTTP and bot entry points
force `cmd/web` to import `internal/application/rulegen` (which transitively
pulls in chromedp) and `internal/infrastructure/artificialintelligence`, and to
read `AI_PRIMARY_DSN` / `AI_FALLBACK_DSN` at startup.

This plan removes both online entry points so that `cmd/web` no longer depends
on any AI client, the rulegen application package, the chromedp fetcher, or the
two AI env vars. Rule (re-)generation continues to work via `cmd/rulegen`,
triggered manually or by a weekly host cron. To support cron usage at scale we
also add `--all` to `cmd/rulegen` so a single invocation iterates every active
row in `rate_sources`, applying the per-source fail-fast / collect-and-continue
pattern already used by `cmd/collector` and `cmd/notifier` (see commit
`3229715`). On per-source failure the binary logs the error and continues;
the process always exits 0 so cron does not page the operator.

No queue table, no admin-trigger UI, no CI/CD invocation, no new env vars on
the web side. The systemd `EnvironmentFile` continues to declare
`AI_PRIMARY_DSN` for `cmd/rulegen`'s benefit — `cmd/web` simply stops reading it.

## Assumptions

- The web HTTP handler `GenerateRules` and the Telegram `/regen` command have
  no other callers (no CI workflow invokes them, no internal tooling shells
  into the bot). This was confirmed by reviewing `.github/workflows/` and the
  current bot command map; if a hidden caller surfaces during review the plan
  must be updated before merge.
- The operator is willing to take a one-time deploy hit: after this ships,
  triggering a rule regeneration requires shell access to the host and running
  `./build/rulegen <name>` or `./build/rulegen --all`. This is a deliberate
  product decision recorded in the goal statement above.
- `cmd/rulegen` will keep its existing single-source behaviour for callers
  that pass a positional `<source-name>`; `--all` is purely additive.
- We deliberately do **not** filter by `active = true` at the database layer
  for `--all`: the existing `RateSourceRepository.ObtainAllRateSources` returns
  every row, and the loop will skip inactive ones in Go. This avoids adding a
  new repository method for what is fundamentally a one-call-per-week tool. If
  reviewer lens E flags this as wasteful, revisit by adding
  `ObtainActiveRateSources` then.
- The `pkg/api.RulegenRequest` / `RulegenResponse` types are only consumed by
  `rulegen_handler.go` / `rulegen_handler_test.go` (verified by grep). They
  are part of the public `pkg/api` surface, so removing them is technically
  a breaking change for any external client — but there is no external client
  inside this repo, and `pkg/api` is consumed only by the WASM frontend which
  has no rulegen UI today. Removal is safe in practice.
- `cmd/web`'s `HttpWriteTimeout` is currently 130 s purely to accommodate the
  rulegen handler's 120 s ceiling. After the handler is gone, the timeout can
  drop back to a normal value. We pick equal-to-`HttpTimeOut` (30 s) for
  symmetry with `ReadTimeout`; this is documented as a deliberate trade-off
  below.
- The `defer release()` lock-manager pattern in `telegramapi_regen.go` is
  rulegen-specific; removing the `/regen` command removes the only consumer
  of `rulegen.LockManager` from the web binary. `cmd/rulegen` itself does
  not need a lock manager because each invocation owns the process.
- All `NewHandler` and `NewTelegramApi` call sites in test files currently
  pass `nil, nil, 0, nil` for the rulegen quadruple (verified by grep). The
  signature change is therefore mechanical — drop the four trailing args at
  each call site.

## Tasks

### Task 1: Add `--all` flag and per-source loop to `cmd/rulegen`

- **Description:** Introduce a `--all` boolean flag on `cmd/rulegen/main.go`.
  When set, the binary must:
  1. Reject any positional argument (usage error, exit 2).
  2. Call `rRateSource.ObtainAllRateSources(ctx)` and iterate the returned
     slice.
  3. Skip rows where `Active == false`, logging the skip at info level.
  4. For each active source, invoke the existing `gen.Generate(ctx, name,
     *forceFallback)` call path. Wrap each call in its own `func() { ... }()`
     so a panic in one source does not abort the whole run; recover, log, and
     mark the source as failed.
  5. After the loop, log a summary line of the form
     `rulegen --all: processed=N succeeded=K failed=M skipped=S` and always
     return exit code 0. Per-source failures are NOT propagated as a non-zero
     exit code (mirrors the resilience pattern from `cmd/collector` and
     `cmd/notifier` introduced in commit `3229715`).
  6. Continue to honour `--force-fallback`, `--max-primary-attempts`, and
     `--max-fallback-attempts` flags — they apply uniformly to every source in
     the `--all` sweep.

  Refactor the existing `run()` function in `cmd/rulegen/main.go` so that the
  dependency-construction block (logger, DSN, DB, AI clients, generator) is
  shared between the single-source and `--all` paths; introduce a private
  helper `runAll(ctx context.Context, gen *rulegen.Generator, srcRepo
  rateSourceLister, forceFallback bool, logger io.Writer) int` so the loop is
  unit-testable without rebuilding the entire dep graph.

  Update the package-level godoc comment (lines 1–39 of the file) to document
  `--all`, the new usage form `rulegen --all [flags]`, and the always-zero
  exit code in that mode. Add a brief note that `--all` and a positional
  source name are mutually exclusive.

- **Acceptance Criteria:**
  - [ ] `./build/rulegen --all` runs without a positional argument and exits 0,
    even when one or more source generations fail.
  - [ ] `./build/rulegen --all my-source` (or any combination with a positional
    arg) exits 2 and prints a usage error to stderr.
  - [ ] Inactive sources (`active = 0` in `rate_sources`) are skipped without
    incrementing the failure counter; the skip is visible in logs.
  - [ ] A summary line in the documented format is emitted on stdout (or
    structured logger at info level) after the loop.
  - [ ] Single-source mode (`./build/rulegen <name>`) keeps its existing exit
    codes (0/1/2/3) — no behavioural change.
  - [ ] New unit test `TestRunAll` (one top-level test, scenarios as
    subtests per CLAUDE.md "One `Test*` per method") covers:
    `all sources succeed` / `one source fails, run still returns 0` /
    `panic in one source is recovered` / `inactive sources are skipped`.
    Test uses a fake `rateSourceLister` and fake generator (interface-typed,
    with a compile-time `var _ rateSourceLister = (*fakeLister)(nil)` assertion
    per CLAUDE.md "Compile-time interface checks").
  - [ ] `make test` passes; `make lint` passes.

- **Pitfalls & edge cases:**
  - The existing `run()` returns the exit code; do not lose that contract when
    refactoring. The shared dep-construction helper should return either an
    error or `(deps, nil)`; `main` translates errors to exit codes.
  - A naive `for _, src := range sources { gen.Generate(...) }` will let a
    panic abort the whole sweep. Wrap each call in a closure with `defer
    recover()` and log the recovered value.
  - `gen.Generate` may take a long time per source (up to
    `maxPrimary + maxFallback` LLM calls). With dozens of sources the total
    wall-clock could be 30+ minutes — fine for cron, but document this in the
    godoc so the operator does not set a tight cron timeout.
  - Do not add a "tolerate first-attempt failure and retry" wrapper — that
    belongs inside `rulegen.Generator`, not in `cmd/rulegen`. The loop is
    intentionally simple.
  - Per CLAUDE.md "No skipped errors": every error returned from `Generate`
    or `ObtainAllRateSources` must be logged with the source name, not
    silently dropped. `_ = err` is forbidden.
  - The new `rateSourceLister` interface in `cmd/rulegen` is package-private
    — keep it narrow (just `ObtainAllRateSources(ctx) ([]domain.RateSource,
    error)`). Do not import `*repository.RateSourceRepository` directly into
    the test.
  - `domain.RateSource` has an `Active bool` field; check before invoking
    the generator. Do not call `gen.Generate` for inactive sources — it would
    waste an LLM call and skew the success/fail counters.

- **Complexity:** Medium

- **Code Example:**
  ```go
  // rateSourceLister is the narrow read-side surface cmd/rulegen needs.
  // Defined locally so tests can fake it without depending on the concrete
  // repository.
  type rateSourceLister interface {
      ObtainAllRateSources(ctx context.Context) ([]domain.RateSource, error)
  }

  // runAll iterates every active rate source and invokes gen.Generate for
  // each. Per-source failures are logged and counted but never propagated;
  // the return value is always 0 so cron does not page on partial failure.
  func runAll(ctx context.Context, gen ruleGenerator, srcs rateSourceLister, forceFallback bool, out io.Writer) int {
      sources, err := srcs.ObtainAllRateSources(ctx)
      if err != nil {
          fmt.Fprintf(out, "FAIL --all reason=list sources: %v\n", err)
          return 0 // still exit 0 — cron must not alarm
      }

      var processed, succeeded, failed, skipped int
      for _, src := range sources {
          if !src.Active {
              skipped++
              fmt.Fprintf(out, "SKIP source=%s reason=inactive\n", src.Name)
              continue
          }
          processed++
          func() {
              defer func() {
                  if r := recover(); r != nil {
                      failed++
                      fmt.Fprintf(out, "FAIL source=%s reason=panic: %v\n", src.Name, r)
                  }
              }()
              res, gerr := gen.Generate(ctx, src.Name, forceFallback)
              if gerr != nil {
                  failed++
                  fmt.Fprintf(out, "FAIL source=%s reason=%v\n", src.Name, gerr)
                  return
              }
              succeeded++
              fmt.Fprintf(out, "OK source=%s rules=%d value=%g attempts=%d\n",
                  src.Name, len(res.Rules), res.Value, res.AttemptsUsed)
          }()
      }
      fmt.Fprintf(out, "rulegen --all: processed=%d succeeded=%d failed=%d skipped=%d\n",
          processed, succeeded, failed, skipped)
      return 0
  }
  ```

## Execution Order

The tasks form a strict dependency chain because each downstream task changes
a function signature that the previous task narrowed.

```
Task 1  (cmd/rulegen --all)            ← independent, can land first or last
   │
   │   ─── parallel with Tasks 2/3/4 (no shared files) ───
   │
Task 2  (delete HTTP handler + route + types) ──┐
                                                │
Task 3  (delete /regen Telegram command)   ─────┤   both narrow signatures
                                                │
                                                ▼
Task 4  (strip wiring from cmd/web/main.go)  ← depends on Tasks 2 AND 3
   │
   ▼
Task 5  (docs)                          ← depends on Tasks 2 + 4 having shipped
```

Recommended execution:
1. **Task 1** — fully independent of the cleanup; landable in isolation. Ship
   first so the operator has the new `--all` flow before the HTTP endpoint
   disappears.
2. **Tasks 2 and 3** — can be done in either order; both narrow constructor
   signatures and update their respective test files. Doing them in
   parallel-but-separate commits aids review; doing them in one commit also
   works because they touch disjoint files except for the shared "shorter
   constructor" theme.
3. **Task 4** — must come after both Task 2 and Task 3 (the `cmd/web/main.go`
   call sites for `gateway.NewGateway` and `service.NewTelegramApi` only
   compile against the narrowed signatures).
4. **Task 5** — last, after all code is in place, so docs match shipped code.

If splitting across multiple PRs: Task 1 in PR1, Tasks 2+3+4+5 in PR2 (they
cannot meaningfully be split because the signature narrowing in 2 and 3
breaks compilation of 4 until all three land together).

## Risks

- **R1 — Operator strips `AI_PRIMARY_DSN` after seeing `cmd/web` no longer
  reads it.** The systemd EnvironmentFile is shared with `cmd/rulegen`. If
  the operator removes the AI vars from the env file, `cmd/rulegen` will fail
  on next cron tick. Mitigation: the inline comment added to `.env.example`
  in Task 5; explicit callout in the PR description; the new `cmd/rulegen
  --all` failure message names the missing env var (`FAIL ... reason=settings
  AI_PRIMARY_DSN: …`) so the diagnosis is obvious from logs.
- **R2 — Hidden caller of `POST /rules/generate`.** If a stakeholder is
  triggering rule generation via a curl script we do not know about, this
  removes their entry point with no migration path other than SSH. Mitigation:
  the goal statement explicitly accepts this trade-off; reviewer lens D
  (security/ops) should re-check the workflow files and the wasm frontend for
  any latent call site.
- **R3 — `cmd/rulegen --all` LLM-spend blowup.** Iterating every active
  source with default attempts (3 primary + 2 fallback) could cost dozens of
  LLM calls in a single cron run. Mitigation: the operator controls cron
  frequency (weekly per goal); the summary line makes spend trivially
  auditable per run.
- **R4 — Concurrent invocations of `cmd/rulegen` (cron overlap).** Two cron
  ticks firing before the previous one finishes will both open the same
  SQLite file. The DB driver serialises writes, but two simultaneous rule
  regenerations for the same source could race. Mitigation: cron interval
  (weekly) is far longer than the worst-case wall-clock (~30 min for
  ~50 sources). If overlap is a real concern later, add a file lock at the
  top of `runAll` — explicitly out of scope here.
- **R5 — Behavioural change of `HttpWriteTimeout` from 130 s to 30 s.** Any
  legitimately slow client of the remaining endpoints (pagination, chart
  aggregation) could now hit a timeout that did not exist before. Mitigation:
  all surviving handlers write < 1 s on typical data volumes; flagged
  separately so reviewer lens E can challenge.
- **R6 — Dead `rulegen.LockManager`.** After Task 3 the `LockManager` type is
  unused inside the web binary. The type still ships as part of the
  `internal/application/rulegen` package and is therefore not dead-code per
  se, but no caller exists. We leave it in place: removing it is a separate,
  reversible cleanup that should not be tangled with this change. Reviewer
  lens E may suggest deletion; defer to a follow-up plan.

## Trade-offs

- **Removed the HTTP endpoint instead of gating it behind a build tag.**
  A build tag would have preserved optionality. Chose deletion because the
  operator has explicitly decided this is a batch-only tool, and a build tag
  would have required keeping the AI imports in `cmd/web` (defeating the
  purpose).
- **Iterate in Go rather than add `ObtainActiveRateSources` SQL.** Cheaper to
  reason about, no new schema-coupled repository method, and the filter
  evaluates once per row — trivial. Reviewer lens E may flag this; the trade
  is that we avoid growing the repository surface for a tool that runs once
  a week.
- **Exit 0 on per-source failure.** Mirrors the pattern in commit `3229715`
  for `cmd/collector` and `cmd/notifier`. The alternative — non-zero on any
  failure — would cause cron to email the operator on transient LLM hiccups,
  which the operator has already rejected for the collector/notifier path.
  Failures remain observable via the summary log line and per-source `FAIL`
  lines.
- **WriteTimeout reduced from 130 s to 30 s.** The 130 s value existed only
  to give the (now-deleted) rulegen handler headroom. Keeping it at 130 s
  would have left a stale config artefact and weakened slow-client
  protection. Trade-off: a hypothetical slow downstream of the remaining
  endpoints loses 100 s of headroom. Documented in R5.
- **Keep `pkg/api.RulegenRequest` / `RulegenResponse` vs. delete.** Deleted.
  These types are internal to a single deleted handler and are not consumed
  by the WASM frontend. Keeping them would be cargo-culting; they would
  confuse future maintainers.
- **One `runAll` helper inside `cmd/rulegen` vs. promote to
  `internal/application/rulegen.RunAll`.** Kept the loop in `cmd/rulegen`.
  Promoting it would mean designing a public package API for a single
  batch caller — premature abstraction. If a second caller ever appears,
  promote then.
- **Do not delete `rulegen.LockManager` even though it becomes unused.** The
  type is a focused utility, easy to keep, and removing it is a separate
  cleanup that can land independently. Bundling it would expand the diff
  and the review surface without changing the user-visible outcome.
