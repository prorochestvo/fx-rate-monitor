# Task Breakdown

## Overview

The subscriber Telegram notification message is built in the **check** agent, not
the dispatch agent. `RateCheckAgent.Run`
(`internal/application/notification/ratecheckagent.go`) folds fired subscriptions
into `alert` values, `buildAlertMessage`
(`internal/application/notification/message.go`) renders them into HTML strings,
those strings are persisted into `rate_user_events.Message`, and
`RateDispatchAgent` (`ratedispatchagent.go`) merely sends the stored string via
`telegramClient.SendHTMLMessage` (ParseMode=HTML). So the only two files whose
logic changes are `message.go` (formatting) and `ratecheckagent.go` (the dedup
that feeds it). **Collection and dispatch sending logic are out of scope.**

This plan reworks three things, all decided by the user (not up for
re-litigation):

1. **Dedup across sources** by `(base, quote, kind)` only — drop `source.Name`
   from the dedup key so a user subscribed to the same pair+kind at multiple
   banks gets ONE row. This requires hoisting the per-user bucket map out of the
   per-source loop so the same key accumulates across sources.
2. **Extremum value selection per pair+kind across sources**: `BID` → MAX price
   (best customer-sell rate), `ASK` → MIN price (best customer-buy rate). The
   delta shown must come from the same winning source so price and delta stay
   coherent. The winning source name is **not** shown (deferred feature).
3. **New message format**: a single monospace `<pre>` table with a header line, a
   timestamp line, and a collapsed trigger-reason badge in the header. Per-line
   trigger labels and the (dead) forecast are removed from the body.

The change is internal and additive to the wire contract in the sense that the
stored `Message` string changes shape but the storage/transport mechanism does
not. No new tables, no new HTTP routes, no new env vars.

## Assumptions

- **BID/ASK semantics are confirmed** (`internal/application/rulegen/prompt.go`):
  `BID` = customer-sell rate (Покупка / Сатып алу), `ASK` = customer-buy rate
  (Продажа / Сату), and `ASK > BID`. From the user's perspective the "best" BID
  is the highest (sells their currency for more) and the "best" ASK is the lowest
  (buys their currency for less). This is why BID→MAX and ASK→MIN.
- **Display inversion is preserved**: `BID` renders the pair as `base/quote`,
  `ASK` renders it as `quote/base` (existing `message.go` behaviour at lines
  88-92).
- The header timestamp is **run time**, passed into the formatter as a `now
  time.Time` parameter (not read via `time.Now()` inside, not a `nowFn` field —
  project preference). It is not the rate's real timestamp; per-rate timestamps
  would differ per winning source and there is no per-row timestamp column. The
  format is `"Mon 2 Jan, 15:04 UTC"` (no year, comma).
- The **first-fire delta guard** `Delta != 0 && Delta != CurrentPrice` is
  load-bearing and must be preserved verbatim in meaning: it suppresses the bogus
  giant delta on a subscription's first-ever fire (`LatestNotifiedRate == 0`,
  so `delta == currentValue`). When the guard suppresses, the delta and arrow
  cells are left blank — never print `0.00`.
- Within a single source, multiple subs on the same `(base, quote, kind)` key can
  carry different deltas; the existing "first fired sub on the key sets
  Delta/CurrentPrice" semantics per source are preserved. The cross-source layer
  then picks the winning source's already-built bucket wholesale (price AND
  delta together), so they stay coherent.
- Subscription advancement (`sub.LatestNotifiedRate = currentValue` + Retain for
  **every** fired subscription) stays per-source / per-subscription and is
  unchanged. This is the highest-impact re-fire-prevention invariant and must not
  regress.
- `make test` (go fmt + go vet + `go test -race ./...`) is the gate. No CGO.
- The existing `internal/tools/labelfmt` package is the home for the new
  thousands-grouping number helper unless it grows the package beyond cohesion;
  `labelfmt` already depends only on `fmt`/`strings`/`internal/domain` and is a
  leaf, so it is a safe host.

## Tasks

### Task 1: Add an ASCII thousands-grouping number formatter in `labelfmt`

- **Description:**
  No grouping helper exists today (`grep` for `Group`/`Thousand`/`FormatNumber`
  across `internal/` returns nothing). Add a pure function to
  `internal/tools/labelfmt/label.go`:

  ```go
  // GroupThousands formats v with exactly two decimal places and groups the
  // integer part in threes using an ASCII space (0x20) as the separator.
  // The ASCII space does not collapse inside an HTML <pre> block, which is why
  // it is used instead of a thin/no-break space. Negative numbers render with a
  // U+2212 MINUS SIGN (not ASCII '-') and the sign is placed before the grouped
  // digits, e.g. -68382.564 -> "−68 382.56".
  func GroupThousands(v float64) string
  ```

  Behaviour:
  - Always two decimals (round half-to-even or `%.2f`-style — pick `%.2f` for
    simplicity and document it).
  - Group the integer part only; never group the fractional part.
  - `68382.56` → `"68 382.56"`, `1138.09` → `"1 138.09"`, `471.95` → `"471.95"`,
    `0` → `"0.00"`.
  - Negative: format the absolute value, then prepend `"−"` (U+2212), e.g.
    `-74.79` → `"−74.79"`. This helper formats the *value column*; the **delta
    column** has its own sign+arrow rendering (Task 3) — do not conflate them.

- **Acceptance Criteria:**
  - [ ] `GroupThousands` lives in `labelfmt` with a godoc comment starting with
        the identifier name and ending with a period.
  - [ ] Returns two decimals always; integer part grouped in threes with a single
        ASCII space; fractional part never grouped.
  - [ ] Negative values use U+2212, sign before digits.
  - [ ] New `TestGroupThousands` (one top-level test, scenarios as `t.Run`
        subtests, all `t.Parallel()`, `require`) covers: sub-thousand, exactly
        one separator, multiple separators (e.g. 1234567.89 → `"1 234 567.89"`),
        zero, negative with separator, a value whose rounding bumps the integer
        part across a thousands boundary (e.g. 999.999 → `"1 000.00"`).
  - [ ] `make test` and `make lint` clean.

- **Pitfalls & edge cases:**
  - **Rounding before grouping.** Format to 2 decimals FIRST (which may carry into
    the integer part, e.g. 999.999 → "1000.00"), THEN group the integer part. If
    you group the pre-rounded integer you get the wrong separators on the
    boundary case.
  - **The separator is ASCII 0x20, not U+00A0 / U+2009.** A non-breaking or thin
    space would defeat the whole point inside `<pre>`. Add a test that asserts the
    separator byte is `0x20` (`strings.Contains(out, " ")` is not enough — assert
    the exact substring like `"68 382.56"`).
  - **Sign handling.** Use U+2212 for the value column's negative sign so it lines
    up visually with the delta column's minus. Do not use ASCII '-'.
  - **Locale-free.** Do not pull in `golang.org/x/text/message` or any locale
    package — hand-roll the grouping. (It is also an unnecessary dependency.)

- **Complexity:** Easy

- **Code Example:**
  ```go
  func GroupThousands(v float64) string {
      neg := v < 0
      s := strconv.FormatFloat(math.Abs(v), 'f', 2, 64) // "68382.56"
      intPart, frac, _ := strings.Cut(s, ".")
      // group intPart in threes from the right
      var b strings.Builder
      n := len(intPart)
      for i, ch := range intPart {
          if i > 0 && (n-i)%3 == 0 {
              b.WriteByte(' ') // ASCII 0x20 — survives <pre>
          }
          b.WriteRune(ch)
      }
      out := b.String() + "." + frac
      if neg {
          out = "−" + out // U+2212 MINUS SIGN
      }
      return out
  }
  ```

### Task 2: Hoist the bucket map and dedup across sources by `(base, quote, kind)`

- **Description:**
  In `RateCheckAgent.Run` (`ratecheckagent.go`), the `userBuckets` map is
  currently created **inside** the `for _, source := range sources` loop (line
  106) and the dedup key includes `source.Name` (line 108), so it never merges
  across sources. Rework so:

  1. **Hoist** the user-bucket map to before the source loop:
     `userBuckets := make(map[string]map[string]*dedupBucket)` (`userID →
     dedupKey → *dedupBucket`).
  2. **Drop `source.Name` from the dedup key** — key on
     `base + "|" + quote + "|" + string(kind)` only.
  3. **Per-source, per-subscription advancement is unchanged**: still compute
     `delta := currentValue - sub.LatestNotifiedRate`, call `sub.IsDue`, and for
     every fired sub set `sub.LatestNotifiedRate = currentValue` +
     `RetainRateUserSubscription`, aggregating errors into `errs[source.Name]`
     exactly as today. Do NOT move retain out of the per-source loop.
  4. When folding a fired sub into a bucket: if the bucket for the key does not
     exist yet, create it from this source's `currentValue` / `delta` / source
     fields (as today). If it already exists (possibly from a *different*
     source), apply the **extremum selection rule** from Task 4 to decide whether
     this source's price+delta replace the stored one. Trigger collapsing
     (`collapseConditionValue` into `b.triggers`) continues to accumulate across
     all sources contributing to the key.
  5. **Flatten after the source loop** (not inside it): iterate the hoisted
     `userBuckets`, set `b.a.Triggers = sortedTriggers(b.triggers)`, and append
     to `telegramBotAlerts[userID]`.

- **Acceptance Criteria:**
  - [ ] `userBuckets` is declared once, before the source loop; the flatten step
        runs once, after the source loop.
  - [ ] The dedup key contains exactly `base|quote|kind` — no source name. Verify
        by grep that `source.Name + "|"` no longer appears in the key
        construction.
  - [ ] A user subscribed to the same `(base, quote, kind)` at two different
        sources produces exactly ONE alert row (one bucket).
  - [ ] `RetainRateUserSubscription` is still called once per fired subscription
        (across all sources), and each retained sub has
        `LatestNotifiedRate == currentValue` of its own source. A test asserts the
        retained-call count equals the number of fired subs across both sources.
  - [ ] The existing "two sources same user — consolidated into one event"
        subtest still passes. NOTE: that test uses two sources with **different**
        pairs (USD/KZT and EUR/KZT), so they remain two rows — it tests
        same-event consolidation, not same-row dedup. Confirm it still expects two
        distinct rows in one message (it asserts both bank titles today; under the
        new format bank titles are gone — see Task 6 for how this assertion is
        rewritten to check pair strings instead).
  - [ ] `make test` passes with `-race`.

- **Pitfalls & edge cases:**
  - **Loop-variable aliasing.** The bucket stores price/delta/source fields by
    value into `dedupBucket.a`; do not store `&sub` or a pointer to the loop
    variable anywhere that outlives the iteration. Go ≥ 1.22 per-iteration scope
    helps, but the bucket must hold copies.
  - **Cross-source trigger accumulation.** Two sources feeding the same key means
    `collapseConditionValue` now merges trigger values from different sources into
    one bucket. That is intended (the badge in Task 5 collapses them anyway), but
    make sure `b.triggers` is initialised once at bucket creation and not reset
    when a later source updates the winning price.
  - **Don't regress retain on collapsed subs.** A sub whose trigger collapses into
    an existing bucket (same condition type already present) must STILL be
    retained. Keep retain unconditional after `IsDue` returns true, separate from
    the bucket-fold logic. This is the single most dangerous regression: a missed
    retain means that sub re-fires every run forever.
  - **`singleSourceName` / event attribution.** With cross-source dedup, a single
    row can now represent multiple sources, so `singleSourceName(alerts)` will
    increasingly return `""` (NULL attribution). That is acceptable and already
    handled (comment at lines 195-200). Do not try to attribute a multi-source row
    to one source.

- **Complexity:** Hard

### Task 3: Replace `buildAlertMessage` body rendering with the monospace `<pre>` table

- **Description:**
  Rewrite the rendering half of `buildAlertMessage` in `message.go` to emit the
  Variant-A layout. Target:

  ```
  📊 FX rates
  Δ ⏰ 🕒 Sat 23 May, 03:00 UTC

  <pre>
  USD/KZT        471.95   +2.60 ↑
  KZT/USD        473.35   +2.60 ↑
  EUR/KZT        547.64   +2.60 ↑
  KZT/EUR        549.66   +2.68 ↑
  GOLD/KZT    68 382.56  −74.79 ↓
  SILVER/KZT   1 138.09   −4.89 ↓
  </pre>
  ```

  Rules:
  - **Header (outside `<pre>`):** line 1 = `📊 FX rates`; line 2 = the collapsed
    trigger badge (Task 5) immediately followed by `🕒 ` + the timestamp, i.e.
    `{Δ }{⏰ }🕒 <timestamp>`. Emoji are allowed ONLY in the header, never inside
    `<pre>`.
  - **Header timestamp** = `now` (passed in as a parameter — see below) in UTC,
    formatted with the Go layout `"Mon 2 Jan, 15:04 UTC"` (no year; a comma before
    the time). Example: `Sat 23 May, 03:00 UTC`.
  - **`now` is a parameter:** the formatter signature becomes
    `buildAlertMessage(now time.Time, alerts ...alert)`. The caller in
    `ratecheckagent.go` passes the run's `now`. Do NOT read `time.Now()` inside the
    formatter and do NOT add a `nowFn` struct field (project preference); tests
    pass a fixed `now`.
  - **Each row inside `<pre>`** has three columns:
    - **pair** (left-aligned): `base/quote` for BID, `quote/base` for ASK.
    - **value** (right-aligned): `labelfmt.GroupThousands(CurrentPrice)`.
    - **delta** (right-aligned) + a single space + arrow: only when the first-fire
      guard `Delta != 0 && Delta != CurrentPrice` holds. Sign uses ASCII `+` for
      positive and U+2212 for negative; arrow `↑` (U+2191) if `Delta > 0`, `↓`
      (U+2193) if `Delta < 0`. When suppressed, the delta and arrow cells are
      blank (do not print `0.00`).
  - **Dynamic alignment:** compute the max rune-width of each column across all
    rows in the part, pad pair to the left, value and delta to the right. Column
    separator is **2 spaces**. The arrow follows the delta after one space. No
    hardcoded widths.
  - **Inside `<pre>`: ASCII only + arrows ↑ ↓ + U+2212 + the ASCII-space
    thousands separator.** No emoji, no `·`.
  - Sort rows deterministically (e.g. by pair string) so output is stable for
    tests.

- **Acceptance Criteria:**
  - [ ] Output contains a `<pre>` … `</pre>` block; the header (`📊 FX rates`,
        `🕒 …`) sits outside it.
  - [ ] Columns are aligned by computed max width, not constants. A test with
        rows of differing widths (e.g. `USD/KZT` vs `SILVER/KZT`, `471.95` vs
        `68 382.56`) asserts the decimal points / column starts line up — assert
        on exact expected lines.
  - [ ] Pair inversion: BID → `base/quote`, ASK → `quote/base` (regression test).
  - [ ] Delta cell blank when `Delta == 0` OR `Delta == CurrentPrice`
        (first-fire); no `0.00` and no arrow in that row's delta region.
  - [ ] Positive delta uses ASCII `+` and `↑`; negative uses U+2212 and `↓`.
  - [ ] Thousands separator is ASCII space and survives inside `<pre>`
        (assert exact `"68 382.56"` substring present in the output).
  - [ ] Timestamp matches layout `"Mon 2 Jan, 15:04 UTC"` (no year, comma); the
        trigger badge (`Δ`/`⏰`) precedes the `🕒` clock on the same line.
  - [ ] `make test` passes with `-race`.

- **Pitfalls & edge cases:**
  - **Width by runes, not bytes.** `↑`, `↓`, `−`, the grouped value's ASCII
    spaces — `utf8.RuneCountInString` (or `[]rune` length) for padding math.
    Padding by byte length will misalign every row that contains a multibyte rune
    in the value/delta column. Note: the value column's spaces ARE width-bearing
    (they occupy a cell), so include them in the width.
  - **The delta column width** must be computed over the *rendered* delta string
    (sign + grouped magnitude), excluding the trailing space+arrow, so the arrows
    line up in their own implicit column. Decide explicitly whether the arrow is
    part of the delta column or a separate 4th column; the target render shows the
    arrow hanging one space after a right-aligned delta — implement it as
    `<right-padded delta>` + `" "` + `<arrow-or-blank>`. Rows with a suppressed
    delta still need their delta cell padded to width so the (absent) arrow column
    does not shift.
  - **HTML escaping.** Currency codes and the digits are safe, but be deliberate:
    nothing inside `<pre>` should contain `<`, `>`, or `&`. Pair strings come from
    `BaseCurrency`/`QuoteCurrency` (ASCII codes like `USD`, `GOLD`) so escaping is
    not required today, but add a comment noting the assumption so a future
    free-text source name does not silently break the HTML.
  - **Trailing whitespace per row.** Right-padding the delta column on a row with
    no arrow can leave trailing spaces before the newline. Telegram tolerates it,
    but trim trailing spaces per line to keep test fixtures clean and avoid
    invisible diffs.
  - **Empty input.** `buildAlertMessage()` with no alerts must still return an
    empty slice (existing "no alerts — empty slice" test). Do not emit an empty
    `<pre>`.

- **Complexity:** Hard

### Task 4: Implement BID-MAX / ASK-MIN extremum selection with coherent delta

- **Description:**
  Inside the bucket-fold step (Task 2), when a fired sub maps to a key whose
  bucket already exists, decide whether THIS source's price replaces the stored
  one:
  - `kind == BID` → replace if `currentValue > b.a.CurrentPrice` (keep the MAX).
  - `kind == ASK` → replace if `currentValue < b.a.CurrentPrice` (keep the MIN).

  When the price is replaced, the **delta must be replaced together with it** from
  the same source (`b.a.Delta = delta`, `b.a.CurrentPrice = currentValue`, and the
  source-identifying fields if any are still carried) so the displayed delta
  always belongs to the winning source. Triggers continue to accumulate
  regardless of which source wins the price.

- **Acceptance Criteria:**
  - [ ] For a BID key fed by two sources at prices `P1 < P2`, the bucket holds
        `P2` and `P2`'s delta.
  - [ ] For an ASK key fed by two sources at prices `P1 < P2`, the bucket holds
        `P1` and `P1`'s delta.
  - [ ] The shown delta always comes from the same source as the shown price
        (assert delta value matches the winning source's delta, not the loser's).
  - [ ] Ties (equal prices) are stable and deterministic — pick a documented
        tiebreak (e.g. first-seen wins) and test it does not flap.
  - [ ] Source name is NOT present in the rendered output (it is a deferred
        feature). A test asserts no source title/name string leaks into the
        message.
  - [ ] `make test` passes with `-race`.

- **Pitfalls & edge cases:**
  - **Delta/price coherence is the whole point.** The classic bug is updating
    `CurrentPrice` to the winning source but leaving `Delta` from the first source
    that created the bucket. Update both atomically in the same branch, or store a
    small struct and swap it wholesale.
  - **First-fire delta on the winning source.** If the winning source's sub is
    first-fire (`LatestNotifiedRate == 0`), its `delta == currentValue`, which the
    render guard (Task 3) suppresses. That is correct — but make sure selection
    operates on PRICE, not on delta, so a first-fire winner still wins on price
    and simply shows a blank delta cell.
  - **Kind comparison constant.** Compare against `domain.RateSourceKindBID` /
    `domain.RateSourceKindASK` (string consts), not raw `"BID"`/`"ASK"` literals.
  - **Mixed kinds never collide** because `kind` is part of the key — a BID row
    and an ASK row for the same pair are two separate buckets. Do not attempt to
    cross-compare them.

- **Complexity:** Medium

### Task 5: Compute and render the collapsed trigger-reason badge in the header

- **Description:**
  Drop per-line trigger labels and the forecast from the table body. Instead,
  compute ONE collapsed reason badge for the whole message, placed in the header
  line next to `📊 FX rates`.

  Collapse the union of fired condition types across all rows in the part into two
  buckets:
  - `delta` → a delta indicator.
  - `{interval, daily, cron}` → a single "time/schedule" indicator.

  Usually only one bucket is present ("обычно один — или дельта или время" /
  "usually one — either delta or time"). If both are present, show both.

  **Badge rendering (user-confirmed):** the badge sits on the TIMESTAMP line,
  immediately BEFORE the `🕒` clock icon — NOT next to `📊 FX rates`. Glyphs:
  `Δ` (U+0394) for the delta bucket, `⏰` (U+23F0 alarm clock) for the
  time/schedule bucket (any of interval/daily/cron). Line 2 layout:
  - delta-only:    `Δ 🕒 Sat 23 May, 03:00 UTC`
  - schedule-only: `⏰ 🕒 Sat 23 May, 03:00 UTC`
  - both:          `Δ ⏰ 🕒 Sat 23 May, 03:00 UTC`
  - none (defensive — should not happen if a row fired):
    `🕒 Sat 23 May, 03:00 UTC` (no leading glyph).

  Each present glyph is followed by a single space; the order is always delta
  (`Δ`) then schedule (`⏰`) then the `🕒` clock. Introduce a
  `badgeIconSchedule = "⏰"` constant alongside the reused `Δ` delta glyph
  (formerly `triggerIconDelta`). `Δ` reads consistently with the rest of the bot;
  `⏰` collapses interval/daily/cron into one "time-based fired" signal (the user
  does not want per-type detail). All glyphs are daughter-safe.

  Decide WHICH of the existing trigger machinery survives:
  - **Keep & repurpose:** `triggerOrder`, `sortedTriggers`, `collapseConditionValue`,
    `cronWeekdayDigit`, the `dedupBucket.triggers` map, the `alertTrigger` type —
    these still drive the per-bucket trigger accumulation that the badge reads
    from. (The badge needs to know which condition *types* fired; the collapsed
    *values* are no longer rendered but the machinery that records "this type
    fired" is reused.)
  - **Delete (now dead):** `triggerLabel` and its test `TestTriggerLabel`
    (no longer rendered per line); `telegramBotForecast`, `alert.ForecastPrice`,
    `alert.ForecastMethod` and the forecast block in `buildAlertMessage`
    (forecast was never populated — confirmed dead code); the per-line
    `triggerIconInterval` / `triggerIconDaily` / `triggerIconCron` constants if
    the badge uses a single clock glyph instead. Keep `triggerIconDelta` (renamed
    or reused as the badge's `Δ`). The old wide emoji `telegramBotArrowUp` /
    `telegramBotArrowDown` (🔼/🔽) are replaced by the in-table `↑`/`↓` and can be
    deleted.
  - **Confirm before deleting:** grep each symbol's references across the repo
    before removing — `triggerLabel`, the forecast fields, and the arrow constants
    may be referenced from tests in `message_test.go` / `ratecheckagent_test.go`
    that this plan rewrites in Task 6. Remove the symbol and its now-orphaned test
    in the same change; keep the deletion tight (no unrelated cleanup).

  To compute the badge, derive the fired condition-type set from the alerts in the
  part (each `alert.Triggers` carries `ConditionType`s). A small helper
  `reasonBadge(alerts []alert) string` returning the badge fragment is the clean
  shape; it folds every alert's trigger types into the two buckets and emits the
  glyphs.

- **Acceptance Criteria:**
  - [ ] Badge appears in the header on line 1 next to `📊 FX rates`, never inside
        `<pre>`.
  - [ ] delta-only set → `Δ`; schedule set (any of interval/daily/cron) → `⏰`;
        both → `Δ ⏰`; empty → no leading glyph. Badge sits before `🕒` on the
        timestamp line.
  - [ ] The schedule bucket collapses interval+daily+cron into a single glyph (no
        per-type icons in the new format).
  - [ ] `triggerLabel`, `TestTriggerLabel`, the forecast fields/block, and the
        wide-emoji arrow constants are removed; `make lint`/`go vet` report no
        unused symbols and no dangling references.
  - [ ] A `TestReasonBadge` (or equivalent subtests under the message test)
        covers: delta-only, interval-only, daily-only, cron-only, mixed
        time-types collapse to one clock, delta+time both shown, empty input.
  - [ ] `make test` passes with `-race`.

- **Pitfalls & edge cases:**
  - **Badge is per-message-part, not per-row.** When the message splits (Task 7),
    each part should reflect the reasons of the rows IT contains, so compute the
    badge from the rows in that part, not globally. (Simplest: compute per part
    after splitting.)
  - **Dead-code removal scope creep.** Only remove symbols this rework genuinely
    orphans. Do not "while I'm here" refactor unrelated parts of `message.go` or
    the agent.
  - **`triggerIconDelta` reuse.** If you keep the `Δ` glyph but rename the
    constant (e.g. to `badgeIconDelta`), update every reference and the test
    assertions in one pass. If you keep the name, ensure its only remaining use is
    the badge.
  - **Clock glyph resolved.** The schedule glyph is `⏰` (user-confirmed) and the
    delta glyph is `Δ`; the badge sits before the `🕒` timestamp clock. If the
    glyph ever changes it is a one-constant edit with no logic impact.

- **Complexity:** Medium

## Risks

- **Re-fire regression (highest impact).** Hoisting the bucket map and changing
  the dedup key must NOT move or skip the per-fired-subscription
  `RetainRateUserSubscription`. A sub whose trigger collapses into a bucket fed by
  another source, if not retained, re-fires on every run and spams the user.
  Mitigation: Task 6's cross-source retain-count assertion is the canary; keep
  retain unconditional and separate from the fold.
- **Delta/price incoherence.** The extremum selection (Task 4) is the easy place
  to leave a stale delta from the bucket-creating source while swapping in a
  different source's price. Mitigation: swap price and delta atomically; Task 6
  asserts the shown delta equals the winning source's.
- **Misalignment from byte-vs-rune width math.** The table contains multibyte
  runes (`↑`, `↓`, `−`) and width-bearing ASCII spaces inside grouped numbers.
  Computing column widths by byte length silently misaligns. Mitigation:
  rune-count everything; Task 6 asserts exact multi-line blocks.
- **`<pre>` HTML validity across splits.** A `<pre>` split across two messages
  renders as broken HTML in Telegram. Mitigation: Task 7 opens/closes `<pre>` per
  part and tests for balance.
- **First-fire guard removed by accident during the rewrite.** Replacing the old
  `(%.2f %s)` delta block is the moment the `Delta != 0 && Delta != CurrentPrice`
  guard can get dropped. Mitigation: Task 3 acceptance + a dedicated first-fire
  subtest in Task 6.
- **Badge glyph not to the user's taste.** Resolved: `Δ` + `⏰`, badge before the
  `🕒` timestamp clock (user-confirmed). One-constant change if ever revisited.
- **Thousands separator collapsing.** If a non-breaking/thin space sneaks in
  (autocorrect, copy-paste), grouping collapses inside `<pre>`. Mitigation: Task 1
  asserts the exact ASCII-space substring.

## Trade-offs

- **GroupThousands in `labelfmt` vs. a new tools package.** Chose `labelfmt` —
  it is already the leaf formatting package on the notification path, depends only
  on stdlib + domain, and a new package for one function is over-engineering.
  Revisit only if number-formatting grows several variants.
- **Compute badge per part vs. once globally.** Chose per part — a split message's
  second part should advertise only its own rows' reasons. Slightly more work; far
  less misleading than a global badge stamped on every part.
- **Recompute column widths per part vs. once globally.** Chose per part — tighter
  alignment and avoids one wide row in part 1 padding every row in part 2. Cost is
  trivial (small row counts).
- **Collapse interval/daily/cron into one clock glyph vs. keep per-type icons.**
  Chose one clock, per explicit user decision ("usually one — delta or time").
  Loses per-type granularity in the badge; the user does not want it, and the
  table body no longer carries it either.
- **Keep run-time clock for the header vs. surface per-rate timestamps.** Chose
  run-time. Per-rate timestamps would differ per winning source and there is no
  per-row timestamp column in the table; a single header timestamp at send time is
  honest about when the digest was assembled.
- **Delete dead forecast/`triggerLabel` code now vs. leave it.** Chose delete —
  the forecast was never populated and `triggerLabel` is orphaned by the new
  format; leaving dead code rots. Scope kept tight: only symbols this rework
  orphans, verified by grep before removal.
- **Winning source name shown vs. hidden.** Chose hidden, per user — "which bank
  is best" is a deferred future feature. Noted here so the next iteration knows
  the selection already computes the winner and only the render is withheld.
