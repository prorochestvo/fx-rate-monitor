# Task Breakdown

## Overview

Introduce a new pure-Go binary `cmd/sourceaudit` that audits every production rate
source defined in the seed SQL against its live page. The tool dedupes by URL,
fetches each URL once with the exact production HTTP behavior (User-Agent, timeout,
status check), applies each source's `regex` or `jsonpath` rule to the fetched body,
and prints a per-source `name | url | side | value | status` table. Two modes:

- `--check` (default): print the report and exit non-zero on any MISS.
- `--refresh`: same report; additionally rewrite `testdata/sources/<slug>.<ext>` for
  every successfully-fetched URL and `<slug>_expected.json` derived from this run.

The work is split into three deliverables: a thin `cmd/sourceaudit/main.go`
composition root, a unit-testable `internal/application/sourceaudit/` core that
takes a `Fetcher` interface, and `Makefile` targets `source-audit` /
`source-audit-refresh`. A small refactor of `internal/tools/rateextractor` exposes
the regex and JSON-path matching as pure helpers so both the production extractor
and the new audit tool share one source of truth.

The plan deliberately excludes: any DB access (the live `active` flag is not read,
admin toggles in production are out of scope and noted as a known limitation),
service-layer wiring, network-touching tests, fixture cleanup for `bankffin.html` /
`halykbank.html` (those are leftovers from the parked `ruledoctor` work and stay
untouched), and any Claude skill wrapper.

## Assumptions

- Source list is parsed from `./migrations/*.seed*.sql`. Admin-toggled `active`
  flags stored in the live DB are not reflected; this is an explicit known
  limitation called out in tool output. Sources whose seed `active` column is `0`
  are still audited (the audit is about extractor correctness, not deployment state).
- All seed `INSERT OR IGNORE INTO rate_sources` rows follow the fixed positional
  column order `(name, vendor, base, quote, url, interval, side, active,
  headers_json, rules_json)` as in `202605.007.rate_sources.seed_initial.sql`.
- Rules JSON is an array of `{"method": "regex"|"json"|"parse_float"|"store_as_rate",
  "pattern": "..."}` matching `domain.RateSourceRule`. The audit tool supports
  `regex` and `json` (the two methods that actually inspect a fetched body); it
  records `UNSUPPORTED_METHOD` for `parse_float` / `store_as_rate` so future seed
  entries that need them aren't silently skipped. Today all 32 seeded sources use
  `regex`.
- Concurrency: sequential fetches. With 4 unique URLs today there is no benefit to
  parallelism, and sequential output keeps the report deterministic. Parallel
  fetching is explicitly out of scope for this plan.
- HTTP behavior matches the production extractor: 1-minute timeout, User-Agent
  `FXRateMonitor/1.0 (+https://github.com/seilbekskindirov/monitor)`, fail on
  non-2xx status. No proxy support in the audit tool (production extractor allows a
  proxy URL but production currently runs without one and the audit tool is a
  developer utility, not a deployable service).
- `testdata/sources/` retains fixtures unrelated to the current production seed
  (`bankffin.html`, `halykbank.html`, their `_expected.json` siblings). `--refresh`
  only writes for URLs currently in the seed; it never deletes.
- Slug derivation: hostname with dots replaced by `_`, joined to a `_`-cleaned
  path. Examples: `https://www.bcc.kz/kz/` → `www_bcc_kz_kz`,
  `https://www.bcc.kz/en/personal/currency-rates/` → `www_bcc_kz_en_personal_currency_rates`,
  `https://gateway.prod.qazpost.kz/mail-app/info/remote/currencies/ops` →
  `gateway_prod_qazpost_kz_mail_app_info_remote_currencies_ops`. Slugs that exceed
  120 characters are truncated and suffixed with the first 8 hex chars of the SHA-256
  of the full URL to keep collisions impossible.
- Fixture extension: `.json` when response `Content-Type` starts with
  `application/json` OR when the body trimmed of whitespace begins with `{` or `[`;
  otherwise `.html`.
- Project rules from `CLAUDE.md`: pure Go, `CGO_ENABLED=0`, no clock func fields
  (pass `now time.Time` as a parameter where needed), no `_ = err` discards, no
  section-divider comments, compile-time interface assertions for all stubs, one
  `Test*` per method with subtests, testify, `-race`. Persisted docs in English.

  ```go
  func ApplyRegex(pattern string, payload []byte) ([]byte, error) {
      re, err := regexp.Compile(pattern)
      if err != nil {
          err = fmt.Errorf("compile pattern %q: %w", pattern, err)
          return nil, errors.Join(err, internal.NewTraceError())
      }

      matches := re.FindSubmatch(payload)
      if len(matches) < 2 {
          err = fmt.Errorf("invalid regex pattern %q", pattern)
          return nil, errors.Join(err, internal.NewTraceError())
      }

      return matches[1], nil
  }

  func (extractor *RateExtractor) fetchRegexPage(_ context.Context, pattern string, payload []byte) ([]byte, error) {
      return ApplyRegex(pattern, payload)
  }
  ```

### Task 2: Add `internal/application/sourceaudit/seedparser.go` — parse seed SQL into source records

- **Description:** Create a new package `internal/application/sourceaudit` with a
  file `seedparser.go` exposing:

  ```go
  type SeededSource struct {
      Name      string
      Vendor    string
      Base      string
      Quote     string
      URL       string
      Interval  string
      Side      string            // "BID" | "ASK"
      Active    bool
      Rules     []domain.RateSourceRule
      Origin    string            // "<seed-file-basename>:<line-number>" for diagnostics
  }

  func ParseSeedFiles(fsys fs.FS, glob string) ([]SeededSource, error)
  ```

  Implementation uses an `fs.FS` (so tests inject `fstest.MapFS` and production
  uses `os.DirFS(".")`). For each file matching `glob`, read line-by-line; for
  each line that begins (after trimming whitespace) with
  `INSERT OR IGNORE INTO rate_sources VALUES(` extract the parenthesized
  comma-separated SQL literal list and decode the 10 columns.

  Use a hand-rolled SQL value tokenizer (not a generic SQL parser dependency).
  Rules:
  - String literals are single-quoted; `''` inside a string is a literal `'`
    (standard SQLite escape).
  - Integers are bare (no quotes).
  - Top-level commas separate columns; commas inside single-quoted strings are
    part of the value.
  - The `rules_json` column is a single-quoted JSON string that itself contains
    JSON `"` characters; the SQL tokenizer treats the whole single-quoted span
    as one value and the result is then JSON-decoded into
    `[]domain.RateSourceRule`.

  Skip blank lines, lines starting with `--`, and any non-matching `INSERT`
  variants. Surface a wrapped error if the column count is not 10 or if rule JSON
  fails to decode, including the file path and line number.

- **Acceptance Criteria:**
  - [x] `ParseSeedFiles` returns 32 records for the current production seed
        (`202605.007.rate_sources.seed_initial.sql`).
  - [x] A unit test `TestParseSeedFiles` with subtests exercises:
        - happy path on a minimal in-memory file with two rows (one regex rule,
          one JSON rule) → both decoded fields and rule arrays.
        - escaped quote in vendor column (`'O''Brien Bank'`) → exact value `O'Brien Bank`.
        - malformed row (9 columns instead of 10) → returns an error mentioning
          file path and line number.
        - malformed rules JSON → returns an error mentioning file path and line
          number.
        - blank lines, comment lines, non-`INSERT` statements → ignored without
          error.
        - glob matches multiple `*.seed*.sql` files → records concatenated in
          lexicographic file order, preserving each file's internal line order.
  - [x] Tests use `testing/fstest.MapFS`; no disk I/O in tests.
  - [x] `Active` correctly maps `1` → `true`, `0` → `false`; any other integer is
        an error.

- **Pitfalls:**
  - SQLite accepts `1` and `0` for booleans — do not treat the integer column as
    a quoted string.
  - The `rules_json` column contains regex patterns with double-quotes, backslashes,
    and curly braces. Test against a real seed line from
    `202605.007.rate_sources.seed_initial.sql` (e.g. the BCC FX regex with
    `[\\s\\S]{0,300}?`) to confirm tokenization round-trips correctly.
  - Don't try to parse the headers JSON — the `headers_json` column is `'{}'`
    everywhere today; store the raw string in case future seeds populate it, but
    don't attempt to apply headers in the fetch step in Task 4. If a future
    headers JSON is non-empty, the audit tool emits `UNSUPPORTED_HEADERS` for
    that source (mirrors the `UNSUPPORTED_METHOD` policy from Task 5).
  - Do not commit a SQL-parser dependency to `go.mod`; the tokenizer is ~60 LOC
    of stdlib `strings`/`bytes`.

- **Complexity:** Medium

- **Code Example:**

  ```go
  // tokenizeSQLValues splits "'a','b''c',1,'[]'" into ["'a'", "'b''c'", "1", "'[]'"].
  func tokenizeSQLValues(s string) ([]string, error) {
      var out []string
      var buf strings.Builder
      inString := false
      for i := 0; i < len(s); i++ {
          c := s[i]
          switch {
          case inString && c == '\'' && i+1 < len(s) && s[i+1] == '\'':
              buf.WriteByte('\''); buf.WriteByte('\''); i++
          case c == '\'':
              inString = !inString
              buf.WriteByte(c)
          case c == ',' && !inString:
              out = append(out, buf.String())
              buf.Reset()
          default:
              buf.WriteByte(c)
          }
      }
      if inString {
          return nil, fmt.Errorf("unterminated string literal")
      }
      if buf.Len() > 0 {
          out = append(out, buf.String())
      }
      return out, nil
  }
  ```

### Task 4: Add `internal/application/sourceaudit/audit.go` — orchestrate fetch + extract per source

- **Description:** Add the audit core in `internal/application/sourceaudit/audit.go`:

  ```go
  type ProbeStatus string

  const (
      StatusOK                   ProbeStatus = "OK"
      StatusFetchError           ProbeStatus = "FETCH_ERROR"
      StatusHTTPNon2xx           ProbeStatus = "HTTP_NON_2XX"
      StatusRegexNoMatch         ProbeStatus = "REGEX_NO_MATCH"
      StatusJSONPathError        ProbeStatus = "JSON_PATH_ERROR"
      StatusValueParseError      ProbeStatus = "VALUE_PARSE_ERROR"
      StatusUnsupportedMethod    ProbeStatus = "UNSUPPORTED_METHOD"
      StatusUnsupportedHeaders   ProbeStatus = "UNSUPPORTED_HEADERS"
  )

  type ProbeResult struct {
      Source  SeededSource
      URL     string
      Status  ProbeStatus
      Value   string  // populated only when Status == StatusOK; raw extracted value, no float normalization
      Detail  string  // diagnostic string for non-OK statuses
      Body    []byte  // shared pointer to the cached fetch body for this URL (nil for FETCH_ERROR / HTTP_NON_2XX)
      Content string  // Content-Type of the cached body (empty when Body is nil)
  }

  type Auditor struct {
      Fetcher Fetcher
  }

  func (a *Auditor) Run(ctx context.Context, sources []SeededSource) ([]ProbeResult, error)
  ```

  Steps inside `Run`:
  1. Group sources by URL preserving first-seen order (sequential traversal of
     the input slice keeps output deterministic).
  2. For each unique URL: call `Fetcher.Fetch`. Cache the `*FetchResult` keyed by
     URL string in a local `map[string]*FetchResult`. On error, every source for
     that URL gets `StatusFetchError` with `Detail` set to `err.Error()`.
  3. For each source, look up the cached result. Apply rules in order using
     `rateextractor.ApplyRegex` / `rateextractor.ApplyJSONPath` from Task 1.
     `domain.MethodParseFloat` and `domain.MethodStoreToRate` produce
     `StatusUnsupportedMethod`.
  4. After the rule chain, run the same post-processing as the production
     extractor: `bytes.ReplaceAll(payload, []byte(","), []byte("."))` then
     `bytes.ReplaceAll(payload, []byte(" "), []byte(""))` (handles the non-
     breaking and regular spaces from BCC's `70 534.67`). Then `strconv.ParseFloat`
     into a float64 for sanity; reject NaN/Inf and values ≤ 0 or `> math.MaxInt32`
     just like `RateExtractor.Run` (lines 130-149).
  5. On success, set `Value` to the **extracted string after post-processing but
     before float parsing** (so the report shows `468.95`, not `468.950000`).
     The float sanity check still runs; failure becomes `StatusValueParseError`.
  6. Sources whose `Active` flag is `false` are still audited; `ProbeResult.Status`
     reflects the extraction outcome. The active flag is preserved on `Source`
     for the table renderer to mark inactive rows.

- **Acceptance Criteria:**
  - [x] `Auditor.Run` is deterministic — same input slice → identical output
        slice (including `ProbeResult.Body` pointers being equal for sources
        sharing a URL).
  - [x] Unit test `TestAuditor_Run` with subtests covers:
        - happy path with a single source whose regex matches → `StatusOK`,
          `Value` matches expectation.
        - two sources sharing one URL → fetcher invoked exactly once
          (assert via a counting fake fetcher).
        - regex with no match → `StatusRegexNoMatch`.
        - fetch error → `StatusFetchError` for all sources of that URL; other
          URLs unaffected.
        - JSON-path rule on a JSON body → `StatusOK`.
        - rule with `domain.MethodParseFloat` → `StatusUnsupportedMethod`.
        - extracted value parses as NaN / 0 / negative → `StatusValueParseError`.
        - thousand-separator value (e.g. `70 534.67` with a regular space) →
          `StatusOK`, `Value == "70534.67"`.
  - [x] Fake fetcher used in tests has a compile-time assertion
        `var _ Fetcher = (*fakeFetcher)(nil)` at the top of the test file.
  - [x] No `_ = err` discards in production code; no clock func fields.

- **Pitfalls:**
  - Don't share the post-processed value between sources — make sure the
    `payload` variable is reset per source from `result.Body` before each rule
    chain. Otherwise sources sharing a URL pollute each other.
  - The production extractor's `Run` mutates `payload` in place across rules.
    Mirror that exactly: each rule's output becomes the next rule's input. Keep
    `bytes.TrimSpace` between rules.
  - `StatusValueParseError` should include the offending payload in `Detail`
    (truncated to 80 chars) — without it, BCC parser regressions are extremely
    painful to diagnose.
  - `errors.Is(err, context.DeadlineExceeded)` from the fetcher must surface
    cleanly as `StatusFetchError` with a recognizable detail string, not get
    flattened into a generic "could not read html page" message.

- **Complexity:** Medium

### Task 5: Add `internal/application/sourceaudit/report.go` — table renderer and exit-code logic

- **Description:** Render results as a fixed-width plain-text table to an
  `io.Writer`:

  ```
  name                                 | url                                                                    | side | value     | status
  -------------------------------------+------------------------------------------------------------------------+------+-----------+----------------
  KZ_BCC_BID_USD_KZT                   | https://www.bcc.kz/kz/                                                 | BID  | 466.00    | OK
  KZ_BCC_FX_ASK_USD_KZT                | https://www.bcc.kz/en/personal/currency-rates/                         | ASK  | 470.35    | OK
  KZ_QAZPOST_ASK_USD_KZT               | https://gateway.prod.qazpost.kz/mail-app/info/remote/currencies/ops    | ASK  | 470.9     | OK
  KZ_NATIONALBANK_BID_USD_KZT          | https://nationalbank.kz/en/exchangerates/...                           | BID  | 463.91    | OK
  ```

  Column widths are derived from the data with hard caps (URL truncated to 72
  chars with trailing `...` if longer). After the table, print a summary line
  `audited <N> sources across <M> URLs: <K> OK, <K> MISS`. When any source has
  a non-OK status, also dump a "MISS DETAILS" section with one line per failed
  source: `<name>\t<status>\t<detail>`.

  Add:

  ```go
  func WriteReport(w io.Writer, results []ProbeResult) (failures int, err error)
  ```

  The function returns the count of non-OK results; the caller maps `>0` to exit
  code 1 in `--check` mode. Take all formatting helpers from stdlib — no
  `text/tabwriter` (the trailing-space behavior of tabwriter is awkward for
  truncated URL cells) and no third-party table library.

- **Acceptance Criteria:**
  - [x] `TestWriteReport` covers: all-OK input → returns 0 failures, output
        contains every source name once; mixed input → returns correct failure
        count, output includes a MISS DETAILS block for each failure; empty input
        → no panic, prints header and `audited 0 sources across 0 URLs`.
  - [x] Output is stable across runs (sources rendered in input order).
  - [x] URL column truncates at 72 chars with `...` suffix when wider.
  - [x] Inactive sources (`Source.Active == false`) get a `[inactive]` suffix
        appended to their name column.

- **Pitfalls:**
  - Truncate by runes, not bytes — a future Cyrillic vendor name would otherwise
    corrupt the table alignment. (Today all source names are ASCII so this is
    cheap insurance.)
  - Long detail strings can contain newlines (joined errors from
    `internal.NewTraceError`). Replace `\n` with ` | ` before printing in the
    MISS DETAILS block so the table doesn't fracture.

- **Complexity:** Easy

### Task 6: Add `internal/application/sourceaudit/refresh.go` — fixture rewrite for `--refresh`

- **Description:** Implement the `--refresh` write path:

  ```go
  type RefreshOptions struct {
      TestdataDir string
  }

  func WriteFixtures(opts RefreshOptions, results []ProbeResult) error
  ```

  Behavior:
  1. Group successful results by URL. Skip URLs whose every result is non-OK
     (no point refreshing a fixture we couldn't extract from).
  2. For each URL with at least one OK result:
     - Slug the URL per the rule in Assumptions.
     - Determine extension: `.json` if the cached `Content` starts with
       `application/json` OR if `bytes.TrimSpace(Body)[0]` is `{` or `[`;
       `.html` otherwise.
     - Write `<TestdataDir>/<slug><ext>` with `Body`. Use `os.WriteFile` with
       mode `0o644`. Create `TestdataDir` with `os.MkdirAll(..., 0o755)` first.
     - Write `<TestdataDir>/<slug>_expected.json` whose schema is:
       ```json
       {
         "_generated_at": "2026-05-13T12:34:56Z",
         "_url": "https://...",
         "_note": "Auto-generated by cmd/sourceaudit --refresh. Do not edit by hand.",
         "pairs": [
           { "name": "KZ_BCC_FX_ASK_USD_KZT", "side": "ASK", "value": "470.35" },
           ...
         ]
       }
       ```
       Only entries with `StatusOK` are included. Sort entries by `name` for
       deterministic diffs.
     - `_generated_at` is rendered from a `time.Time` argument **passed in by the
       caller** — do not call `time.Now()` inside this function. The flag-parsing
       main passes `time.Now().UTC()` once at the start of the run.
  3. The function does not delete any file. `bankffin.html`, `halykbank.html`,
     and their `_expected.json` siblings are left untouched because their URLs
     are not in the current seed.

  Add the signature variant:

  ```go
  func WriteFixturesAt(opts RefreshOptions, results []ProbeResult, now time.Time) error
  ```

  to keep the `time.Time` parameter explicit and obvious to callers (per the
  user-memory rule against clock func fields).

- **Acceptance Criteria:**
  - [x] `TestWriteFixturesAt` covers:
        - one URL with two OK results writes one `.html` (or `.json`) plus one
          `_expected.json` with two pairs sorted by name.
        - one URL with one OK and one MISS writes the fixture and includes only
          the OK pair.
        - one URL with all MISS results writes nothing for that URL.
        - URL with JSON content type writes a `.json` extension; URL with HTML
          content writes `.html`.
        - URL slug longer than 120 chars produces a truncated slug with a hash
          suffix and the file is still written.
        - Existing unrelated files in the target directory (`bankffin.html`)
          remain on disk untouched after a refresh — test creates a sentinel
          file then asserts it still exists.
  - [x] `_generated_at` in the output equals the `now` argument formatted as
        RFC3339Nano.
  - [x] Test uses `t.TempDir()` exclusively — no writes outside the test
        sandbox.

- **Pitfalls:**
  - JSON sniffing must happen on `bytes.TrimSpace(Body)`, not raw `Body`. The
    QazPost endpoint returns the response with a leading whitespace prefix in
    some configurations.
  - `os.WriteFile` truncates by default — that is the desired behavior here, but
    note that a panic between writing the body and writing the expected JSON
    would leave them out of sync. Acceptable given this is a developer tool, not
    a production write path.
  - Slug truncation must be deterministic: the SHA-256 hash is over the full
    URL string, so the same URL always slugs to the same file.

- **Complexity:** Medium

### Task 7: Add `cmd/sourceaudit/main.go` — composition root, flag parsing, exit code

- **Description:** New binary entry point at `cmd/sourceaudit/main.go`:

      exclusive with `--check`: when `--refresh` is set, `--check` is forced to
      `false`. Document this in the flag's help string.
    - `--seed-glob` (string, default `./migrations/*.seed*.sql`).
    - `--testdata-dir` (string, default `./testdata/sources`).
    - `--tmp-dir` (string, default `./tmp/source_probe`) — in `--check` mode the
      tool also writes raw fetched bodies to this directory using the same slug
      rules, so a developer can `diff` fixtures against the live response
      without re-running.
    - `--only` (string, default `""`) — when non-empty, treat as a regex
      compiled with `regexp.Compile` and filter `SeededSource.Name` matches.
  - Open the seed files with `os.DirFS(".")` (relative to the user's working
    directory — `make source-audit` runs from the repo root).
  - Build an `Auditor{Fetcher: NewHTTPFetcher(time.Minute)}`.
  - Run, render the report to `os.Stdout`.
  - In `--check` mode: also dump raw bodies to `--tmp-dir` (per unique URL,

  Do not introduce a `loginjector.Logger` or DSN parsing — this binary uses

## Risks

- **Production parity drift.** If a future change to `RateExtractor.Run`
  introduces post-processing the audit tool doesn't mirror (e.g. a new step
  between regex match and float parse), the audit tool will report green while
  production silently breaks. **Mitigation:** Task 4's float-sanity step is
  copied verbatim from `RateExtractor.Run` lines 130-149; if that block ever
  changes, the engineer must update Task 4's mirror in the same PR. A reviewer
  comment on `rateextractor.go` referencing `internal/application/sourceaudit`
  is acceptable belt-and-suspenders.
- **Seed file format drift.** A future seed migration could introduce a new
  column or change column order. The hand-rolled tokenizer in Task 2 hard-codes
  positional column indices. **Mitigation:** Task 2's malformed-row test catches
  column-count mismatches at the per-file boundary. If columns are reordered we
  fail loudly, not silently.
- **Live site HTML change between merge and next deploy.** A page redesign that
  invalidates the seed regex is exactly what this tool exists to catch — but if
  the redesign lands during the audit-tool PR's review window, the audit tool
  will block its own merge. **Mitigation:** if Task 9 reports MISS for a real
  upstream-changed page, file a separate "fix-seed-regex-for-X" plan and
  proceed with the audit-tool PR once the seed-fix plan is merged. The audit
  tool merging is not contingent on every upstream parser being green forever.
- **Network test ban escape hatch.** A junior engineer may be tempted to add a
  `httptest.NewServer` smoke test for `httpFetcher`. That is fine and is **not**
  a network test in the policy sense (the listener is loopback only). The
  policy rule is "no test that hits a real upstream"; in-process test servers
  are encouraged.

## Trade-offs

- **Decision 1: parse seed SQL, do not query the live DB.** Chose seed-parsing
  because (a) the tool runs in CI / on developer machines without DB access,
  (b) the audit is about "does the seeded regex still match the live page",
  not "what does the running service currently believe". Cost: admin-toggled
  `active` flags in production are not reflected. Documented as a known
  limitation in the report output.

- **Decision 2: extract `ApplyRegex` / `ApplyJSONPath` from the extractor (Task
  1) rather than duplicate ~30 LOC.** Chose extraction because a single source
  of truth for the extraction algorithm means audit findings track production
  behavior exactly. Cost: a small public-API addition to
  `internal/tools/rateextractor`. Cheaper than the alternative — two parallel
  implementations drifting silently is exactly the class of bug this tool
  exists to catch.

- **Decision 3: sequential fetches, not parallel.** With 4 unique URLs today,
  parallel adds complexity (errgroup, ordered result collection) for ~3 seconds
  of wall-clock savings on a tool a developer runs by hand. Trade-off
  revisitable when the seed grows past ~30 unique URLs; not before. Plan does
  not include any parallelism task.

- **Decision 4: ProbeStatus as enum of strings, not error sentinels.** Chose
  strings because they serialize cleanly into the table column and the future
  `_expected.json` "_unverified_reason" field (if we add one). Sentinel error
  values would require an extra mapping layer at every render site.

- **Decision 5: `cmd/sourceaudit` lives under `cmd/` like every other binary
  (`cmd/web`, `cmd/collector`, `cmd/notifier`, `cmd/migrator`, `cmd/wasm`) and
  the audit core lives under `internal/application/sourceaudit/`.** Considered
  `internal/tools/` (sibling to `rateextractor`), but the audit logic
  orchestrates fetch + extract across many sources — that is application-layer
  composition, not a low-level utility. `internal/tools/` stays the home of
  reusable, single-concern helpers.

- **Decision 6: developer-utility CLI, not a service.** Chose a one-shot binary
  with `log.Printf` over the `loginjector`/`dsninjector` machinery used by
  `cmd/web` and friends. The audit tool has no DB, no env vars, no DSN, no
  lifecycle — promoting it to a "service" with rotating logs and a
  configurable verbosity flag would be ceremony without benefit. If we later
  want CI to run it on a schedule, a separate plan can lift it.
