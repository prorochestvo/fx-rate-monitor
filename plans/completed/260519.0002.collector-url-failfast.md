# Task Breakdown

## Overview

Add an in-process, per-URL negative cache ("fail-fast tombstone") inside both rate
extractors so that the second and subsequent sources sharing a failing URL in a single
`RateAgent.Run` short-circuit instead of repeating a 60s HTTP wait (or 30s headless
Chrome attempt). The cache lives on the extractor struct, has **no TTL**, and is
recreated implicitly on the next cron tick because `cmd/collector` is a short-lived
one-shot binary.

The fix targets the BCC outage pattern: ~20 `KZ_BCC_FX_*` sources all point at
`https://www.bcc.kz/en/personal/currency-rates/`. When that origin is down, today's
collector spends ~20 min wedged on the same URL. After this change, the same outage
costs one timeout per cron invocation.

The change is local to the `rateextractor` package ŌĆö no schema migration, no domain
changes, no new exported types. Both extractors keep independent state because a source
uses one `FetcherKind`, never both, so cross-extractor coordination buys nothing.

## Assumptions

- `cmd/collector` continues to be a one-shot binary invoked by cron. If that ever
  becomes a long-lived process, this design needs a TTL (out of scope here).
- Sources sharing a URL share **exactly** the raw `source.URL` string. URL
  canonicalisation (trailing slash, query-string order, scheme case) is not in scope:
  if two sources spell the same origin two different ways, they get two tombstone slots
  ŌĆö acceptable.
- The agent processes sources sequentially today (`for _, source := range sources` in
  `RateAgent.execution`). We still mutex-protect the map for forward compatibility and
  to match the idiom used by `RateExtractor.cache` (`threadsafe.Cache` is internally
  mutexed).
- `chromedp.Run` errors are always cache-worthy. We do not try to distinguish "browser
  crashed" from "navigation timed out from a slow server" ŌĆö both indicate the URL is
  not productive to retry inside the current Run.
- Successful fetches do **not** populate the negative map. The existing 30-minute
  positive cache (`extractor.cache`) already short-circuits repeated successful reads;
  the negative map only exists to short-circuit repeated **failed** reads inside one
  process lifetime.
- "Skipped because previous attempt failed" is a real failure for `execution_history`
  purposes ŌĆö every source that shares the failing URL still gets a row written, with
  an `error` column that names the original cause. This matches today's behaviour:
  there is no concept of "deferred" in the schema, and inventing one is out of scope.

## Tasks

### Task 1: Add the negative-cache field and helpers to `RateExtractor`

- **Description:** In `internal/tools/rateextractor/rateextractor.go`, add a
  `failedURLs map[string]error` field plus a `failedURLsMu sync.Mutex` field to the
  `RateExtractor` struct. Initialise the map in both
  `NewRateExtractorWithHTTPClient` (constructor) and **not** in any reset path ŌĆö
  it must live for the full extractor lifetime.

  Add two private helpers on `*RateExtractor`:
  - `loadFailedURL(url string) (error, bool)` ŌĆö read under the mutex.
  - `recordFailedURL(url string, err error)` ŌĆö write under the mutex (overwrite is
    fine; we only ever care that a tombstone exists, not which exact error it carries).

  Modify `fetchHtmlPage` so that:
  1. **Before** the existing positive-cache `Fetch` call, check `loadFailedURL(rawURL)`.
     If present, log one line via `extractor.logger` (format below) and return a
     wrapped error: `fmt.Errorf("skipping %s: previous attempt in this run failed: %w", rawURL, cached)` joined with `internal.NewTraceError()`.
  2. **On every error-return path inside `fetchHtmlPage`**, call
     `recordFailedURL(rawURL, err)` immediately before `return nil, err`. The four
     paths are:
     - `http.NewRequestWithContext` error (malformed URL),
     - `extractor.httpClient.Do` error (transport/timeout/DNS/connection refused),
     - non-2xx status code (`resp.StatusCode < 200 || resp.StatusCode >= 300`),
     - `io.ReadAll(resp.Body)` error.

  **Do not** record on the `cache.Push` error log path ŌĆö that branch returns the body
  with `nil` error and represents a degraded but successful fetch.

  Log line format: `_, _ = fmt.Fprintf(extractor.logger, "rate_extractor: short-circuit url=%s prior_error=%v\n", rawURL, cached)`

- **Acceptance Criteria:**
  - [ ] `RateExtractor` struct has fields `failedURLs map[string]error` and
        `failedURLsMu sync.Mutex` declared after the existing `cache` field.
  - [ ] `NewRateExtractorWithHTTPClient` initialises `failedURLs` to a non-nil empty
        map.
  - [ ] `loadFailedURL` and `recordFailedURL` are private methods on
        `*RateExtractor`, both godoc-commented.
  - [ ] All four error-return paths in `fetchHtmlPage` write to the map before
        returning.
  - [ ] A short-circuit return wraps the original error with `%w`, mentions the URL,
        and is joined with `internal.NewTraceError()` to match local conventions.
  - [ ] Short-circuit emits exactly one log line via `extractor.logger`.
  - [ ] `make test` passes with no new lint findings.
- **Pitfalls:**
  - Forgetting to record the tombstone on the `NewRequestWithContext` path ŌĆö
    unlikely to fire in production but the linter will flag the inconsistency.
  - Holding the mutex across the HTTP call (deadlock risk if anyone ever adds
    recursion). The helpers must `Lock`/`defer Unlock` only around the map access.
  - Wrapping the cached error with `errors.Join(cached, internal.NewTraceError())`
    on every short-circuit ŌĆö this is correct but ensure the test assertion uses
    `errors.Is(err, cached)` or `ErrorContains` against the original message rather
    than equality.
  - Do **not** add a clock field; this cache has no TTL (per project rule
    `feedback_no_clock_func_field`).
  - Do **not** use `_ = err` to swallow any error introduced here.
- **Complexity:** Easy
- **Code Example:**

```go
// Inside fetchHtmlPage, before the positive-cache lookup:

if cached, ok := extractor.loadFailedURL(rawURL); ok {
    _, _ = fmt.Fprintf(extractor.logger,
        "rate_extractor: short-circuit url=%s prior_error=%v\n", rawURL, cached)
    err := fmt.Errorf("skipping %s: previous attempt in this run failed: %w", rawURL, cached)
    err = errors.Join(err, internal.NewTraceError())
    return nil, err
}

// Each existing error path becomes:
if resp.StatusCode < 200 || resp.StatusCode >= 300 {
    err = fmt.Errorf("fetch %s: unexpected status %d (%s)", rawURL, resp.StatusCode, resp.Status)
    err = errors.Join(err, internal.NewTraceError())
    extractor.recordFailedURL(rawURL, err)
    return nil, err
}

// Helpers:

// loadFailedURL returns the cached error for url and true if url was previously
// recorded as failed during the current process lifetime.
func (extractor *RateExtractor) loadFailedURL(url string) (error, bool) {
    extractor.failedURLsMu.Lock()
    defer extractor.failedURLsMu.Unlock()
    e, ok := extractor.failedURLs[url]
    return e, ok
}

// recordFailedURL stores err as the tombstone for url. Subsequent fetches of url
// inside the same process short-circuit and return a wrapped form of err.
func (extractor *RateExtractor) recordFailedURL(url string, err error) {
    extractor.failedURLsMu.Lock()
    defer extractor.failedURLsMu.Unlock()
    extractor.failedURLs[url] = err
}
```

### Task 3: Tests for plain extractor fail-fast behaviour

- **Description:** Add a new top-level test `TestRateExtractor_failFast` in
  `internal/tools/rateextractor/rateextractor_test.go` (project rule: one `Test*`
  per method; this is a new behaviour with its own scenarios, so it gets its own
  top-level test). All subtests run in parallel.

  The four subtests below mirror real production scenarios. Each subtest constructs
  one `*RateExtractor`, registers an `httptest.Server` (or no server, for the
  unreachable-host case), and runs the extractor twice against sources that share
  the URL (or use different URLs for the third case).

  Use the existing `mockRateValueRepository` and `threadsafe.NewBuffer(nil)` logger
  patterns. Use an `atomic.Int32` (`sync/atomic`) counter for `callCount` so the
  race detector stays clean even though the agent is sequential today.

- **Acceptance Criteria:**
  - [ ] `TestRateExtractor_failFast` exists with four parallel subtests:
    - [ ] `"shared URL 500 hits server once"` ŌĆö two sources, same URL, server
      returns 500, increment counter on every request. After both `ext.Run`
      calls, assert `callCount == 1` and the second error contains the
      substring `"previous attempt"`.
    - [ ] `"shared URL 200 still hits server once"` ŌĆö regression guard for the
      positive cache. Two sources, same URL, server returns 200 with a valid
      payload. Both runs succeed, `callCount == 1`, `rateRepo.retained`
      contains 2 entries (one per source ŌĆö `applyRulesAndStore` runs per
      source even when the body is cached).
    - [ ] `"unrelated URLs failure does not poison other URL"` ŌĆö two
      `httptest.Server` instances. First returns 500, second returns 200.
      Both endpoints are hit exactly once each; the 500 source returns an
      error, the 200 source succeeds.
    - [ ] `"unreachable host short-circuits second attempt"` ŌĆö both sources
      point at `http://127.0.0.1:1` (nothing listening). First `Run` returns a
      transport error; second `Run` returns an error containing
      `"previous attempt"` and the second call must be observably faster
      (assert `time.Since(start) < 100*time.Millisecond` on the second call
      while the first uses a `&http.Client{Timeout: 500*time.Millisecond}` ŌĆö
      this is the most direct way to prove the network was not retried).
  - [ ] No `_ = err` swallowing.
  - [ ] All subtests use `t.Parallel()`.
  - [ ] Log output is captured in a `threadsafe.Buffer` and the short-circuit
        subtests assert the buffer contains `"short-circuit"` to lock in the
        operator-facing log contract.
- **Pitfalls:**
  - `httptest.Server` callbacks must not race on `callCount`; use
    `atomic.Int32` not a plain `int`.
  - The "200 still hits server once" case is easy to misread as redundant ŌĆö
    it locks in the existing positive-cache behaviour so a future refactor of
    the cache can't silently break it.
  - The unreachable-host case must use a port that is reliably closed
    (`127.0.0.1:1` is the convention already used in the existing
    `"http request fails"` subtest at line 224). Do **not** use a hostname that
    requires DNS ŌĆö DNS failures behave differently across environments and may
    add seconds of latency that confuse the timing assertion.
  - When asserting "second call short-circuits", capture wall-clock around
    only the second `Run`, not both. The first call legitimately waits for the
    transport timeout.
- **Complexity:** Medium
- **Code Example:**

```go
func TestRateExtractor_failFast(t *testing.T) {
    t.Parallel()

    logger := threadsafe.NewBuffer(nil)

    t.Run("shared URL 500 hits server once", func(t *testing.T) {
        t.Parallel()

        var callCount atomic.Int32
        srv := httptest.NewServer(http.HandlerFunc(func(w http.ResponseWriter, _ *http.Request) {
            callCount.Add(1)
            w.WriteHeader(http.StatusInternalServerError)
        }))
        defer srv.Close()

        ext, err := NewRateExtractorWithHTTPClient(
            &mockRateValueRepository{},
            &http.Client{Timeout: 5 * time.Second},
            logger,
        )
        require.NoError(t, err)

        srcA := &domain.RateSource{Name: "a", URL: srv.URL, Rules: []domain.RateSourceRule{{Method: domain.MethodRegex, Pattern: `.+`}}}
        srcB := &domain.RateSource{Name: "b", URL: srv.URL, Rules: []domain.RateSourceRule{{Method: domain.MethodRegex, Pattern: `.+`}}}

        err1 := ext.Run(t.Context(), srcA)
        require.Error(t, err1)

        err2 := ext.Run(t.Context(), srcB)
        require.Error(t, err2)
        require.ErrorContains(t, err2, "previous attempt")

        require.Equal(t, int32(1), callCount.Load(), "the failing URL must be hit only once across both runs")
    })

    // ... additional subtests
}
```

## Risks

- **Cross-source error contamination.** When URL X fails for source A, sources B
  and C that share URL X now also fail with a "previous attempt" error. The
  `execution_history` rows for B and C will reference A's failure mode rather
  than their own attempt. Operators reading `execution_history.error` need to
  understand the `"previous attempt in this run failed"` prefix as "see the
  earlier row for the real cause". The log line in Task 1 helps but does not
  surface in the DB.
  - *Mitigation:* the error message includes the URL, which uniquely identifies
    the failing origin. The pattern is greppable.
- **Tombstone too sticky inside a single Run.** If a transient blip causes the
  first source to fail but a retry 30s later would succeed, the remaining N-1
  sources still get short-circuited for the rest of the Run. This is the
  intended trade ŌĆö the alternative is "hammer the dead origin N times" which
  is the bug we're fixing.
- **Process-lifetime scope assumes one-shot cron.** If anyone ever runs
  `cmd/collector` in a loop or daemon mode, tombstones would persist
  indefinitely. The godoc on `recordFailedURL` must explicitly state "lifetime:
  extractor lifetime, intended for one-shot processes."
- **Map growth is unbounded.** Each unique failing URL adds one entry. In
  practice the URL count is in the low hundreds, the map never gets cleaned,
  and the process exits within minutes. Not a leak in this deployment model.
  If the deployment ever changes, this needs a size cap or TTL.
- **Forgotten chromedp record path.** `chromedp.Run` is the only failure path
  today, but `chromedp.NewExecAllocator` / `chromedp.NewContext` can in
  principle fail (they don't return errors today). If future chromedp upgrades
  add error returns, this plan won't catch them automatically. Reviewer should
  flag any newly-introduced error returns in `fetchRenderedPage`.

## Trade-offs

- **No TTL vs. TTL'd cache.** Chose plain map over `threadsafe.Cache` because
  the process is short-lived (cron tick ŌåÆ exit). A TTL'd implementation would
  add ~10 lines of code and offer no real-world benefit; future-proofing for a
  long-running collector is YAGNI today.
- **Independent maps per extractor vs. shared cache.** Chose independent maps
  because no source uses both `plain` and `chromedp` for the same URL ŌĆö they
  pick one `fetcher_kind`. Sharing would require lifting the map into a new
  package-level helper for zero correctness benefit.
- **Record raw error vs. wrapped error.** Chose to record whatever the
  extractor was about to return (the wrapped form). This means the tombstone
  replay says `"previous attempt in this run failed: chromedp fetch ...: ..."`
  with the operator-friendly prefix preserved, at the cost of nesting one
  extra `: ` in the message. Net win for log readability.
- **`failedURLs` as a field vs. a package-global.** Chose field for testability
  ŌĆö each test gets a fresh extractor and therefore a fresh tombstone state. A
  package-global would force tests to either reset it or run sequentially.
- **No new `urlTombstoneCache` type.** Chose to inline two methods per struct
  (~12 lines total duplication) rather than introduce a new exported-or-internal
  type whose only consumers would be these two extractors. If a third caller
  ever appears, extract then.
- **Log on short-circuit vs. silent.** Chose to log one line per short-circuit
  so operators investigating "why didn't N sources update?" can grep for
  `short-circuit` and see the cascade. Cost: extra log volume during outages
  ŌĆö acceptable because outages are the case we want maximum observability for.
