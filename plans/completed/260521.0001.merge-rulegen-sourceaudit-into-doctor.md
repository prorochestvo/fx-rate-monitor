# Task Breakdown

## Overview

Today two operator-only CLI binaries live side-by-side under `cmd/`:

- `cmd/rulegen` — LLM-driven extraction-rule generator (single-source and `--all` modes).
  Built into `./build/rulegen` by `make build`.
- `cmd/sourceaudit` — read-only seed auditor that probes live pages to verify rules.
  Invoked only via `go run` from the `make audit` target.

Both are local maintenance tools. Neither is shipped by `.github/workflows/{stage,prime}.yml`
nor invoked by any systemd unit in `configs/` or `deploy/`. They share one transitive
dependency: `cmd/rulegen` already imports `internal/application/sourceaudit` for its
plain HTTP fetcher.

This plan merges both into a single umbrella binary `cmd/doctor` with `flag.NewFlagSet`-
based subcommand dispatch (no cobra). The internal packages
(`internal/application/{rulegen,sourceaudit}`) are not touched — only the `cmd/` shell
and the docs/Makefile around it. The old `cmd/rulegen` and `cmd/sourceaudit`
directories are deleted in the same commit that introduces `cmd/doctor`; no
backward-compatibility shims.

Final UX:
```
doctor audit   [--source <name>|--all] [--seed-glob ...] [--only <regex>] [-v]
doctor rulegen <source> [--force-fallback ...]
doctor rulegen --all [--force-fallback ...]
doctor --help                                 # exit 0
doctor <unknown>                              # exit 2
doctor                                        # exit 2
```

## Assumptions

1. **Internal packages are stable.** `internal/application/rulegen/` and
   `internal/application/sourceaudit/` keep their existing exported surface
   (`Generator`, `Result`, `NewHTTPFetcher`, `Auditor`, `ParseSeedFiles`,
   `WriteReport`, `SeededSource`, `Fetcher`, …). The merge is a `cmd/` refactor.
2. **`go.mod` does not change.** No new third-party deps; `flag` from stdlib is
   sufficient. The cobra/urfave-cli question is closed (decided: stdlib).
3. **No operator-cron migration is needed.** Grep across the repo shows no
   `rulegen`/`sourceaudit` references in `configs/`, `deploy/`, or `.github/`.
   The only callers are `Makefile`, `README.md`, `CLAUDE.md`, and `cmd/rulegen/README.md`.
   `plans/HANDOFF-2026-05-17-llm-rulegen.md` mentions old paths but is a frozen
   historical doc — leave it untouched (history file, not a live reference).
4. **Build artifacts.** Replace `./build/rulegen` with `./build/doctor`. No
   `./build/sourceaudit` exists today (sourceaudit is `go run`-only), so nothing
   to delete there. `clean` target's explicit `rm -f` list should be updated
   accordingly.
5. **`make audit`'s ergonomics are preserved.** Today operators run
   `make audit ARGS="--only ^halyk_"`. The replacement should keep the
   `ARGS=...` pass-through pattern (`make audit ARGS="--source halyk_usd"`).
6. **Exit codes are a public contract.** They are documented in
   `cmd/rulegen/main.go` godoc and a Telegram bot consumer parses output; they
   must be preserved verbatim per subcommand. Top-level `doctor` adds only one
   new exit code (`2` for unknown subcommand / no subcommand), which already
   collides with the existing rulegen "usage error" code 2 — by design, since
   they mean the same thing semantically.
7. **No log-format change.** Stdout/stderr lines (`OK source=…`, `FAIL source=…`,
   `rulegen --all: processed=N succeeded=…`, sourceaudit's report block) are
   preserved verbatim, including the literal string `rulegen --all:` in the
   summary line. The `runall_test.go` assertions and any external `grep` cron
   greps continue to match.

## Tasks

### Task 1: Create `cmd/doctor/main.go` skeleton with subcommand dispatch
- **Description:** Add a new `cmd/doctor/main.go` file containing the umbrella
  entrypoint. It owns only argument parsing and dispatch — all real work is
  delegated to subcommand functions defined in sibling files (`audit.go`,
  `rulegen.go`). The dispatch table is hand-written, no reflection.
  Package godoc starts with `// Command doctor ...` per CLAUDE.md. The file
  declares the same `BuildVersion`/`BuildTime`/`BuildHash`/`LogsDir`/
  `LogVerbosity` link-time variables that `cmd/rulegen` declares today so
  `-ldflags ${BUILD_OPTIONS}` continues to work without Makefile changes other
  than the binary path.

  Dispatch contract:
  - `os.Args[1] == "audit"` → call `runAudit(os.Args[2:])`
  - `os.Args[1] == "rulegen"` → call `runRulegen(os.Args[2:])`
  - `os.Args[1] == "--help" || "-h" || "help"` → print combined usage to
    stdout, exit 0.
  - `os.Args[1] == ""` or `len(os.Args) < 2` → print usage to stderr, exit 2.
  - any other value → print `"doctor: unknown subcommand %q"` to stderr +
    usage, exit 2.
  - `doctor audit --help` and `doctor rulegen --help` are handled by each
    subcommand's own `FlagSet` (stdlib `flag` prints defaults to its
    configured output and returns `flag.ErrHelp`; the subcommand recognises
    that and exits 0).
- **Acceptance Criteria:**
  - [ ] `cmd/doctor/main.go` exists with `// Command doctor ...` godoc covering
        subcommands, exit codes (per subcommand, plus top-level 2), and a one-
        line example for each subcommand.
  - [ ] `func main()` is a 2-liner: `os.Exit(run(os.Args[1:], os.Stdout, os.Stderr))`.
  - [ ] `run([]string, io.Writer, io.Writer) int` is testable (no global state
        reads except env vars and the link-time `Build*` vars). Writers default
        to `os.Stdout`/`os.Stderr` in `main`.
  - [ ] `BuildVersion`, `BuildTime`, `BuildHash`, `LogsDir`, `LogVerbosity` are
        declared as package-level vars matching `cmd/rulegen`'s shape so
        `-ldflags "-X main.BuildVersion=…"` continues to work.
  - [ ] Combined help (`doctor --help`) lists both subcommands, with a one-line
        description for each, and points to `doctor <sub> --help` for details.
  - [ ] `cmd/doctor/main_test.go` covers: unknown subcommand → exit 2, no args →
        exit 2, `--help` → exit 0, `audit --help` → exit 0, `rulegen --help` →
        exit 0. One top-level `TestRun` with subtests per CLAUDE.md convention.
  - [ ] `var _` compile-time interface checks present in test files where
        fakes/mocks exist (none expected in this task; required for tasks 2/3).
- **Pitfalls & edge cases:**
  - `flag.NewFlagSet(name, flag.ContinueOnError)` is required so we can detect
    `flag.ErrHelp` instead of exiting via stdlib's default `flag.ExitOnError`.
    Each subcommand's FlagSet must have its `SetOutput(io.Writer)` wired to
    the test-controllable writer.
  - Do NOT call `flag.Parse()` from this binary at package level — `cmd/rulegen`'s
    current code uses the default global `FlagSet`, which conflicts with a
    subcommand-style binary the moment two subcommands declare same-named flags.
    Use a fresh `FlagSet` per subcommand.
  - The unknown-subcommand error message goes to stderr, not stdout. Tests must
    assert on both writers separately.
  - Single-dash short options exist today (`-v` in sourceaudit). `flag` accepts
    both `-v` and `--v` so we don't need to choose; document `--source` /
    `--all` style in help text for consistency.
- **Complexity:** Easy

### Task 2: Implement `doctor audit` subcommand in `cmd/doctor/audit.go`
- **Description:** Port the body of today's `cmd/sourceaudit/main.go` into a
  function `runAudit(args []string, out, errOut io.Writer) int` in a new file
  `cmd/doctor/audit.go`. The subcommand owns its own `flag.NewFlagSet` and
  delegates all real work to `internal/application/sourceaudit` (no logic
  changes inside that package).

  Flag surface:
  - `--seed-glob string`  (default `"migrations/*.seed*.sql"`) — same as today.
  - `--only string`       — regex filter, same as today. KEPT for power users.
  - `--source string`     — new convenience flag; exact-match shortcut.
    Internally rewritten as `--only=^<regexp.QuoteMeta(source)>$`.
  - `--all bool`          — new explicit "audit every source" flag.
  - `-v bool`             — verbose, same as today.

  Mode resolution (validated up-front before any I/O):
  | `--all` | `--source` | `--only` | Outcome |
  |---------|------------|----------|---------|
  | false   | ""         | ""       | **error** "specify --all, --source, or --only" → exit 2 |
  | true    | ""         | ""       | audit every source |
  | true    | non-empty  | any      | usage error → exit 2 (mutually exclusive) |
  | true    | ""         | non-empty | usage error → exit 2 (mutually exclusive) |
  | false   | non-empty  | ""       | exact-match audit |
  | false   | ""         | non-empty | regex audit (today's `--only`) |
  | false   | non-empty  | non-empty | usage error → exit 2 (mutually exclusive) |

  Exit codes (preserved from `cmd/sourceaudit`):
  - `0` — all probes OK.
  - `1` — at least one MISS (matches today's behaviour).
  - `2` — usage error (NEW exit code for the audit subcommand; sourceaudit
    today uses `log.Fatalf` which always exits 1 on usage failures. This is a
    behaviour change called out in Risks; it makes audit consistent with
    rulegen and the top-level dispatcher).
  - `3` — infrastructure error (e.g. seed-glob parse failure or auditor.Run
    returns an error). Also a NEW code — sourceaudit today calls
    `log.Fatalf` (exit 1) for these. Again called out in Risks.

  Note: the engineer may keep behaviour bug-for-bug compatible (collapse all
  non-zero codes to `1`) if the reviewer pipeline pushes back on the new codes.
  The argument for the split is operator UX symmetry with `rulegen`; the
  argument against is preserving a documented contract. Default to the split
  unless review objects; the rename of the binary is itself a contract break
  signal so this is the right window to widen the codes.
- **Acceptance Criteria:**
  - [ ] `cmd/doctor/audit.go` contains `runAudit(args []string, out, errOut io.Writer) int`.
  - [ ] All flag definitions live inside a fresh `flag.NewFlagSet("audit",
        flag.ContinueOnError)`; `fs.SetOutput(errOut)`.
  - [ ] Mutually-exclusive flag combinations are validated BEFORE any seed file
        is opened.
  - [ ] `--source NAME` is internally rewritten to a regex of the form
        `^<regexp.QuoteMeta(NAME)>$` (so special chars like `.` in source names
        are escaped). A subtest covers `--source "src.with.dots"` matching only
        that exact name and not e.g. `srcXwithXdots`.
  - [ ] When neither `--all`, `--source`, nor `--only` is supplied, the
        subcommand prints `"doctor audit: specify --all, --source, or --only"`
        to errOut and returns 2.
  - [ ] All existing sourceaudit report text (`OK`, `MISS`, `DETAILS`, the
        per-source table under `-v`) is byte-identical to today — only the
        program name "sourceaudit" in any error prefix changes to "doctor audit".
  - [ ] `cmd/doctor/audit_test.go` has one `TestRunAudit` with subtests for:
        no-flags error, mutually-exclusive errors, `--source` exact match,
        `--all` happy path, infrastructure error path. Tests stub the
        `sourceaudit.Auditor.Fetcher` so they don't hit the live web.
- **Pitfalls & edge cases:**
  - `regexp.QuoteMeta` is mandatory for `--source` — source names can contain
    `.` or `_`. Failing to escape would silently broaden the match.
  - `sourceaudit.ParseSeedFiles` returns `nil, nil` when no files match the
    glob; the audit subcommand must keep today's "no sources found matching
    glob %q" error and return a non-zero code.
  - The `--seed-glob` default is relative to the process CWD. Document this in
    the godoc on `runAudit` — the operator must run `doctor audit` from the
    repo root just like today.
  - `auditor.Run` returns `error` only on context/cancellation paths today;
    don't collapse the "infrastructure" branch with the "at least one MISS"
    branch — they have different operator semantics.
- **Complexity:** Medium

## Risks

1. **Exit-code widening on `doctor audit`.** Today `sourceaudit` uses
   `log.Fatalf` for both usage errors and infra errors, all of which exit
   with code 1 — the same code used for "at least one MISS". Task 2 splits
   these into 2 (usage) and 3 (infra). Any wrapper script that distinguishes
   "audit reported a real MISS" from "audit failed to start" would have been
   forced to grep output text today; with the split, exit codes alone suffice.
   But a script that today does `if ! make audit; then alert; fi` keeps
   working identically — only the *meaning* of non-zero broadens. **Mitigation:**
   call out in PR description; revert to all-1 if any reviewer dissents.

2. **Help-text quality drift.** stdlib `flag` produces minimal `-h` output
   ("Usage of audit: ..." plus a flat list of flags). It does NOT produce the
   structured "Usage / Flags / Exit codes / Env vars" block that the current
   `cmd/rulegen/main.go` godoc carries. **Mitigation:** override each FlagSet's
   `.Usage` field with a hand-written function that prints the structured
   block (mirror today's godoc shape).

3. **Logger filename collision.** Both subcommands could in principle write to
   `<logsDir>/<prefix>.YYYYMMDD.log`. Today only rulegen writes a log file
   (audit prints to stdout only). Task 3 preserves the `"rulegen"` prefix.
   Audit does not gain a log file. **Mitigation:** none needed unless a
   reviewer pushes for `"doctor.rulegen"` and `"doctor.audit"` — push back
   citing operator log paths as stable surface.

4. **Cron operator surprise (theoretical).** If any operator has
   `./build/rulegen --all` baked into a personal crontab, that crontab breaks
   silently after the binary is removed. Empirical check: `grep` across the
   repo confirms zero deploy/cron references. **Mitigation:** none — this is a
   local-only tool; the user is the only operator; they know about the
   rename.

5. **External stdout-grepping consumers.** The Telegram bot's `/regen` handler
   parses `rulegen` output strings. Task 3 preserves the literal `rulegen --all:`
   summary prefix. **Mitigation:** task 3 acceptance criterion locks this in.

6. **Stale build artefact.** If an operator has `./build/rulegen` on disk
   from before the merge, it keeps working until `make clean` is run, masking
   the deprecation. **Mitigation:** task 5 adds `./build/rulegen` to the
   `clean` target's rm list for one release cycle (a "transitional cleanup"
   line — easy to remove later).

7. **`make audit` build-dep timing.** Switching `make audit` from
   `go run ./cmd/sourceaudit` to `./build/doctor audit` introduces a stale-
   binary failure mode if the operator forgets to rebuild. **Mitigation:**
   the new target carries an explicit `build` dependency (task 5 acceptance
   criterion).

## Trade-offs

1. **Stdlib `flag` vs `cobra` — chose stdlib.**
   Pro: zero new dependencies, smaller binary, no `go.mod` churn, fits the
   existing project's "small surface" aesthetic. Con: weaker help-text
   generation, no automatic completion. Acceptable for an operator-only
   tool with two subcommands.

2. **`doctor audit` with no flags errors out, instead of defaulting to
   `--all` (today's implicit default).**
   Pro: explicit is better than implicit; matches `doctor rulegen`'s need
   for an explicit `--all` flag; eliminates the "I thought it would only
   audit the one I just changed" surprise. Con: breaks muscle memory for
   `make audit` (without `ARGS=`). **Mitigation:** the Makefile's `audit`
   target could default `ARGS` to `--all` so `make audit` keeps behaving
   the same as today. Leave that decision to the engineer with this plan
   as guidance.

3. **Keep both `--source` and `--only` on `doctor audit`.**
   Pro: `--source halyk_usd` is the 90% case, `--only ^halyk_.*$` covers
   the long tail of "audit a regex-defined cohort". Removing `--only` would
   force users into either single-source mode or `--all`, with no middle
   ground. Con: two flags doing related things invites confusion. Mitigated
   by mutual-exclusivity validation and clear `--help` text. **Decision:
   keep both**, document `--source` as the exact-match shorthand, document
   `--only` as the regex form.

4. **Keep the `rulegen --all:` summary-line prefix literal (not changing to
   `doctor rulegen --all:`).**
   Pro: preserves the operator/cron grep contract zero-effort; preserves
   `runall_test.go` assertions zero-effort. Con: confusing for someone who
   discovers the log line cold and wonders why it says `rulegen` when the
   binary is `doctor`. Acceptable: a single sentence in `cmd/doctor/README.md`
   notes this is a deliberate cron-stability decision.

5. **Keep `internal/application/{rulegen,sourceaudit}` separate.**
   Pro: the merge is purely a `cmd/` reorganisation, blast radius is tiny,
   reviewer cost is low. Con: misses the chance to deduplicate the
   `sourceAuditFetcherAdapter` shim. **Decision:** ship the small merge
   now; a follow-up plan can promote shared types into `internal/` if a
   second consumer ever appears.

6. **Subcommand-style UX (`doctor rulegen ...`) vs flag-mode
   (`doctor --mode=rulegen ...`).**
   Decided up-front with the user: subcommand-style. Pro: standard, what
   `git`/`go`/`kubectl` do, more discoverable via `--help`. Con: per-
   subcommand `FlagSet` boilerplate. Worth it.

7. **Logger filename prefix override (post-pipeline)**: the plan originally specified preserving the `"rulegen"` log-file prefix for compatibility (`<logsDir>/rulegen.YYYYMMDD.log`). During the engineer stage, the user overrode this: log filenames follow the binary's name, so `cmd/doctor` writes to `<logsDir>/doctor.YYYYMMDD.log` regardless of subcommand. Reason: hard rule that "log filenames always match binary/service name". Operators previously tailing `logs/rulegen.*.log` must update to `logs/doctor.*.log`. The in-line summary text `"rulegen --all: ..."` is preserved verbatim (separate concern — test/grep compatibility).
