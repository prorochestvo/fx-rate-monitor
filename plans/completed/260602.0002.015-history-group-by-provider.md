# Task Breakdown

## Overview

The per-pair history modal in the Telegram Mini App leaks the storage model
into the UI: each provider that publishes both BID and ASK for the same pair
is configured in `rate_sources` as two distinct rows (e.g. `KZ_BCC_FX_BID_USD_KZT`
and `KZ_BCC_FX_ASK_USD_KZT`, both with `title="Center Credit Bank (FX)"`).
Today the history service in `internal/application/chart/history.go` groups by
`(SourceName, Timestamp)` and the WASM client builds chips from the
`SourceName → SourceTitle` map. Result: the chip row shows the same title
twice (one per direction), and a single scrape moment is rendered as two rows
(one BID-only, one ASK-only).

This plan rewrites the grouping unit from `SourceName` to `SourceTitle`
end-to-end so the **provider title** is the user-facing identity. Each chip
appears once, each timestamp collapses BID and ASK from sibling sources into
a single row carrying both columns. The change touches: DTO, history service,
HTTP handler, route query param (`source_name` → `source_title`), WASM
apiclient and URL builder, WASM state and renderer. The internal source name
is dropped from the wire — there are no third-party clients and the contract
is one day old (commits `722fa58`, `531d4a5`, `5ee73ce`), so we replace
without a back-compat shim.

The `Total` accounting is the trickiest part. The repository's
`ObtainHistoryForPairsPaged` returns a row-level total counted in SQL (one
count per `rate_values` row). After we group by `(SourceTitle, Timestamp)`
in memory, that total overshoots — two rate_values rows at the same scrape
moment for the same provider become one history row. Pagination math then
shows phantom next pages. We resolve this in the service layer by computing
the grouped Total in Go and overriding the repo's row-count (see Task 2).

## Assumptions

These hold in the current schema and seeds and are the load-bearing
invariants for the rewrite. If any of them break in the future, the model
collapses.

1. Each provider exposes at most one BID source and one ASK source per
   `(BaseCurrency, QuoteCurrency)` pair. The grouping logic assumes there
   are never two BID sources for one provider on one pair — if there were,
   they would silently collapse and the second one's price would overwrite
   the first within a `(title, timestamp)` group.
2. `SourceTitle` is unique per provider across all the sources a single user
   is subscribed to. Two unrelated providers sharing the literal string
   `"Center Credit Bank (FX)"` would be merged into one chip and one
   timestamped row. The seeds today enforce this by spelling each provider
   title once; document the invariant in the service godoc so future seed
   edits do not violate it.
3. Title matching is case-sensitive and byte-for-byte (no Unicode folding,
   no whitespace normalisation). The seeds use one canonical spelling per
   provider; if a user is subscribed to a typoed duplicate, the chip row
   will show both spellings as distinct chips. Acceptable for v1.
4. A Mini-App-only API change is acceptable. The dashboard at `/admin/`
   does not call `/api/me/rates/history`. The wire change ships in lockstep
   with the WASM bundle in the same deploy.
5. The v1 delta-chain limitation (no cross-page anchoring) is preserved.
   First row in any `(title, direction)` chain within a page has nil delta.

### Task 2: Group history by `(SourceTitle, Timestamp)` in the service

- **Description:** Rewrite the grouping pass in
  `internal/application/chart/history.go` `ObtainMeHistory`. Replace the
  current `groupKey{sourceName, timestamp}` with `groupKey{sourceTitle,
  timestamp}`. Replace the per-source delta-state map keyed by
  `SourceName` with a per-title map. Carry both BID and ASK across two
  underlying `rate_sources` rows (different `SourceName`, same `Title`)
  into one `MeHistoryRowResult`.

  Drop the `SourceName` field from `MeHistoryRowResult` so the service-
  layer output reflects the wire contract.

  Resolve the `Total` accounting. The repository's `total` is the count of
  `rate_values` rows matching the WHERE clause — it overcounts after
  in-memory grouping because two underlying sources at the same scrape
  moment collapse. **Decision:** keep `ObtainHistoryForPairsPaged`'s
  signature unchanged; do not push grouping into SQL (it would require a
  join against `rate_sources` and a `COUNT(DISTINCT title, timestamp)`
  expression that defeats the existing tuple-IN strategy). Instead,
  compute the grouped Total in Go.

  The grouped total cannot be derived from one page alone — page 1 might
  not contain every distinct `(title, timestamp)` combination. The
  simplest correct fix is to fetch all matching rows (no LIMIT/OFFSET) for
  the count step. That risks unbounded reads in pathological accounts.
  The pragmatic v1: continue to paginate at the repository layer over
  `rate_values` rows, then map the row-level total to a grouped total
  using the observed grouping ratio on this page. That math is fragile
  (the last page may have a different ratio than middle pages).

  The chosen approach is straightforward and correct:
  add one extra repository call (or a new repository method) to count
  `COUNT(DISTINCT timestamp)` joined to `rate_sources.title` for the same
  WHERE. Concretely: add
  `CountDistinctTitleTimestampForPairs(ctx, []SourcePairKey) (int64, error)`
  to the `HistoryValuesLoader` interface and implement it in
  `internal/repository/ratevalue.go` against
  `SELECT COUNT(DISTINCT rs.title || '|' || rv.timestamp) FROM rate_values rv
  JOIN rate_sources rs ON rs.name = rv.source_name WHERE (rv.source_name,
  rv.base_currency, rv.quote_currency) IN (...)`. Use this as the new
  `Total`. The pipe-delimited concatenation is safe because titles do not
  contain the pipe character in any current seed (call it out in a code
  comment; if it ever becomes a concern, swap to a CTE with `GROUP BY`).

  Filter input changes from `sourceName` to `sourceTitle`. Apply the
  filter by intersecting `matchingKeys` with the set of source names whose
  `sourceMeta[name].Title == sourceTitle`. Preserve the existing
  semantics: empty filter → no filter; unknown title → return empty page
  with `Total=0` (200 OK).

- **Acceptance Criteria:**
   - [ ] `ObtainMeHistory` signature becomes
         `ObtainMeHistory(ctx, userID, pair, sourceTitle string, page, limit int64) (*MeHistoryResult, error)`.
   - [ ] `MeHistoryRowResult.SourceName` is removed; `SourceTitle`
         remains and is the grouping key.
   - [ ] When the page contains both BID and ASK rate_values rows at the
         same `Timestamp` from two sibling sources sharing a title, the
         result has exactly one row with both `Bid` and `Ask` populated.
   - [ ] When the page contains only BID for a `(title, timestamp)`, the
         row's `Ask` is nil; same in reverse.
   - [ ] `Total` reflects the count of distinct `(title, timestamp)`
         tuples matching the WHERE — not row count, not BID-row count
         alone. A page 1 with 2 timestamps × (BID+ASK) reports `Total=2`,
         not `Total=4`.
   - [ ] Filter with a known title narrows results to that provider's
         sources only.
   - [ ] Filter with an unknown title returns
         `Total=0`, `Items=[]`, no error.
   - [ ] Delta computation keys by `(SourceTitle, direction)`. Two
         provider chains are independent.
   - [ ] The godoc on `ObtainMeHistory` documents Assumption 1
         (one-BID-one-ASK per provider per pair) and Assumption 2 (titles
         are unique per provider) as load-bearing invariants.
- **Pitfalls:**
  - Forgetting to update the delta state map's key — leaving it keyed by
    `SourceName` while the rows are keyed by title would re-introduce
    duplicate chains for the BID side and ASK side of one provider.
  - Forgetting the `seenSources` dedup still applies at the source-name
    level (we need every distinct source name in `matchingKeys` because
    the repository's tuple-IN filter is by name). Do not switch
    `seenSources` to dedup by title — that would drop one of the two
    sibling sources from the SQL WHERE.
  - The pipe-delimiter `COUNT(DISTINCT)` trick: make sure the `||`
    expression is `rs.title || '|' || rv.timestamp` and not the other
    direction; SQLite's string ordering only matters if you later add
    `GROUP BY`. Add a unit test asserting `Total` with a title containing
    a literal `|` to catch future regressions.
  - `Total` is now computed independently of the row-list query. If the
    `COUNT(DISTINCT)` call fails after the SELECT succeeds, surface the
    error and return — do not return a partial result with a wrong Total.
  - Empty `matchingKeys` after filtering must short-circuit before the
    `COUNT(DISTINCT)` call (the existing `if len(matchingKeys) == 0`
    guard already handles this — verify it still runs first).
- **Complexity:** Hard
- **Code Example:**
  ```go

      filtered := matchingKeys[:0]
      for _, k := range matchingKeys {
          if src, ok := sourceMeta[k.SourceName]; ok && src.Title == sourceTitle {
              filtered = append(filtered, k)
          }
      }
      matchingKeys = filtered
  }

  // Grouping key (replaces the old groupKey{sourceName, timestamp}).

      timestamp   time.Time
  }
  ```

## Risks

1. **Two providers sharing a title would silently collapse.** The
   `feedback_minimize_pii_in_fxmonitor` and the existing seeds enforce
   uniqueness today, but a future seed addition could violate it. The
   service-level godoc must call this out so the next person adding a
   provider knows. Detection in production would be confusing: one user
   sees only one chip and one merged row when there should be two
   providers. Mitigation: add a `doctor audit` check (out of scope for
   this plan; flag as a follow-up) that warns when two active sources
   share a `(title, base, quote)` tuple.

2. **`Total` accounting bug breaks pagination.** If
   `CountDistinctTitleTimestampForPairs` returns a value out of sync
   with the rows the SELECT actually produces (e.g. via a timing race
   between two reads in `ReadOnlyTransaction`), the next/prev buttons
   render incorrectly. The two queries run in the same read transaction,
   so the snapshot is consistent; verify by reading
   `ReadOnlyTransaction` semantics in `sqlclient.go` before relying on
   this.

3. **The pipe-delimiter trick in `COUNT(DISTINCT)`.** If a future title
   includes a literal `|`, the count under-reports. The unit test in
   Task 3 locks this in; a CTE-based rewrite is the fallback if titles
   ever legitimately contain pipes.

4. **WASM bundle / server skew during deploy.** Browsers caching the
   old `app.wasm` would call the old endpoint variant (`source_name`)
   and the new server would silently ignore the filter (treating it as
   empty). For one deploy cycle, plan to bump the WASM bundle's
   cache-busting path or accept "stale Mini-App users see the unfiltered
   view until they refresh." Acceptable — the contract is one day old,
   no production user has trained muscle memory on filters.

5. **The `SourceName`-gone leak for debugging.** Operator tooling that
   reads the `/api/me/rates/history` response (none exists today) would
   need the underlying source name to correlate with the `rate_sources`
   table. Acceptable trade-off documented under Trade-offs.

## Trade-offs

- **Drop `SourceName` from the wire entirely vs keep it for debugging.**
  Chose drop. The Mini App does not display it, the operator dashboard
  does not consume it, and keeping it would re-introduce the same
  duplicate-chip footgun on any future renderer that forgets to dedup.
  Operators correlate via `rate_values` directly when needed.

- **Group in SQL vs group in Go.** Chose Go. SQL grouping would require
  a `JOIN rate_sources` plus a `GROUP BY rs.title, rv.timestamp` with
  pagination over the grouped result. The repository currently does its
  IN-tuple filter on `rate_values` alone and assumes the SELECT shape
  matches the row-level model — pushing grouping into SQL would force a
  larger rewrite (and complicate the existing `rateValueSqlSelect`
  scaffolding). In-memory grouping over a single page is O(rows-per-page)
  with no allocations beyond the result slice. Cost: one extra
  read-only query per page for the grouped Total.

- **Add a new repository method vs change the existing one's return
  shape.** Chose new method. Changing `ObtainHistoryForPairsPaged`'s
  return shape to drop the row-level total would ripple into
  `internal/repository/ratevalue_test.go` for no functional gain; that
  helper is fine as-is. The cost is a second-method-per-call pattern,
  matching how the rest of the repo already mixes row queries and count
  queries.

- **`source_title` query param vs `provider` or `source` alias.** Chose
  `source_title`. Mechanical rename keeps the change reviewable; the
  `source_*` prefix groups consistently with the response field name
  `SourceTitle`. A more neutral `provider=` would require renaming the
  DTO field too, which is bigger churn for the same UX.

- **`map[string]struct{}` vs keeping `map[string]string` for
  `KnownSources`.** Chose set. After the rename, the value would equal
  the key — the map degenerates into a set, and an explicit set type
  prevents the next reviewer asking why the values are identical to
  the keys. Cost: one ripple into the renderer's iteration code, which
  is a one-line change.

- **Rename `SelectedSource` → `SelectedSourceTitle` vs keep the field
  name and change only the semantics.** Chose rename. A field named
  `SelectedSource` carrying a title would be a footgun for the next
  contributor; the cost of `git grep` + replace is small.
