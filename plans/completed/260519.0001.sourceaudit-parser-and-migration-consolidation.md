# Task Breakdown

## Overview

Two related but independent concerns land together in one commit:

1. **`sourceaudit` seed parser is silently blind** to the 13-column rows
   introduced by migrations `202605.009` (add `rule_metadata`),
   `202605.010` (add `fetcher_kind`), and `202605.014` (column-list
   `INSERT OR IGNORE INTO rate_sources (col1, ...) VALUES (...)` form).
   The parser at `internal/application/sourceaudit/seedparser.go:30`
   hardcodes the positional 10-column prefix
   `INSERT OR IGNORE INTO rate_sources VALUES(` and rejects anything
   else as "not an INSERT" — without logging or erroring. The four
   Jusan/Halyk rows in `202605.014.rate_sources.seed_jusan_halyk_usd_kzt.sql`
   are therefore invisible to `make audit`, which reports 32 sources
   instead of the real 36.

2. **Migrations 007–014 are unpushed** and can be consolidated
   before they reach `origin/main`. Migrations 001–006 are immutable
   (already in `origin/main`) and must not be touched. The
   consolidation collapses 8 files into 4 while producing a
   bit-identical final schema and data set vs. the current sequence.
   The one explicit behavioural simplification is dropping the
   contents of `202605.012` (BCC `wait_selector` UPDATE) — by the
   time `202605.013` deactivates that row, the wait_selector is
   useless cosmetic state on an inactive row.

This plan covers planning only. Implementation, review, and the
plan move into `completed/` follow the standard pipeline.

## Assumptions

- Migrations `202605.001`–`202605.006` are immutable and pushed to
  `origin/main` (verified via `git branch -r --contains fc1ad9d`).
  They will not be touched.
- Migrations `202605.007`–`202605.014` are unpushed (verified via
  `git branch -r --contains 1e896ee 1295273` returning no remote
  refs). They are safe to delete, rename, or rewrite.
- Production deployments have never run 007–014, so the
  `__schema_migrations` table on prod is unaffected.
- The user's **local** database at
  `build/infrastructure/monitor.sqlite` has 007–014 recorded under
  their current filenames in `__schema_migrations`. The user has
  agreed to wipe the local DB before re-running.
- Dropping the `wait_selector` update from old `202605.012` is the
  only intentional behavioural difference. Final state for
  `KZ_BCC_BID_USD_KZT` after the consolidated migrations: row
  exists with `active = 0`, `options = '{}'`, `fetcher_kind = 'chromedp'`,
  `rule_metadata = '{}'`. Final state under old chain: same row
  with `active = 0`, `options = '{"wait_selector":"div.text-lg"}'`,
  `fetcher_kind = 'chromedp'`, `rule_metadata = '{}'`. Since the
  row is inactive nothing consumes `options`; the divergence is
  inert.
- Post-migration `rate_sources` schema column order is:
  `name, title, base_currency, quote_currency, url, interval,
  kind, active, options, rules, rule_metadata, fetcher_kind`
  (12 columns). The "13-column" figure in the task brief includes
  an implicit row-id; SQLite's `INSERT VALUES(...)` positional form
  takes exactly these 12 user-defined columns. The seedparser must
  therefore handle a positional row of 12 tokens (legacy 10 plus
  `rule_metadata` and `fetcher_kind`) and a column-list row of any
  subset, mapped by name.
- The `cmd/sourceaudit` glob default is `migrations/*.seed*.sql`
  (verified at `cmd/sourceaudit/main.go:22`). Any consolidated file
  containing seed `INSERT`s must keep the substring `seed` in its
  filename or the audit tool will not match it. ALTER-only files
  (no inserts) do not need to match.
- "Headers" column (`options` per the schema) in legacy 10-column
  positional inserts maps to the `options` field. The current
  parser comments call it `headers_json` (seedparser.go:186) but
  never uses the value; the field name in the consolidated parser
  should be `options` to match the schema.
- The engineer will run `make test` and `make audit` after the
  consolidation; the reviewer gates approval on both being clean.

## Tasks

### Task 1: Extend `sourceaudit` seed parser to accept both INSERT forms with named column mapping

- **Description:**
  Rewrite `parseSeedFile` and friends in
  `internal/application/sourceaudit/seedparser.go` so that:

  1. It recognises both shapes:
     - Positional: `INSERT OR IGNORE INTO rate_sources VALUES(<v1>, ..., <vN>);`
     - Column-list: `INSERT OR IGNORE INTO rate_sources (col1, col2, ...) VALUES (<v1>, ..., <vN>);`
     The detection is a strict prefix match on
     `INSERT OR IGNORE INTO rate_sources` (with one or more spaces
     allowed before the `(` or `VALUES`). Any whitespace tolerance
     uses `strings.Fields` / regex, not ad-hoc trimming.

  2. The new constants `insertPrefixWithColumns` and
     `insertPrefixPositional` replace the single `insertPrefix` at
     `seedparser.go:30`. The recogniser function returns one of three
     verdicts: `noMatch`, `positional`, `columnList` along with the
     parenthesised payload(s).

  3. When the form is **column-list**, parse the column-name list
     (comma-separated, unquoted SQL identifiers, whitespace-tolerant)
     and the value list independently. Build a
     `map[string]string` of column name to raw token, then read each
     `SeededSource` field by looking up its expected column name
     in that map. Unknown column names are **skipped with a one-line
     stderr WARN** (e.g. `sourceaudit: %s:%d: ignoring unknown column %q`)
     — this surfaces schema drift but does not fail the parse.

  4. When the form is **positional**, fall back to a defined
     legacy column order:
     `[]string{"name", "title", "base_currency", "quote_currency",
     "url", "interval", "kind", "active", "options", "rules",
     "rule_metadata", "fetcher_kind"}`. Accept either 10 tokens
     (legacy pre-009 rows) or 12 tokens (post-010 rows). The slice
     length determines which suffix of the column list is filled;
     missing trailing fields default to `rule_metadata = "{}"` and
     `fetcher_kind = "plain"` (matching the migration default
     clauses). Reject any other arity loudly.

  5. The function used to build a `SeededSource` from the
     name-to-token map (call it `seededSourceFromColumns`) extracts
     these expected names and ignores everything else (with the
     WARN above):
     - Required: `name`, `title` (→ `Vendor`), `base_currency`,
       `quote_currency`, `url`, `interval`, `kind` (→ `Side`),
       `active`, `rules`.
     - Optional: `options`, `rule_metadata`, `fetcher_kind` (parsed
       but not yet surfaced on `SeededSource` — leave them as
       struct fields only if a future task needs them; do **not**
       add fields just for symmetry).

  6. **Fail loudly** when a line begins with
     `INSERT OR IGNORE INTO rate_sources` but the parenthesised
     payload cannot be tokenized, the column list is malformed,
     a required column is missing, or `active` is not 0/1. The
     error chain returns a `%s:%d: <reason>` formatted error and
     the caller (`ParseSeedFiles`) propagates it — the existing
     `_, err := ParseSeedFiles(...)` callers will see it. This is
     the behaviour change that prevents the original silent-skip
     bug from recurring.

  7. Lines that do **not** start with the `INSERT OR IGNORE INTO
     rate_sources` prefix (other tables, `CREATE TABLE`, `UPDATE`,
     `ALTER`, comments, blank lines) are still silently ignored —
     keep the existing test `blank lines comment lines non-INSERT
     statements ignored` green.

  The reformat must keep the existing exported surface (`ParseSeedFiles`,
  `SeededSource`) identical and the existing `Origin` field semantics
  unchanged (`<basename>:<lineNum>`).

- **Acceptance Criteria:**
  - [ ] `internal/application/sourceaudit/seedparser.go` no longer
        contains the `const insertPrefix = "INSERT OR IGNORE INTO
        rate_sources VALUES("` declaration; replaced by the

- **Pitfalls:**
  - The legacy `realSeedLine` fixture (line 14 of `seedparser_test.go`)
    is verbatim 10-column form. It must still parse — the positional
    path with 10 tokens has to keep working.
  - The `tokenizeSQLValues` helper handles single-quoted strings
    with `''` escapes but **not** backslash escapes inside JSON.
    The Jusan/Halyk rules contain `\\\\` (four backslashes in the SQL
    source, representing JSON's `\\` which is regex's `\`). Verify
    the tokenizer doesn't trip — it shouldn't, because backslashes
    aren't special inside a SQL single-quoted string, but the
    regression test forces the issue.
  - Column-list parsing must be whitespace-tolerant. The real file
    uses `(name, title, base_currency, ...)` with single spaces, but

    to stdout — `cmd/sourceaudit` prints its report to stdout and
    mixing channels would corrupt machine-readable consumption.

    the regression subtest's assertion; do not change it.

- **Complexity:** Medium.

- **Code Example:**

  ```go
  // recogniseInsert classifies a SQL line into one of three forms.
  // Returns the parenthesised payload(s) when applicable. For the
  // column-list form, columnsPayload is the comma-separated column
  // names and valuesPayload is the comma-separated value tokens.
  type insertForm int

  const (
      noMatch insertForm = iota
      positional
      columnList
  )

  // Matches:
  //   INSERT OR IGNORE INTO rate_sources VALUES(...)
  //   INSERT OR IGNORE INTO rate_sources (col, col, ...) VALUES (...)
  // with flexible whitespace between tokens.
  var (
      reInsertPositional = regexp.MustCompile(
          `^INSERT\s+OR\s+IGNORE\s+INTO\s+rate_sources\s+VALUES\s*\((.*)\)\s*;\s*$`)
      reInsertColumnList = regexp.MustCompile(
          `^INSERT\s+OR\s+IGNORE\s+INTO\s+rate_sources\s*\(([^)]*)\)\s+VALUES\s*\((.*)\)\s*;\s*$`)
      reLooksLikeInsert = regexp.MustCompile(
          `^INSERT\s+OR\s+IGNORE\s+INTO\s+rate_sources\b`)
  )
  ```

  The recogniser tries column-list first (longer match), then
  positional. If neither matches but `reLooksLikeInsert` does, the
  caller returns a "malformed insert" error — this is the loud-fail
  path.

  6. **Deactivation effective:** `SELECT active FROM rate_sources
     WHERE name = 'KZ_BCC_BID_USD_KZT';` → expect `0`.

  7. **Audit parser count check:** `make audit ARGS='-only=^IMPOSSIBLE$'`
     to dry-run the parser without any HTTP probing. Pipe output
     through `grep -c 'sources matching glob'` or, simpler, run a
     temporary Go test (see code example) that invokes
     `sourceaudit.ParseSeedFiles(os.DirFS(repoRoot), "migrations/*.seed*.sql")`
     and prints `len(sources)`. Expect **36** (32 BCC/NBK/QazPost
     + 4 Jusan/Halyk).

  8. **Make test green:** `make test` passes end-to-end, including
     the new sourceaudit subtests from Task 1.

  Engineer captures each command and its actual output, pastes the
  block into the commit message body or into the plan-completion
  hand-off. Reviewer compares against expected.

  **Optional but recommended:** add a tiny test
  `internal/application/sourceaudit/seedparser_count_test.go` (one
  function) that invokes `ParseSeedFiles` against the real
  `migrations.MigrationsFS` and asserts `len(sources) == 36`.
  This converts the manual step 7 into a permanent regression
  guard. If the engineer adds it, name it
  `TestParseSeedFiles_EmbeddedMigrations` with a single subtest
  `embedded migrations enumerate 36 sources`. This test would have
  caught the original silent-skip bug.

- **Acceptance Criteria:**
  - [ ] Engineer's commit message (or plan-completion note)
        contains the output of all eight verification steps above.
  - [ ] `make test` exits 0.
  - [ ] `make audit` exits 0 (or returns the live-probe verdict,
        whatever that is at the moment — the binary's parse phase
        must succeed and report >= 36 sources before probing).
        Captured output names 36 sources, not 32.
  - [ ] If the optional test is added: `CGO_ENABLED=0 go test
        -race -run TestParseSeedFiles_EmbeddedMigrations
        ./internal/application/sourceaudit/...` passes.
  - [ ] No new test fixtures are committed under the repo root
        (per memory `feedback_no_scratch_files_in_root`); any
        scratch files live under `./tmp/` and are not added to
        git.

- **Pitfalls:**
  - The `make audit` default behaviour fires live HTTP requests
    against every source — this can take minutes and may flap on
    rate-limited endpoints. For the count check only, the engineer
    should use a no-match `-only` regex (step 7) to short-circuit
    the probing phase, OR add the optional test which sidesteps
    HTTP entirely.
  - If the optional test is added, it must be a **build-tag-free**
    test that runs in the default `go test ./...` set. Don't gate
    it behind `//go:build integration` — the whole point is to
    catch silent regressions automatically.
  - The test must use the `migrations.MigrationsFS` embed, not
    `os.DirFS("../../../migrations")`. Using the embed keeps the
    test working regardless of CWD and matches how production
    actually consumes the files.
  - The expected count (36) is load-bearing on Task 2 producing
    exactly 32 + 4 sources. If Task 2 changes any row count, the
    expected number in the test must update in lockstep — both
    in this plan and in the test.

- **Complexity:** Easy.

- **Code Example** (optional regression test sketch):

  ```go

      t.Parallel()

          t.Parallel()
  ```

## Execution Order

1. **Task 1 first.** Land the parser fix before the migration
   consolidation. Reason: the new column-list-aware parser must be
   in place to validate Task 2's output via Task 3's audit count
   check. Doing it the other way leaves a window where the engineer
   can't tell whether a new migration row parsed correctly.

2. **Task 2 next.** With the parser updated, consolidate the
   migrations. The 8-delete-4-add operation is staged in a single
   commit so the bisect surface is one revision, not eight.

3. **Task 3 last.** Run the verification sequence. If any step
   fails, loop back to Task 1 or Task 2 with specific findings; do
   not paper over by mutating the expected counts.

Tasks 1 and 2 can technically be parallelised (different files),
but the verification ordering still requires Task 1 to land before
Task 3's "audit reports 36" can be observed. Recommend serial
execution by the engineer for cognitive simplicity.

## Risks

### R1 — Local SQLite DB wipe required after pull

**Cause:** the user's local `build/infrastructure/monitor.sqlite`
records 007–014 under their old filenames in `__schema_migrations`.
On next `make migrate`, the migrator sees 4 "new" filenames
(`202605.007.rate_sources.seed_initial.sql`,
`202605.008.rate_sources.add_rule_metadata_and_fetcher_kind.sql`,
`202605.009.rate_sources.rename_headless_to_chromedp.sql`,
`202605.010.rate_sources.seed_jusan_halyk_and_deactivate_bcc.sql`)
and tries to apply them. File B's
`ALTER TABLE rate_sources ADD COLUMN rule_metadata` will fail
because that column already exists. The migrator aborts; the user
sees a confusing duplicate-column error reminiscent of the
"page is nil" saga.

**Mitigation / recovery procedure** (must appear in the engineer's
hand-off / commit message body):

```
# 1. Stop the running services (if any) so the DB file isn't held open.
make stop  # or: kill the collector/notifier/web processes manually

# 2. Back up the existing DB to ./tmp/ in case the user wants
#    to inspect subscription rows added after the seed.
mkdir -p ./tmp
cp build/infrastructure/monitor.sqlite ./tmp/monitor.pre-consolidation.sqlite

# 3. Wipe the live DB.
rm build/infrastructure/monitor.sqlite \
   build/infrastructure/monitor.sqlite-shm \
   build/infrastructure/monitor.sqlite-wal

# 4. Re-apply migrations from a clean state.
make migrate

# 5. Sanity-check.
sqlite3 build/infrastructure/monitor.sqlite \
   'SELECT COUNT(*) FROM rate_sources;'   # expect 36
sqlite3 build/infrastructure/monitor.sqlite \
   'SELECT filename FROM __schema_migrations ORDER BY filename;'
   # expect 001..010, ten rows
```

The engineer **asks the user before running step 3** per memory
`feedback_ask_before_destructive_tmp_ops`. The wipe is destructive
for any subscriptions the user added beyond the seeded admin set
since the original seed ran — the backup in step 2 lets the user
manually re-insert them if needed.

**Why prod is unaffected:** none of 007–014 reached `origin/main`
(verified via `git branch -r --contains`), so prod's
`__schema_migrations` does not have those rows. CI will apply
001–010 fresh.

### R2 — Behavioural divergence on inactive BCC row

**Cause:** the consolidation drops old 012's `wait_selector` UPDATE.
Final `options` value for `KZ_BCC_BID_USD_KZT` is `'{}'` instead of
`'{"wait_selector":"div.text-lg"}'`. The row is inactive so
collector and notifier never read it, but a future reactivation
without re-adding `wait_selector` would regress to the
pre-`wait_selector`-fix behaviour.

**Mitigation:** the leading comment in File D names the dropped
behaviour explicitly. If/when the row is reactivated, the
reactivator must restore `wait_selector` themselves. Acceptable
risk because the row is parked indefinitely; if it stays parked,
the absence is moot.

### R3 — `rate_user_subscriptions` column-list mismatch

**Cause:** old 008's positional inserts have 9 tokens. The author
of the original file assumed a particular column order; that order
is whatever `202605.003.rate_user_subscriptions.table_initiate.sql`
declares. Rewriting to column-list form requires reading that
schema. If the engineer guesses the order wrong, the rewritten
inserts will land columns under wrong names (e.g. `threshold` value
in `mode` column) and the subscription engine will malfunction.

**Mitigation:** engineer reads
`202605.003.rate_user_subscriptions.table_initiate.sql` **before**
writing File A, copies the column list literally, and verifies one
admin row's effective fields with a `SELECT` after migration.

### R4 — `make audit` glob mismatch on new filenames

**Cause:** the audit glob `migrations/*.seed*.sql` matches by
substring `.seed`. Filenames lacking that substring are silently
ignored. If the engineer names File D something like
`202605.010.rate_sources.jusan_halyk_and_deactivate_bcc.sql`
(without `seed`), the four Jusan/Halyk inserts will be invisible
to the audit tool again and the regression returns.

**Mitigation:** Task 2 acceptance criterion mandates the exact
filename including `seed_jusan_halyk_and_deactivate_bcc`. Task 3
acceptance verifies count == 36.

### R5 — Test fixtures for new parser break legacy fixtures

**Cause:** the new column-list path could accidentally short-circuit
the positional path if both detection regexes match the same line
(e.g. if column-list regex is greedy on payload that contains
`VALUES`).

**Mitigation:** Task 1 requires the recogniser to try column-list
first with a strict bracketed column-name match `\(([^)]*)\)`. The
legacy positional regex requires `VALUES\s*\(` immediately after
`rate_sources`, which the column-list regex cannot accidentally
match. The existing `realSeedLine` test exercises the positional
path; all current subtests must remain green.

## Trade-offs

### T1 — Column-list vs positional INSERT form in consolidated files

**Decision:** use column-list form in Files A and D (the data files).

**Reason:** column-list is robust against future `ALTER TABLE ADD
COLUMN` migrations. Today the table has 12 columns; tomorrow it may
have 14. With column-list inserts, future schema additions don't
break the seed rows. With positional inserts, every `ADD COLUMN`
that lands between the seed file and a fresh apply forces a
rewrite of every seed row.

The cost is verbosity (~50 more bytes per insert) and the parser
must understand both forms — which Task 1 already requires anyway.

**Alternative considered:** keep positional in File A (matches old
007 byte-for-byte) and use column-list only in File D. **Rejected**
because mixing the two styles within a single consolidation makes
the rationale opaque to future readers and provides no upside.

### T2 — Fold old 011 (`headless → chromedp` rename) into the new 008, or keep as separate 009?

**Decision:** keep as separate File C (`202605.009`), but make it a
no-op against the post-consolidation seed.

**Reason for keeping separate:** the user explicitly requested it
("011 — kept (data-only rename inside `kind`; conceptually
separate)"). The conceptual separation (schema additions vs.
enum-value rename) is real and surfaces in commit history /
filename / standalone reviewability. The cost of a no-op UPDATE
running once at apply time is negligible (<1ms) and the
defensive value is non-zero: if a future plan re-introduces a
`headless` value in error, this migration silently normalises it
on the next deploy.

**Alternative considered:** fold into File B, so File B reads
`ADD COLUMN fetcher_kind TEXT NOT NULL DEFAULT 'plain'` followed by
`UPDATE rate_sources SET fetcher_kind = 'chromedp' WHERE name =
'KZ_BCC_BID_USD_KZT'` — effectively replacing the old 010+011
chain in a single statement. **Rejected** on user instruction, but
worth noting as the leaner alternative if priorities change.

A third alternative — keep File C but make it conditional on
`EXISTS (SELECT 1 ... WHERE fetcher_kind = 'headless')` —
**rejected** as cute-but-not-useful: SQLite executes the UPDATE
in O(table) regardless; the conditional saves nothing and adds
clutter.

### T3 — Drop old 012 (`wait_selector` UPDATE on inactive row)

**Decision:** drop entirely.

**Reason:** user has explicitly confirmed the row is permanently
inactive ("012 wait_selector бесполезный на деактивной строке —
опускаем"). Carrying dead data into a fresh apply chain adds
nothing.

**Acknowledged divergence:** final state of
`KZ_BCC_BID_USD_KZT.options` is `'{}'` instead of
`'{"wait_selector":"div.text-lg"}'`. Row is inactive; collector
and notifier never read it. See R2 for mitigation.

### T4 — Heavy E2E schema-equality test vs. manual verification

**Decision:** manual verification (Task 3 sequence) plus optional
embedded-migrations count test.

**Reason:** building an automated harness that runs both the old
and new migration chains against ephemeral SQLite files and
compares `.schema` + row-by-row content would consume more effort
than the entire rest of this plan. The user's stated minimum is
"manual verification sequence the engineer must walk through and
report results." The optional regression test (`TestParseSeedFiles_
EmbeddedMigrations`) covers the original silent-skip bug class at
near-zero cost, which is the highest-value automation here.

**Alternative considered:** golden-file comparison of `sqlite3
test.db .dump` output between old and new chains. **Rejected**
because the test would need to check out the previous git
revision to obtain the old chain, doubling the CI complexity and
adding `git` as a test dependency. Defer until a second
consolidation is in scope.

### T5 — Add new fields to `SeededSource` for `rule_metadata` / `fetcher_kind`?

**Decision:** do **not** add new fields in this plan.

**Reason:** the auditor doesn't read either value; adding fields
"because they exist in the schema" violates the project's "no
speculative abstractions" rule. The parser accepts the columns
(via the WARN-on-unknown fallback in Task 1) so future audit
features can add the fields when they actually need them, without
needing another parser pass.

**Alternative considered:** add `RuleMetadata string` and
`FetcherKind string` to `SeededSource` now. **Rejected** for the
reason above; revisit when a downstream consumer materialises.
