# Task Breakdown

## Overview

Delete the `internal/bootstrap` package entirely and inline its three
responsibilities (logger construction, DB open + schema gate, repository
construction) into each consuming `cmd/*/main.go`. The three current importers
are `cmd/web`, `cmd/collector`, and `cmd/notifier`. After all three are
inlined, the package and its test file are deleted and `CLAUDE.md` is updated.

This is a **behavior-preserving refactor**. Runtime behavior must be unchanged:
same daily log filename prefix per binary, same schema gate, same set of
repositories per binary, same fatal-on-missing-config semantics, same startup
log-marker sequence (`logger -> settings -> dependencies -> repositories -> ...`).

The guiding principle (locked with the user, not up for debate): the three
service binaries look similar today but must be free to diverge — different DBs,
different dependency sets — without a shared "startup" layer imposing a contract.
A few duplicated trivial lines per binary is the accepted, intended cost. **No
replacement helper is introduced** (no `OpenServiceClient`, no
`OpenMigratedClient`, nothing). The pattern to mirror already exists in the
repo: `cmd/migrator/main.go` and `cmd/doctor/rulegen.go` both inline this exact
wiring today and import nothing from `internal/bootstrap`.

What `internal/bootstrap` bundles and how each is inlined:

1. **`NewLogger`** — a thin wrapper over `internal.NewLogger`. CAVEAT: the
   argument order is flipped. `bootstrap.NewLogger(binaryName, logsDir, verbosity)`
   delegates to `internal.NewLogger(logsDir, binaryName, verbosity)`. Verified
   signature: `internal.NewLogger(logsDir, name string, printerMinLevel loginjector.LogLevel)`
   (`internal/logger.go:38`). The inline call is therefore
   `internal.NewLogger(LogsDir, "<binary>", LogVerbosity)` — note `LogsDir`
   first, the binary name second. Getting this backwards silently corrupts the
   daily log filename prefix, which is a hard project rule
   (`feedback_log_filename_matches_binary`). `cmd/migrator/main.go:48` already
   does it correctly: `internal.NewLogger(LogsDir, "migrator", LogVerbosity)`.

2. **`NewDB`** — inline as: `dsninjector.Unmarshal(<envVar>)` →
   `sqlitedb.NewSQLiteClient(dsn, logger)` → `sqlitedb.RequireMigratedSchema(ctx, db)`,
   each error handled with `log.Fatalf`. The `close-on-error errors.Join` dance
   in `bootstrap.NewDB` (`bootstrap.go:61-66`) is **dropped**. That dance exists
   only because `NewDB` is a *returning constructor* that must not leak a
   half-open handle before `return`. In `main.go` you do not return — you
   `log.Fatalf`, the process exits, and the OS reclaims the fd; on a schema-check
   failure there is nothing to checkpoint (read-only). `cmd/doctor/rulegen.go:227-241`
   already inlines exactly this chain without the close-join, which is the
   proof-of-concept for this simplification. The single real invariant — "do not
   boot on unmigrated schema" — stays intact because it is already a public
   one-liner: `sqlitedb.RequireMigratedSchema`.

3. **`Repositories` struct + `NewRepositories`** — inline: each `main.go`
   constructs ONLY the repositories it actually uses (verified per binary below),
   each with its own `log.Fatalf` on the constructor error. The
   `bootstrap.Repositories` struct goes away.

## Assumptions

- Branch is `refactorin`; commit freely, no push (per task brief and
  `feedback_commit_freely_off_main`). Each task is its own atomic commit.
- `make test` runs `CGO_ENABLED=0 go vet ./... && CGO_ENABLED=0 go test -race ./...`
  (Makefile:110-113). `make build` builds all six binaries with `CGO_ENABLED=0`
  (Makefile:89-97). Both must be green at the end.
- The success-path `defer db.Close()` already present in all three main.go files
  is kept verbatim. It is understood and intended that `log.Fatalf` ⇒
  `os.Exit(1)` ⇒ deferred `Close` does NOT run on the fatal path. This matches
  current behavior (today's `defer` sits below the `bootstrap.NewDB` Fatalf the
  same way) — there is no regression. The defer only runs on clean shutdown.
- All three binaries read `SQLITEDB_DSN` today (via `bootstrap.SQLiteDSNEnv`).
  The env-var name will live as a local package-level `const` in each binary,
  matching the existing convention: `cmd/migrator` uses `envDsnSqliteDB = "SQLITEDB_DSN"`
  and `cmd/doctor` uses `envDsnSqliteDB = "SQLITEDB_DSN"`. The string literal
  `"SQLITEDB_DSN"` is a literal data token (env-var key) — preserve byte-for-byte.
- `cmd/doctor` and `cmd/migrator` do NOT import `internal/bootstrap` (verified by
  grep — only `cmd/web`, `cmd/collector`, `cmd/notifier` do). No changes to those
  two binaries.
- Verified per-binary repository usage (read from each `main.go`):
  - **cmd/web** uses ALL FIVE: `ExecutionHistory`, `Source`, `RateValue`,
    `UserSubscription`, `UserEvent` (`main.go:119-125`, `:137`, `:153`).
  - **cmd/collector** uses THREE: `Source`, `ExecutionHistory`, `RateValue`
    (consumed inside `buildRunners`, `main.go:150-157`).
  - **cmd/notifier** uses FOUR: `Source`, `RateValue`, `UserSubscription`,
    `UserEvent` (`main.go:82-93`). It does NOT use `ExecutionHistory`.
  - These confirm the preliminary findings in the brief exactly — no
    contradiction.

## Execution Order

Strictly sequential — each task is a separate atomic commit, and the deletion
(Task 4) MUST come after every importer is gone or the tree won't compile.

1. **Task 1 — cmd/notifier** (warm-up: four repos, no helper-function rewiring).
2. **Task 2 — cmd/web** (all five repos; more call sites but same shape).
3. **Task 3 — cmd/collector** (last of the three because of the `buildRunners`
   signature change — the one non-mechanical edit; do it once the pattern from
   Tasks 1-2 is locked in).
4. **Task 4 — delete `internal/bootstrap` + relocate schema-gate test + update
   `CLAUDE.md`** (only after Tasks 1-3 remove the last importer).

Tasks 1-3 are independent of each other (different files, no shared symbols once
each drops the bootstrap import) and could in principle be parallelized, but
sequential is recommended so the inline pattern is validated on the simplest
binary first and the diffs stay reviewable. After each of Tasks 1-3, run
`CGO_ENABLED=0 go build` for that one binary and `go vet` it before committing —
the package still compiles between tasks because the remaining importers keep
`internal/bootstrap` alive until Task 4.

Verification gate after Task 4: `make build` and `make test` both green
(`CGO_ENABLED=0`, `-race`). Behavior must be runtime-equivalent: same log
filename prefix per binary, same schema gate, same per-binary repository set,
same fatal-on-missing-config semantics, same startup-marker order.

## Risks

- **Flipped `NewLogger` argument order** (highest risk). `internal.NewLogger`
  takes `(logsDir, name, level)` but `bootstrap.NewLogger` took
  `(name, logsDir, level)`. Both leading args are strings, so swapping them
  compiles cleanly and silently breaks the daily log filename prefix — a hard
  project rule. Mitigation: every task's acceptance criteria pins the exact call;
  `cmd/migrator:48` and `cmd/doctor:209` are the reference call sites to copy.
- **Dropped close-on-error path being mistaken for a regression.** A reviewer
  unaware of the locked decision may flag the missing `errors.Join` close-on-error
  as a leak. It is intentional — on `log.Fatalf` the process exits and the OS
  reclaims the fd; the schema gate is read-only so there is nothing to checkpoint.
  Mitigation: the rationale is documented in the Overview and in each task.
- **Silent loss of the schema-gate test.** If `bootstrap_test.go` is deleted
  without relocating the unmigrated-gate assertion, `RequireMigratedSchema` loses
  its only behavioral test. Mitigated by Task 4's `TestRequireMigratedSchema`.
- **Missed `repos.*` call site** (cmd/web especially). Low impact — it surfaces
  as a compile error, not a runtime bug — but worth a careful pass.
- **`buildRunners` signature drift.** Wrong param order to `collection.NewRateAgent`
  would pass the wrong repo into the wrong slot. Compile-checked by Go's type
  system only if the repo types differ; Source/History/Value are distinct types,
  so a mis-order is caught at compile time. Still, double-check positions.

## Trade-offs

- **Inline-and-duplicate over a shared helper.** Chose to duplicate ~10 trivial
  lines per binary over keeping (or replacing) `internal/bootstrap`, because the
  user wants the service binaries free to diverge (different DBs, dependency sets)
  without a shared startup layer imposing a contract. Cost: three near-identical
  DB-open blocks that can drift. Accepted: the wiring is stable, the duplication
  is small, and `cmd/migrator` / `cmd/doctor` already live this way without
  issue. A replacement helper was explicitly rejected by the user and is not
  proposed.
- **Per-binary repository construction over a full bundle.** Each binary now
  builds only the repositories it uses (notifier: 4, web: 5, collector: 3). This
  drops one needless allocation in notifier (ExecutionHistory) and collector
  (UserSubscription + UserEvent) and makes each binary's true dependency set
  visible at its composition root. Cost: more lines than a single
  `NewRepositories(db)` call. Accepted as the point of the refactor.
- **Relocating one test vs. dropping all bootstrap tests.** Kept the
